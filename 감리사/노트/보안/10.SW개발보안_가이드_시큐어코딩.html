① 입력데이터 검증 및 표현 — 핵심 취약점

SQL 인젝션 (SQL Injection)
CWE-89 / 입력값으로 SQL 구조 변조
⭐최중요
▾
// ❌ 취약한 코드 (문자열 연결) String query = "SELECT * FROM users WHERE id='" + userId + "'"; // 입력: ' OR '1'='1 → WHERE절 무력화 // ✅ 안전한 코드 (Prepared Statement) PreparedStatement ps = conn.prepareStatement( "SELECT * FROM users WHERE id = ?"); ps.setString(1, userId); // 파라미터 바인딩

대응: ①Prepared Statement ②입력값 유효성 검증 ③최소권한 DB 계정 ④오류 메시지 은닉 ⑤WAF

크로스사이트 스크립팅 XSS (Cross-Site Scripting)
CWE-79 / 악성 스크립트 삽입
⭐최중요
▾
// ❌ 취약한 코드 (입력값 그대로 출력) out.println("<b>" + request.getParameter("name") + "</b>"); // ✅ 안전한 코드 (출력 인코딩) String name = request.getParameter("name"); out.println("<b>" + HtmlUtils.htmlEscape(name) + "</b>"); // < → &lt; / > → &gt; / & → &amp; / " → &quot;

대응: ①출력 HTML 인코딩 ②CSP 헤더 ③HttpOnly 쿠키 ④입력 화이트리스트

운영체제 명령어 인젝션 (OS Command Injection)
CWE-78 / 외부 입력으로 시스템 명령 실행
⭐고위험
▾
// ❌ 취약한 코드 (입력값으로 시스템 명령 실행) Runtime.getRuntime().exec("ping " + ip); // 입력: 127.0.0.1; rm -rf / → 명령 추가 실행 // ✅ 안전한 코드 (화이트리스트 검증) if (!ip.matches("^[0-9]{1,3}(\\.[0-9]{1,3}){3}$")) throw new Exception("Invalid IP"); // 또는 exec() 대신 API 사용

대응: ①화이트리스트 입력 검증 ②Runtime.exec() 사용 금지 ③라이브러리 API 사용

경로 조작 및 자원 삽입 / 디렉터리 트래버설
CWE-22 / 경로 이동으로 임의 파일 접근
경로
▾
// ❌ 취약한 코드 String path = "/uploads/" + filename; // filename: ../../etc/passwd → 시스템 파일 접근 // ✅ 안전한 코드 (정규화 후 검증) File file = new File(baseDir, filename).getCanonicalFile(); if (!file.getPath().startsWith(baseDir)) throw new SecurityException("Invalid path");

대응: ①입력값 ../·..\\ 필터링 ②경로 정규화(Canonical Path) ③화이트리스트 검증

버퍼 오버플로 / 포맷 스트링
CWE-121·134 / 메모리 경계 초과 · 포맷 문자열 악용
메모리
▾
  • 버퍼 오버플로: 경계를 초과하는 입력으로 인접 메모리 덮어쓰기 → strcpy → strncpy / gets → fgets 사용
  • 포맷 스트링: printf(user_input) 형태에서 %s, %x 등 포맷 문자 악용 → printf("%s", user_input) 형태로 수정
  • 대응: ①안전한 함수(strncat, strncpy) ②컴파일러 방어(Stack Canary, ASLR) ③입력 길이 제한
입력검증 핵심: SQL→PreparedStatement / XSS→HTML인코딩 / OS명령→화이트리스트 / 경로→정규화 검증

② 보안기능 — 인증·인가·세션 관리

인증(Authentication) 보안
적절한 인증 수행 / 자격증명 강화
⭐핵심
▾
취약점안전한 구현
취약한 패스워드 정책최소 8자, 대소문자+숫자+특수문자 조합 / 패스워드 복잡도 정책 강제 적용
평문 패스워드 저장솔트(Salt) + 강력한 해시(bcrypt·PBKDF2·Argon2)로 저장 — SHA-1/MD5 단독 사용 금지
무차별 대입 허용계정 잠금(5회 실패 → 30분 잠금), CAPTCHA, 지수 백오프
단일 인증중요 서비스에는 MFA(다단계 인증) 적용
자동완성 허용비밀번호 입력 필드에 autocomplete="off" 또는 new-password
// ❌ 취약한 패스워드 저장 String hash = MD5(password); // MD5 단독 해시 → 레인보우 테이블 취약 // ✅ 안전한 저장 (bcrypt) String hash = BCrypt.hashpw(password, BCrypt.gensalt(12)); // gensalt(12): cost factor 12 → 충분히 느린 해시
세션 관리 (Session Management)
세션 ID 보안 / 타임아웃 / 고정화 방지
⭐핵심
▾
취약점안전한 구현
예측 가능한 세션 ID암호학적 난수(SecureRandom)로 충분한 길이(128비트 이상) 세션 ID 생성
세션 고정화(Fixation)로그인 성공 후 반드시 새 세션 ID 발급 (기존 세션 무효화)
세션 탈취 위험HttpOnly + Secure 쿠키 속성 / TLS 사용 / SameSite=Strict
세션 타임아웃 없음비활동 타임아웃(idle timeout) + 절대 타임아웃 적용 (예: 30분·8시간)
URL 세션 IDURL에 세션 ID 노출 금지 — 쿠키로만 관리
// 세션 고정화 방지 (로그인 후 세션 재생성) HttpSession oldSession = request.getSession(false); if (oldSession != null) oldSession.invalidate(); // 기존 세션 무효화 HttpSession newSession = request.getSession(true); // 새 세션 생성 // 새 세션에 인증 정보 저장
인가(Authorization) 및 접근통제
최소권한 원칙 / 수직·수평 권한 상승 방지
접근통제
▾
  • 최소 권한 원칙(Least Privilege): 사용자·프로세스에게 필요한 최소한의 권한만 부여
  • 수직 권한 상승: 일반 사용자 → 관리자 권한 — 서버 측 권한 검증 필수
  • 수평 권한 상승: 다른 사용자의 자원에 접근 — 자원 소유자 검증
  • 안전하지 않은 직접 개체 참조(IDOR): URL에 타 사용자 ID 입력으로 접근 → 간접 참조 또는 서버 권한 확인
  • 접근 통제 누락: 민감 URL에 인증 미확인 — 모든 요청에서 서버 측 인가 검증
// IDOR 취약 예시 GET /user/getInfo?userId=12345 → userId를 13579로 바꿔 타인 정보 접근 // ✅ 안전한 처리 (소유자 검증) int requestedId = Integer.parseInt(req.getParameter("userId")); int currentUserId = (int) session.getAttribute("userId"); if (requestedId != currentUserId) throw new AccessDeniedException();
인증: bcrypt/Argon2+Salt / 세션: 로그인 후 세션ID 재발급+HttpOnly+Secure / 인가: 최소권한+IDOR 차단

암호화 / 민감정보 보호

항목❌ 취약한 방법✅ 안전한 방법
대칭키 암호DES, 3DES, RC4AES-256 (CBC 또는 GCM 모드) — ECB 모드 절대 금지
해시 함수MD5, SHA-1 (충돌 취약)SHA-256 이상 (패스워드는 bcrypt·PBKDF2·Argon2)
비대칭키RSA-512, RSA-1024RSA-2048 이상, ECDSA P-256 이상
난수 생성Math.random(), java.util.RandomSecureRandom (CSPRNG)
전송 보안HTTP, SSL 2.0/3.0, TLS 1.0/1.1HTTPS (TLS 1.2 이상), TLS 1.3 권장
하드코딩 키소스코드에 암호키·비밀번호 직접 기재환경변수, HSM, 비밀 관리 서비스(Vault)
// ❌ 취약: ECB 모드 AES (같은 블록 = 같은 암호문) Cipher cipher = Cipher.getInstance("AES/ECB/PKCS5Padding"); // ✅ 안전: GCM 모드 AES (인증 암호화 제공) Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding"); GCMParameterSpec paramSpec = new GCMParameterSpec(128, iv); cipher.init(Cipher.ENCRYPT_MODE, secretKey, paramSpec); // ❌ 취약: 하드코딩 키 String secretKey = "mySecretKey12345"; // 소스코드에 노출 // ✅ 안전: 환경변수에서 로드 String secretKey = System.getenv("ENCRYPTION_KEY");
암호화: AES-256+GCM / SHA-256↑ / RSA-2048↑ / SecureRandom / TLS1.2↑ / 하드코딩 절대금지

④ 에러 처리 취약점

취약점대응
상세 오류 노출DB 구조·스택 트레이스를 사용자에게 노출 금지 → 일반적 오류 메시지만 표시
빈 catch 블록예외를 무시하지 않고 적절히 처리·로깅
오류 코드 노출HTTP 응답에서 실제 오류 원인 은닉
// ❌ 취약: 상세 오류 노출 } catch (Exception e) { out.println(e.getMessage()); // 내부 정보 노출 } // ✅ 안전: 로깅하고 일반 메시지만 } catch (Exception e) { logger.error("Error", e); // 내부 로그 out.println("서비스 오류가 발생했습니다."); // 일반 메시지 }

⑤ 코드 품질 취약점

  • Null Pointer 역참조: 객체 사용 전 null 검사 필수
  • 자원 미해제: DB 연결·파일·스트림은 try-with-resources 또는 finally에서 반드시 close()
  • 불충분한 초기화: 변수·배열은 사용 전 초기화
  • 경쟁 조건(Race Condition): 공유 자원 접근 시 동기화(synchronized) 필수
  • TOCTOU: 검사 시점(Time of Check)과 사용 시점(Time of Use) 불일치 — 원자적 연산 사용
// ✅ try-with-resources (자원 자동 해제) try (Connection conn = ds.getConnection(); PreparedStatement ps = conn.prepareStatement(sql)) { // 자동으로 close() 호출 }
에러처리: 상세 오류 숨김+로깅 / 코드품질: null검사+자원해제+동기화+TOCTOU 주의

OWASP Top 10 (2021 기준)

Open Web Application Security Project — 웹 애플리케이션 가장 심각한 보안 위험 Top 10

순위취약점핵심 내용대응
A01 접근 통제 실패
(Broken Access Control)		 IDOR, 수직·수평 권한 상승, URL 직접 접근 서버 측 인가 검증, 최소권한 원칙
A02 암호화 실패
(Cryptographic Failures)		 평문 전송, 취약 알고리즘(MD5·SHA-1·DES·RC4), 하드코딩 키 TLS1.2↑, AES-256, SHA-256↑, 안전한 키 관리
A03 인젝션
(Injection)		 SQL·OS·LDAP 인젝션, XSS, NoSQL 인젝션 Prepared Statement, 출력 인코딩, 입력 검증
A04 안전하지 않은 설계
(Insecure Design)		 설계 단계 보안 요구사항 누락, 위협 모델링 미수행 보안 설계 원칙, 위협 모델링, SDL 적용
A05 보안 설정 오류
(Security Misconfiguration)		 기본 계정·비밀번호, 불필요한 기능 활성화, 과도한 오류 정보 보안 강화된 기본 설정, 불필요 기능 비활성화
A06 취약하고 오래된 구성요소
(Vulnerable Components)		 취약점 있는 라이브러리·프레임워크·OS 사용 소프트웨어 구성 분석(SCA), 정기 패치 관리
A07 식별 및 인증 실패
(Auth Failures)		 취약한 패스워드, 세션 관리 실패, 자격증명 무차별 대입 MFA, 강력한 패스워드 정책, 계정 잠금
A08 SW·데이터 무결성 실패
(Integrity Failures)		 CI/CD 파이프라인 보안 취약, 역직렬화 취약점 디지털 서명, SCA, 안전한 역직렬화

정적/동적 분석 도구

SAST (정적 분석)

  • 시점: 소스코드·바이너리를 실행 없이 분석
  • 장점: 개발 초기 단계에서 취약점 발견 / 전체 코드 커버리지
  • 한계: 런타임 취약점 탐지 어려움, 거짓 양성(False Positive) 多
  • 도구: SonarQube, Checkmarx, Fortify, Coverity, FindBugs
  • 국내: 행안부 지정 소프트웨어 보안약점 분석도구 (KISA 인증)

DAST (동적 분석)

  • 시점: 실행 중인 애플리케이션을 블랙박스 방식으로 분석
  • 장점: 실제 런타임 취약점 탐지 / 소스코드 불필요
  • 한계: 코드 커버리지 제한, 발견 시점 늦음
  • 도구: OWASP ZAP, Burp Suite, Nikto, AppScan
  • 퍼지테스팅: 비정상 입력으로 예기치 않은 동작 탐지 (AFL, libFuzzer)
