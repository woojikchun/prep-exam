블록 암호 운용 모드란?

블록 암호(AES·DES 등)는 고정 크기 블록(예: 128비트)만 처리할 수 있음.
실제 데이터는 블록보다 길므로, 여러 블록을 어떻게 연결·처리할지를 정의하는 것이 운용 모드(Mode of Operation).
모드 선택에 따라 보안성·성능·병렬 처리 가능 여부가 달라짐.

ECB (Electronic CodeBook) — 전자 코드북

가장 단순한 모드. 각 블록을 완전히 독립적으로 같은 키로 암호화.
동일한 평문 블록 → 동일한 암호문 블록. 패턴이 암호문에 그대로 드러남.

ECB 동작 원리

ECB의 치명적 취약점

동일한 평문 블록이 동일한 암호문 블록으로 변환됨 → 패턴(반복)이 그대로 노출. 대표 사례: "ECB 펭귄" — 비트맵 이미지를 ECB로 암호화해도 윤곽선이 그대로 보임. 빈도 분석·패턴 공격에 취약. 실무·감리에서 ECB 사용은 즉각 지적 사항.

CBC (Cipher Block Chaining) — 암호 블록 체이닝

각 평문 블록을 암호화하기 전에 이전 암호문 블록과 XOR하는 체이닝 방식.
첫 번째 블록은 IV(초기화 벡터)와 XOR. 동일 평문이어도 IV가 다르면 다른 암호문 생성.
과거 가장 널리 사용된 모드. 현재는 패딩 오라클 공격으로 GCM에 자리를 내줌.

CTR (CounTeR) — 카운터 모드

Nonce(랜덤값) + 카운터(0,1,2…)를 블록 암호로 암호화해 키스트림 생성, 평문과 XOR.
블록이 완전히 독립적 → 암호화·복호화 모두 병렬 처리 가능.
스트림 암호처럼 동작. 패딩 불필요. 현재 고성능 암호화의 기반.

CTR vs CBC 성능 비교

CTR은 병렬 처리로 CBC 대비 암호화 속도가 수 배 빠름. AES-NI(하드웨어 가속) + CTR은 현대 고성능 암호화의 표준 조합. 단, 무결성 보장 없음 → GCM(CTR + GHASH)으로 보완
한 줄 암기CTR = Nonce + 카운터 → 키스트림 → XOR. 완전 병렬(암호화·복호화 모두), 패딩 불필요, 랜덤 접근 가능. Nonce 재사용은 치명적(키스트림 동일 → XOR로 평문 노출)

GCM (Galois/Counter Mode) — 갈루아 카운터 모드

CTR 모드 + GHASH(갈루아 필드 MAC)의 결합.
기밀성(CTR 암호화) + 무결성·인증(Authentication Tag) 동시 제공 → AEAD(Authenticated Encryption with Associated Data).
TLS 1.2/1.3, IPSec, SSH 등 현대 보안 프로토콜의 표준 암호 스위트.

GCM Nonce 재사용의 치명성

GCM에서 동일 (Key, Nonce) 쌍으로 두 번 암호화하면 GHASH 인증 키가 노출되어 공격자가 임의 태그를 위조할 수 있음. 기밀성뿐 아니라 무결성(AEAD의 핵심 장점)까지 붕괴. Nonce를 순차 카운터 또는 암호학적 난수로 관리하고, 절대 재사용 금지

기출 예상 Q&A

ECB 모드의 취약점을 설명하고 왜 사용하면 안 되는가? ▾
ECB는 동일한 평문 블록이 항상 동일한 암호문 블록으로 변환됨. 암호문에서 패턴(반복)이 그대로 노출되어 평문 구조를 추론 가능. 대표 사례가 "ECB 펭귄" — 비트맵 이미지를 ECB로 암호화해도 이미지 윤곽이 보임. 빈도 분석·블록 치환 공격에 취약. 감리 관점: ECB 사용 발견 시 즉각 지적, AES-CBC 또는 AES-GCM으로 교체 권고.
CBC 모드의 패딩 오라클 공격이란? ▾
CBC 복호화 시 패딩 오류 여부를 오류 메시지나 응답 시간으로 구분할 수 있을 때, 공격자가 암호문 블록을 조작하며 반복 시도해 패딩 검증 성공/실패 피드백으로 평문을 1바이트씩 역산하는 공격. SSL 3.0의 POODLE, TLS 1.0의 BEAST 공격이 이 원리 기반. 대응: 패딩 오류와 MAC 오류를 동일한 메시지로 응답(오류 구분 금지), MAC-then-Encrypt 대신 Encrypt-then-MAC 또는 AEAD(GCM) 사용.
CTR 모드에서 Nonce 재사용이 왜 치명적인가? ▾
CTR에서 동일 (Key, Nonce) 쌍으로 두 평문 P1, P2를 암호화하면 C1=P1⊕KS, C2=P2⊕KS(KS는 같은 키스트림). C1⊕C2=P1⊕P2가 되어 두 암호문을 XOR하면 키스트림이 제거된 평문 XOR이 노출됨. 공격자가 P1을 알면 P2를 즉시 복원 가능. 이를 "Two-Time Pad" 공격이라 함(One-Time Pad 재사용과 동일 원리). GCM에서는 Nonce 재사용 시 GHASH 인증 키까지 노출되어 무결성도 붕괴.
GCM의 AEAD란 무엇이며 Authentication Tag의 역할은? ▾
AEAD(Authenticated Encryption with Associated Data)는 기밀성(암호화)과 무결성·인증을 하나의 알고리즘으로 동시에 제공하는 암호화 방식. GCM은 CTR 모드로 암호화(기밀성)하고, GHASH로 Authentication Tag(128비트)를 생성(무결성·인증). 수신자는 복호화 전 태그를 검증해 데이터 변조 여부를 확인. 태그 불일치 시 복호화 거부. AAD(추가 인증 데이터)는 암호화 없이 태그 계산에만 포함되어 헤더 정보 등의 무결성도 보호.
각 운용 모드의 병렬 처리 가능 여부와 이유를 설명하시오 ▾
ECB: 암·복호화 모두 병렬 가능. 블록이 완전 독립. CBC: 암호화는 이전 암호문 블록이 필요해 순차(직렬). 복호화는 이전 암호문을 바로 참조 가능해 병렬 가능. CFB: 암호화 순차, 복호화 병렬. OFB: 암·복호화 모두 순차(키스트림이 순차 생성). CTR·GCM: 암·복호화 모두 병렬 가능. 카운터로 각 블록의 키스트림을 독립 생성. 성능 측면에서 CTR·GCM이 가장 유리.
TLS 1.3에서 CBC 모드를 제거하고 GCM을 선택한 이유는? ▾
CBC 모드는 패딩 오라클 공격(POODLE·BEAST·Lucky13 등) 취약점으로 인해 안전한 구현이 어렵고, 별도 MAC을 추가해도 MAC-then-Encrypt 방식에서 취약점이 남음. GCM은 AEAD로 암호화와 인증을 단일 패스로 처리해 패딩 오라클 공격 불가. CTR 기반 병렬 처리로 성능 우수. Authentication Tag로 무결성 보장. TLS 1.3은 보안성이 검증된 AEAD(AES-128-GCM, AES-256-GCM, ChaCha20-Poly1305)만 허용하고 CBC·RC4·DES 등 취약 모드를 완전 제거.
