블록체인 보안 특성 vs 취약점

✅ 블록체인 보안 강점

  • 불변성: 해시 체인으로 과거 블록 변조 어려움
  • 분산성: 단일 실패 지점(SPOF) 없음
  • 투명성: 모든 거래 공개 검증 가능 (퍼블릭)
  • 암호학적 보안: SHA-256, ECDSA 등 강력한 암호화
  • 탈중앙화: 중앙 서버 없이 합의로 운영

⚠️ 블록체인 취약 영역

  • 스마트컨트랙트: 코드 버그 = 취약점 → 자산 탈취
  • 개인키 관리: 키 분실·도용 = 자산 복구 불가
  • 합의 메커니즘: 해시파워 집중 → 51% 공격
  • 오라클 문제: 외부 데이터 신뢰성 보장 어려움
  • 프라이버시: 퍼블릭 체인에서 거래 추적 가능
블록체인 강점=불변성·분산 / 취약점=스마트컨트랙트 버그·개인키·합의 집중·오라클

합의 계층 주요 공격

51% 공격 (Majority Attack)
PoW 네트워크에서 해시파워 과반 장악
⭐핵심
▾
  • 원리: 공격자가 전체 네트워크 해시파워의 51% 이상을 확보 → 블록 생성 속도를 조작하여 체인 재편
  • 가능한 공격: ①이중지불(Double Spending) ②트랜잭션 검열(특정 거래 배제) ③블록 보상 독점
  • 불가능한 것: 다른 주소의 코인 탈취, 발행량 변경, 기존 확정 블록 변조(수백 블록 이상)
  • 발생 조건: 소규모 PoW 코인(해시파워 집중 쉬움) — 비트코인은 현실적으로 어려움
  • 대응: 확인 수(Confirmation) 증가, Merge Mining, PBFT 등 다른 합의 사용
// 51% 공격 이중지불 시나리오 1. 공격자 → 거래소에 코인 전송 (공개 체인에 기록) 2. 동시에 비밀 체인에서 동일 코인을 자신에게 전송하는 블록 생성 3. 거래소가 확인 후 상품 제공 4. 비밀 체인(더 긴 체인) 공개 → 공개 체인 무효화 → 결과: 코인도 살아있고 상품도 받음 (이중지불 성공)

대응: 충분한 블록 확인 수 대기 (비트코인: 6 Confirmations 권장) / PoS 전환 / Checkpoint

이중지불 (Double Spending)
동일 코인을 두 번 사용하는 사기
⭐핵심
▾
  • Race Attack: 동일 UTXO로 두 거래를 동시 전파 → 0-Confirmation 거래를 노림
  • Finney Attack: 사전 채굴된 블록에 자신으로의 거래를 포함 → 공격자가 채굴자인 경우
  • 51% 기반: 과반 해시파워로 다른 체인 재편
  • 대응: 0-Confirmation 거래 수락 금지 / 충분한 Confirmation 대기 / 소액 거래만 즉시 수락
셀피시 마이닝 (Selfish Mining)
채굴한 블록을 숨겨 다른 채굴자 낭비 유도
전략적 공격
▾
  • 원리: 채굴에 성공해도 블록을 숨기고(비공개 체인 유지) 상대방이 다음 블록을 채굴하면 숨긴 블록을 공개 → 상대방 작업 무효화
  • 효과: 실제 해시파워보다 높은 블록 보상 비율 획득 — 25% 이상 해시파워로 이익 가능
  • 조건: 50% 미만 해시파워로도 이익 추구 가능 (네트워크 지연 악용)
  • 대응: 타임스탬프 검증, α(알파) 임계치 설정, Fork Resolution 개선
Nothing-at-Stake (PoS 취약점)
PoS에서 포크 시 손실 없이 모든 체인 검증
PoS 취약
▾
  • 원리: PoS에서 포크 발생 시 검증자가 모든 포크 체인에 동시 베팅 → 어느 쪽이 되든 이득 (PoW는 해시파워 분산 시 손해)
  • 결과: 이중지불 가능, 합의 불안정
  • 대응: 슬래싱(Slashing) — 이중 검증 시 스테이킹 자산 몰수 / Casper FFG(이더리움)
51%=해시파워 과반장악→이중지불 / 셀피시마이닝=블록 숨겨 25%로 이익 / NaS=PoS 포크 악용→슬래싱 대응

스마트컨트랙트 주요 취약점 (SWC 기반)

재진입 공격 (Reentrancy Attack)
The DAO 해킹 — 함수 실행 중 재귀 호출 악용
⭐최중요
▾
  • 원리: 컨트랙트가 외부 주소로 ETH를 전송할 때, 수신 컨트랙트의 fallback 함수가 원래 컨트랙트를 재귀 호출 → 잔액 차감 전 반복 출금
  • 실제 사례: The DAO 해킹 (2016) — 약 360만 ETH 탈취
// 취약한 코드 (잔액 차감 전 전송) function withdraw() { uint amount = balance[msg.sender]; msg.sender.call{value: amount}(""); // 재진입 가능! balance[msg.sender] = 0; // 차감이 너무 늦음 } // 안전한 코드 (CEI 패턴: Check-Effects-Interactions) function withdraw() { uint amount = balance[msg.sender]; balance[msg.sender] = 0; // 먼저 차감 (Effects) msg.sender.call{value: amount}(""); // 그 다음 전송 (Interactions) }

대응: ①CEI 패턴(Check-Effects-Interactions) ②ReentrancyGuard(재진입 방지 modifier) ③pull 패턴 사용

정수 오버플로/언더플로
Integer Overflow / Underflow
⭐핵심
▾
  • 오버플로: uint256 최대값 + 1 = 0 (랩어라운드)
  • 언더플로: uint256의 0 - 1 = 2^256-1 (엄청나게 큰 수)
  • 악용: 잔액을 언더플로로 천문학적 숫자로 만들어 대량 출금
// 취약한 코드 uint256 balance = 0; balance -= 1; // 언더플로 → 2^256-1 = 엄청나게 큰 값! // 대응 1: SafeMath 라이브러리 사용 (Solidity 0.7 이하) // 대응 2: Solidity 0.8+ 는 오버플로 자동 감지 후 revert

대응: ①Solidity 0.8+ 사용 (기본 오버플로 검사) ②SafeMath 라이브러리

프론트러닝 (Front-Running)
Mempool 트랜잭션 관찰 후 선행 처리
MEV 공격
▾
  • 원리: 공개 Mempool의 미확정 트랜잭션을 관찰 → 더 높은 가스비(Gas Price)를 제시하여 먼저 처리되도록 유도
  • DEX 공격: 대형 스왑 거래 탐지 → 먼저 같은 방향 거래 실행 → 가격 올린 후 매도 (샌드위치 공격)
  • MEV(Maximal Extractable Value): 블록 포함·순서 조작으로 이익 추출
  • 대응: ①Commit-Reveal 패턴 ②최소 허용 슬리피지 설정 ③Flashbots(MEV 차단) ④영지식 증명
오라클 조작 / 플래시론 공격
외부 데이터 신뢰 취약점 · 무담보 순간 대출 악용
DeFi 공격
▾
  • 오라클 조작: 단일 DEX 가격 오라클에 의존 → 순간적 가격 조작으로 잘못된 데이터 주입 → 담보 가치 오판
  • 플래시론 공격: 무담보 순간 대출(같은 트랜잭션 내 상환 조건) → 대량 유동성으로 가격 조작 → 이익 실현 → 상환
  • 대응 오라클: Chainlink 등 탈중앙화 오라클 / TWAP(시간가중평균가격) 사용
  • 대응 플래시론: 동일 블록 내 가격 조작 탐지 / 다중 오라클 가격 사용

P2P 네트워크 계층 공격

이클립스 공격 (Eclipse Attack)
피해 노드의 P2P 연결을 공격자 노드로 독점
P2P
▾
  • 원리: 피해자 노드의 모든 P2P 연결을 공격자 노드로 채움 → 피해자가 정상 네트워크와 격리
  • 결과: 피해자에게 위조된 블록체인 정보 제공 → 이중지불 유도, 51% 공격 보조
  • 대응: 무작위 피어 선택, 아웃바운드 연결 증가, 피어 다양성 확보
시빌 공격 (Sybil Attack)
다수의 가짜 신원(노드)으로 네트워크 영향력 장악
P2P
▾
  • 원리: 하나의 실제 공격자가 수많은 가짜 노드(신원)를 생성하여 P2P 네트워크를 점령
  • 결과: 네트워크 투표·합의 조작, 이클립스 공격 실행
  • 대응: PoW(작업증명)·PoS(지분증명)으로 노드 생성 비용 부과 / 신원 검증

블록체인 보안 강화 방법론

스마트컨트랙트 보안

  • 형식 검증(Formal Verification)
  • 외부 보안 감사(Security Audit)
  • OpenZeppelin 검증 라이브러리 사용
  • Bug Bounty 프로그램 운영
  • 업그레이드 가능 프록시 패턴
  • 일시 정지(Pause) 기능 구현

네트워크·합의 보안

  • 다중 서명(Multi-signature) 적용
  • 충분한 Confirmation 수 요구
  • Checkpoint 설정으로 롤백 방지
  • 슬래싱으로 PoS 비잔틴 행동 억제
  • 탈중앙화된 오라클 사용
  • 크로스체인 브리지 보안 강화

키·지갑 보안

  • 하드웨어 지갑(Ledger·Trezor) 사용
  • 다중 서명 지갑 (2-of-3 등)
  • MPC(다자간 계산) 기반 키 분산
  • HSM(하드웨어 보안 모듈)
  • 니모닉 오프라인 분산 백업
  • 양자 내성 암호 전환 준비

프라이버시 보안

  • 영지식 증명(ZKP, Zero-Knowledge Proof)
  • 링 서명(Ring Signature) — Monero
  • Stealth Address 사용
  • CoinJoin으로 거래 믹싱
  • 프라이빗 체인/퍼미션드 블록체인
  • zk-SNARKs · zk-STARKs 활용
