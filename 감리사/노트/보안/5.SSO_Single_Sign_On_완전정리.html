SSO (Single Sign-On) 정의

SSO(Single Sign-On)란 사용자가 한 번만 인증(로그인)하면, 연결된 여러 시스템·애플리케이션에 추가 로그인 없이 접근할 수 있도록 하는 인증 통합 메커니즘

✅ 장점

  • 사용자 편의성 향상 (한 번만 로그인)
  • 비밀번호 관리 부담 감소
  • IT 관리 비용 절감 (비밀번호 재설정 요청↓)
  • 강력한 중앙 인증 정책 적용 가능
  • 계정 프로비저닝·디프로비저닝 일원화

⚠️ 단점 / 위험

  • 단일 실패 지점(SPOF): IdP 장애 시 모든 서비스 접근 불가
  • 계정 탈취 위험 증폭: 한 계정 도용 → 모든 서비스 접근
  • 구현 복잡도 높음
  • 레거시 시스템과의 통합 어려움
  • 세션 관리 복잡 (세션 타임아웃·공유)
SSO = 한 번 인증 → 여러 시스템 접근 / SPOF 위험 (IdP 장애 → 전체 서비스 차단)
🏛️

SSO 핵심 구성 요소

구성요소역할예시

IdP(인증) ↔ SP(서비스) / 토큰·어설션으로 신뢰 전달 / IdP = SSO의 핵심 SPOF

SAML (Security Assertion Markup Language)

항목내용
정의XML 기반 인증·권한 부여 정보를 IdP와 SP 사이에서 교환하는 개방형 표준
버전SAML 2.0 (현행 / 2005년 OASIS 표준)
데이터 형식XML (무겁고 복잡하지만 엔터프라이즈 환경에 강함)
전송 방식HTTP Redirect / HTTP POST / HTTP Artifact Binding
주요 용도기업 환경 Federated SSO — 서로 다른 조직/도메인 간 인증 연동
보안XML 디지털 서명(xmldsig) + XML 암호화(xmlenc)로 어설션 무결성·기밀성 보장
SAML = XML 기반 / OASIS 표준 / 엔터프라이즈 Federation SSO / IdP↔SP 어설션 교환
📦

SAML 3가지 어설션 유형

어설션 유형내용포함 정보
Authentication Assertion
(인증 어설션)		사용자가 IdP에서 언제, 어떻게 인증했는지 증명인증 시각, 인증 방법(비밀번호·MFA 등)
Attribute Assertion
(속성 어설션)		사용자의 속성 정보 전달이름, 이메일, 역할(Role), 부서 등
Authorization Decision Assertion
(인가 결정 어설션)		특정 자원에 대한 접근 권한 결정 결과허용/거부 결정, 자원 URI
SAML 3대 어설션: ①인증(언제·어떻게) ②속성(이름·역할) ③인가결정(허용/거부)
🔄

SAML SSO 동작 흐름 (SP-Initiated)

① 사용자가 SP(서비스 앱)에 접근 시도
② SP가 SAML AuthnRequest 생성 → 브라우저를 통해 IdP로 리다이렉트
③ IdP에서 사용자 로그인 (이미 SSO 세션 있으면 자동 통과)
④ IdP가 SAML Response(어설션 포함) 생성 → 서명 후 브라우저를 통해 SP로 POST
⑤ SP가 SAML Response 서명 검증 + 어설션 파싱
⑥ SP가 세션 생성 → 사용자에게 서비스 제공

IdP-Initiated: IdP에서 먼저 SAML Response를 SP에 Push하는 방식 (SP로의 리다이렉트 없음)

SAML 흐름: SP→AuthnRequest→IdP 로그인→SAML Response(서명)→SP 검증→서비스

OAuth 2.0 — 인가(Authorization) 프레임워크

항목내용
정의제3자 애플리케이션이 자원 소유자(사용자) 동의 하에 자원에 접근하도록 권한을 위임하는 프레임워크
핵심인가(Authorization) 프레임워크 — 인증(Authentication)이 아님 ⭐
토큰Access Token (자원 접근권) + Refresh Token (재발급용)
데이터 형식JSON + HTTPS (SAML의 XML보다 경량)
역할Resource Owner(사용자) / Client(앱) / Authorization Server(IdP) / Resource Server(API)
OAuth 2.0 = 인가 위임 프레임워크 (인증 ❌) / Access Token 발급 / JSON+HTTPS
📋

OAuth 2.0 — 4가지 Grant Type

① Authorization Code Grant
웹 앱 · 가장 안전 · 권장
▾
  • 흐름: 클라이언트 → 인가코드 요청 → 사용자 동의 → 인가코드 발급 → Access Token 교환
  • 특징: Access Token이 브라우저에 직접 노출되지 않음 — 가장 안전
  • 용도: 서버 사이드 웹 애플리케이션
  • PKCE: Public Client(모바일·SPA)에는 PKCE(Proof Key for Code Exchange) 필수 추가
② Implicit Grant
브라우저 SPA · ⚠️ 보안 취약 · 사용 지양
▾
  • Authorization Code 없이 Access Token을 URL Fragment로 직접 발급
  • 브라우저 히스토리에 Token 노출 위험 → 보안상 취약
  • OAuth 2.1에서 삭제 예정 → PKCE 적용 Authorization Code로 대체
③ Client Credentials Grant
서버↔서버 · 사용자 없음
▾
  • 사용자 없이 클라이언트 자체 자격증명으로 Access Token 발급
  • Machine-to-Machine(M2M) 통신에 사용 (백엔드 API 간 통신)
  • 사용자 동의 불필요
④ Resource Owner Password Credentials
⚠️ 비권장 · 사용자 비밀번호 직접 노출
▾
  • 사용자의 ID/비밀번호를 클라이언트에게 직접 제공 → 클라이언트가 Token 요청
  • 비밀번호가 클라이언트에 노출되어 보안상 비권장
  • OAuth 2.1에서 삭제 예정
Grant Type: Authorization Code(권장) / Implicit(지양) / Client Credentials(M2M) / ROPC(비권장)

OIDC (OpenID Connect) — OAuth 2.0 위에 인증 추가

항목내용
정의OAuth 2.0을 기반으로 사용자 인증(Authentication) 기능을 추가한 프로토콜
핵심 토큰ID Token (JWT 형식) — 사용자 신원 정보 포함
관계OIDC = OAuth 2.0 + ID Token + UserInfo Endpoint
용도소셜 로그인(Google·Kakao·Naver), 웹/모바일 앱 인증
UserInfo/userinfo 엔드포인트로 사용자 프로필 조회

⭐ OAuth 2.0 vs OIDC vs SAML 핵심 구분

구분OAuth 2.0OIDCSAML 2.0
목적인가 (Authorization)인증 (Authentication)인증 + 인가
토큰Access TokenID Token (JWT)XML Assertion
형식JSONJSON / JWTXML
주요 환경API 접근 위임소셜 로그인, SPA기업 Enterprise
OIDC = OAuth 2.0 + 인증 / ID Token(JWT) 추가 / "소셜 로그인 = OIDC" / OAuth는 인가만

Kerberos — 티켓 기반 네트워크 인증

항목내용
정의MIT 개발 네트워크 인증 프로토콜 — 대칭키 기반 티켓으로 SSO 구현
버전Kerberos v5 (RFC 4120, 현행)
핵심 서버KDC (Key Distribution Center) = AS + TGS로 구성
ASAuthentication Server — 사용자 인증 후 TGT(Ticket Granting Ticket) 발급
TGSTicket Granting Server — TGT를 받아 서비스 접근 티켓(Service Ticket) 발급
특징비밀번호가 네트워크로 전송되지 않음 / 상호 인증(클라이언트↔서버) / 시간 동기화 필수(±5분)
환경Active Directory(AD) 도메인 내 Windows SSO의 기반 프로토콜
단점단일 실패 지점(KDC) / 시간 동기화 필수 / 방화벽 통과 어려움
Kerberos = KDC(AS+TGS) / TGT→Service Ticket / 대칭키 / 상호 인증 / AD 기반 SSO
🎫

Kerberos 동작 흐름 (5단계)

① 클라이언트 → AS(인증서버): 사용자 ID 전송 (비밀번호 미전송)
② AS → 클라이언트: TGT(Ticket Granting Ticket) + 세션키 발급 (사용자 비밀키로 암호화)
③ 클라이언트 → TGS(티켓부여서버): TGT + 서비스 요청 전송
④ TGS → 클라이언트: Service Ticket 발급 (서비스 서버 비밀키로 암호화)
⑤ 클라이언트 → 서비스 서버: Service Ticket 제시 → 서비스 이용
Kerberos 흐름: AS→TGT → TGS→Service Ticket → 서비스서버 제시 / 비밀번호 네트워크 미전송

LDAP (Lightweight Directory Access Protocol)

항목내용
정의네트워크 디렉터리 서비스 접근 프로토콜 — 사용자·그룹·자원 정보 저장·조회
포트389(평문) / 636(LDAPS, TLS)
구조DIT(Directory Information Tree) — 계층적 트리 구조, DN(Distinguished Name)으로 객체 식별
역할SSO에서 신원 저장소(Identity Store) — IdP 뒷단에서 사용자 계정 관리
예시Microsoft AD(Active Directory), OpenLDAP
LDAP = 디렉터리 서비스 / 계층적 트리 / DN 식별 / SSO의 신원 저장소
📡

RADIUS (Remote Authentication Dial-In User Service)

항목내용
정의중앙화된 네트워크 접근 인증·권한부여·계정관리(AAA) 프로토콜
포트1812(인증) / 1813(계정)
AAAAuthentication·Authorization·Accounting — 인증·권한부여·로그
용도VPN, 무선 LAN(Wi-Fi 802.1X), 네트워크 장비 접근 제어
vs TACACS+RADIUS=UDP·인증/인가 통합 / TACACS+=TCP·AAA 분리·Cisco 독점
RADIUS = AAA(인증·인가·계정) / UDP 1812/1813 / VPN·WiFi 802.1X / vs TACACS+=TCP·분리

핵심 암기 포인트

  • OAuth 2.0 ≠ 인증: 인가(Authorization) 프레임워크 — 인증은 OIDC
  • OIDC = OAuth 2.0 + ID Token: JWT 형식의 ID Token으로 인증 추가
  • SAML: XML, OASIS 표준, Enterprise Federated SSO
  • Kerberos: KDC(AS+TGS), TGT→Service Ticket, 대칭키, ±5분 시간동기
  • SSO 최대 위험: SPOF — IdP 장애 시 모든 서비스 불가
  • Implicit Grant: 보안 취약 → PKCE+Auth Code로 대체
  • LDAP: 디렉터리 서비스·신원 저장소 / RADIUS: AAA·VPN·WiFi
🎯

자주 혼동 포인트

  • OAuth ≠ SSO: OAuth는 인가 위임, SSO는 인증 통합 — OIDC가 SSO 역할
  • SAML vs OIDC: 기업·레거시=SAML / 모바일·API·소셜=OIDC
  • Kerberos 상호 인증: 클라이언트와 서버 모두 인증 — 단방향 아님
  • RADIUS vs TACACS+: RADIUS=UDP·인증인가통합 / TACACS+=TCP·AAA분리
  • TGT vs Service Ticket: TGT=TGS 접근권 / ST=실제 서비스 접근권
  • LDAP vs RADIUS: LDAP=계정정보저장소 / RADIUS=네트워크접근 AAA
