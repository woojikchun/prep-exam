이메일 보안 위협 & 프로토콜 체계

이메일 보안 위협

  • 스팸(Spam): 대량 발송 광고·피싱 메일
  • 피싱(Phishing): 발신자 위조로 개인정보 탈취
  • 스푸핑(Spoofing): 발신 도메인·IP 위조
  • 메일 위변조: 전송 중 내용 변조
  • 도청(Eavesdropping): 전송 중 내용 도청

이메일 보안 솔루션 분류

  • 메일 내용 보안: PEM, PGP, S/MIME
  • 발신자 인증: SPF, DKIM, DMARC
  • 전송 보안: STARTTLS, SMTPS
  • 전자결제: SET (Secure Electronic Transaction)
이메일 보안 2대 축: ①내용 보호(PEM·PGP·S/MIME) ②발신자 인증(SPF·DKIM·DMARC)

🔵 PEM (Privacy Enhanced Mail)
IETF RFC 1421~1424 / 중앙 집중식 CA 계층 구조
▾
항목내용
정의IETF가 제안한 초기 이메일 보안 표준 — 중앙 집중식 PKI 기반
암호화 방식DES(대칭키) + RSA(비대칭키) 혼합 방식
신뢰 모델계층적 CA(Certificate Authority) 구조 — 최상위 CA에서 하위 CA로 신뢰 전파
보안 서비스기밀성, 무결성, 인증, 부인방지
한계중앙 집중식 구조로 확장성 제한, 인터넷 보급으로 PGP·S/MIME에 밀려 사실상 폐기
인코딩Base64 인코딩 사용 — ASCII 텍스트로 변환
PEM = 최초 이메일 보안 표준 / 계층적 CA / DES+RSA / Base64 / 실제 사용↓
🟢 PGP (Pretty Good Privacy)
Phil Zimmermann 개발 / 분산형 신뢰 모델 (Web of Trust)
▾
항목내용
정의1991년 Phil Zimmermann이 개발한 이메일 암호화 소프트웨어
신뢰 모델Web of Trust (신뢰망) — 중앙 CA 없이 사용자 간 상호 신뢰 (분산형)
암호화 과정① 세션키(대칭키) 생성 → ② 메시지를 세션키로 암호화 → ③ 세션키를 수신자 공개키로 암호화 → ④ 전송
사용 알고리즘대칭: IDEA·3DES·AES / 비대칭: RSA·DSA·El Gamal / 해시: MD5·SHA-1
디지털 서명해시(메시지) → 발신자 개인키로 서명 → 수신자가 공개키로 검증
키 관리공개키 링(Public Key Ring), 개인키 링(Private Key Ring)으로 관리
OpenPGPPGP 기반 표준 (RFC 4880) — GnuPG(GPG) 등 오픈소스 구현체
// PGP 암호화 과정
평문 → [세션키(대칭)]로 암호화 → 암호문
세션키 → [수신자 공개키(비대칭)]로 암호화 → 암호화된 세션키
전송: {암호문 + 암호화된 세션키 + 디지털 서명}
PGP = Web of Trust(분산) / 하이브리드 암호화 / 세션키→공개키 암호화 / OpenPGP
🟡 S/MIME (Secure/MIME)
RSA Security 개발 → IETF 표준 / RFC 8551 / 계층적 CA 기반
▾
항목내용
정의MIME(Multipurpose Internet Mail Extensions)에 보안 기능을 추가한 이메일 보안 표준
신뢰 모델계층적 PKI(CA) — PEM과 유사, 현재 기업 환경에서 가장 많이 사용
암호화CMS(Cryptographic Message Syntax) 기반 — 하이브리드 암호화
사용 알고리즘대칭: 3DES·AES / 비대칭: RSA / 해시: SHA-256·SHA-384 / 서명: DSA·RSA
기능기밀성(암호화), 무결성·인증·부인방지(디지털 서명), 첨부파일 포함 전체 보안
인증서X.509 인증서 사용 — 공인된 CA에서 발급
PGP와 차이S/MIME = 계층적 CA / PGP = Web of Trust / S/MIME이 기업 이메일 클라이언트(Outlook 등) 표준
S/MIME = MIME+보안 / X.509 인증서 / 계층적 CA / 기업 이메일 표준 / Outlook 지원

🔵 SPF (Sender Policy Framework)
RFC 7208 / DNS TXT 레코드 / 발신 IP 검증
▾
항목내용
목적이메일 발신 서버 IP 주소가 도메인 소유자가 허가한 서버인지 검증 — 발신 IP 스푸핑 방지
동작 방식도메인 DNS TXT 레코드에 허가된 메일 서버 IP 목록을 등록 → 수신 서버가 확인
검증 위치메일 봉투(Envelope)의 Return-Path(MAIL FROM) 헤더 도메인 기반 검증
한계메일 전달(Forwarding) 시 원래 IP와 달라져 SPF 실패 가능 / 헤더 From 주소는 검증 안 함
// DNS TXT 레코드 예시
v=spf1 ip4:203.0.113.0/24 include:_spf.example.com ~all
// v=spf1: SPF 버전 / ip4: 허가 IP / include: 외부 서비스 포함
// ~all: Softfail(의심) / -all: Hardfail(거부) / +all: 모두 허용
SPF = DNS TXT에 허가 IP 등록 / 발신 IP 검증 / 포워딩 시 실패 가능 / Return-Path 기반
🟢 DKIM (DomainKeys Identified Mail)
RFC 6376 / 디지털 서명 기반 / 메일 내용 무결성 검증
▾
항목내용
목적메일 헤더·본문에 디지털 서명을 추가하여 전송 중 위변조 탐지 + 발신 도메인 인증
동작 방식발신 서버가 개인키로 서명 → DNS에 공개키 등록 → 수신 서버가 공개키로 서명 검증
서명 위치이메일 헤더에 DKIM-Signature 헤더 추가
장점포워딩(Forwarding)에도 서명은 유지 — SPF 한계 보완
검증 도메인From 헤더의 도메인 기반 (d= 태그)
알고리즘RSA + SHA-256 (기본)
// DKIM-Signature 헤더 예시
DKIM-Signature: v=1; a=rsa-sha256; d=example.com; s=mail;
h=From:To:Subject:Date; bh=base64(body_hash); b=base64(signature)
// d: 서명 도메인 / s: 셀렉터(공개키 위치) / bh: 본문 해시 / b: 서명값
DKIM = 개인키 서명 + DNS 공개키 등록 / DKIM-Signature 헤더 / 포워딩에도 유효 / From 도메인 인증
🟡 DMARC (Domain-based Message Authentication, Reporting and Conformance)
RFC 7489 / SPF + DKIM 통합 정책 / 처리 방침 지정
▾
항목내용
목적SPF·DKIM 검증 결과를 종합하여 인증 실패 메일에 대한 처리 정책 지정 + 보고서 수신
정책 (p=)none(모니터링만), quarantine(스팸함), reject(거부)
정렬 (Alignment)From 헤더 도메인이 SPF의 Return-Path 도메인 또는 DKIM의 d= 도메인과 일치해야 통과
보고서집계 보고서(rua=): 통계 / 실패 보고서(ruf=): 개별 실패 메일 상세
역할SPF·DKIM이 없으면 DMARC도 의미 없음 — SPF·DKIM의 통합 관리 레이어
// DMARC DNS TXT 레코드 예시
_dmarc.example.com TXT "v=DMARC1; p=reject; rua=mailto:dmarc@example.com"
// p=reject: 인증 실패 메일 거부 / p=quarantine: 스팸함 / p=none: 모니터링
// rua: 집계 보고서 수신 주소 / ruf: 실패 메일 보고서 수신 주소
DMARC = SPF+DKIM 통합 정책 / p=none·quarantine·reject / From 정렬 검증 / 보고서 제공

SPF · DKIM · DMARC 동작 흐름

발신 서버
DKIM 서명 추가
수신 서버
수신 서버 검증 순서:
① SPF 검증: 발신 IP가 DNS TXT(SPF)에 허가된 IP인가?
② DKIM 검증: DNS 공개키로 DKIM-Signature 유효성 확인
③ DMARC 정책 적용: SPF·DKIM 결과 + From 정렬 확인 → p= 정책에 따라 처리
④ 보고서 발송: rua/ruf 주소로 인증 결과 보고
검증 순서: SPF(IP확인) → DKIM(서명확인) → DMARC(정책적용·보고)

SET (Secure Electronic Transaction)

항목내용
정의Visa·MasterCard가 공동 개발한 전자상거래 지불 프로토콜 (1996)
목적인터넷 신용카드 결제 보안 — 카드번호 보호, 상점에게 카드번호 노출 방지
핵심 기술이중 서명 (Dual Signature) — 구매 정보(상점)와 결제 정보(은행)를 분리 서명
참여자고객 - 상점 - 결제 게이트웨이 - 발급 은행 - 브랜드(Visa 등)
이중 서명 효과상점은 결제 금액만, 은행은 구매 내역을 알 수 없음 → 프라이버시 보호
현재복잡성·비용으로 실제 배포 실패 → SSL/TLS 기반 결제로 대체
SET = Visa·MasterCard / 이중 서명으로 카드번호 보호 / 복잡성으로 실패 → TLS로 대체

SSL/TLS (Secure Sockets Layer / Transport Layer Security)

항목내용
위치전송계층과 응용계층 사이 (4.5계층) — HTTPS·SMTPS·IMAPS 등에 활용
버전SSL 1.0~3.0(폐기) → TLS 1.0·1.1(폐기) → TLS 1.2·1.3(현행)
핵심 기능핸드셰이크(인증·키 교환) + 레코드(암호화·무결성)
TLS 1.3 특징불필요한 암호 알고리즘 제거, 0-RTT(Zero Round-Trip), 순방향 비밀성(PFS) 필수화
인증서X.509 인증서, CA 계층 구조
이메일 적용STARTTLS(SMTP 보안 업그레이드), SMTPS(465), IMAPS(993), POP3S(995)
TLS 1.3 = 현행 표준 / 0-RTT·PFS 필수 / STARTTLS로 SMTP 보안 업그레이드

핵심 암기 포인트

  • PEM: 계층적CA, DES+RSA, Base64, 사실상 폐기
  • PGP: Web of Trust(분산), 하이브리드, OpenPGP
  • S/MIME: X.509 인증서, 계층적CA, 기업 표준
  • SPF: DNS TXT에 허가 IP 등록, Return-Path 검증, 포워딩 실패
  • DKIM: 개인키 서명+DNS 공개키, DKIM-Signature 헤더, 포워딩 유지
  • DMARC: SPF+DKIM 통합, p=none/quarantine/reject, 보고서
🎯

자주 나오는 구분 포인트

  • PGP vs S/MIME: PGP=Web of Trust(분산) / S/MIME=X.509(계층적CA)
  • SPF vs DKIM: SPF=IP검증 / DKIM=서명검증 / SPF는 포워딩 실패
  • DMARC 역할: SPF·DKIM 결과로 처리 정책 결정 (none/quarantine/reject)
  • SET 이중 서명: 상점=주문정보만, 은행=결제정보만 알 수 있게 분리
  • TLS 1.3: 0-RTT, PFS 필수, 불안전 알고리즘 제거
