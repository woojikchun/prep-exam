GoF 디자인 패턴 개요 ⭐

Gang of Four(GoF): Erich Gamma·Richard Helm·Ralph Johnson·John Vlissides가 1994년 출판한 «Design Patterns: Elements of Reusable Object-Oriented Software»에서 23가지 객체지향 디자인 패턴을 정리. 패턴 = 반복적으로 나타나는 문제에 대한 검증된 해결책의 템플릿

생성 패턴 (Creational Patterns) — 객체 생성 캡슐화 ⭐

① Singleton — 인스턴스 1개 보장 ⭐⭐
▾
  • 목적: 클래스의 인스턴스가 오직 하나만 생성되고 전역에서 접근 가능하도록 보장
  • 구현: private 생성자 + static 인스턴스 + static getInstance()
  • 스레드 안전 ⭐: double-checked locking / synchronized / enum Singleton
  • 문제점: 전역 상태 → 테스트 어려움 / SRP 위반 가능 / 안티패턴 논란
  • 사용: Logger·DB 커넥션풀·설정 관리·캐시
// Java — Double-Checked Locking class Singleton { private static volatile Singleton instance; private Singleton() {} public static Singleton getInstance() { if (instance == null) { synchronized(Singleton.class) { if (instance == null) instance = new Singleton(); } } return instance; } }
② Factory Method — 서브클래스가 생성 결정 ⭐⭐
▾
  • 목적: 객체 생성을 서브클래스에 위임 — 어떤 클래스를 생성할지 결정은 서브클래스가
  • 핵심: Creator(추상)의 createProduct() 팩토리 메서드를 서브클래스가 오버라이드
  • OCP 달성: 새 제품 추가 시 ConcreteCreator만 추가
  • 사용: Java Collection.iterator() / Spring BeanFactory
  • vs Abstract Factory: Factory Method=1개 제품 / Abstract Factory=제품군
// 추상 Creator abstract class Dialog { abstract createButton(): Button; render() { createButton().render(); } } // 구체 Creator class WinDialog extends Dialog { createButton() { return new WinButton(); } } class MacDialog extends Dialog { createButton() { return new MacButton(); } }
③ Abstract Factory — 관련 객체 군(family) 생성 ⭐
▾
  • 목적: 관련 있는 여러 객체를 일관성 있게 생성하는 인터페이스 제공
  • 핵심: AbstractFactory 인터페이스 → 구체 팩토리들이 제품군 생성
  • 예시: UI 테마(Windows팩토리→WinButton+WinCheckbox / Mac팩토리→MacButton+MacCheckbox)
  • 사용: GUI 툴킷·크로스플랫폼 라이브러리
interface UIFactory { createButton(): Button; createCheckbox(): Checkbox; } class WinFactory implements UIFactory { createButton() { return new WinButton(); } createCheckbox() { return new WinCheckbox(); } }
④ Builder — 단계적 복잡 객체 생성 ⭐
▾
  • 목적: 복잡한 객체의 생성 과정을 단계별로 분리 — 동일한 생성 과정으로 다른 표현 생성
  • 핵심: Builder 인터페이스 + Director(순서 지정) + ConcreteBuilder
  • Java 사용: StringBuilder·Stream.builder()·Lombok @Builder
  • vs Factory: Builder=복잡한 단계적 생성 / Factory=단순 생성 결정
// 메서드 체이닝 스타일 Builder class Pizza { private size, cheese, pepperoni; static class Builder { setSize(s) { this.size=s; return this; } setCheese() { this.cheese=true; return this; } setPepperoni() { this.pepperoni=true; return this; } build() { return new Pizza(this); } } } // 사용: new Pizza.Builder().setSize("L").setCheese().build()
⑤ Prototype — 복사(clone)로 생성 ⭐
▾
  • 목적: 기존 객체를 복사(clone)하여 새 객체 생성 — 생성 비용이 비쌀 때 유용
  • 핵심: Cloneable 인터페이스 / clone() 메서드 구현
  • 얕은 복사 vs 깊은 복사: 참조 타입 필드는 깊은 복사 필요
  • 사용: Java Object.clone() / 게임 유닛 생성 / DB 결과 복사
  • vs Singleton: Prototype=복사 多 / Singleton=하나만
패턴핵심 의도SOLID 연관대표 사용

구조 패턴 (Structural Patterns) — 클래스·객체 구성 ⭐

① Adapter — 인터페이스 변환·호환 ⭐⭐
▾
  • 목적: 호환되지 않는 인터페이스를 가진 클래스들을 함께 동작하도록 변환
  • 별칭: Wrapper — 기존 클래스를 감싸서 새 인터페이스 제공
  • Class Adapter: 다중 상속 활용 (C++ 주로)
  • Object Adapter ⭐: 컴포지션으로 감싸기 (Java 주로)
  • 사용: 레거시 코드 통합 / 서드파티 라이브러리 연동
// 220V → 110V 어댑터 interface Target { request110V(); } class Adaptee { request220V() {/*기존*/} } class Adapter implements Target { private adaptee: Adaptee; request110V() { // 220V 호출 후 변환 this.adaptee.request220V(); } }
② Decorator — 기능 동적 추가·래핑 ⭐⭐
▾
  • 목적: 객체에 동적으로 새 기능을 추가 — 상속 없이 기능 확장
  • 핵심: Component 인터페이스를 구현하면서 동시에 Component 참조를 가짐
  • OCP 달성: 기존 컴포넌트 수정 없이 데코레이터 추가로 기능 확장
  • 사용: Java I/O (BufferedInputStream·DataInputStream) / 로그·암호화·압축
  • vs Adapter: Decorator=기능 추가 / Adapter=인터페이스 변환
interface Coffee { getCost(): number; } class SimpleCoffee implements Coffee { getCost() { return 1000; } } class MilkDecorator implements Coffee { constructor(private c: Coffee) {} getCost() { return this.c.getCost() + 300; } } // new MilkDecorator(new SimpleCoffee()).getCost()
③ Facade — 복잡한 서브시스템 단순화 ⭐⭐
▾
  • 목적: 복잡한 서브시스템에 단순한 통합 인터페이스 제공
  • 핵심: 외부 클라이언트는 Facade만 알면 됨 — 서브시스템 숨김
  • 결합도 감소: 클라이언트-서브시스템 간 직접 의존 제거
  • 사용: 홈씨어터 컨트롤러 / 은행 API / 프레임워크 API
  • 계층형 Facade: 각 계층마다 Facade로 상위에 단순 API 제공
④ Composite — 트리 구조·전체-부분 ⭐
▾
  • 목적: 객체를 트리 구조로 구성하여 단일 객체와 복합 객체를 동일하게 취급
  • Component: Leaf와 Composite의 공통 인터페이스
  • Leaf: 자식 없는 끝 노드
  • Composite: 자식을 가진 내부 노드 — 자식에게 작업 위임
  • 사용: 파일 시스템(파일/디렉터리) / GUI 컴포넌트 트리 / 메뉴 구조
interface Component { display(); } class Leaf implements Component { display() { /* 파일 출력 */ } } class Composite implements Component { children: Component[] = []; add(c: Component) { children.push(c); } display() { children.forEach(c => c.display()); // 위임 } }
⑤ Proxy · ⑥ Bridge · ⑦ Flyweight
▾
패턴목적핵심 구조사용 예
Proxy ⭐ 실제 객체의 대리자 — 접근 제어·지연 로딩·로깅 Subject 인터페이스 → RealSubject와 Proxy가 모두 구현 / Proxy가 RealSubject 참조 Spring AOP·원격 프록시·가상 프록시·보안 프록시

Flyweight 공유 객체로 메모리 절약 — 다수의 유사 객체 공유 가능한 내부 상태(Intrinsic)와 비공유 외부 상태(Extrinsic) 분리 텍스트 에디터 글자 객체 / 게임 나무·풀 오브젝트

행위 패턴 I — Observer · Strategy · Command · Template Method · Iterator · Chain

① Observer — 이벤트 발행-구독 ⭐⭐
▾
  • 목적: 객체의 상태 변화를 여러 의존 객체에 자동 통보 — 1:N 의존 관계
  • Subject(Publisher) ⭐: 상태를 가지고 Observer 목록 관리·통보
  • Observer(Subscriber) ⭐: Subject 변화를 수신하는 인터페이스
  • 느슨한 결합: Subject는 Observer 인터페이스만 알면 됨
  • 사용 ⭐: Event Listener / MVC Model-View / Redux / RxJS / EDA Pub/Sub
  • Push vs Pull: Push=Subject가 데이터 전달 / Pull=Observer가 요청
interface Observer { update(data): void; } class EventSource { observers: Observer[] = []; subscribe(o: Observer) { observers.push(o); } notify(data) { observers.forEach(o => o.update(data)); } } // Observer들은 update()만 구현 // Subject는 Observer 구체 타입을 모름
Observer vs Mediator: Observer=1:N 방송 / Mediator=N:N 조율(중재자가 조율)
② Strategy — 알고리즘 교체 가능 ⭐⭐
▾
  • 목적: 알고리즘 군을 정의하고 캡슐화 — 런타임에 교체 가능
  • Context ⭐: Strategy 인터페이스를 가지며 실행 위임
  • ConcreteStrategy: 구체 알고리즘 구현
  • OCP 달성 ⭐: 새 알고리즘 추가 시 Context 수정 없음
  • 사용: 정렬 알고리즘 교체 / 결제 방법 / 경로 탐색 / 압축 방식
  • vs Template Method: Strategy=구성(위임) / Template=상속
interface SortStrategy { sort(arr): void; } class BubbleSort implements SortStrategy { ... } class QuickSort implements SortStrategy { ... } class Sorter { // Context constructor(private strategy: SortStrategy) {} setStrategy(s: SortStrategy) { strategy = s; } sort(arr) { strategy.sort(arr); } // 위임 }
③ Template Method — 알고리즘 골격 정의 ⭐⭐
▾
  • 목적: 알고리즘의 골격(skeleton)을 부모가 정의하고, 세부 단계를 서브클래스가 구현
  • templateMethod() ⭐: final로 선언 — 순서 고정·서브클래스 변경 불가
  • Hook 메서드: 서브클래스가 선택적으로 오버라이드
  • OCP + DIP: 고정된 골격(추상화)에 다양한 구현
  • 사용: 커피/티 제조 과정 / JUnit setUp/tearDown / HTTP 요청 처리
  • vs Strategy: Template=상속(정적) / Strategy=구성(동적, 런타임 교체)
abstract class Beverage { // 골격 — final로 고정 final prepare() { boilWater(); // 공통 brew(); // 추상 — 서브클래스 구현 addCondiments(); // Hook (선택적) } abstract brew(): void; addCondiments() {} // 기본 빈 구현 } class Coffee extends Beverage { brew() { /* 커피 내리기 */ } }
④ Command · ⑤ Iterator · ⑥ Chain of Responsibility
▾
패턴목적 & 핵심사용 예
Command ⭐ 요청을 객체로 캡슐화 — 실행 취소(Undo)·재실행·요청 큐 가능 / Invoker → Command → Receiver 텍스트 편집기 Ctrl+Z / 트랜잭션 / 매크로 / 원격 제어
Iterator ⭐ 컬렉션 내부 구조를 노출하지 않고 순차 접근 / hasNext()·next() 인터페이스 Java Iterator / for-each / Python __iter__ / DB 커서
Chain of Responsibility ⭐ 요청을 처리할 수 있는 핸들러 체인으로 전달 — 처리 가능한 핸들러가 처리, 불가능 시 다음으로 전달 이벤트 버블링 / 로그 레벨 체인 / Spring Security 필터 체인 / 미들웨어
// Command 패턴 — Undo 가능 interface Command { execute(): void; undo(): void; } class WriteCommand implements Command { constructor(private editor, private text) {} execute() { editor.insert(text); } undo() { editor.delete(text); } }
Observer=이벤트발행구독 / Strategy=알고리즘교체 / Template Method=골격정의·final / Command=요청객체화·Undo / Chain=처리체인

행위 패턴 II — State · Mediator · Memento · Visitor · Interpreter

① State — 상태에 따른 행위 변경 ⭐
▾
  • 목적: 객체의 내부 상태가 바뀔 때 행동이 바뀌도록 — if/else 없이 상태 패턴으로 분기 제거
  • Context: 현재 State를 참조하고 State에 행위 위임
  • State 인터페이스: 상태별 행위 정의
  • 사용: 신호등 / 자판기 / 주문 상태 / TCP 연결 상태
  • vs Strategy: State=상태에 따라 자동 전환 / Strategy=외부에서 선택
interface State { insert(): void; eject(): void; } class NoMoney implements State { insert() { /* 돈 넣기 → GotMoney로 전환 */ } eject() { /* 돈 없어서 불가 */ } } class GotMoney implements State { eject() { /* 반환 → NoMoney로 전환 */ } }
② Mediator — 중재자·직접 소통 제거 ⭐
▾
  • 목적: 객체들 간의 직접 통신을 제거하고 중재자를 통해서만 소통 — 결합도 감소
  • Mediator ⭐: 모든 컴포넌트가 Mediator를 통해 통신
  • 동료 객체(Colleague): 서로를 모름 / Mediator만 알면 됨
  • 사용: 채팅방 / 항공 관제 시스템 / UI 폼 요소 간 조율
③ Memento · ④ Visitor · ⑤ Interpreter
▾
패턴목적 & 핵심사용 예
Memento ⭐ 객체의 내부 상태를 스냅샷으로 저장하고 나중에 복원 — 캡슐화 위반 없이 Undo/Redo 구현 / Originator·Memento·Caretaker 텍스트 편집기 Undo / 게임 세이브 / 데이터베이스 롤백
Visitor 객체 구조에서 알고리즘을 분리 — 기존 클래스 수정 없이 새 연산 추가 / double dispatch AST(추상 구문 트리) 연산 / 세금 계산 / 파일 시스템 탐색
Interpreter 언어의 문법 규칙을 클래스로 표현하고 해석기 구현 / Terminal·Nonterminal Expression SQL 파서 / 정규표현식 / 스크립트 언어 인터프리터
// Memento 패턴 — Undo 구현 class Editor { // Originator private text = ""; save() { return new Memento(text); } restore(m: Memento) { text = m.getText(); } } class Memento { constructor(private text: string) {} getText() { return this.text; } } // Caretaker: history 배열로 Memento 관리

혼동하기 쉬운 패턴 비교 ⭐ — 감리사 핵심

비교 쌍패턴 A패턴 B핵심 차이
Adapter vs Decorator 인터페이스 변환 (다른 인터페이스로) 기능 추가 (같은 인터페이스 유지) Adapter=변환·호환 / Decorator=기능확장
Adapter vs Facade 1개 객체의 인터페이스 변환 여러 서브시스템을 단순화 Adapter=1:1 변환 / Facade=N:1 단순화
Adapter vs Proxy 다른 인터페이스 제공 동일 인터페이스 유지·접근 제어 Adapter=인터페이스 변경 / Proxy=동일 인터페이스
Strategy vs Template Method ⭐ 구성(Composition)·런타임 교체 상속(Inheritance)·컴파일 타임 결정 Strategy=위임·동적 / Template=상속·정적
Strategy vs State 외부에서 알고리즘 선택 상태 변화 시 자동 전환 Strategy=의도적 선택 / State=자동 전환
Observer vs Mediator 1:N 방향·이벤트 방송 N:N 중재·직접 통신 차단 Observer=Subject→Observer들 / Mediator=모든쌍 조율
Factory Method vs Abstract Factory 1개 제품 생성·서브클래스 결정 관련 제품군 일관 생성 FM=1개 / AF=군(family) 전체
Decorator vs Composite 1개 컴포넌트 래핑·기능 추가 트리 구조·전체-부분 처리 Decorator=래핑체인 / Composite=트리 n개 자식
Command vs Strategy 요청 객체화·Undo·큐 알고리즘 교체·인터페이스 통일 Command=요청·실행취소 / Strategy=알고리즘 선택
