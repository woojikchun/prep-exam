XP (eXtreme Programming) ⭐

XP 5대 가치

  • 의사소통 (Communication): 팀 내 지속적 소통
  • 단순성 (Simplicity): 가장 단순한 해결책
  • 피드백 (Feedback): 짧은 주기 피드백 루프
  • 용기 (Courage): 리팩토링·변경 두려움 없음
  • 존중 (Respect): 팀원 상호 존중

XP 핵심 실천법 ⭐

  • TDD (테스트 주도 개발) ⭐: 테스트 먼저 → 코드 → 리팩토링
  • 페어 프로그래밍 ⭐: 2명이 1컴퓨터에서 협력
  • CI (지속적 통합) ⭐: 코드 변경 시마다 빌드·테스트
  • 리팩토링 ⭐: 기능 변경 없이 코드 구조 개선
  • 소규모 릴리즈: 자주 작은 단위로 배포
  • 공동 코드 소유: 누구든 코드 수정 가능
  • 40시간 주당 근무: 지속 가능한 페이스
  • 온사이트 고객: 고객이 개발팀과 함께
TDD: 테스트 먼저 → 코드 → 리팩토링 순서! (코드 먼저가 아님) / XP=기술실천법 중심, Scrum=프로세스 중심
XP 핵심: TDD·페어프로그래밍·CI·리팩토링·소규모릴리즈 / TDD는 XP의 실천법이자 독립 기법

Kanban & 린 SW 개발 ⭐

Kanban 핵심 ⭐

  • WIP 제한 ⭐: 진행 중 작업 수 제한 → 흐름 최적화
  • 작업 시각화 ⭐: Kanban Board
  • 타임박스 없음: 스프린트·이터레이션 없음
  • 역할 정의 없음: PO·SM 개념 없음
  • 리드타임: 고객 요청→배포까지 총 시간
  • 사이클타임: 작업 시작→완료까지 시간
  • Little's Law: WIP = 처리량 × 사이클타임

린 SW 개발 7대 원칙 ⭐

  • 낭비 제거
  • 품질 내재화
  • 지식 창출
  • 결정 지연 (마지막 책임 시점)
  • 빠른 전달
  • 팀 존중
  • 전체 최적화
Kanban=WIP제한+작업시각화·타임박스·역할 없음 / 린=낭비제거·결정지연·전체최적화 / Kanban≠Scrum

DSDM (Dynamic Systems Development Method) ⭐

DSDM: 1994년 영국에서 RAD(신속 애플리케이션 개발)를 기반으로 탄생한 애자일 프레임워크 — 고정된 일정과 예산 내에서 범위(기능)를 유연하게 조정하는 방식. 현재는 Agile Project Framework(APF)로 발전
DSDM 8대 원칙 ⭐
▾
원칙 01
비즈니스 필요성에 집중 — 모든 결정을 비즈니스 가치 기준으로
원칙 02
정시 납품 ⭐ — 기간과 예산은 고정, 범위(기능)를 조정
원칙 03
협업 ⭐ — 비즈니스·기술팀 긴밀한 협력
원칙 04
품질 저하 없음 ⭐ — 기준 아래로 품질 타협하지 않음
원칙 05
견고한 기반 위에 점진적 구축 — 탄탄한 기반 위 증분적 개발
원칙 06
반복적·점진적 개발 ⭐ — Timeboxing 기반 반복
원칙 07
지속적인 명확한 의사소통 — 이해관계자 참여 필수
원칙 08
제어 시현 — 계획 대비 실제를 측정·관리

파운데이션 ⭐비즈니스 목표·요구사항·아키텍처 기반 수립비즈니스 케이스·우선순위화 요구사항·시스템 아키텍처

DSDM 핵심 기법 ⭐
▾
  • Timeboxing ⭐: 고정된 기간(Timebox) 내에 완료 가능한 최고 우선순위 기능에 집중 — 기간은 고정, 범위는 유연
  • MoSCoW 우선순위 ⭐: Must Have(필수) / Should Have(중요) / Could Have(있으면 좋음) / Won't Have(이번에는 제외)
  • 프로토타이핑 ⭐: 비즈니스·기술 요구사항 검증을 위한 반복적 프로토타입 개발
  • DSDM 팀 역할: Executive Sponsor(비즈니스 비전) / Project Manager / Business Analyst / Solution Developer / Solution Tester
  • 워크숍(Workshop): 비즈니스와 기술팀이 함께 참여하는 협업적 요구사항 정의
  • 일일 스탠드업: Scrum의 Daily Scrum과 유사한 짧은 팀 동기화
DSDM의 핵심 철학: "기간·예산은 고정, 기능(범위)은 협상 가능" — 전통 방법론(범위 고정)과 반대!

TDD (Test-Driven Development, 테스트 주도 개발) ⭐⭐

TDD: Kent Beck이 XP의 실천법으로 제안 — 코드를 작성하기 전에 테스트를 먼저 작성하고, 테스트를 통과하는 최소한의 코드를 작성한 뒤 리팩토링하는 반복적 개발 방식
🔴
Red
실패 테스트 작성
→
🟢
Green
최소 코드로 통과
→
🔵
Refactor
코드 개선·정리
→ 반복
TDD 3단계 (Red → Green → Refactor) ⭐
▾
단계색상행동목적
① Red ⭐🔴 빨강아직 없는 기능에 대한 실패하는 테스트 먼저 작성요구사항 명확화·설계 유도
② Green ⭐🟢 초록테스트를 통과시키는 최소한의 코드만 작성기능 구현·테스트 통과
③ Refactor ⭐🔵 파랑기능 변경 없이 중복 제거·가독성·구조 개선코드 품질 향상·기술부채 제거
// 예시: 계산기 더하기 기능 TDD // ① Red — 먼저 테스트 작성 (아직 add() 없음) @Test void testAdd() { Calculator calc = new Calculator(); assertEquals(5, calc.add(2, 3)); // → 컴파일 오류/실패 } // ② Green — 최소 코드 작성 int add(int a, int b) { return a + b; } // 테스트 통과! // ③ Refactor — 코드 정리 (기능 동일 유지)
TDD 효과·도구·한계 ⭐
▾

TDD 효과

  • 회귀 버그 방지: 기존 테스트가 안전망 역할
  • 설계 개선: 테스트하기 쉬운 코드 = 좋은 설계 유도
  • 문서화: 테스트 코드 자체가 실행 가능한 명세
  • 리팩토링 자신감: 테스트가 있어 변경이 두렵지 않음
  • 디버깅 시간 단축: 오류를 작은 단계에서 조기 발견

TDD 도구 & 한계

  • Java: JUnit, Mockito
  • JavaScript: Jest, Jasmine, Mocha
  • Python: pytest, unittest
  • C#: NUnit, xUnit
  • 한계: 초기 개발 속도 저하 / 테스트 설계 스킬 필요 / UI·레거시 코드 적용 어려움
TDD=테스트먼저(Red)→최소코드(Green)→리팩토링(Blue) 반복 / Kent Beck·XP / 코드먼저가 아님! / JUnit·Jest·pytest

BDD (Behavior-Driven Development, 행위 주도 개발) ⭐⭐

BDD: Dan North가 TDD를 개선하여 2003년 제안 — 비기술적 이해관계자도 이해할 수 있는 자연어(Given-When-Then)로 시스템의 행위(Behavior)를 명세하고 이를 실행 가능한 테스트로 만드는 개발 방식
BDD 핵심 — Given-When-Then 문법 ⭐
▾
키워드의미대응 개념
Given ⭐사전 조건·초기 상태 설정Arrange (테스트 준비)
When ⭐특정 행동/이벤트 발생Act (테스트 실행)
Then ⭐기대 결과·검증Assert (결과 검증)
And / But추가 조건·결과 연결다수 조건 연결
# Gherkin 문법 예시 (Feature 파일) Feature: 사용자 로그인 Scenario: 올바른 자격증명으로 로그인 성공 Given 사용자가 "user@test.com" 이메일로 가입되어 있다 When 사용자가 이메일 "user@test.com"과 비밀번호 "pass123"으로 로그인 시도 Then 로그인이 성공하고 대시보드 페이지로 이동한다 Scenario: 잘못된 비밀번호로 로그인 실패 Given 사용자가 "user@test.com" 이메일로 가입되어 있다 When 사용자가 잘못된 비밀번호 "wrong"으로 로그인 시도 Then "비밀번호가 일치하지 않습니다" 오류 메시지가 표시된다

▾

  • 개발자: "어떻게(How)" — 기술 구현
  • 테스터: "무엇이 잘못될 수 있나(What if)" — 엣지 케이스
  • 세 역할이 함께 시나리오 작성 → 공통 이해 형성

SAFe 핵심 구성 요소 ⭐
▾
레벨구성역할·특징
Portfolio LevelLean Portfolio Management / Epic전략·투자 의사결정 / 가치 흐름 관리
Program Level ⭐ART(Agile Release Train) / PI Planning여러 팀 동기화 / 8~12주 PI 단위 계획
Team Level ⭐Scrum / Kanban 팀2주 스프린트 / 개별 팀 개발
  • ART (Agile Release Train) ⭐: 5~12개 팀이 동기화하여 PI 단위로 가치 전달
  • PI (Program Increment) ⭐: 8~12주 단위 계획 주기 (4~5개 스프린트 포함)
  • PI Planning ⭐: 모든 팀이 모여 다음 PI 목표·의존성 계획 — 2일 핵심 이벤트
  • RTE (Release Train Engineer): ART 레벨의 Scrum Master
