클린 코드 정의 & 핵심 철학 ⭐

Clean Code: Robert C. Martin(Uncle Bob)이 2008년 정리한 소프트웨어 장인 정신 — 읽기 쉽고, 이해하기 쉽고, 유지보수하기 쉬운 코드를 작성하는 원칙과 실천법의 집합

저명한 개발자들의 클린 코드 정의 ⭐

Bjarne Stroustrup (C++ 창시자)

"우아하고 효율적인 코드를 좋아한다. 논리가 간단해야 버그가 숨어들기 힘들다. 한 가지 일만 제대로 한다."

Grady Booch (OOA/OOD)

"단순하고 직접적이다. 설계자의 의도를 숨기지 않는다. 명확한 추상화로 가득하다."

Ward Cunningham (기술부채 창시자)

"코드를 읽으면서 짐작했던 대로 돌아가는 코드. 아름다운 코드는 당신을 가장 단순한 방법으로 이끈다."

Robert C. Martin (Clean Code 저자)

"나중에 코드를 읽는 사람이 자신이라는 사실을 기억하라. 읽기 쉬운 코드를 작성하는 것이 전문가의 의무다."

클린 코드의 핵심 가치 ⭐

  • 가독성(Readability) ⭐: 코드는 쓰는 것보다 읽는 시간이 10:1 — 읽기 쉬운 코드가 좋은 코드
  • 단순성(Simplicity): 복잡성은 모든 문제의 근원 — KISS(Keep It Simple Stupid)
  • 의도 표현(Expressive): 코드가 개발자의 의도를 명확히 전달
  • 유지보수성: 변경하기 쉬운 코드 — 수정 비용 최소화
  • 중복 제거 ⭐: DRY(Don't Repeat Yourself) 원칙

나쁜 코드의 비용 ⭐

  • 생산성 저하: 레거시 코드 이해에 시간 대부분 소비
  • 나쁜 코드가 나쁜 코드를 낳음: 급히 작성한 코드에 급히 코드 추가
  • 기술 부채 누적: 리팩토링 없이 기능만 추가 시 복잡도 급증
  • 재작성 유혹: 처음부터 다시 짜고 싶은 충동 → 더 나빠지는 경우 多
  • LeBlanc의 법칙 ⭐: "나중은 결코 오지 않는다(Later equals never)"
클린코드=가독성·단순성·의도표현·DRY / LeBlanc의법칙="나중은 결코 오지 않는다" / 코드 읽기:쓰기=10:1

범위(Scope) 별 이름 길이

  • 넓은 범위(전역·클래스): 긴 이름 — 의미 명확히
  • 좁은 범위(지역변수·루프): 짧은 이름 허용 — i, j, k
  • 개념에 맞는 한 단어: fetch·retrieve·get 중 하나만 일관성 있게
  • 해법 영역 이름 ⭐: CS 용어 허용 (JobQueue, AccountVisitor)
  • 문제 영역 이름: 도메인 용어 우선 (Order, Invoice, Customer)

함수 (Functions) 원칙 ⭐⭐

함수 핵심 원칙 7가지 ⭐
▾

① 작게 만들어라 (Small) ⭐

함수는 작을수록 좋다 — 20줄 이하 권장. if/else·while 문 안의 블록은 1줄(함수 호출)이어야 한다. 들여쓰기는 1~2단계가 최대.

② 한 가지만 해라 (Do One Thing) ⭐⭐

"함수는 한 가지를 해야 한다. 그 한 가지를 잘 해야 한다. 그 한 가지만을 해야 한다." — 함수 내부를 여러 섹션으로 나눌 수 있다면 한 가지 이상 하는 것

③ 추상화 수준을 일치시켜라 (One Level of Abstraction) ⭐

한 함수 내의 모든 문장은 동일한 추상화 수준을 유지해야 한다 — 고수준(getHtml())과 저수준(buffer.append("\n"))을 섞지 않기

④ 내려가기 규칙 (The Stepdown Rule)

코드는 신문처럼 위에서 아래로 읽혀야 한다 — 함수 추상화 수준이 위→아래로 낮아짐

⑤ 매개변수 최소화 ⭐

0개가 이상적, 1개(단항), 2개(이항), 3개 이상은 피하라. 플래그 매개변수(boolean)는 금지 — 함수가 2가지 일을 한다는 신호

render(bool isSuite) // 금지! 2가지 일 renderForSuite(); // 두 함수로 분리 renderForSingleTest();

⑥ 부수 효과 없애라 (No Side Effects) ⭐

함수 이름과 다른 숨겨진 동작 금지 — checkPassword()가 세션도 초기화하면 안 됨

⑦ 명령과 조회 분리 (CQS: Command-Query Separation) ⭐

함수는 무언가를 수행하거나(Command) 대답하거나(Query) 둘 중 하나만 — 둘 다 하면 혼란
if (set("username","bob")) // 설정하고 성공여부도?

함수: ①작게(20줄) ②한가지만 ③추상화수준일치 ④내려가기규칙 ⑤매개변수최소화(플래그금지) ⑥부수효과없음 ⑦CQS(명령조회분리)

주석 (Comments) 원칙 ⭐

"주석은 나쁜 코드를 보완하지 못한다. 코드로 의도를 표현하라." — 좋은 코드는 주석이 필요 없다. 주석을 달아야 한다면 먼저 코드를 개선할 방법을 찾아라.

✅ 좋은 주석 (허용)

  • 법적 주석: 저작권·라이선스 등
  • TODO 주석: 아직 구현 안 된 것 표시
  • 경고 주석: 특정 이유로 결과가 다를 수 있음
  • 의도 설명: 코드로 표현 불가한 결정 이유
  • 공개 API JavaDoc: 공개 API는 문서 필요

❌ 나쁜 주석 (제거 대상)

  • 코드 자체 반복 ⭐: 코드를 그대로 주석으로
  • 주석 처리한 코드 ⭐: git이 있으므로 삭제하면 됨
  • 너무 많은 설명: 한 줄 코드에 10줄 주석
  • 오래된 주석: 코드와 맞지 않는 주석
  • 의무적 JavaDoc: 모든 함수에 자동 생성 주석
  • 위치 표시: // Actions ///////////// 구분선
// 나쁜 주석 예시 // 직원이 복지 혜택을 받을 수 있는지 검사한다 if ((employee.flags & HOURLY_FLAG) && (employee.age > 65)) // 좋은 코드 — 주석 필요 없음 if (employee.isEligibleForFullBenefits())
주석=나쁜 코드를 보완 못함 / 코드로 의도 표현 / 주석 처리한 코드=즉시 삭제 / 좋은주석=법적·TODO·경고·의도 / 나쁜주석=반복·오래됨

형식 (Formatting) 원칙 ⭐

세로 형식

  • 신문 기사처럼 ⭐: 위쪽=고수준 요약, 아래=세부 사항
  • 개념 분리: 빈 줄로 관련 코드 그룹 구분
  • 관련 코드 밀집: 관련 있는 코드는 세로로 가까이
  • 변수 선언: 사용 위치에 가깝게 선언
  • 인스턴스 변수: 클래스 맨 위에 선언
  • 종속 함수: 호출하는 함수를 호출되는 함수 위에

가로 형식

  • 짧은 행 ⭐: 80~120자 이내 권장
  • 가로 정렬 지양: 타입 정렬 위해 공백 과용 금지
  • 들여쓰기 ⭐: 범위(Scope)를 시각적으로 표현 — 들여쓰기 생략 금지
  • 팀 규칙 ⭐: 개인 취향보다 팀 규칙 우선 — 일관성이 핵심
형식: 신문기사처럼(위=요약·아래=세부) / 빈줄=개념분리 / 행길이 80~120자 / 들여쓰기생략금지 / 팀규칙우선

오류 처리 (Error Handling) 원칙 ⭐

"오류 처리는 중요하다. 하지만 오류 처리 코드로 인해 프로그램 논리를 이해하기 어려워진다면 깨끗한 코드라 부르기 어렵다."
오류 처리 핵심 원칙 ⭐
▾

① 오류 코드보다 예외를 사용하라 ⭐

// 나쁨 — 오류 코드 확인 강제 if (deletePage(page) == ERROR_RESPONSE) { if (registry.delete(page.name) == ERROR_RESPONSE) { ... } } // 좋음 — 예외로 분리 try { deletePage(page); registry.delete(page.name); } catch (Exception e) { logger.log(e.getMessage()); }

② Try-Catch-Finally를 먼저 작성하라 ⭐

예외가 발생할 코드를 먼저 try-catch로 작성 — 어떤 상태든 일관성 있게 동작하도록 범위 정의

③ null을 반환하지 마라 ⭐⭐

null 반환 시 호출자가 null 체크 강제 — NullPointerException의 근원. 대신 빈 컬렉션·Optional·Null Object 패턴 사용

// 나쁨 List<Employee> getEmployees() { return null; } // null 반환 금지! // 좋음 List<Employee> getEmployees() { return Collections.emptyList(); }

④ null을 전달하지 마라 ⭐

함수 호출 시 null을 인수로 전달 금지 — null 전달이 필요하다면 설계 자체가 잘못된 것. 방어적 프로그래밍으로 null 체크보다 null을 원천 차단

⑤ 미확인(Unchecked) 예외를 사용하라 ⭐

확인(Checked) 예외는 OCP 위반 — 저수준 메서드의 예외를 고수준 메서드 시그니처에 추가 강제. RuntimeException(Unchecked) 사용 권장

오류처리: 오류코드→예외 / null반환금지(emptyList·Optional) / null전달금지 / Unchecked예외 사용 / Try-Catch-Finally 먼저 작성

경계 (Boundaries) & 디미터 법칙 ⭐

외부 코드 경계 관리 ⭐

  • 경계 패키지 학습 테스트: 외부 API를 사용하기 전에 테스트 코드로 먼저 학습
  • Adapter 패턴 ⭐: 외부 라이브러리를 Adapter로 감싸서 인터페이스 변환 → 교체 용이
  • 경계 격리: 외부 코드와 내부 코드 사이에 명확한 경계 설정
  • Map 노출 금지 ⭐: Map을 공개 API에 노출하지 말고 클래스로 감싸기

디미터 법칙 (Law of Demeter) ⭐⭐

모듈은 자신이 직접 아는 객체의 메서드만 호출해야 한다 — "친구 친구에게 말을 걸지 마라"

// 위반 — 기차 충돌(Train Wreck) String output = ctxt.getOptions().getScratchDir().getAbsolutePath(); // 준수 — 직접 원하는 것을 요청 String output = ctxt.createScratchFileDirectory();
  • 허용: 자신·매개변수·생성 객체·직접 필드 객체의 메서드
  • 기차 충돌(Train Wreck) ⭐: a.getB().getC().getD() 형태 → 위반
경계: Adapter로 외부라이브러리 감싸기·Map공개금지 / 디미터법칙=친구친구금지 / 기차충돌=a.getB().getC()형태 위반

단위 테스트 & FIRST 원칙 ⭐⭐

"테스트 코드는 실제 코드만큼 중요하다." — 테스트 코드가 지저분하면 변경 어려움 → 테스트가 없으면 코드 변경이 두려움 → 경직된 시스템
FIRST 원칙 상세 & 깨끗한 테스트 규칙 ⭐
▾
원칙전체 이름의미위반 시
F — Fast빠른 실행테스트가 빨리 실행되어야 함느리면 자주 실행 안 함 → 문제 늦게 발견
I — Independent독립적테스트들이 서로 의존하지 않음 / 순서 무관한 테스트 실패가 다른 테스트 연쇄 실패
R — Repeatable반복 가능어떤 환경(로컬·CI·운영)에서도 동일 결과환경 의존 시 테스트 신뢰 불가
S — Self-Validating자가 검증실행 결과가 boolean(Pass/Fail) — 수동 확인 불필요로그를 수동으로 읽어야 한다면 오류
T — Timely적시에테스트는 실제 코드 작성 직전에 작성 (TDD)코드 작성 후 테스트 → 테스트하기 어려운 설계
깨끗한 테스트 코드 & 테스트 당 assert 하나 ⭐
▾
  • 테스트 당 개념 하나 ⭐: 테스트 함수에서 하나의 개념만 검증 — assert가 여러 개여도 같은 개념이면 괜찮음
  • Build-Operate-Check 패턴: 테스트 자료 만들고(Build) → 조작하고(Operate) → 검증(Check)
  • Given-When-Then ⭐: 사전조건(Given) → 행동(When) → 검증(Then) — BDD 스타일
  • 테스트 코드도 리팩토링 ⭐: 중복 제거·명확한 이름·간결하게
// 좋은 테스트 — Given-When-Then @Test void calculateTotalPrice_withDiscount() { // Given Order order = new Order(100, 2); DiscountPolicy policy = new TenPercentDiscount(); // When int total = order.calculateTotal(policy); // Then assertEquals(180, total); }
FIRST=Fast(빠름)·Independent(독립)·Repeatable(반복)·Self-Validating(자가검증)·Timely(적시) / 테스트당 개념하나 / Given-When-Then

  • 클래스도 작게 ⭐: 클래스는 작아야 한다 — 함수와 동일한 원칙
  • 단일 책임 원칙(SRP) ⭐: 변경의 이유가 하나여야 한다
  • 높은 응집도(Cohesion) ⭐: 클래스의 메서드와 변수가 서로 관련되어 있어야 함
  • 낮은 결합도(Coupling): 클래스 간 의존도 최소화
  • 변경으로부터 격리 ⭐: 구체적인 클래스에 의존하지 말고 추상화에 의존 (DIP)

응집도 & 클래스 크기 ⭐

// 응집도 낮은 클래스 (나쁨) class SuperDashboard { // 700줄, 65개 메서드 // 보안·렌더링·비즈니스·DB 모두 담당 } // 응집도 높은 클래스 (좋음) class Version { getMajorVersionNumber(); getMinorVersionNumber(); getBuildNumber(); // 버전 정보만 — 응집도 높음 }
  • LCOM (응집도 결여) ⭐: 낮을수록 좋음 — 메서드들이 같은 필드 사용
클래스: 작게·SRP·높은응집도·낮은결합도·변경격리(DIP) / 응집도=클래스내 메서드·변수 관련성 / LCOM 낮을수록 좋음
✨

창발성 (Emergence) — Kent Beck의 단순한 설계 4규칙 ⭐⭐

창발적 설계: Kent Beck이 제안한 단순한 설계 4가지 규칙을 따르면 우수한 설계가 자연스럽게 창발(Emerge)된다 — 각 규칙은 중요도 순서로 나열
4규칙 상세 & 클린 코드 전체 연결 ⭐
▾
규칙 (우선순위 순)세부 내용관련 원칙
규칙 1: 테스트 실행 ⭐⭐ 테스트 없는 시스템은 검증 불가 / 테스트가 가능한 시스템을 만들면 → SRP·DIP 자연히 달성 / 테스트가 많을수록 리팩토링 두려움 감소 TDD·FIRST·SRP·DIP
규칙 2: 중복 제거 ⭐ 코드 중복은 추가 작업·위험·불필요한 복잡도 / 소규모 중복도 적극 제거 / 규칙 1 통과 후 안전하게 리팩토링 DRY·Extract Method·Pull Up·Template
규칙 3: 의도 표현 ⭐ 코드 작성자의 의도를 분명히 표현 / 좋은 이름·작은 함수·표준 패턴 명칭 / 주석보다 코드로 명명규칙·함수규칙·GoF패턴명
규칙 4: 클래스·메서드 최소화 과한 설계(Over Engineering) 방지 / 불필요한 추상화·클래스 생성 지양 / 실용적 관점 유지 YAGNI·KISS·SRP

Clean Code 전체 원칙 연결 고리 ⭐

좋은 이름(명명) → 코드가 의도를 표현 → 주석 최소화 가능
작은 함수(한 가지) → SRP 달성 → 클래스도 작아짐
오류처리로 null 제거 → 디미터 법칙 준수 쉬워짐
FIRST 테스트 → 창발성 규칙 1 달성 → 안전한 리팩토링 → 중복 제거(규칙 2)
SOLID 원칙 + 창발성 4규칙 = 클린한 설계의 자연스러운 달성
