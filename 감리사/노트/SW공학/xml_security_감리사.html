전송 수준 보안 vs 메시지 수준 보안 (빈출)

전송 수준 (TLS/HTTPS)
채널 암호화
전송 구간(Point-to-Point)만 보호. 중간 노드(프록시·라우터)에서 복호화 후 재암호화. End-to-End 보장 어려움
메시지 수준 (XML-Enc/WS-Security)
메시지 자체 암호화
중간 노드를 거쳐도 메시지 내용 보호. 전송 프로토콜 독립. 부분 암호화·서명 가능. End-to-End 보장
감리 핵심: HTTPS만으로는 중간 노드(프록시·ESB)에서 메시지가 노출될 수 있음. 민감 데이터 교환 시스템에서는 XML Encryption/WS-Security로 메시지 수준 보안을 추가해야 함
한 줄 암기XML 보안 3축: 서명(XML-DSig·무결성·부인방지) + 암호화(XML-Enc·기밀성) + 인증(SAML·SSO). 전송보안(TLS) ≠ 메시지보안(End-to-End)

XML Signature (XML-DSig) 개요

W3C + IETF 공동 표준. XML 문서 또는 특정 요소에 디지털 서명을 추가.
서명 대상이 XML 내부·외부 리소스 모두 가능. 부분 서명(특정 요소만) 지원이 핵심 장점.
보안 목표: 무결성 + 인증 + 부인방지.

서명 위치 유형 (빈출)

Enveloped 서명
서명이 원본 안에 포함
서명 요소가 서명 대상 XML 내부에 자식으로 삽입. 가장 일반적. 원본 구조 유지
Enveloping 서명
원본이 서명 안에 포함
서명 요소가 원본 데이터를 자식으로 포함. 서명이 컨테이너 역할
Detached 서명
서명이 원본과 분리
서명과 원본이 별도 요소·파일. URI 참조로 연결. 외부 리소스(이미지 등) 서명에 적합

XML Signature 구조

<Signature xmlns="http://www.w3.org/2000/09/xmldsig#"> <SignedInfo> <CanonicalizationMethod <!-- 정규화 알고리즘 --> Algorithm="http://www.w3.org/TR/2001/REC-xml-c14n-20010315"/> <SignatureMethod <!-- 서명 알고리즘 (RSA-SHA256 등) --> Algorithm="http://www.w3.org/2001/04/xmldsig-more#rsa-sha256"/> <Reference URI="#order-001"> <!-- 서명 대상 참조 --> <DigestMethod Algorithm="...sha-256"/> <DigestValue>ABC123...</DigestValue> <!-- 해시값 --> </Reference> </SignedInfo> <SignatureValue>XYZ789...</SignatureValue> <!-- 암호화된 서명값 --> <KeyInfo> <!-- 서명 검증용 키 정보 --> <X509Data><X509Certificate>...</X509Certificate></X509Data> </KeyInfo> </Signature>

정규화 (Canonicalization, C14N)가 필요한 이유

XML은 논리적으로 동일한 문서도 공백·속성 순서·네임스페이스 선언 위치에 따라 다른 바이트열이 될 수 있음. 해시를 구하기 전 반드시 정규화(C14N)로 표준 표현으로 변환해야 서명 검증이 일치함. 정규화 알고리즘 불일치는 서명 검증 실패 원인 중 하나
한 줄 암기XML Signature = C14N(정규화) → 해시(Digest) → 개인키 서명(SignatureValue). 서명 위치: Enveloped(안에) / Enveloping(감쌈) / Detached(분리)

XML Encryption (XML-Enc) 개요

W3C 표준. XML 문서의 요소·속성·내용·문서 전체를 선택적으로 암호화.
암호화된 내용은 <EncryptedData> 요소로 대체됨.
보안 목표: 기밀성 (Confidentiality). 전송 구간이 아닌 메시지 자체를 보호.

암호화 대상 범위

암호화 대상설명type 속성값
요소 전체시작·종료 태그 포함 전체 암호화http://...#Element
요소 내용만태그는 유지, 내용(텍스트·자식)만 암호화http://...#Content
문서 전체XML 문서 전체를 바이너리로 암호화없음 (기본)
임의 데이터XML 외부 데이터(이미지 등) 암호화 후 삽입MIME 타입

XML Encryption 구조

<EncryptedData xmlns="http://www.w3.org/2001/04/xmlenc#" Type="http://www.w3.org/2001/04/xmlenc#Element"> <EncryptionMethod <!-- 대칭키 암호화 알고리즘 --> Algorithm="http://...#aes256-cbc"/> <KeyInfo xmlns="http://www.w3.org/2000/09/xmldsig#"> <EncryptedKey> <!-- 대칭키를 공개키로 암호화 --> <EncryptionMethod Algorithm="http://...#rsa-oaep-mgf1p"/> <CipherData> <CipherValue>암호화된 대칭키...</CipherValue> </CipherData> </EncryptedKey> </KeyInfo> <CipherData> <CipherValue>암호화된 실제 데이터...</CipherValue> </CipherData> </EncryptedData>

하이브리드 암호화 방식

원본 데이터
→
대칭키(AES)로 암호화
→
CipherValue
대칭키 자체
→
수신자 공개키(RSA)로 암호화
→

XML Encryption은 대칭키(AES)로 데이터를 암호화하고, 그 대칭키를 수신자의 공개키(RSA)로 암호화하는 하이브리드 방식. 공개키 암호화의 속도 문제와 대칭키의 키 배포 문제를 동시에 해결

XML Signature + XML Encryption 함께 사용

서명 후 암호화(Sign-then-Encrypt) vs 암호화 후 서명(Encrypt-then-Sign) 순서에 따라 보안 속성이 달라짐. 일반적으로 서명 후 암호화가 권장됨 — 서명은 평문에, 암호화는 서명된 전체에 적용하여 기밀성과 무결성을 동시에 보장
한 줄 암기XML Encryption = 요소/내용/문서 선택 암호화. AES(데이터) + RSA(대칭키) 하이브리드. 순서: 서명→암호화(Sign-then-Encrypt) 권장

SAML (Security Assertion Markup Language)

OASIS 표준. 인증·권한부여 정보를 XML로 표현·교환하는 프레임워크.
핵심 사용 사례: SSO(Single Sign-On) — 한 번 로그인으로 여러 서비스 접근.
현재 SAML 2.0이 표준. 기업·공공 SSO에서 사실상 표준으로 사용.

SAML 3대 구성 요소

IdP (Identity Provider)
신원 제공자
사용자 인증 수행 후 SAML Assertion 발급. 예: 사내 Active Directory, Google, Okta
SP (Service Provider)
서비스 제공자
SAML Assertion을 받아 서비스 접근 허용. 예: Salesforce, Office365, 전자정부 시스템
Subject (주체)
인증 대상 사용자
IdP에서 인증받고 SP 서비스에 접근하려는 사용자 또는 시스템

Assertion 유형내용용도

Authorization Decision Assertion특정 리소스에 대한 접근 허용·거부접근 제어

WS-Security (WSS) 개요

OASIS 표준. SOAP 메시지의 헤더(Header)에 보안 정보를 추가하는 메시지 수준 보안 프레임워크.
XML Signature·XML Encryption·SAML을 SOAP에 통합. WS-* 보안 스택의 핵심.

WS-* 보안 스택

WS-Federation
이기종 도메인 간 신뢰 연합. 크로스 도메인 SSO
WS-Trust
보안 토큰 서비스(STS). 토큰 발급·갱신·검증·취소
WS-SecureConversation
세션 기반 보안 컨텍스트. 다중 메시지 교환 시 효율적 보안
WS-SecurityPolicy
보안 정책 기술. 어떤 토큰·알고리즘·암호화 방식을 요구하는지 선언
WS-Security (기반)
SOAP Header에 보안 토큰·서명·암호화 추가. XML-DSig + XML-Enc 통합

WS-Security SOAP 메시지 구조

<soap:Envelope> <soap:Header> <wsse:Security <!-- WS-Security 헤더 --> xmlns:wsse="http://...secext-1.0.xsd" soap:mustUnderstand="1"> <wsse:UsernameToken> <!-- 사용자 자격증명 --> <wsse:Username>admin</wsse:Username> <wsse:Password Type="...PasswordDigest"> 해시된_패스워드 </wsse:Password> <wsu:Nonce>랜덤값</wsse:Nonce> <wsu:Created>2025-04-01T09:00:00Z</wsu:Created> </wsse:UsernameToken> <wsse:BinarySecurityToken> <!-- X.509 인증서 --> Base64인코딩된_인증서 </wsse:BinarySecurityToken> <ds:Signature>...</ds:Signature> <!-- XML Signature --> <xenc:EncryptedData>...</xenc:EncryptedData> <!-- XML Encryption --> </wsse:Security> </soap:Header> <soap:Body> ... </soap:Body> </soap:Envelope>

XXE (XML External Entity) 상세

<!-- 공격 예시: 외부 엔티티로 서버 파일 읽기 --> <?xml version="1.0"?> <!DOCTYPE foo [ <!ENTITY xxe SYSTEM "file:///etc/passwd"> <!-- 서버 파일 참조 --> ]> <user><name>&xxe;</name></user> <!-- 엔티티 사용 → 파일 내용 노출 -->
XXE는 OWASP Top 10(A05:2021 Security Misconfiguration 포함)에 등재된 심각한 취약점. 외부 엔티티 처리를 XML 파서 설정에서 반드시 비활성화해야 함. Java: factory.setFeature("http://apache.org/.../disallow-doctype-decl", true)

XML Bomb (Billion Laughs) 예시

<!DOCTYPE bomb [ <!ENTITY a "AAAA..."> <!-- 4 bytes --> <!ENTITY b "&a;&a;&a;&a;"> <!-- 16 bytes --> <!ENTITY c "&b;&b;&b;&b;"> <!-- 64 bytes --> <!ENTITY d "&c;&c;&c;&c;"> <!-- 256 bytes --> ... (반복) ... <!-- 10단계 → 수 GB --> ]> <root>&d;</root> <!-- 파싱 시 메모리 폭발 -->

전송 수준 보안(TLS)과 메시지 수준 보안(XML Encryption)의 차이는? ▾
TLS(HTTPS)는 전송 채널을 암호화하는 Point-to-Point 방식. 중간 노드(프록시·ESB·로드밸런서)에서 TLS를 종료하면 메시지가 평문으로 노출될 수 있음. XML Encryption은 메시지 자체를 암호화하는 End-to-End 방식. 중간 노드를 거쳐도 메시지 내용이 보호됨. 전송 프로토콜 독립적이며 부분 암호화(특정 요소만)도 가능. 민감 데이터 교환 시스템은 TLS + XML Encryption을 이중으로 적용하는 것이 권장됨.
XML Signature의 3가지 서명 위치 유형을 설명하시오 ▾
Enveloped 서명: 서명 요소가 서명 대상 XML 내부에 자식으로 삽입. 원본 문서 구조 유지. 가장 일반적. Enveloping 서명: 서명 요소가 원본 데이터를 자식으로 포함. 서명이 컨테이너 역할을 함. Detached 서명: 서명과 원본이 별도 요소·파일로 분리. URI 참조로 서명 대상 지정. 여러 문서에 하나의 서명, 외부 리소스(이미지·바이너리) 서명에 적합.
XML Signature에서 정규화(C14N)가 필요한 이유는? ▾
XML은 논리적으로 동일한 문서도 속성 순서, 공백 처리, 네임스페이스 선언 위치 등에 따라 다른 바이트열로 표현될 수 있음. 해시 함수는 동일한 입력에 대해 동일한 출력을 내야 하므로, 서명 전 XML을 표준 형식(Canonical XML)으로 변환하는 정규화(Canonicalization, C14N) 과정이 필수. 정규화 알고리즘이 서명자와 검증자 간에 일치하지 않으면 서명 검증이 실패함.
SAML의 역할과 SSO 동작 흐름을 설명하시오 ▾
SAML(Security Assertion Markup Language)은 인증·권한부여 정보를 XML로 교환하는 OASIS 표준. SSO 동작: ① 사용자가 SP(서비스 제공자)에 접근 요청 ② SP가 AuthnRequest 생성 후 IdP(신원 제공자)로 리디렉트 ③ IdP에서 사용자 인증 수행 ④ IdP가 SAML Assertion 발급(인증·속성·권한 정보 포함) ⑤ 사용자가 Assertion을 SP에 전달 ⑥ SP가 Assertion 서명 검증 후 서비스 접근 허용. Assertion에는 NotBefore/NotOnOrAfter로 유효 기간이 설정됨.
XXE(XML External Entity) 취약점의 원리와 대응 방법은? ▾
XXE는 XML 파서가 외부 엔티티를 처리할 때 발생. 공격자가 DTD에 SYSTEM 키워드로 서버 내부 파일(file:///etc/passwd)이나 내부 네트워크 URL을 참조하는 엔티티를 정의하면, 파서가 해당 내용을 읽어 응답에 포함시킴. SSRF(서버 측 요청 위조)·파일 읽기·DoS로 이어질 수 있어 OWASP Top 10에 포함. 대응: XML 파서에서 외부 엔티티 처리 및 DTD 로딩을 비활성화. 신뢰할 수 없는 XML 입력은 XSD 검증 후 처리. 엔티티 확장 크기·깊이 제한.
WS-Security가 HTTPS만으로 부족한 이유와 추가 보안 기능은? ▾
HTTPS(TLS)는 전송 구간만 보호하는 Point-to-Point 방식. ESB·프록시 등 중간 노드에서 TLS가 종료되면 SOAP 메시지가 평문으로 처리됨. WS-Security는 SOAP Header에 직접 보안 정보를 포함해 End-to-End 메시지 수준 보안을 제공. 추가 기능: UsernameToken/X.509/SAML 토큰으로 인증, XML Signature로 메시지 무결성·부인방지, XML Encryption으로 Body 기밀성 보호, Timestamp+Nonce로 재전송 공격 방지. 전송 프로토콜 독립적.
