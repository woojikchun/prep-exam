리팩토링 정의 & 목적 ⭐

리팩토링(Refactoring): Martin Fowler 정의 — "소프트웨어의 외부 동작을 변경하지 않으면서 내부 구조를 개선하는 것" — 코드를 더 이해하기 쉽고 수정하기 쉽게 만드는 규율 있는 기법

리팩토링의 목적 ⭐

  • 가독성(Readability) 향상: 코드 이해 속도 증가
  • 유지보수성(Maintainability) 향상: 변경 비용 감소
  • 기술 부채(Technical Debt) 상환: 누적된 나쁜 코드 해소
  • 결함 발견: 구조 개선 과정에서 숨은 버그 발견
  • 새 기능 추가 준비: 구조를 개선 후 기능 추가 용이
  • 설계 개선: OOP 원칙·SOLID 준수

리팩토링 ≠ 이것들 ⭐

혼동 개념차이
버그 수정버그 수정=동작 변경 / 리팩토링=동작 유지
성능 최적화최적화=동작은 유지·성능 변경 / 구조 악화 허용
기능 추가새 기능=동작 추가 / 리팩토링은 기능 추가 안 함
재작성(Rewrite)재작성=처음부터 / 리팩토링=점진적 개선

리팩토링 핵심 원칙 & 언제 해야 하나 ⭐

리팩토링 핵심 원칙 (Martin Fowler)

  • 두 모자(Two Hats) ⭐: 기능 추가와 리팩토링을 동시에 하지 말 것 — 한 번에 하나만
  • 테스트 선행 ⭐: 리팩토링 전 테스트 코드 확보 (안전망)
  • 작은 단계 ⭐: 한 번에 하나씩 작은 변경 — 빨리 실패하고 빨리 수정
  • 지속적 통합(CI): 리팩토링 후 즉시 테스트·통합
  • Rule of Three ⭐: 3번 반복되면 리팩토링 신호 (중복 제거)

언제 리팩토링해야 하나

  • 기능 추가 전: 구조가 기능 추가를 어렵게 할 때
  • 버그 수정 중: 코드를 이해하다가 나쁜 코드 발견 시
  • 코드 리뷰 중: 팀원 코드 검토 시
  • 캠핑 규칙 ⭐: "항상 처음보다 깨끗하게 남겨라" (Boy Scout Rule)

언제 하지 말아야 하나

  • API가 완전히 잘못되어 처음부터 작성해야 할 때
  • 배포 마감 직전 (위험 증가)
리팩토링은 외부 동작을 변경하지 않는다 — 테스트가 통과해야만 올바른 리팩토링!
리팩토링=외부동작유지·내부구조개선 / 두모자=기능추가와리팩토링 동시에 하지 말 것 / Rule of Three=3번반복시리팩토링 / 캠핑규칙

코드 스멜 (Code Smells) — 리팩토링이 필요한 신호 ⭐

코드 스멜(Code Smell): Kent Beck이 명명, Martin Fowler가 정리 — 코드에서 리팩토링이 필요함을 알려주는 나쁜 설계의 징후(Surface Indication). 즉각적인 버그는 아니지만 유지보수성·가독성을 저하시킴

핵심 리팩토링 기법 I — 메서드·클래스 추출·인라인 ⭐

① 메서드 추출 (Extract Method) ⭐⭐ — 가장 많이 사용
▾
  • 목적: 긴 메서드의 일부를 별도 메서드로 분리
  • 적용 시점: 메서드가 너무 길 때 / 주석이 필요할 때 / 중복 코드 발견
  • 효과: 코드 재사용 / 가독성↑ / 테스트 용이
  • 역방향: Inline Method (너무 단순한 메서드를 다시 인라인)

Before

void printOrder(order) { // 헤더 출력 console.log("=== ORDER ==="); console.log("Name:" + order.name); // 금액 출력 console.log("Total:" + order.qty*order.price); console.log("============="); }

After

void printOrder(order) { printHeader(); printDetails(order); printFooter(); } void printHeader() { console.log("=== ORDER ==="); } void printDetails(o){ console.log(o.name + o.qty*o.price); }
② 클래스 추출 (Extract Class) ⭐ & 인라인 클래스 (Inline Class)
▾

Extract Class ⭐

  • 목적: 한 클래스가 두 가지 이상의 책임을 가질 때 분리
  • 적용: God Class 발견 시 / SRP 위반 시
  • 방법: 관련 필드·메서드를 새 클래스로 이동
// Before: Person이 전화 관련 책임도 class Person { name; areaCode; number; getTelephoneNumber() {...} } // After: 전화 책임 분리 class TelephoneNumber { areaCode; number; } class Person { name; telephone: TelephoneNumber; }

Inline Class (역방향)

  • 목적: 하는 일이 거의 없는 클래스를 다른 클래스에 통합
  • 적용: Lazy Class 스멜 해결
  • 방법: 모든 기능을 다른 클래스로 이동 후 삭제
③ 메서드 이동 (Move Method) · 필드 이동 (Move Field) ⭐
▾
  • Move Method: 메서드가 자신의 클래스보다 다른 클래스의 기능·데이터를 더 많이 사용 → 해당 클래스로 이동
  • 적용 스멜: Feature Envy(기능 편애) 해결
  • Move Field: 필드가 다른 클래스에서 더 많이 사용될 때 해당 클래스로 이동
  • 효과: 응집도↑ / 불필요한 결합도↓
// Feature Envy: Account가 AccountType 데이터에 의존 class Account { overdraftCharge() { if (this.type.isPremium()) {...} // AccountType 데이터 과다 접근 } } // After: AccountType으로 이동 class AccountType { overdraftCharge() { if (this.isPremium()) {...} } }
④ 변수·필드 이름 변경 (Rename) · 변수 추출 (Extract Variable) ⭐
▾

이름 변경 (Rename)

  • 의도를 드러내지 않는 이름 → 명확한 이름으로
  • 가장 비용 대비 효과가 큰 리팩토링 ⭐

변수 추출 (Extract Variable)

  • 복잡한 표현식을 이름 있는 변수로 분리
  • 역방향: Inline Variable (너무 단순한 변수 제거)
// Extract Variable — 복잡 표현식에 이름 부여 // Before if (order.quantity > 500 && order.itemPrice > 100 && !order.isPriority) {...} // After — 의도가 명확해짐 const isLargeOrder = order.quantity > 500; const isHighValueItem = order.itemPrice > 100; const isRegularOrder = !order.isPriority; if (isLargeOrder && isHighValueItem && isRegularOrder) {...}
⑤ 메서드 올리기·내리기 (Pull Up / Push Down Method) ⭐
▾

Pull Up Method ⭐

  • 서브클래스들에 중복된 메서드 → 슈퍼클래스로 올림
  • 적용: 중복 코드(Duplicate Code) 스멜 해결
  • 효과: 중복 제거 / 상속 계층 활용

Push Down Method

  • 일부 서브클래스만 사용하는 메서드 → 해당 서브클래스로 내림
  • 적용: 불필요한 상속 제거 / 클래스 정리
Extract Method=가장 많이 사용·긴메서드 분리 / Extract Class=God Class 분리 / Move Method=Feature Envy 해결 / Rename=비용대비효과최고
🔧

핵심 리팩토링 기법 II — 조건·데이터·위임·캡슐화 ⭐

① 조건문을 다형성으로 대체 (Replace Conditional with Polymorphism) ⭐⭐
▾
  • 목적: 타입에 따른 switch/if-else → 다형성으로 대체
  • 적용 스멜: Switch Statements 해결 / OCP 달성
  • 효과: 새 타입 추가 시 기존 코드 수정 없이 새 클래스만 추가
  • GoF 연관: Strategy 패턴·State 패턴 적용
// Before — switch 스멜 getSpeed() { switch (this.type) { case "EUROPEAN": return baseSpeed(); case "AFRICAN": return baseSpeed() - loadFactor; } } // After — 다형성 class EuropeanSwallow { getSpeed() { return baseSpeed(); } } class AfricanSwallow { getSpeed() { return baseSpeed()-loadFactor; } }
② 조건문 분해 (Decompose Conditional) · Guard Clause ⭐
▾

조건문 분해 (Decompose Conditional)

  • 복잡한 조건문의 각 부분을 메서드로 추출하여 의도 명확화
if (date.before(SUMMER_START) || date.after(SUMMER_END)) charge = qty * winterRate + winterServiceCharge; // After if (isWinter(date)) charge = winterCharge(qty);

Guard Clause로 중첩 제거 ⭐

  • 중첩 if문을 Guard Clause(조기 반환)로 flatten
// Before — 중첩 if if (isAlive) { if (isActive) { if (hasBalance) { process(); } } } // After — Guard Clause if (!isAlive) return; if (!isActive) return; if (!hasBalance)return; process();
③ 매개변수 객체 도입 (Introduce Parameter Object) · 임시변수→쿼리 ⭐
▾

Introduce Parameter Object ⭐

  • 매개변수가 항상 같이 다닌다면 객체로 묶기
  • 적용 스멜: Long Parameter List·Data Clumps
amountInvoiced(start, end) // Long Parameter List amountReceived(start, end) // After — DateRange 객체로 묶기 amountInvoiced(range: DateRange) amountReceived(range: DateRange)

임시변수를 질의메서드로 (Replace Temp with Query)

  • 임시 변수 대신 메서드 호출로 — 재사용 가능 / 클래스 추출 준비
const basePrice = qty * price; // 임시변수 if (basePrice > 1000) {...} // After — 질의 메서드로 if (basePrice() > 1000) {...} basePrice() { return this.qty * this.price; }
④ 마법 수를 기호 상수로 (Replace Magic Number with Symbolic Constant) ⭐
▾
// Before — 매직넘버: 9.81이 무엇인지 알 수 없음 potentialEnergy = mass * height * 9.81; // After — 의도가 명확한 상수 const GRAVITATIONAL_CONSTANT = 9.81; potentialEnergy = mass * height * GRAVITATIONAL_CONSTANT;
  • 적용 스멜: 코드에 숫자·문자열 리터럴이 직접 사용될 때
  • 효과: 의도 명확 / 변경 시 한 곳만 수정
⑤ 상속을 위임으로 (Replace Inheritance with Delegation) · 위임을 상속으로 ⭐
▾

상속→위임 (Refused Bequest 해결)

  • 서브클래스가 슈퍼클래스 기능의 일부만 사용할 때 (LSP 위반)
  • 상속 대신 구성(Composition)으로 리팩토링

캡슐화: 필드 캡슐화 (Encapsulate Field)

  • public 필드 → private + getter/setter
  • 컬렉션 캡슐화: 반환 시 복사본 제공

리팩토링 & TDD 관계 ⭐⭐

TDD의 Red-Green-Refactor 사이클에서 리팩토링은 3단계의 핵심 — 테스트가 통과한 후 코드를 개선하는 단계

리팩토링과 테스트의 관계 ⭐

  • 테스트=리팩토링 안전망 ⭐: 테스트 없이 리팩토링은 위험 — 실수를 즉시 감지
  • 리팩토링=테스트 용이성↑: 잘 리팩토링된 코드는 테스트하기 쉬움 (의존성 감소)
  • 자동화 테스트 필수 ⭐: 수동 테스트로는 리팩토링 사이클이 너무 느림
  • 테스트 커버리지: 높을수록 안전하게 리팩토링 가능
  • Refactor 단계: TDD 사이클에서 가장 설계가 이루어지는 단계

리팩토링과 SOLID 원칙 ⭐

기술 부채 (Technical Debt) & 정적 분석 도구 ⭐

기술 부채 (Technical Debt) ⭐

  • 정의: Ward Cunningham 제안 — 빠른 개발을 위해 미룬 설계·코드 품질 문제의 누적. 나중에 더 큰 비용(이자)으로 돌아옴
  • 원인: 시간 압박 / 잘못된 설계 / 지식 부족 / 테스트 생략
  • 해결: 리팩토링으로 기술 부채 상환
  • 의도적 부채: 단기 출시 위해 의도적으로 부채 발생시키고 나중에 상환

정적 분석·도구 ⭐

  • 정적 분석(Static Analysis): 코드 실행 없이 소스를 분석하여 문제 탐지
  • Java: SonarQube·PMD·CheckStyle·FindBugs
  • JavaScript: ESLint·TSLint·SonarJS
  • Python: Pylint·Flake8·Black
  • 코드 복잡도 ⭐: McCabe 순환 복잡도 — 분기가 많을수록 높음 → 리팩토링 필요
  • 중복도: 코드 복제율(Duplication) 측정

비교리팩토링대상핵심 차이
리팩토링 vs 재설계(Redesign) 점진적·지속적 내부 구조 개선 기존 코드 기반 유지 리팩토링=작은 단계 / 재설계=큰 변화
리팩토링 vs 재작성(Rewrite) 외부 동작 유지 / 위험 낮음 처음부터 새로 작성리팩토링=안전·점진 / 재작성=위험·단절
리팩토링 vs 성능 최적화 가독성·구조 개선 / 성능 무관 성능 향상 / 구조 악화 허용 목표가 다름 — 동시에 하지 말 것
리팩토링 vs 버그 수정 동작을 변경하지 않음 잘못된 동작을 올바르게 변경 리팩토링 중 버그 발견 시 분리 처리
리팩토링 vs 기능 추가 새 기능 없음 / 구조만 변경 새 기능 추가 / 동작 변경 두 모자(Two Hats) — 동시에 하지 말 것
