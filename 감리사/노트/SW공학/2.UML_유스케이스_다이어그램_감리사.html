유스케이스 다이어그램 정의 & 목적 ⭐

유스케이스 다이어그램(Use Case Diagram): 시스템이 외부 행위자(액터)에게 제공하는 기능(유스케이스)과 그 관계를 시각적으로 표현하는 UML 행위 다이어그램 — 요구사항을 비기술적 이해관계자와 공유하고 시스템 경계를 명확히 하는 데 사용

목적 & 활용

  • 시스템 범위(Scope) 정의: 시스템이 하는 것과 하지 않는 것의 경계 명확화
  • 요구사항 도출·문서화: 사용자 관점의 기능 요구사항 표현
  • 이해관계자 소통: 비기술자도 이해할 수 있는 시각적 언어
  • 테스트 케이스 기반: 각 유스케이스 → 시스템 테스트 시나리오
  • 개발 계획: 유스케이스 단위로 반복 개발(애자일) 계획 수립

유스케이스 다이어그램 5대 구성 요소 ⭐

구성 요소표기법설명핵심 규칙
액터 (Actor) ⭐ 🧍 시스템과 상호작용하는 외부 개체 — 사람·외부 시스템·시간·이벤트 시스템 외부에 위치 / 시스템 경계 밖
유스케이스 (Use Case) ⭐ 시스템이 액터에게 제공하는 기능·서비스 단위 — 동사+명사 형태 타원(Ellipse)으로 표기 / 동사형 이름 / 시스템 경계 내부
시스템 경계 (System Boundary) ⭐ ⬜ 시스템의 범위를 나타내는 사각형 박스 — 유스케이스는 내부, 액터는 외부 사각형에 시스템 이름 표기 / Subject라고도 함
연관 관계 (Association) ───── 액터와 유스케이스 사이의 상호작용 — 실선으로 연결 액터 ↔ 유스케이스만 / 화살표 없는 실선 (양방향)
관계 (Relationships) ⭐ - - - -→ 유스케이스 간 관계 — include / extend / generalization 점선 화살표 + 스테레오타입 표기
5대 구성요소: 액터(스틱맨)·유스케이스(타원)·시스템경계(사각형)·연관(실선)·관계(점선) / 액터=시스템 외부·유스케이스=시스템 내부
👤

액터 (Actor) 상세 ⭐

액터: 시스템 외부에서 시스템과 상호작용하는 모든 개체 — 반드시 사람일 필요는 없으며 외부 시스템·디바이스·시간도 액터가 될 수 있음
액터 유형 ⭐
▾
유형설명예시
주 액터 (Primary Actor) ⭐유스케이스를 직접 시작(trigger)하는 액터 / 시스템의 도움을 원하는 주체고객, 관리자, 일반 사용자
보조 액터 (Secondary Actor)유스케이스 수행 시 시스템이 필요로 하는 외부 서비스 제공자결제 게이트웨이, 이메일 서버, 외부 API
사람 액터실제 시스템을 사용하는 사람·역할회원, 비회원, 관리자, 운영자
시스템 액터시스템과 연동하는 외부 시스템결제 시스템, 인증 서버, ERP, 레거시 시스템
시간 액터특정 시간에 자동으로 유스케이스를 trigger하는 타이머·이벤트자정 배치 처리, 월말 정산, 스케줄러
액터 일반화 (Actor Generalization) ⭐
▾
  • 정의: 액터 간 상속 관계 — 자식 액터는 부모 액터의 모든 유스케이스를 상속
  • 표기: 자식 액터 → 부모 액터 방향의 속이 빈 삼각형 화살표(실선)
  • 예시: 관리자 →(상속)→ 일반회원 / 관리자는 회원의 모든 기능 + 추가 관리 기능
  • 목적: 역할 간 공통 기능 공유 표현 / 다이어그램 단순화
🧍 일반회원 ──(로그인)──▷ 주문조회 ──(상품검색)
△
│ (일반화, 상속)
🧍 관리자 ──▷ 주문관리·회원관리·통계조회 (+ 상속된 모든 기능)
액터 식별 방법 & 유의사항
▾
  • 식별 질문: "누가 시스템을 사용하는가?" / "시스템이 어떤 외부 시스템과 연동하는가?" / "시스템을 자동으로 실행시키는 이벤트가 있는가?"
  • 역할 기준: 개인이 아닌 역할(Role)로 정의 — 동일 사람이 두 역할(회원+관리자)일 수 있음
  • 시스템 내부 구성요소는 액터가 아님: DB·서버·모듈·클래스는 시스템 내부 → 액터 X
  • 액터 수 최소화: 불필요하게 세분화하지 않고 역할 중심으로 통합
주액터=유스케이스 시작(고객) / 보조액터=시스템이 이용(외부결제서버) / 시스템 내부구성요소(DB·서버)는 액터 아님! / 역할 기준으로 정의

유스케이스 관계 (Relationships) ⭐ — 시험 핵심

① include 관계 ⭐ — 반드시 포함
▾
  • 정의: 기본 유스케이스가 수행될 때 포함 유스케이스가 반드시 실행됨
  • 의미: 공통 기능을 별도 유스케이스로 분리하여 재사용
  • 표기: 기본 UC → 포함 UC 방향 점선 / «include» 스테레오타입
  • 방향: 기본 유스케이스 → 포함 유스케이스 (화살표 방향 주의!)
  • 필수성: 항상(무조건) 실행 — 조건 없이
(주문하기) ──«include»──→ (로그인 확인)
(결제하기) ──«include»──→ (로그인 확인)

← 로그인 확인은 공통 기능
← 무조건 실행됨
include 화살표 방향: 기본UC → 포함UC / "기본이 포함을 호출" / 화살표가 포함되는 쪽을 가리킴
② extend 관계 ⭐ — 조건부 확장
▾
  • 정의: 특정 조건(확장 포인트)이 충족될 때만 선택적으로 추가 기능 실행
  • 의미: 기본 유스케이스에 선택적·예외적 기능을 추가로 표현
  • 표기: 확장 UC → 기본 UC 방향 점선 / «extend» 스테레오타입
  • 방향: 확장 유스케이스 → 기본 유스케이스 (include와 반대!)
  • 선택성: 조건부 실행 — 항상 실행되지 않음
  • 확장 포인트: 어떤 조건에서 확장되는지 명시 가능
(할인 쿠폰 적용) ──«extend»──→ (결제하기)
(적립금 사용) ──«extend»──→ (결제하기)

← 쿠폰이 있을 때만 실행
← 결제하기는 항상 실행
extend 화살표 방향: 확장UC → 기본UC (include와 반대!) / "확장이 기본을 가리킴" / 선택적·조건부 실행
③ 일반화 (Generalization) ⭐ — 유스케이스 상속
▾
  • 정의: 자식 유스케이스가 부모 유스케이스를 상속하여 특수화
  • 의미: 공통 행위를 추상화하여 상위 유스케이스로 표현
  • 표기: 자식 UC → 부모 UC 방향 / 속이 빈 삼각형 화살표(실선)
  • 활용: 유사한 유스케이스 중 공통 부분을 추상 UC로 추출
(카드 결제) ──△──→ (결제하기)
(계좌 이체) ──△──→ (결제하기)
(카카오페이) ──△──→ (결제하기)

△ = 속이 빈 삼각형
결제하기 = 추상 유스케이스

관계실행 조건화살표 방향표기선 종류핵심 키워드

유스케이스 명명 규칙 & 수준
▾
  • 동사+목적어 형태 ⭐: "상품 검색", "주문하기", "결제 처리" — 명사형 금지("로그인 기능" X → "로그인" O)
  • 사용자 목표 관점: 기술적 구현이 아닌 사용자가 달성하려는 목표 중심
  • 적절한 추상화 수준: 너무 세밀(버튼 클릭)하거나 너무 추상적(시스템 사용)이지 않게
  • 사용자 스토리와 대응: "~(으)로서, ~하고 싶다" 형태와 1:1 매핑 가능

감리 관점 체크포인트 ⭐
▾
  • 완전성 검토: 식별된 모든 요구사항이 유스케이스로 표현되었는가 / 누락된 기능이 없는가
  • 시스템 경계 명확성: 경계가 명확히 정의되었는가 / 외부 시스템과의 인터페이스가 모두 식별되었는가
  • 액터 완전성: 모든 이해관계자(인간·시스템·시간)가 식별되었는가 / 역할 기준으로 정의되었는가
  • 관계 정확성 ⭐: include/extend 관계가 올바르게 사용되었는가 / 화살표 방향이 맞는가
  • 유스케이스 명세: 각 유스케이스의 사전·사후 조건, 기본·대안 흐름이 작성되었는가
  • 추적성: 유스케이스가 설계·구현·테스트 케이스로 추적 가능한가 (RTM)
  • 비기능 요구사항: 성능·보안·가용성 요구사항이 별도 명세되었는가
  • 일관성: 다이어그램과 명세서의 내용이 일치하는가
