① 계층형 아키텍처 (Layered / N-Tier) ⭐

  • 단방향 의존 ⭐: 상위→하위만 의존 (역방향 금지)
  • Layer vs Tier: Layer=논리적 분리 / Tier=물리적 분리
  • Sinkhole 안티패턴 ⭐: 처리 없이 단순 통과(pass-through)
  • 장점: 이해 단순·관심사 분리·테스트 용이
  • 단점: 대규모 확장 한계·계층 통과 오버헤드

MSA (마이크로서비스 아키텍처) ⭐⭐
▾
  • 독립 배포 ⭐: 서비스별 독립 빌드·배포

헥사고날 (Ports & Adapters) ⭐⭐
▾
  • 도메인 중심 ⭐: 도메인이 기술에 의존하지 않음
  • 포트(Port) ⭐: 도메인의 인터페이스(계약)
  • 어댑터(Adapter) ⭐: 포트의 구체 구현체
  • Primary: 외부→도메인 (REST·CLI)
  • Secondary: 도메인→외부 (DB·Kafka)
  • DIP 적용 ⭐: 도메인은 추상화에만 의존
  • 클린과 관계: 동일 철학 — 의존성 역전·도메인 보호

⑪ DDD (Domain-Driven Design, 도메인 주도 설계) ⭐⭐ ★NEW

DDD: Eric Evans가 2003년 저서 «Domain-Driven Design»에서 체계화 — 복잡한 비즈니스 도메인을 소프트웨어에 충실히 반영하기 위해 도메인 전문가와 개발자가 공유하는 언어·모델로 설계하는 방법론
전략적 설계 (Strategic Design) ⭐⭐
▾

유비쿼터스 언어 (Ubiquitous Language) ⭐

도메인 전문가와 개발자가 동일한 용어를 사용하여 소통 — 코드·문서·대화에서 일관된 언어 사용. 언어의 차이가 모델과 코드의 불일치를 만든다.

바운디드 컨텍스트 (Bounded Context) ⭐⭐

도메인 모델이 유효한 경계 — 동일 용어가 다른 컨텍스트에서 다른 의미를 가질 수 있음. MSA의 서비스 경계와 대응. 예) 주문 컨텍스트의 "상품"과 물류 컨텍스트의 "상품"은 다른 모델.

Context Map 패턴 ⭐
▾
패턴설명
Shared Kernel두 컨텍스트가 공유하는 코드·모델 영역
Customer/SupplierUpstream(공급)과 Downstream(소비) 관계
ACL (Anti-Corruption Layer) ⭐외부 모델이 내부 도메인을 오염시키지 않도록 변환 계층 삽입 — 헥사고날의 Secondary Adapter와 유사

전술적 설계 (Tactical Design) — 빌딩 블록 ⭐⭐
▾
엔티티 (Entity) ⭐
식별자(ID)로 구분되는 객체
동일 ID면 속성이 달라도 같은 객체 / 라이프사이클을 가짐
예) Order(주문ID=100)
값 객체 (Value Object) ⭐
속성의 집합, ID 없음·불변
내용이 같으면 동일 / 불변(Immutable)
예) Money(100, "KRW"), Address
애그리게이트 (Aggregate) ⭐⭐
일관성 경계를 가진 엔티티 군집
Aggregate Root를 통해서만 접근 / 트랜잭션 단위
예) Order(Root) + OrderItem
Aggregate Root ⭐⭐
애그리게이트의 진입점(단일 Root)
외부에서 Aggregate 내부 접근 시 반드시 Root 경유 / ID로 참조
예) Order가 Root → OrderItem에 직접 접근 금지
도메인 서비스 (Domain Service)
특정 엔티티에 속하지 않는 도메인 로직
상태 없음(Stateless) / 여러 엔티티 협력
예) TransferService(계좌 간 이체)
도메인 이벤트 (Domain Event) ⭐
도메인에서 발생한 사건
과거형 이름 / EDA·CQRS·이벤트소싱과 결합
예) OrderPlaced, PaymentCompleted
리포지토리 (Repository) ⭐
Aggregate 영속성 관리 추상화
도메인이 DB 기술을 모르도록 / 인터페이스로 정의
예) OrderRepository.findById()
팩토리 (Factory)
복잡한 Aggregate 생성 로직 캡슐화
생성 로직이 복잡할 때 별도 Factory 사용
GoF Factory Method 패턴과 연관
DDD와 다른 패턴의 관계 ⭐⭐
▾

⑫ Reactive 아키텍처 (반응형 선언) ⭐⭐ ★NEW

Reactive Manifesto (반응형 선언): 2013년 Jonas Bonér 등이 주도하여 발표 — 현대적 분산 시스템의 4가지 핵심 원칙을 정의. 비동기·논블로킹·메시지 기반으로 높은 응답성과 탄력성을 달성

Responsive (응답성) ⭐

요청에 즉시·일관되게 응답 — 응답 시간 한계를 정의하고 준수. 문제를 빠르게 감지·처리

→ 고객 신뢰 기반 / 사용 편의성

Resilient (탄력성) ⭐

장애 발생 시에도 응답성을 유지 — 복제·격리·위임·격납(Containment)으로 장애 격리. Circuit Breaker

→ 단일 장애점 없음 / 자기 치유

Elastic (유연성) ⭐

부하 변화에 따라 자원을 자동 확장·축소 — 병목 없음·중복 없음. 예측적·반응적 스케일링

→ Auto-Scaling / 비용 최적화

Message Driven (메시지 주도) ⭐⭐

컴포넌트 간 비동기 메시지 전달로 느슨한 결합 — 위치 투명성·논블로킹·Back-Pressure

→ 나머지 3원칙의 기반

💡 Message Driven이 나머지 3원칙(Responsive·Resilient·Elastic)의 기반 — 메시지 기반이 없으면 나머지 3가지 달성 어려움
Reactive Streams & Back-Pressure · 구현 기술 ⭐
▾

Reactive Streams 핵심 개념 ⭐

  • 비동기·논블로킹 ⭐: 스레드를 블로킹하지 않고 콜백·이벤트로 처리
  • Back-Pressure ⭐⭐: 소비자가 처리 속도에 맞게 생산자에게 데이터 요청량 조절 — Publisher가 Subscriber를 압도하지 않도록
  • Publisher: 데이터 생산자 — subscribe() 요청 대기
  • Subscriber: 데이터 소비자 — onNext/onError/onComplete
  • Processor: Publisher + Subscriber 역할 동시

⑬ 공간 기반 아키텍처 (Space-Based Architecture, SBA) ⭐ ★NEW

공간 기반 아키텍처(SBA): 데이터베이스 병목을 제거하기 위해 메모리 내 분산 공유 공간(Tuple Space)에 데이터를 저장·처리하는 아키텍처 — 극한 확장성과 단일 장애점 제거가 목표
SBA 핵심 구성 요소 & 특징 ⭐
▾

핵심 구성 요소

구성 요소역할
처리 단위 (Processing Unit) ⭐비즈니스 로직 + In-Memory 데이터 복사본 포함 / 자동 확장 단위
가상화 미들웨어 (VM) ⭐메시징·데이터·처리·배포 그리드 통합 관리
Messaging Grid ⭐요청 라우팅·세션 관리·처리 단위 조율
Data Grid (IMDG) ⭐⭐분산 In-Memory 데이터 저장·복제·파티셔닝
Deployment Manager ⭐부하 기반 Processing Unit 자동 확장·축소
DB Persistence비동기로 In-Memory 데이터를 영구 저장소에 복사

SBA 장단점 & 적합 상황

  • 극한 확장성 ⭐: DB 병목 없음 — 수평 확장(Scale-Out) 용이
  • 단일 장애점 없음 ⭐: PU 복제로 고가용성
  • 낮은 지연 ⭐: 모든 데이터가 메모리 내 — 빠른 응답
  • 단점: 메모리 비용 높음 / 복잡성↑ / 영속성 관리
  • 적합 상황 ⭐: 피크 트래픽(쇼핑몰·티켓팅) / 금융 거래 / 실시간 경매·게임

조합시너지대표 사용
DDD + 헥사고날 + CQRS ⭐BC=서비스경계 / 헥사고날=도메인보호 / CQRS=읽기쓰기분리복잡 도메인 마이크로서비스
MSA + EDA + DDD ⭐BC=서비스경계 / Domain Event=비동기통신 / 느슨한결합대규모 이커머스
CQRS + 이벤트소싱 ⭐Write=이벤트저장 / Read=Projection생성 — 자연스러운 결합금융·감사추적
Reactive + MSA + EDA비동기·논블로킹 / 이벤트기반 / 서비스독립실시간 스트리밍
SBA + CQRS메모리기반 고속 쓰기 / Read Model 별도 최적화초고빈도 거래 시스템
DDD + 이벤트소싱Domain Event를 이벤트 스토어에 영구 기록감사추적·이력관리
감리 체크포인트 ⭐
▾
  • 아키텍처 결정 근거: 품질 속성 기준 선택 여부
  • DDD: BC 경계 명확성 / Aggregate Root 준수 / 유비쿼터스 언어 일관성
  • Reactive: Back-Pressure 처리 방안 / 예외 처리 전략
  • SBA: 데이터 일관성 보장 방안 / 영속성 동기화 주기
  • MSA: DB per Service 준수 / 서비스 경계 적절성
  • CQRS: 최종 일관성 허용 범위 / 동기화 지연 관리
  • 이벤트소싱: 이벤트 스키마 버전 관리 방안
  • EDA: 이벤트 멱등성(Idempotent) 보장 여부
DDD+헥사고날+CQRS=복잡도메인의 황금조합 / MSA+EDA+DDD=대규모시스템 / Reactive=I/O집약·고동시성 / SBA=피크트래픽·DB병목제거
