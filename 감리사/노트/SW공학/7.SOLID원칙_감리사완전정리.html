SOLID 원칙 개요 ⭐

SOLID: Robert C. Martin (Uncle Bob)이 정리한 객체지향 소프트웨어 설계의 5대 원칙 — 변경에 강하고, 이해하기 쉬우며, 재사용 가능한 소프트웨어를 만들기 위한 가이드라인. 각 원칙의 앞 글자를 따서 SOLID라 부름.
원칙정의핵심 키워드위반 시 문제
S SRP 클래스는 변경의 이유가 하나여야 한다 단일 책임 / 응집도↑ / 하나의 역할 변경 시 연쇄 영향 / 거대 클래스
O OCP 확장에 열려 있고 수정에 닫혀 있어야 한다 추상화 / 다형성 / 인터페이스 확장 기능 추가 시 기존 코드 수정 필요
L LSP 자식 클래스는 부모를 완전히 대체할 수 있어야 한다 is-a 관계 / 행위 호환성 / 계약 instanceof 남발 / 상속 오용
I ISP 클라이언트는 사용하지 않는 메서드에 의존하지 말아야 한다 인터페이스 분리 / 작은 인터페이스 뚱뚱한 인터페이스 / 빈 구현 강제
D DIP 고수준 모듈이 저수준 모듈에 의존하지 말고 추상화에 의존해야 한다 의존성 역전 / 추상화에 의존 / IoC 구체 클래스 직접 의존 / 교체 어려움
SOLID = 단일책임(SRP)·개방폐쇄(OCP)·리스코프치환(LSP)·인터페이스분리(ISP)·의존역전(DIP) / Robert C. Martin(Uncle Bob)
1️⃣

SRP — Single Responsibility Principle (단일 책임 원칙) ⭐

S — "클래스를 변경해야 하는 이유는 오직 하나여야 한다"
하나의 클래스는 하나의 책임(역할)만 가져야 한다. 책임 = "변경의 이유(Reason to Change)". 하나의 클래스가 두 가지 이유로 변경되어야 한다면 두 개의 클래스로 분리해야 한다.

❌ SRP 위반 예시

// 한 클래스에 여러 책임이 혼재 class Employee { // 책임 1: 업무 계산 calculatePay() { ... } getHours() { ... } // 책임 2: DB 저장 save() { ... } ← DB 팀 변경 시 // 책임 3: 보고서 출력 printReport() { ... } ← 보고서 팀 변경 시 } // 변경 이유: ①인사팀 ②DB팀 ③보고서팀

✅ SRP 준수 — 책임별 분리

// 책임별로 클래스 분리 class Employee { ← 업무 계산 calculatePay() { ... } getHours() { ... } } class EmployeeRepo { ← DB 저장 save(emp) { ... } } class EmployeeReporter { ← 보고서 printReport(emp) { ... } } // 각 클래스는 하나의 이유로만 변경됨
SRP의 핵심 — "책임 = 변경의 이유" ⭐
▾
  • 책임(Responsibility)의 정의: 변경의 이유(Reason to Change) — "이 클래스를 바꿔야 할 이유가 몇 가지인가?"
  • 액터(Actor) 기반 식별 ⭐: 변경을 요청하는 이해관계자(액터)가 다르면 → 다른 책임 → 분리 대상
  • 응집도(Cohesion)와 관계: SRP를 지키면 클래스 응집도(내부 요소들이 같은 목적)가 높아짐
  • 결합도(Coupling)와 관계: 책임 분리로 다른 클래스와의 결합도가 낮아짐
  • 파사드 패턴과 SRP: 분리된 클래스들을 Facade로 통합하여 사용성 유지
SRP 실전 적용 & 감리 포인트
▾
위반 징후해결 방법
클래스가 수백~수천 줄 (God Class)기능 그룹별로 클래스 분리
메서드 이름이 두 가지 다른 동사 영역책임 경계 파악 후 분리
한 클래스가 UI·비즈니스·DB 모두 담당계층별·역할별 분리
변경 1건이 여러 다른 기능에 영향책임 분리 후 인터페이스로 연결
테스트가 지나치게 복잡테스트하기 어려운 책임이 섞인 신호
SRP=하나의 클래스·하나의 변경이유 / 책임=변경의이유 / 액터기반식별 / 응집도↑ 결합도↓ / God Class=SRP위반 징후
2️⃣

OCP — Open/Closed Principle (개방 폐쇄 원칙) ⭐

O — "확장에는 열려(Open) 있고, 수정에는 닫혀(Closed) 있어야 한다"
새로운 기능을 추가할 때 기존 코드를 수정하지 않고(Closed) 새 코드를 추가(확장, Open)하는 방식으로 구현해야 한다. 핵심 도구: 추상화(인터페이스·추상 클래스)와 다형성

❌ OCP 위반 — 분기로 처리

class DiscountCalculator { calculate(order, type) { if (type === "VIP") { return order * 0.8; } else if (type === "MEMBER") { return order * 0.9; } // 새 할인 추가 시 이 코드 수정 필요! } }

✅ OCP 준수 — 추상화로 확장

interface DiscountPolicy { calculate(order): number; } class VipDiscount implements DiscountPolicy { calculate(o) { return o * 0.8; } } class MemberDiscount implements DiscountPolicy { calculate(o) { return o * 0.9; } } // 새 할인 추가 = 새 클래스만 추가! // 기존 코드 수정 없음
OCP 구현 메커니즘 ⭐
▾
메커니즘방법예시
상속 기반 ⭐부모 클래스를 상속하여 새 동작 추가AbstractShape → Circle, Rectangle
인터페이스 기반 ⭐인터페이스를 구현하여 새 타입 추가DiscountPolicy → VipDiscount, NewDiscount
전략 패턴 (Strategy) ⭐알고리즘을 인터페이스로 캡슐화·교체SortStrategy → BubbleSort, QuickSort
데코레이터 패턴기존 기능을 감싸서 새 기능 추가BasicLogger → TimestampLogger
플러그인 아키텍처새 기능을 플러그인으로 추가IDE 플러그인, 브라우저 확장
  • OCP와 추상화의 관계 ⭐: 변하는 것(구현)을 추상화 뒤에 숨기고, 변하지 않는 것(인터페이스)에 의존
  • if/else·switch 분기문 ⭐: 새 타입 추가 시 반드시 수정 필요 → OCP 위반 신호
  • 완전한 OCP는 불가능: 모든 변경에 대해 닫혀있을 수는 없음 → 예상 변경 지점에만 적용
OCP=확장Open·수정Closed / 추상화+다형성이 핵심 수단 / if/else·switch=OCP위반 신호 / 전략패턴·템플릿메서드가 대표 구현
3️⃣

LSP — Liskov Substitution Principle (리스코프 치환 원칙) ⭐

L — "자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다"
Barbara Liskov(1987) 제안 — 부모 타입을 사용하는 모든 곳에서 자식 타입으로 교체해도 프로그램의 동작이 올바르게 유지되어야 한다. 상속 관계의 올바른 사용을 보장하는 원칙.
LSP 위반 — 유명한 직사각형-정사각형 문제 ⭐
▾
class Rectangle { setWidth(w) { this.width = w; } setHeight(h) { this.height = h; } area() { return width * height; } } class Square extends Rectangle { // 정사각형: 가로=세로 강제 setWidth(w) { this.width = this.height = w; } setHeight(h) { this.width = this.height = h; } } // Rectangle r = new Square(); // r.setWidth(5); r.setHeight(3); // r.area() → 9 (예상: 15) → LSP 위반!

현실에서는 정사각형이 직사각형이지만, 코드 행위(Behavior)가 다름 — 수학적 is-a ≠ 행위적 is-a

// 해결 방법 1: 공통 추상화 interface Shape { area(): number; } class Rectangle implements Shape { ... } class Square implements Shape { ... } // 상속 대신 공통 인터페이스 사용
LSP 준수 조건 (행위적 계약) ⭐
▾
조건설명위반 예시
사전 조건(Precondition) 약화 ⭐자식 클래스의 사전 조건은 부모보다 같거나 약해야 함부모는 양수만, 자식은 양수+음수 → OK / 부모는 양수, 자식은 양수만(더 제한) → 위반
사후 조건(Postcondition) 강화 ⭐자식 클래스의 사후 조건은 부모보다 같거나 강해야 함부모가 보장하던 반환값 범위를 자식이 축소 → 위반
예외 규칙부모가 발생시키지 않는 새로운 예외를 자식이 던지면 안 됨부모: 예외 없음, 자식: UnsupportedOperationException → 위반
불변 조건(Invariant) 유지부모 클래스의 불변 조건을 자식도 유지해야 함부모의 항상 양수 보장을 자식이 음수 허용으로 깨뜨림
  • instanceof 남발 ⭐: if (obj instanceof Specific) 패턴 → LSP 위반 신호 / 다형성으로 대체해야
  • 빈 메서드 오버라이드 ⭐: 자식이 부모 메서드를 빈 메서드(throw UnsupportedOperationException)로 오버라이드 → LSP 위반
  • UML 클래스 다이어그램과 LSP: 상속 관계(▷)는 반드시 LSP를 만족해야 올바른 설계
LSP=자식으로 부모 대체 가능·행위 호환성 / 직사각형-정사각형=LSP위반 대표 사례 / instanceof남발·빈메서드=LSP위반 신호
4️⃣

ISP — Interface Segregation Principle (인터페이스 분리 원칙) ⭐

I — "클라이언트가 사용하지 않는 메서드에 의존하도록 강제해서는 안 된다"
하나의 뚱뚱한(Fat) 인터페이스를 여러 개의 작고 특화된 인터페이스로 분리해야 한다. 클라이언트는 자신이 사용하는 메서드만 포함된 인터페이스에 의존해야 한다.

❌ ISP 위반 — 뚱뚱한 인터페이스

interface Worker { work(); eat(); ← 로봇에겐 불필요 sleep(); ← 로봇에겐 불필요 } class RobotWorker implements Worker { work() { ... } eat() { /* 아무것도 안 함 */ } sleep() { /* 아무것도 안 함 */ } } // 로봇이 eat/sleep 구현 강제 → ISP 위반

✅ ISP 준수 — 인터페이스 분리

interface Workable { work(); } interface Eatable { eat(); } interface Sleepable { sleep(); } class HumanWorker implements Workable, Eatable, Sleepable { ... } class RobotWorker implements Workable { work() { ... } // 필요한 것만 구현 }
ISP 위반 징후 & 핵심 개념 ⭐
▾
  • 뚱뚱한 인터페이스(Fat Interface) ⭐: 너무 많은 메서드를 포함하여 모든 구현체가 일부 메서드를 빈 구현으로 강제되는 인터페이스
  • 빈 구현(Empty Implementation) ⭐: 인터페이스 메서드를 구현하지만 내용이 없거나 예외를 던지는 것 → ISP 위반 신호
  • 인터페이스 오염: 클라이언트에 관계없는 메서드가 포함된 인터페이스 → 변경 시 불필요한 영향
  • 역할 인터페이스(Role Interface) ⭐: ISP 준수 방향 — 역할별로 작은 인터페이스 정의
  • SRP와의 관계 ⭐: ISP는 SRP의 인터페이스 버전 — "인터페이스도 단일 책임을 가져야 한다"
위반 징후해결
빈 메서드(throw Not Implemented) 다수사용하는 메서드만 포함된 작은 인터페이스로 분리
인터페이스 메서드가 수십 개역할·책임에 따라 인터페이스 분리
구현 클래스마다 일부 메서드만 의미 있음공통 부분과 특화 부분을 별도 인터페이스로
ISP=뚱뚱한인터페이스→작은인터페이스로분리 / 빈구현(throw)=ISP위반신호 / 역할인터페이스·SRP의인터페이스버전
5️⃣

DIP — Dependency Inversion Principle (의존 역전 원칙) ⭐

D — "고수준 모듈이 저수준 모듈에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다"
두 가지 규칙:
① 고수준 모듈(비즈니스 로직)은 저수준 모듈(구현 세부사항)에 의존하면 안 된다 — 둘 다 추상화에 의존
② 추상화는 세부사항에 의존하면 안 된다 — 세부사항이 추상화에 의존해야 한다
DIP 핵심 개념 & IoC·DI와의 관계 ⭐
▾
  • 고수준 모듈(High-level): 비즈니스 정책·로직 / 의미 있는 행동을 하는 모듈
  • 저수준 모듈(Low-level): 구체적인 세부사항 / DB·네트워크·파일 I/O
  • 의존 역전(Inversion)의 의미 ⭐: 원래는 고수준→저수준 의존이 당연 → 역전하여 둘 다 추상화에 의존
  • IoC (제어의 역전) ⭐: 객체 생성·관리의 제어권을 프레임워크(Spring)에 넘김
  • DI (의존성 주입) ⭐: IoC를 구현하는 방법 — 외부에서 의존 객체를 주입
// DI 3가지 방식 // ① 생성자 주입 (권장) class OrderService { constructor( private repo: OrderRepository ) {} } // ② Setter 주입 setRepo(repo: OrderRepository) { ... } // ③ 필드 주입 (비권장) @Autowired private OrderRepository repo;
개념관계
DIP설계 원칙 — 무엇을 해야 하는가(추상화에 의존)
IoC패턴 — 제어권을 역전시키는 아이디어
DIIoC 구현 방법 — 외부에서 의존 객체 주입
DI 컨테이너DI를 자동화하는 프레임워크 — Spring IoC, Guice
DIP와 다른 원칙의 관계 & 헥사고날 아키텍처
▾
  • OCP와의 관계 ⭐: DIP를 통해 OCP를 달성 — 추상화에 의존하면 구현 교체 시 고수준 모듈 수정 불필요
  • 헥사고날 아키텍처 ⭐: 도메인이 포트(인터페이스)에만 의존 = DIP의 아키텍처적 적용
  • 클린 아키텍처 ⭐: 의존성 방향 규칙(안쪽으로만) = DIP의 대규모 적용
  • Mock 테스트 용이성: DIP 준수 시 인터페이스의 Mock 구현체로 테스트 가능
DIP=고수준·저수준 모두 추상화에 의존 / 역전=원래 고→저 의존을 뒤집음 / IoC=제어권역전 / DI=의존성주입·생성자주입권장
