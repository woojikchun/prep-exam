V&V (Verification & Validation) 개요 ⭐

소프트웨어 검증·확인(V&V): SW가 명세를 충족하고 의도된 목적에 맞는지 평가하는 모든 활동 — IEEE 1012에서 체계화. 결함을 예방·발견·수정하여 소프트웨어 품질을 보장

Verification (검증) ⭐⭐

"Are we building the product RIGHT?"

제품을 올바르게 만들고 있는가?
명세서·계획·표준 대비 일치 여부 확인

  • 시점: 개발 각 단계 내 (중간 산출물)
  • 기법: 코드 리뷰·인스펙션·단위 테스트·정적 분석
  • 기준: 요구사항 명세서·설계 문서
  • 질문: "이 코드가 설계를 따르는가?"

Validation (확인) ⭐⭐

"Are we building the RIGHT product?"

올바른 제품을 만들고 있는가?
사용자 요구사항·의도된 사용 충족 여부

  • 시점: 개발 완료 후 또는 최종 단계
  • 기법: 인수 테스트·사용자 테스트·시스템 테스트
  • 기준: 사용자 요구사항·비즈니스 목표
  • 질문: "이 시스템이 사용자가 원하는 것인가?"

V 모델 (V-Model) ⭐⭐ — 개발 단계와 테스트 레벨의 대응

테스트의 7가지 원칙 (ISTQB) ⭐

  • 원칙 1: 테스트는 결함을 발견한다 (완전 제거 불가)
  • 원칙 2: 완전 테스트(Exhaustive)는 불가능
  • 원칙 3 ⭐: 조기 테스트 (Shift Left) — 결함 발견 비용 절감
  • 원칙 4 ⭐: 결함 집중 (결함은 일부 모듈에 집중)
  • 원칙 5 ⭐: 살충제 역설 (같은 테스트 반복 시 새 결함 발견 못함)
  • 원칙 6: 테스트는 컨텍스트에 의존
  • 원칙 7: 오류 부재의 궤변 (결함 없어도 사용자 요구 불충족 가능)

테스트 분류 체계 ⭐

분류 기준종류
실행 여부동적(Dynamic) 테스트 / 정적(Static) 테스트
코드 가시성블랙박스 / 화이트박스 / 그레이박스
테스트 레벨단위·통합·시스템·인수
테스트 유형기능·비기능(성능·보안·사용성)
재테스트회귀(Regression) / 재확인(Re-test)
V=명세대비 올바르게(RIGHT way) / V=요구대비 올바른것(RIGHT thing) / V모델=개발단계↔테스트레벨 대응 / 살충제역설·조기테스트·결함집중=핵심원칙

테스트 레벨 (Test Levels) 4단계 ⭐⭐

① 단위 테스트 (Unit Test) ⭐⭐
▾
  • 목적: 개별 코드 단위(함수·메서드·클래스)의 동작 검증
  • 수행자: 개발자 — 코드 작성과 동시 수행
  • 특징: 빠른 실행·독립적·자동화 용이
  • TDD ⭐: 테스트 먼저 작성 후 코드 작성 (Red-Green-Refactor)
  • Mock/Stub ⭐: 의존 모듈을 대체하는 가짜 객체
  • 도구: JUnit(Java)·pytest(Python)·Jest(JS)·NUnit(.NET)
  • 커버리지 기준 ⭐: 구문·분기·경로 커버리지 / 80% 이상 권장
// JUnit 5 단위 테스트 예시 @Test void calculateDiscount_withVipCustomer() { // Given Customer vip = new Customer(CustomerType.VIP); Order order = new Order(10000); // When int discounted = pricingService.calculate(vip, order); // Then assertEquals(8000, discounted); // 20% 할인 } // Mock 사용 (Mockito) when(repository.findById(1L)) .thenReturn(Optional.of(mockOrder));
② 통합 테스트 (Integration Test) ⭐⭐
▾
  • 목적: 단위들이 결합되었을 때 인터페이스·상호작용 검증
  • 결함 유형: 인터페이스 불일치·데이터 변환 오류·타이밍

  • 수행자: 독립 테스트 팀
  • 유형 ⭐: 기능 테스트·성능 테스트·보안 테스트·사용성·이식성·신뢰성

블랙박스 테스트 설계 기법 (Black-Box) ⭐⭐

블랙박스 테스트: 내부 코드 구조를 보지 않고 입력·출력·명세서 기반으로 테스트 케이스를 설계 — 동적 테스트의 대표 기법
① 동등 분할 (Equivalence Partitioning) ⭐⭐
▾
  • 목적: 동일한 결과를 산출하는 입력값을 그룹(파티션)으로 분류하여 각 그룹에서 대표값 1개 선택
  • 핵심 ⭐: 동등 클래스 내에서는 어떤 값을 선택해도 결과가 같다고 가정
  • 분류: 유효 동등 클래스 + 무효 동등 클래스

예시: 나이 입력 (18~65 유효)

무효 클래스 1: age < 18 → 대표: 10 유효 클래스: 18 ≤ age ≤ 65 → 대표: 40 무효 클래스 2: age > 65 → 대표: 70 → 3개 테스트 케이스로 충분!

② 경계값 분석 (BVA) ⭐⭐

  • 목적: 경계 근처에서 결함이 자주 발생 — 경계값과 바로 옆 값 테스트
  • 규칙 ⭐: 각 경계에서 min·min+1·max-1·max 4개 값
// 18~65 유효 범위 경계값 테스트: 17, 18, 19, 64, 65, 66 // 강조 = 경계 포함 직전·직후 // 3-point BVA (일반) min-1, min, min+1 max-1, max, max+1 // 2-point BVA (간소화) min, min+1, max-1, max
③ 결정 테이블 · ④ 상태 전이 · ⑤ 유스케이스 테스트 ⭐
▾

결정 테이블 (Decision Table) ⭐

  • 조건(Condition)의 모든 조합에 따른 행동(Action) 정의
  • 복잡한 비즈니스 규칙 테스트에 적합
  • 조건 n개 → 최대 2^n 가지 규칙
┌──────────┬──┬──┬──┬──┐ │ 조건/행동 │R1│R2│R3│R4│ ├──────────┼──┼──┼──┼──┤ │ 로그인? │Y │Y │N │N │ │ 권한있음?│Y │N │Y │N │ ├──────────┼──┼──┼──┼──┤ │ 접근허용 │✓ │✗ │✗ │✗ │ │ 에러메시지│ │✓ │✓ │✓ │ └──────────┴──┴──┴──┴──┘

상태 전이 (State Transition) ⭐

  • 상태(State)·이벤트(Event)·전이(Transition)·행동(Action) 기반
  • 각 상태·전이를 커버하는 테스트 케이스 설계
  • 적용: 로그인 상태·주문 상태·ATM 상태

⑤ 유스케이스 테스트 · ⑥ 페어와이즈(Pairwise) · ⑦ 탐색적 테스트

기법특징적합 상황
유스케이스 테스트UC 시나리오(정상·대안·예외 흐름) 기반기능 테스트 전반
페어와이즈(All-Pairs) ⭐모든 쌍(pair) 조합 최소 1회 커버 — 경우의 수 감소다중 파라미터 조합 테스트
탐색적 테스트 ⭐사전 스크립트 없이 학습·실행·동시 수행 / 경험 기반요구사항 불명확·시간 제약
오류 추측 (Error Guessing)경험·직관으로 오류 발생 가능 곳 테스트경험 있는 테스터
블랙박스: 동등분할(대표값1개)·경계값분석(min·max±1)·결정테이블(2^n)·상태전이·유스케이스·페어와이즈·탐색적·오류추측

화이트박스 테스트 설계 기법 (White-Box) ⭐⭐

화이트박스 테스트: 내부 코드 구조·논리 흐름을 분석하여 테스트 케이스 설계 — 구조 기반(Structure-Based) 테스트. 개발자가 주로 수행, 단위 테스트에서 주로 활용
커버리지 기법 계층 구조 ⭐⭐
▾
커버리지기준강도
구문(Statement) ⭐모든 실행 문장 1회 이상 실행가장 약함
결정(Decision/Branch) ⭐모든 분기(참/거짓) 1회 이상중간
조건(Condition) ⭐각 조건식의 참/거짓 1회 이상중간
MC/DC ⭐⭐각 조건이 독립적으로 결과에 영향항공·안전 분야
경로(Path)모든 실행 경로 커버가장 강함·비현실적
데이터 흐름변수 정의~사용 경로 커버높음
구문 커버리지 100%라도 분기 커버리지 100%가 아닐 수 있다 — 커버리지 ≠ 결함 없음!

MC/DC (Modified Condition/Decision Coverage) ⭐⭐

  • 정의: 각 개별 조건(condition)이 최종 결정(decision) 결과에 독립적으로 영향을 미침을 보임
  • 요구 TC 수: 조건 n개 → 최소 n+1개 테스트 케이스
  • 적용 분야 ⭐: DO-178B/C(항공), IEC 61508(기능 안전), 자동차 ISO 26262
  • 의의: 안전 크리티컬 시스템의 표준 커버리지
// 예: (A && B) || C // A=T,B=T,C=F → 결과T (기본) // A=F,B=T,C=F → 결과F (A 영향 증명) // A=T,B=F,C=F → 결과F (B 영향 증명) // A=F,B=F,C=T → 결과T (C 영향 증명) // → 4개 TC (조건3개+1)
제어 흐름·데이터 흐름 테스트 ⭐
▾

기본 경로 테스트 (Basis Path Testing) ⭐

  • McCabe 순환 복잡도(V(G)) 계산 후 독립 경로 수만큼 TC 도출
  • V(G) = 분기문 수 + 1 = 필요 TC 최솟값
  • 제어 흐름 그래프(CFG) 기반

데이터 흐름 테스트 (Data Flow Testing)

  • 변수 정의(def)와 사용(use) 쌍을 추적
  • All-defs: 모든 정의에서 최소 1회 사용 도달
  • All-uses: 모든 def-use 쌍 커버
  • All-du-paths: 모든 def→use 경로 커버

정적 테스트 (Static Testing) ⭐⭐

정적 테스트: 소프트웨어를 실행하지 않고 코드·문서·명세서를 검토하여 결함을 발견하는 기법 — Verification의 핵심 수단
정적 검토 기법 (Review) ⭐⭐
▾
기법형식성주도자특징산출물
비공식 검토 가장 낮음 작성자 자유로운 검토·구두·이메일 / 문서화 미흡 비공식 피드백
워크스루 (Walkthrough) ⭐ 낮음 작성자 작성자가 참가자들에게 문서 설명 / 제안 중심·비판 지양 검토 의견
기술 검토 (Technical Review) 중간 진행자 동료 기술 전문가 검토·표준 준수 확인 결함 목록·개선안
인스펙션 (Inspection) ⭐⭐ 가장 높음 전문 리더 공식적·훈련된 팀·체크리스트·결함 추적·메트릭 수집 결함 로그·메트릭

AI/ML 모델 자체의 품질·신뢰성을 검증하는 특수 테스트 기법

뉴런 커버리지 (Neuron Coverage) ⭐⭐
▾
  • 정의: DNN(심층신경망)에서 코드의 구문 커버리지에 대응하는 개념 — 활성화된 뉴런의 비율
  • NC (Neuron Coverage) ⭐: 테스트 케이스 실행 시 임계값 이상 활성화된 뉴런 비율
  • KMNC (K-Multisection Neuron Coverage) ⭐: 뉴런 활성화 범위를 K개 구간으로 나누어 커버리지 측정
  • NBC (Neuron Boundary Coverage) ⭐: 각 뉴런의 활성화 상한·하한 경계 도달 여부
  • SNAC (Strong Neuron Activation Coverage): 뉴런 최대 활성화 구간 커버
  • 제안자: Kuchnik & Pei (DeepXplore, 2017)
// DNN 뉴런 커버리지 개념 Layer 1: [n1=0.9, n2=0.1, n3=0.7] Layer 2: [n4=0.3, n5=0.8] // 임계값 θ = 0.5로 설정 활성화: n1, n3, n5 (3개) 전체: 5개 NC = 3/5 = 60% // 일반 SW: 구문 커버리지 // AI 모델: 뉴런 커버리지 // → 개념적으로 대응
DeepXplore · 변이 테스트 · 메타모픽 테스트 ⭐
▾

DeepXplore (2017) ⭐

  • 제안: Pei et al. — DNN 첫 화이트박스 체계적 테스트 프레임워크
  • 목표: 뉴런 커버리지를 최대화하는 입력 자동 생성
  • Joint Optimization: 여러 DNN의 뉴런 커버리지를 동시 최대화하여 서로 다른 예측이 나오는 입력 탐색 (차분 테스트)
  • 도메인 특화: 자율주행·악성코드 탐지·의료 이미지

변이 테스트 (Mutation Testing for AI) ⭐

  • DNN 파라미터(가중치·활성함수)를 의도적으로 변형(변이) 생성
  • 변이 모델과 원본 모델의 동작 차이를 감지하는 TC 우수성 평가
  • Mutation Score: 탐지된 변이 / 전체 변이

메타모픽 테스트 (Metamorphic Testing) ⭐⭐

  • 배경: AI 모델은 정답 오라클 없음 → 출력 검증 어려움
  • 방법: 입력 변환 시 출력이 만족해야 할 메타모픽 관계(MR) 정의 후 검증
  • 예시: "이미지를 좌우 반전해도 분류 결과 동일해야" → 다르면 결함
  • 의의: Oracle Problem 해결의 핵심 기법

결함(Defect) 분석 · 생명주기 ⭐
▾

결함 관련 용어 구분 ⭐⭐

용어정의
오류(Error)개발자의 실수·잘못된 행동
결함(Defect/Fault) ⭐오류로 인해 코드에 존재하는 문제
고장(Failure) ⭐결함이 실행되어 관찰된 잘못된 동작
버그(Bug)결함의 비공식 명칭

결함 생명주기 ⭐

신규(New) → 할당(Assigned) → 진행중(In Progress) → 수정됨(Fixed) → 재테스트(Re-testing) → 검증됨(Verified) → 종료(Closed) 또는 → 재오픈(Reopen)

결함 분석 기법 ⭐

  • 근본 원인 분석(RCA) ⭐: 5 WHY·Fish-bone(특성요인도) / 결함 근본 원인 제거
  • 결함 클러스터링 ⭐: 80%의 결함이 20%의 모듈에 집중 (파레토 법칙)
  • 결함 밀도 ⭐: 결함 수 / KLOC — 모듈별 품질 비교
  • ODC (Orthogonal Defect Classification): IBM 결함 직교 분류 — 결함 유형·원인·영향 분류
  • 살충제 역설 ⭐: 같은 테스트 반복 시 새로운 결함 발견 감소 → 주기적 TC 갱신
