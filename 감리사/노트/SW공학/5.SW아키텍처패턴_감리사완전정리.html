소프트웨어 아키텍처 패턴 개요 ⭐

아키텍처 패턴(Architecture Pattern): 소프트웨어 시스템의 전체적인 구조를 구성하는 반복적으로 검증된 설계 결정의 집합 — 디자인 패턴보다 상위 레벨의 광범위한 구조적 지침

아키텍처 패턴 vs 디자인 패턴

구분아키텍처 패턴디자인 패턴
범위시스템 전체 구조모듈·클래스 수준
추상화더 높음더 낮음
예시MSA, 계층형, MVCSingleton, Factory, Observer
결정 주체아키텍트개발자

아키텍처 품질 속성 (QA) ⭐

  • 성능(Performance): 응답 시간·처리량
  • 확장성(Scalability): 부하 증가 대응
  • 가용성(Availability): 장애 없이 서비스
  • 유지보수성(Maintainability): 변경 용이성
  • 보안(Security): 인증·인가·데이터 보호
  • 테스트 용이성(Testability): 단위 테스트 가능

① 계층형 아키텍처 (Layered / N-Tier Architecture) ⭐

계층형 아키텍처: 시스템을 수평적 계층으로 분리하고, 각 계층이 바로 아래 계층에만 의존하는 방식 — 가장 널리 사용되는 전통적 아키텍처 패턴

계층형 아키텍처 특징 ⭐

  • 단방향 의존: 상위 계층→하위 계층만 의존 (역방향 금지)
  • 계층 격리 (Isolation): 각 계층은 인접 계층만 알면 됨
  • 관심사 분리: 각 계층이 하나의 역할만 담당
  • Layer vs Tier: Layer=논리적 분리 / Tier=물리적 분리

장단점

장점단점
이해·개발 단순계층 통과 오버헤드
관심사 분리Sinkhole 안티패턴: 단순 pass-through
교체 용이대규모에서 모놀리스 한계
테스트 용이계층 간 강한 결합 가능
Sinkhole 안티패턴: 요청이 아무런 처리 없이 계층을 그냥 통과(pass-through)하는 것 — 불필요한 계층화의 낭비
계층형: 단방향의존(상위→하위)·관심사분리·Layer=논리/Tier=물리 / Sinkhole=단순통과 안티패턴

② MVC / MVP / MVVM 패턴 ⭐

MVC (Model-View-Controller) ⭐
▾
  • Model: 데이터·비즈니스 로직 / 상태 관리 / DB 통신
  • View: UI 표현 / 사용자에게 데이터 시각화 / HTML·Template
  • Controller: 사용자 입력 처리 / Model 업데이트 / View 선택
  • 흐름: User → Controller → Model → View → User
  • 활용: Spring MVC·Django·Rails·ASP.NET
  • 단점: Controller 비대화(Fat Controller) / View-Model 직접 의존
MVP (Model-View-Presenter) ⭐
▾
  • Presenter: View와 Model 사이의 중재자 / 뷰에서 모든 로직 제거
  • View: UI만 담당 / Presenter에 완전 의존 / 수동적(Passive View)
  • 흐름: User → View → Presenter → Model → Presenter → View
  • 특징: View-Model 완전 분리 → 테스트 용이
  • 활용: Android(구버전)·Windows Forms·WinUI
  • MVC와 차이: Presenter가 View를 직접 업데이트 / View는 수동적
MVVM (Model-View-ViewModel) ⭐
▾
  • ViewModel: View를 위한 데이터·명령(Command) 제공 / 상태 관리
  • 데이터 바인딩 ⭐: View와 ViewModel이 양방향 자동 동기화
  • View: ViewModel 상태를 자동으로 반영 / UI 선언적 작성
  • 활용: WPF·XAML·Vue.js·Angular·React(유사) / Android(현재)
  • 특징: View 코드 최소화 / 테스트 용이 / 선언적 UI
  • 단점: 데이터 바인딩 디버깅 어려움 / 작은 앱엔 오버스펙

MVC / MVP / MVVM 비교 ⭐

항목MVCMVPMVVM
중간 역할ControllerPresenterViewModel
View 능동성능동 (일부 로직)완전 수동선언적 (바인딩)
View-Model 관계알 수 있음완전 분리바인딩으로 연결
테스트 용이성중간높음높음
대표 프레임워크Spring MVC, DjangoAndroid(구), WinFormsWPF, Vue.js, Angular
MVC=Controller가 Model·View 연결 / MVP=View수동·Presenter가 직접업데이트 / MVVM=양방향데이터바인딩·선언적UI

③ 마이크로서비스 아키텍처 (MSA) ⭐⭐

MSA: 단일 애플리케이션을 작고 독립적으로 배포 가능한 서비스들의 집합으로 구성 — 각 서비스는 자체 프로세스에서 실행되며 경량 API(REST·gRPC)로 통신
MSA 핵심 특징 & 원칙 ⭐
▾
  • 서비스별 독립 배포 ⭐: 다른 서비스 영향 없이 독립 배포
  • 서비스별 독립 DB ⭐: Database per Service — 다른 서비스 DB 직접 접근 금지
  • 기술 다양성: 서비스별 최적 언어·DB 선택 자유
  • 장애 격리 (Fault Isolation): 한 서비스 장애가 전체로 전파 방지
  • API Gateway ⭐: 단일 진입점 / 인증·라우팅·로드밸런싱
  • Service Discovery ⭐: 서비스 주소 동적 탐색 (Eureka·Consul)
  • Circuit Breaker ⭐: 연쇄 장애 방지 (Hystrix·Resilience4j)
  • Saga Pattern ⭐: 분산 트랜잭션 처리

MSA 단점

  • 분산 복잡성: 네트워크 지연·장애 처리 복잡
  • 데이터 일관성: 분산 트랜잭션 어려움 (Saga 필요)
  • 운영 복잡성: 많은 서비스·컨테이너·오케스트레이션
  • 테스트 어려움: 통합 테스트 복잡
  • 서비스 간 통신 오버헤드
Monolith vs MSA 비교 ⭐
▾
항목모놀리스 (Monolith)MSA
배포 단위전체 애플리케이션서비스별 독립 배포
스케일링전체 스케일 업서비스별 독립 스케일 아웃
개발 팀단일 팀 / 강한 결합서비스별 독립 팀 (Conway's Law)
기술 스택단일 스택서비스별 최적 스택 선택
장애 영향전체 서비스 장애해당 서비스만 영향
초기 개발빠름·단순느림·복잡
데이터 일관성ACID 트랜잭션 쉬움Eventually Consistent·Saga 필요
MSA=독립배포·DB per Service·API Gateway·Service Discovery·Circuit Breaker·Saga / 단점=분산복잡성·데이터일관성

④ SOA (Service-Oriented Architecture) ⭐

SOA 핵심 개념

  • ESB (Enterprise Service Bus) ⭐: 서비스 간 통신을 중앙에서 관리하는 미들웨어 버스
  • 서비스 재사용 ⭐: 비즈니스 기능을 서비스로 캡슐화·재사용
  • 느슨한 결합: 서비스 간 인터페이스(WSDL·SOAP)로만 통신
  • 레지스트리: UDDI — 서비스 검색·등록 디렉터리
  • WS-* 표준: SOAP·WSDL·UDDI 기반 웹 서비스

SOA vs MSA 비교 ⭐

항목SOAMSA
서비스 크기크고 굵은 서비스작고 세밀한 서비스
중심 허브ESB 중앙집중분산·경량 API
통신 방식SOAP / XMLREST / gRPC / 이벤트
데이터공유 DB 허용DB per Service
적합 환경기업 통합·레거시클라우드 네이티브
SOA=ESB중앙집중·서비스재사용·SOAP/WSDL·레거시통합 / MSA=SOA의 발전·경량·분산·REST·DB per Service

⑤ 이벤트 기반 아키텍처 (EDA) ⭐⭐

EDA (Event-Driven Architecture): 시스템 컴포넌트들이 이벤트를 발생시키고 구독하는 방식으로 느슨하게 결합하여 비동기 통신하는 아키텍처 패턴
EDA 핵심 구성 요소 ⭐
▾
  • 이벤트 생산자 (Producer) ⭐: 이벤트를 발생시켜 브로커에 발행
  • 이벤트 브로커 (Broker) ⭐: 이벤트를 수신·저장·전달 — Kafka·RabbitMQ·SNS
  • 이벤트 소비자 (Consumer) ⭐: 관심 이벤트를 구독하여 처리
  • 이벤트 채널 (Topic/Queue): 이벤트가 흐르는 논리적 경로
  • 이벤트 스토어 ⭐: 이벤트 영구 저장·재처리 가능
EDA 주요 패턴 & 특징 ⭐
▾
패턴설명
Pub/Sub (발행-구독) ⭐생산자는 채널에 발행, 소비자는 채널 구독 — 1:N 팬아웃
Event Streaming이벤트를 순서 있게 저장하여 재처리 — Kafka 로그 기반
Event Sourcing ⭐상태가 아닌 이벤트를 저장 → 상태 재구성 가능
CQRS + EDA명령(쓰기)은 이벤트 발행, 조회(읽기)는 이벤트 구독으로 Read Model 유지
Saga Pattern ⭐분산 트랜잭션을 이벤트 체인으로 처리 — Choreography Saga
  • 장점: 느슨한 결합·확장성·비동기·장애 격리
  • 단점: 이벤트 순서 보장 어려움·중복 처리·디버깅 복잡·최종 일관성(Eventual Consistency)
EDA=Producer·Broker·Consumer / Pub/Sub·이벤트스트리밍·Saga / 장점=느슨한결합·비동기 / 단점=순서보장·최종일관성

⑥ 파이프 & 필터 (Pipe and Filter) ⭐

파이프 & 필터: 데이터를 처리하는 필터(Filter)들을 파이프(Pipe)로 연결하여 순차적으로 데이터를 변환·처리하는 패턴 — Unix 파이프라인이 대표 사례
  • 필터(Filter): 데이터를 받아 처리·변환하는 독립 컴포넌트 — 입력→처리→출력
  • 파이프(Pipe): 필터 간 데이터를 전달하는 채널
  • 독립성: 각 필터는 다른 필터를 알지 못함 → 교체·재조합 용이
  • 적합: 컴파일러·ETL·유닉스 파이프·ML 전처리·이미지 처리
장점단점
필터 재사용·교체 용이필터 간 데이터 변환 오버헤드
병렬 처리 가능공유 상태 불가
테스트 용이배치 처리에만 적합
파이프&필터: Input→Filter1→Filter2→...→Output / 필터=독립컴포넌트·재사용 / Unix파이프·컴파일러·ETL·ML전처리
🔷

⑦ 헥사고날 아키텍처 (Hexagonal / Ports & Adapters) ⭐⭐

헥사고날 아키텍처: Alistair Cockburn(2005) 제안 — 도메인 로직을 중심에 두고, 외부 세계와 포트(Port)와 어댑터(Adapter)로 연결하는 패턴. 도메인이 외부 기술에 의존하지 않도록 격리
포트 & 어댑터 상세 ⭐
▾
개념설명예시
포트 (Port) ⭐도메인이 외부와 소통하는 인터페이스(계약) — 기술 무관한 추상화OrderRepository(인터페이스), NotificationPort
Primary Adapter외부가 도메인을 호출하는 어댑터 (Driving) — 도메인 사용자REST Controller, GraphQL Resolver, CLI
Secondary Adapter도메인이 외부를 호출하는 어댑터 (Driven) — 도메인이 사용하는 인프라JPA Repository, SMTP Mail, Kafka Publisher
  • 의존성 역전(DIP) ⭐: 도메인은 포트(인터페이스)에만 의존 — 어댑터 교체 시 도메인 변경 없음
  • 테스트 용이성 ⭐: 어댑터를 Mock으로 교체 → 도메인 순수 단위 테스트
  • DDD와 궁합: 도메인 모델이 중심 → DDD·Clean Architecture와 결합
⑧ 클린 아키텍처 (Clean Architecture) ⭐
▾

Robert C. Martin (Uncle Bob) 제안

  • 동심원 구조: 안쪽 = 더 추상적 = 더 안정적
  • 의존성 규칙(Dependency Rule) ⭐: 소스 코드 의존성은 반드시 안쪽을 향해야 — 바깥에서 안으로만
  • Entities(핵심): 가장 안쪽 / 엔터프라이즈 비즈니스 규칙
  • Use Cases: 애플리케이션 비즈니스 규칙
  • Interface Adapters: Controller·Presenter·Gateway
  • Frameworks & Drivers: DB·UI·외부 프레임워크 — 가장 바깥
  • 헥사고날과 관계: 같은 철학 — 도메인 보호·의존성 역전
헥사고날: 도메인 중심·포트(인터페이스)·어댑터(구현체) / Primary=외부→도메인 / Secondary=도메인→외부 / 클린=의존성방향=안쪽만

⑨ CQRS (Command Query Responsibility Segregation) ⭐⭐

CQRS: Greg Young이 CQS(Command-Query Separation) 원칙을 확장 — 읽기(Query) 모델과 쓰기(Command) 모델을 완전히 분리하여 각각 최적화하는 패턴
CQRS=읽기(Query)와 쓰기(Command) 모델 완전분리 / Write DB(MySQL)·Read DB(Redis·ES) / 최종일관성 / Event Sourcing과 함께 자주 사용

⑩ 이벤트 소싱 (Event Sourcing) ⭐⭐

이벤트 소싱: 객체의 현재 상태를 저장하는 대신 상태를 변경시킨 이벤트의 시퀀스를 저장하고, 필요 시 이벤트를 재생하여 현재 상태를 재구성하는 패턴
이벤트 소싱 핵심 개념 & 단점 ⭐
▾
  • 이벤트 스토어(Event Store) ⭐: 이벤트를 순서대로 영구 저장하는 불변(Immutable) 저장소
  • Aggregate ⭐: 이벤트를 발생시키고 재생하는 도메인 객체
  • 스냅샷(Snapshot): 성능 최적화 — 특정 시점 상태를 저장하여 전체 재생 방지
  • Projection: 이벤트 스트림을 읽어 Read Model 생성
  • 시간 여행(Time Travel) ⭐: 특정 시점의 상태를 이벤트 재생으로 복원
  • CQRS와 결합 ⭐: Write=이벤트 스토어·Read=Projection으로 Read Model 생성

이벤트 소싱 단점

  • 복잡성: 이벤트 스키마 관리·진화(Schema Evolution)
  • 쿼리 어려움: 현재 상태 조회 위해 CQRS 필요
  • 이벤트 스토어 기술: EventStoreDB·Kafka 등 전문 기술 필요
  • 이벤트 버전 관리: 구버전 이벤트 처리 복잡
이벤트소싱=상태가 아닌 이벤트 저장·재생으로 상태재구성 / 감사추적·시간여행·CQRS와 결합 / 단점=복잡성·스키마진화
