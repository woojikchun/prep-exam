Git 개요 및 핵심 특징

정의
분산형(Distributed) 버전 관리 시스템(VCS). 로컬에 전체 히스토리를 복제하여 네트워크 없이도 작업 가능
창시
Linus Torvalds (2005), Linux 커널 개발을 위해 설계
핵심 모델
스냅샷(Snapshot) 기반 — 변경된 파일만 저장하는 델타 방식이 아니라, 매 커밋 시점의 전체 파일 상태를 스냅샷으로 저장

Git 5대 특징

① 분산 저장소
모든 개발자가 전체 저장소의 복사본 보유. 서버 없이도 버전 관리 가능
② 스냅샷 기반 저장
변경점이 아닌 파일 전체 상태를 저장. 변경 없는 파일은 이전 스냅샷 링크로 처리
③ 무결성 (SHA-1/SHA-256)
모든 객체는 해시값으로 관리. 내용 위·변조 즉시 탐지 가능
④ 경량 브랜칭
브랜치 = 포인터(40바이트). 생성·삭제·이동이 O(1)로 매우 빠름
⑤ 스테이징 영역 (Index)
작업 디렉터리 → 스테이징 → 저장소의 3단계. 커밋 내용을 세밀하게 제어
⑥ 비선형 개발
여러 브랜치를 병렬로 운영하며 독립적으로 실험·개발 후 병합
한 줄 암기: Git = 분산형 + 스냅샷 + SHA 무결성 + 경량 브랜치 + 3단계 작업흐름

Git 4대 객체

Blob
파일 내용 그 자체. 파일명·권한 정보 없음. SHA 해시로 식별
Tree
디렉터리 구조. Blob 및 다른 Tree 를 가리키는 포인터 목록
Commit
Tree 포인터 + 부모 커밋 + 작성자/시간 + 메시지. 불변 객체
Tag
특정 커밋에 이름 부여. Lightweight(포인터만) vs Annotated(서명 포함)

파일 상태 생명주기

Untracked
→ git add →
Staged
→ git commit →
Unmodified
→ 편집 →
Modified
→ git add →
Staged
git rm → Unmodified에서 삭제 후 Staged(삭제됨) 상태로 전환
한 줄 암기: Git 저장소 = Working → Staging → Local → Remote, 객체 = Blob(내용) + Tree(구조) + Commit(메타) + Tag(이름)

① 초기화 / 설정

git init
현재 디렉터리를 Git 저장소로 초기화 (.git 생성)
git clone <url>
원격 저장소 전체를 로컬로 복제. 히스토리 포함
git config
사용자 이름·이메일·에디터 등 환경 설정git config --global user.email "a@b.com"
git remote
원격 저장소 별칭 관리git remote add origin <url>

② 기본 작업 (add / commit)

git status
작업 디렉터리와 스테이징 영역의 현재 상태 확인
git add <파일>
파일을 스테이징 영역에 추가git add . (전체 추가)
git commit -m
스테이징 내용을 로컬 저장소에 커밋git commit -m "feat: 로그인 기능"
git commit --amend
직전 커밋 메시지 또는 내용 수정 (히스토리 재작성 주의)
git diff
작업 디렉터리 vs 스테이징 차이git diff --staged: 스테이징 vs 최신 커밋
git log
커밋 히스토리 조회git log --oneline --graph --all

③ 브랜치 / 병합

git branch
브랜치 목록 조회 / 생성 / 삭제git branch -d feature/login
git checkout / switch
브랜치 전환. switch는 Git 2.23+ 권장git switch -c feature/new
git merge <브랜치>
대상 브랜치를 현재 브랜치에 병합. Fast-forward / 3-way merge
git rebase <브랜치>
커밋을 다른 브랜치 위로 재배치. 히스토리 선형화 (공개 브랜치 사용 금지)
git cherry-pick
특정 커밋만 선택적으로 현재 브랜치에 적용
git stash
현재 작업 내용을 임시 저장하고 워킹 디렉터리를 깨끗이 정리git stash pop: 복원

④ 원격 저장소

git fetch
원격 변경사항을 가져오되 로컬 브랜치에 병합 안 함
git pull
fetch + merge 결합. origin/main → 로컬 main 자동 병합
git push
로컬 커밋을 원격 저장소로 전송git push -u origin main
git push --force-with-lease
강제 push 시 타인 커밋 덮어쓰기 방지. --force보다 안전

⑤ 되돌리기

git revert <커밋>
안전되돌리는 새 커밋 생성. 히스토리 보존. 공개 브랜치에 사용
git reset
주의커밋 포인터 이동. --soft(스테이징 유지) / --mixed(기본) / --hard(완전 삭제)
git restore
작업 디렉터리 파일 복원 (Git 2.23+)git restore --staged: 언스테이징
git reflog
HEAD 이동 이력 전체 조회. reset --hard 실수 복구에 활용
한 줄 암기: revert=새커밋으로 안전 취소, reset=포인터 이동(히스토리 재작성 위험), rebase=선형 히스토리(공개 브랜치 금지)

주요 브랜치 전략 비교

전략주요 브랜치특징적합 대상
Git Flowmain, develop, feature, release, hotfix엄격한 릴리즈 관리. 롤백 용이정기 배포 제품
GitHub Flowmain + feature/*main 항상 배포 가능. PR 중심 단순 흐름CI/CD, SaaS
GitLab Flowmain + environment 브랜치환경별(staging, production) 브랜치 추가다중 환경 배포
Trunk-Basedmain (trunk)모든 개발자가 trunk에 소규모 커밋. Feature Flag 활용고속 CI/CD

Git Flow 상세 브랜치 역할

main (master)
항상 프로덕션 배포 상태. 직접 커밋 금지. 태그로 버전 관리
develop
통합 브랜치. 다음 릴리즈 준비. feature 브랜치의 병합 대상
feature/*
기능 개발. develop에서 분기 → develop으로 병합
release/*
QA·버그픽스만. main + develop 동시 병합 후 삭제
hotfix/*
프로덕션 긴급 패치. main에서 분기 → main + develop 동시 병합

Merge vs Rebase

git merge
병합 커밋(Merge Commit) 생성. 히스토리에 병합 사실 기록. 안전하고 추적 가능
장점: 히스토리 보존 | 단점: 히스토리 복잡
git rebase
커밋을 대상 브랜치 끝에 재배치. 선형 히스토리. 공개 브랜치에 절대 사용 금지
장점: 깔끔한 히스토리 | 단점: SHA 변경, 협업 위험
한 줄 암기: Git Flow=5브랜치 엄격 관리, GitHub Flow=main+feature 단순화, rebase=로컬만/공개브랜치 금지

감리사 핵심 점검 항목

감리 포인트 ① 접근 통제 및 권한 관리
· main/release 브랜치에 직접 Push 가능한 계정 최소화 (Branch Protection Rule)
· 코드 리뷰 없는 Merge 차단 (Required Reviewers 설정 확인)
· 외부 기여자·퇴직자 권한 즉시 회수 여부 확인
· SSH Key / Token 관리 정책 (만료 기한, 범위 최소 원칙)
감리 포인트 ② 이력 무결성
· git rebase / git commit --amend / git push --force로 공개 히스토리 재작성 여부
· 커밋 SHA 변조 가능성 → Signed Commit(GPG 서명) 적용 확인
· 태그(Tag) 삭제·이동 방지 정책 (Annotated Tag + 보호 규칙)
· Audit Log 활성화 여부 (GitHub Enterprise / GitLab Audit Events)
감리 포인트 ③ 민감 정보 유출
· 비밀번호·API Key·개인정보가 커밋 히스토리에 포함 여부 (git log --all 검색)
· .gitignore 미설정으로 환경설정 파일(.env) 업로드 여부
· Secret Scanning / SAST 도구 CI 파이프라인 통합 여부
· 이미 올라간 민감 정보는 BFG Repo Cleaner / git filter-repo로 완전 삭제 필요
감리 포인트 ④ 브랜치 전략 준수
· 정의된 브랜치 전략(Git Flow 등) 실제 운영 준수 여부
· 장기 미병합 Feature 브랜치 존재 여부 (분기 이탈·방치 브랜치)
· PR/MR 리뷰 프로세스 준수 및 리뷰 없이 병합된 케이스 확인
· 배포 가능 상태 보장: main 브랜치 CI 파이프라인 통과 여부
감리 포인트 ⑤ 백업 및 복구
· 원격 저장소 자체 백업 주기 및 보관 정책
· 재해복구(DR) 시나리오에서 Git 저장소 복구 절차 문서화 여부
· Bare Repository 미러 운영 여부 (git clone --mirror)
· Large File Storage(LFS) 사용 시 별도 백업 정책 확인
감리 핵심: 접근통제(Branch Protection) + 이력무결성(Force Push 금지) + 민감정보(Secret Scanning) + 브랜치전략 준수 + 백업정책

감리사 시험 예상 Q&A

Q1. Git의 분산형 VCS 구조가 집중형 대비 가지는 가용성 측면 장점은?
모든 클라이언트가 전체 히스토리를 포함한 저장소 복사본을 보유하므로, 중앙 서버 장애 시에도 로컬에서 커밋·브랜치·이력 조회가 가능하다. 또한 어느 클라이언트든 원격 저장소의 역할을 대신할 수 있어 단일 장애점(SPOF) 위험이 없다.
Q2. git revert와 git reset의 차이를 설명하고, 공개(원격) 브랜치에 적합한 방법은?
git revert는 취소 내용을 담은 새 커밋을 추가하여 기존 히스토리를 보존한다. git reset은 커밋 포인터를 이전으로 이동시켜 히스토리를 재작성한다. 공개 브랜치에는 타 개발자의 로컬과 충돌을 방지하기 위해 반드시 git revert를 사용해야 한다.
Q3. Git 저장소에서 민감 정보(API Key) 커밋 후 조치 방법은?
① 즉시 해당 Key를 무효화(Revoke)한다. ② git filter-repo 또는 BFG Repo Cleaner로 히스토리 전체에서 해당 파일·문자열을 제거한다. ③ 모든 협력자가 저장소를 재클론(re-clone)한다. ④ .gitignore 및 Secret Scanning 도구를 도입하여 재발 방지한다.
Q4. Git 4대 내부 객체를 열거하고 각각의 역할을 설명하라.
① Blob: 파일 내용(바이너리)을 SHA 해시로 저장. 파일명 없음. ② Tree: 디렉터리 구조를 표현. Blob과 하위 Tree의 포인터 목록. ③ Commit: Tree 포인터 + 부모 커밋 SHA + 작성자·시각·메시지 포함. ④ Tag: 특정 Commit에 사람이 읽을 수 있는 이름을 부여. Annotated Tag는 GPG 서명 포함 가능.
Q5. Git Flow에서 hotfix 브랜치를 사용하는 이유와 병합 대상은?
프로덕션(main) 환경의 긴급 결함을 develop 브랜치와 무관하게 즉시 수정하기 위해 main에서 직접 분기한다. 수정 완료 후 main(프로덕션 반영)과 develop(이후 릴리즈에 반영) 양쪽에 모두 병합해야 하며, 태그로 버전을 명시한다.
Q6. Staging Area(Index)의 역할과 감리 관점에서의 의미는?
Staging Area는 커밋에 포함할 변경 사항을 선별하는 중간 준비 공간이다. 감리 관점에서는 개발자가 의도적으로 포함·제외할 파일을 제어할 수 있으므로, 민감 파일이 실수로 커밋되기 전 최후 검토 지점이다. git diff --staged로 커밋 직전 변경 내용을 검증하는 습관이 중요하다.
