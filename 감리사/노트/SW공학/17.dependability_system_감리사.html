Dependability란?

시스템이 믿을 수 있는 서비스를 제공할 수 있는 능력의 총체적 개념.
Avizienis 등이 정의한 신뢰성 공학의 핵심 프레임워크로, 단일 지표가 아닌
속성(Attributes) + 위협(Threats) + 수단(Means) 3축 구조로 체계화됨.

3축 구조 전체 맵

속성 (Attributes)
시스템이 가져야 할 특성
Availability, Reliability, Safety, Integrity, Maintainability (+ Security, Performability)
위협 (Threats)
Dependability를 해치는 요인
Fault(결함) → Error(오류) → Failure(장애) 인과 체인
수단 (Means)
Dependability 달성 방법
Fault Prevention, Tolerance, Removal, Forecasting (예방·허용·제거·예측)
Dependability는 결과(outcome)가 아닌 능력(capability) 개념. "서비스가 얼마나 믿을 수 있는가"를 다면적으로 측정하는 종합 프레임워크
한 줄 암기Dependability = 속성(RAMAS) + 위협(결함→오류→장애) + 수단(예방·허용·제거·예측) 3축 구조

수단 (Means) — 4가지 달성 방법

Fault Prevention (결함 예방)
결함이 생기지 않도록 사전 차단
정형 명세(Formal Specification), 정적 분석, 코드 리뷰, 엄격한 개발 방법론 적용
Fault Tolerance (결함 허용)
결함이 있어도 서비스 계속 제공
이중화(Redundancy), 체크포인트·롤백, 예외 처리, N-버전 프로그래밍, 페일오버
Fault Removal (결함 제거)
존재하는 결함을 찾아 제거
검증(Verification), 유효성확인(Validation), 테스팅, 디버깅, 소프트웨어 검사
Fault Forecasting (결함 예측)
결함 발생 가능성·영향 예측
정성적: FMEA, FTA, HAZOP
정량적: 신뢰도 모델, 마르코프 체인, 결함 예측 모델

신뢰성 측정 핵심 공식

가용성(A) = MTBF / (MTBF + MTTR)
신뢰도 함수 R(t) = e^(-λt) [지수분포, λ = 고장률]
고장률(λ) = 1 / MTBF
MTBF = 총 운영시간 / 고장 횟수
MTTR = 총 복구시간 / 고장 횟수

직렬·병렬 시스템 신뢰도

직렬 시스템
모두 정상이어야 작동
Rs = R1 × R2 × ... × Rn
구성요소 중 하나라도 실패하면 전체 장애. 개수 늘수록 신뢰도 감소
병렬(이중화) 시스템
하나만 정상이면 작동
Rp = 1 - (1-R1)(1-R2)...(1-Rn)
모두 실패해야 전체 장애. 개수 늘수록 신뢰도 증가

욕조 곡선 (Bathtub Curve) — 고장률 시간 변화

구간명칭고장률원인 및 대응
초기유아 사망기 (Burn-in)감소 (DFR)초기 불량·설계 결함. 번인 테스트·초기 검수로 조기 제거
중기정상 운영기 (Useful Life)일정 (CFR)우발 고장. MTBF로 예측 가능. 지수분포 적합

지수분포(Exponential Distribution)는 정상 운영기(CFR)에서 고장률이 일정할 때 적용. 메모리리스(Memoryless) 특성으로 감리사 시험 단골 출제
한 줄 암기욕조곡선: 초기 DFR(유아사망) → 중기 CFR(지수분포·메모리리스) → 말기 IFR(노화·교체)

기출 예상 Q&A

Fault, Error, Failure의 차이를 설명하시오 ▾
Fault(결함)는 오류를 유발할 수 있는 잠재적 원인으로, 활성화 전에는 휴면 상태. Error(오류)는 결함이 활성화돼 시스템 내부 상태가 올바르지 않은 상태. Failure(장애)는 오류가 서비스 경계를 넘어 외부에서 관찰 가능한 이상 동작으로 나타난 것. Fault → Error → Failure 인과 체인으로 연결됨.
Dependability의 4가지 수단(Means)을 설명하시오 ▾
① Fault Prevention(결함 예방): 정형명세·코드 리뷰로 결함이 생기지 않게 차단. ② Fault Tolerance(결함 허용): 결함이 있어도 이중화·N-버전·체크포인트로 서비스 유지. ③ Fault Removal(결함 제거): 테스팅·검증·디버깅으로 존재하는 결함 제거. ④ Fault Forecasting(결함 예측): FMEA·FTA·신뢰도 모델로 결함 발생 가능성·영향 예측.
가용성(Availability)과 신뢰성(Reliability)의 차이는? ▾
가용성은 특정 시점에 서비스가 작동하고 있을 확률(MTBF/(MTBF+MTTR))로, 다운→복구를 반복해도 평균 가동률이 높으면 가용성이 높음. 신뢰성은 연속적으로 중단 없이 동작하는 확률(R(t)=e^(-λt))로, 자주 재시작하는 시스템은 가용성이 높아도 신뢰성은 낮을 수 있음.
N-버전 프로그래밍과 Recovery Block의 차이는? ▾
N-버전 프로그래밍은 N개의 독립적으로 개발된 버전을 동시에 실행하고 투표(Voting)로 결과를 선택하는 병렬 방식. Recovery Block은 주 블록 실행 후 수용 테스트 실패 시 대체 블록으로 순차 전환하는 방식. 전자는 높은 신뢰성, 후자는 낮은 오버헤드가 장점.
욕조 곡선(Bathtub Curve)의 3구간을 설명하시오 ▾
① 유아 사망기(Burn-in): 초기 고장률 감소(DFR), 설계·제조 결함. 번인 테스트로 조기 제거. ② 정상 운영기(Useful Life): 고장률 일정(CFR), 우발 고장, 지수분포 적용·MTBF 예측 가능. ③ 노화기(Wear-out): 고장률 증가(IFR), 마모·노후화. 예방 교체 및 리엔지니어링 필요.
비잔틴 장애(Byzantine Failure)란 무엇인가? ▾
시스템 구성요소가 단순히 동작을 중단하는 것이 아니라, 잘못된 정보를 다른 구성요소에 임의로 전송하는 최악의 장애 유형. 장애 감지 자체가 어려움. 분산 시스템에서 비잔틴 내결함성 합의 알고리즘(PBFT 등)이나 N-버전 프로그래밍으로 대응.
직렬 vs 병렬 시스템 신뢰도 차이 및 계산 방식은? ▾
직렬: Rs = R1 × R2 × ... × Rn. 모든 구성요소가 정상이어야 시스템 정상. 요소 추가 시 신뢰도 감소. 병렬(이중화): Rp = 1-(1-R1)(1-R2)...(1-Rn). 모든 요소가 동시에 실패해야 시스템 장애. 요소 추가 시 신뢰도 증가. 예: R=0.9인 요소 2개 병렬 → 1-(0.1×0.1)=0.99.
