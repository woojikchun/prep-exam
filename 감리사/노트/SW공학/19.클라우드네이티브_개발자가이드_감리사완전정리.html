클라우드 네이티브 개요 & CNCF ⭐

클라우드 네이티브(Cloud-Native): 클라우드 환경의 이점을 최대화하도록 설계·개발·운영되는 시스템 — CNCF(Cloud Native Computing Foundation) 정의: "컨테이너·MSA·불변 인프라·선언적 API를 활용해 복잡성을 분산하고 확장성과 회복력이 높은 시스템 구축"
12-Factor App 방법론 ⭐⭐ — 클라우드 네이티브 앱의 설계 원칙
▾

Adam Wiggins(Heroku, 2011) 제안 — SaaS 애플리케이션을 올바르게 구축하기 위한 12가지 원칙

#원칙핵심 내용적용 예
I코드베이스 (Codebase) ⭐하나의 코드베이스 → 여러 환경 배포 / Git 단일 저장소mono repo or per-service repo
II의존성 (Dependencies) ⭐명시적 선언·격리 — 시스템에 암묵적 의존 금지package.json·pom.xml·requirements.txt
III설정 (Config) ⭐⭐환경에 따라 변하는 모든 것을 환경변수에 저장ENV var·K8s ConfigMap·Secret
IV백엔드 서비스 (Backing Services)DB·메시지큐·캐시를 교체 가능한 리소스로 취급URL로 연결·어태치/디태치 가능
V빌드·릴리즈·실행 (Build/Release/Run) ⭐세 단계 엄격히 분리 — 빌드산출물을 그대로 배포Docker 이미지 불변 artifact
VI프로세스 (Processes) ⭐무상태(Stateless) · 아무것도 공유 안 함 — 세션은 외부 저장소Redis session·DB 상태 저장
VII포트 바인딩 (Port Binding)서비스를 포트 바인딩으로 공개 — 내장 서버Spring Boot 내장 Tomcat
VIII동시성 (Concurrency)프로세스 모델로 수평 확장(Scale-Out)K8s replicas 증가
IX폐기 가능성 (Disposability) ⭐빠른 시작·정상 종료(Graceful Shutdown) — 언제든 대체 가능SIGTERM 처리·readiness probe
X개발/운영 일치 (Dev/Prod Parity) ⭐개발·스테이징·운영 환경 최대한 동일하게 유지Docker Compose 로컬 개발
XI로그 (Logs) ⭐로그를 이벤트 스트림으로 취급 — 파일 저장 아닌 stdout 출력EFK Stack 수집
XII어드민 프로세스 (Admin Processes)일회성 관리 작업을 일반 프로세스처럼 실행K8s Job·CronJob

Dockerfile 작성 예시 ⭐

# Multi-stage Build (최적화) FROM maven:3.9-openjdk-17 AS build WORKDIR /app COPY pom.xml . RUN mvn dependency:go-offline COPY src ./src RUN mvn package -DskipTests # 실행 이미지 — 최소화 FROM eclipse-temurin:17-jre-alpine WORKDIR /app COPY --from=build /app/target/*.jar app.jar EXPOSE 8080 # non-root 실행 (보안) USER nobody ENTRYPOINT ["java","-jar","app.jar"]
  • Multi-stage Build ⭐: 빌드 도구 제외한 최소 이미지 생성
  • Layer 캐싱: 변경 없는 레이어 재사용 → 빌드 속도↑
  • non-root 실행: 보안 모범 사례
컨테이너 격리 기술 & 이미지 레이어 ⭐
▾

Linux 컨테이너 기반 기술

  • Namespace ⭐: 프로세스 격리 — PID·NET·MNT·UTS·IPC·USER 네임스페이스
  • cgroup ⭐: 자원(CPU·메모리·I/O) 제한 및 통계
  • Union FS ⭐: 여러 레이어를 하나의 파일시스템으로 합침 — OverlayFS
  • seccomp: 시스템 콜 필터링 — 컨테이너 보안 강화

OCI 표준 ⭐

  • OCI (Open Container Initiative): 컨테이너 표준 — Image Spec + Runtime Spec
  • OCI Runtime: runc(기본)·crun·gVisor
  • CRI (Container Runtime Interface): K8s와 런타임 연결 인터페이스
  • containerd: Docker 내부 런타임 → K8s가 직접 사용
VM=Guest OS별도(수GB) / 컨테이너=커널공유(수MB)·빠른시작·높은이식성 / Dockerfile=Multi-stage build / Namespace+cgroup=컨테이너격리기반 / OCI=컨테이너표준

Kubernetes (K8s) 핵심 ⭐⭐

Kubernetes: Google이 2014년 오픈소스로 공개한 컨테이너 오케스트레이션 시스템 — 컨테이너화된 애플리케이션의 자동화 배포·확장·관리를 담당. 현재 CNCF 관리
K8s 핵심 오브젝트 ⭐⭐
▾
오브젝트역할핵심 특징
Pod ⭐⭐K8s 최소 배포 단위1개 이상 컨테이너 포함 / 같은 네트워크·볼륨 공유 / 임시적(Ephemeral)
Deployment ⭐⭐Pod 집합 선언적 관리롤링 업데이트·롤백 / ReplicaSet 관리 / 가장 많이 사용
Service ⭐⭐Pod에 안정적 네트워크 접근ClusterIP(내부)·NodePort(외부)·LoadBalancer(클라우드)·ExternalName
ConfigMap ⭐설정 데이터 저장환경변수·파일로 Pod에 주입 / 12-Factor III 구현
Secret ⭐민감 데이터 저장base64 인코딩(암호화 아님) / 별도 권한 관리
Ingress ⭐외부 HTTP/S 트래픽 라우팅L7 라우팅·TLS·가상 호스팅 / Nginx·Traefik
PersistentVolume (PV/PVC) ⭐영구 스토리지 추상화PV=스토리지 공급 / PVC=스토리지 요청 / StorageClass
Namespace클러스터 논리적 분리팀·환경별 분리 (dev·staging·prod)
StatefulSet ⭐상태 있는 Pod 관리고정된 네트워크ID·순서 있는 배포 / DB·Kafka
DaemonSet모든 노드에 Pod 1개 배포로그 수집·모니터링 에이전트
HPA ⭐수평 자동 확장CPU·메모리 메트릭 기반 Pod 수 자동 조절

DevOps · CI/CD · GitOps · IaC ⭐⭐

CI/CD 파이프라인 전체 흐름 ⭐⭐
▾

CI (Continuous Integration) ⭐

  • 목적: 코드 변경을 자주·자동으로 통합하여 조기 결함 발견
  • 핵심 활동: 자동 빌드·단위 테스트·정적 분석·코드 품질 측정
  • 도구 ⭐: Jenkins·GitHub Actions·GitLab CI·CircleCI
  • 트리거: Git push·PR(Pull Request) 생성 시 자동 실행

CD (Continuous Delivery/Deployment) ⭐

  • Continuous Delivery ⭐: 언제든 운영 배포 가능 상태 유지 — 배포는 수동 승인
  • Continuous Deployment ⭐: 승인 없이 자동 운영 배포
  • 도구: ArgoCD·Flux·Spinnaker·Helm
  • 전략: 카나리·블루그린·롤링 업데이트
GitOps · IaC · DevSecOps ⭐
▾

GitOps ⭐⭐

  • 정의: Git을 단일 진실의 원천(SSOT)으로 사용 — 원하는 상태를 Git에 선언
  • 핵심 원칙 ⭐: 선언적 인프라·Git 버전 관리·자동 동기화·지속적 reconciliation
  • Push vs Pull ⭐: GitOps는 Pull 방식 — ArgoCD가 Git 변경을 감지하여 K8s에 적용
  • 도구: ArgoCD·FluxCD

IaC (Infrastructure as Code) ⭐

  • 정의: 인프라를 코드로 정의·관리 — 버전 관리·재현 가능
  • 도구 ⭐: Terraform(다중 클라우드)·Pulumi·AWS CDK·Ansible
  • 장점: 드리프트 감지·환경 일관성·빠른 복구

DevSecOps ⭐

  • 정의: DevOps에 보안(Security)을 통합 — Shift Left Security
  • SAST ⭐: 정적 앱 보안 테스트 — 코드 분석 (SonarQube·Semgrep)
  • DAST ⭐: 동적 앱 보안 테스트 — 실행 중 공격 시뮬레이션 (OWASP ZAP)
  • SCA ⭐: 오픈소스 의존성 취약점 분석 (Snyk·Dependabot)
  • 컨테이너 스캔 ⭐: 이미지 취약점 스캔 (Trivy·Clair)
  • Secrets 스캔: 코드에서 키·비밀번호 탐지 (git-secrets·truffleHog)
CI=자동빌드·테스트·통합 / CD Delivery=수동승인·Deployment=자동배포 / GitOps=Git이 SSOT·Pull방식·ArgoCD / IaC=인프라코드화(Terraform) / DevSecOps=Shift Left Security

서비스 메시: MSA에서 서비스 간 통신을 관리하는 인프라 계층 — 서비스 코드 변경 없이 트래픽 관리·보안·관찰가능성·회복력을 제공

Control Plane vs Data Plane ⭐

  • Control Plane: 정책 배포·인증서 관리·서비스 디스커버리 (Istiod)
  • Data Plane: 실제 트래픽 처리 — Sidecar 프록시(Envoy)
서비스메시: Sidecar(Envoy) 자동 주입 / 트래픽관리·mTLS·관찰가능성·정책 코드변경없이 제공 / Istio=가장 널리사용 / Linkerd=경량 / ControlPlane↔DataPlane(Sidecar)

Observability(관찰가능성): 시스템 외부 출력만으로 내부 상태를 이해하는 능력 — 클라우드 네이티브 환경에서는 로그(Logs)·메트릭(Metrics)·추적(Traces) 세 가지 신호가 핵심

OpenTelemetry (OTEL) ⭐⭐

  • 정의: CNCF 프로젝트 — 로그·메트릭·추적 수집의 오픈 표준
  • 구성: API + SDK + Collector(수집기)
  • 의의 ⭐: 벤더 중립 — Jaeger·Datadog·Zipkin 등 다양한 백엔드로 전송
  • SDKs: Java·Python·Go·Node.js 등 다수

SRE 핵심 지표 (Google SRE) ⭐

지표설명
SLI ⭐서비스 수준 지표 — 실제 측정값 (가용성·응답시간)

RBAC (Role-Based Access Control) ⭐⭐

# Role 정의 apiVersion: rbac.authorization.k8s.io/v1 kind: Role metadata: namespace: production name: pod-reader rules: - apiGroups: [""] resources: ["pods"] verbs: ["get","list","watch"] # 최소 권한 원칙

Secrets 관리 모범 사례 ⭐

  • K8s Secret 한계: base64 인코딩 — 암호화 아님
  • 외부 Secrets 관리 ⭐: HashiCorp Vault·AWS Secrets Manager·Sealed Secrets
  • External Secrets Operator: 외부 저장소 → K8s Secret 동기화
  • 절대 금지: 코드·Git에 시크릿 하드코딩

NetworkPolicy ⭐

  • 기본: 모든 Pod 간 통신 허용 (Zero-Trust 불가)
  • NetworkPolicy로 인그레스/이그레스 명시 제한
  • CNI 플러그인 지원 필요: Calico·Cilium·Weave
감리 체크포인트 & 비용 최적화 ⭐
▾

클라우드 네이티브 감리 체크포인트

  • 컨테이너 보안: non-root·읽기전용FS·이미지스캔 적용 여부
  • K8s 보안: RBAC·NetworkPolicy·PodSecurityAdmission 설정
  • Secrets 관리: 코드 내 하드코딩 없음·외부 Secrets 저장소 사용
  • CI/CD 보안: DevSecOps(SAST·SCA·DAST) 파이프라인 통합
  • Observability: 로그·메트릭·추적 3가지 신호 수집 여부
  • SLO 설정: 서비스 수준 목표·Error Budget 관리 여부
  • 12-Factor: 설정을 환경변수로 관리·상태 없는 앱 구현
  • IaC: 인프라 코드화·GitOps로 버전 관리

클라우드 비용 최적화 (FinOps) ⭐

  • 리소스 Request/Limit ⭐: 적절히 설정하여 낭비 방지
  • HPA/VPA ⭐: 부하에 따른 자동 스케일링
  • Spot/Preemptible 인스턴스: 가격 70~90% 절감
  • Namespace 별 비용 추적: Kubecost·OpenCost
  • 이미지 최적화: Multi-stage build로 용량 최소화
  • Right-Sizing: 실제 사용량 기반 리소스 조정
