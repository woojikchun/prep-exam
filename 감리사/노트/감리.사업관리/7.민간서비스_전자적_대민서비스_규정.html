행정안전부 고시 제2023-26호 디지털플랫폼정부

민간서비스 활용을 통한 전자적 대국민서비스 제공에 관한 규정

시행 2023.4.18 · 행정안전부 고시 제2023-26호 · 최초 제정 · 전자정부법 제12조의2 근거

📌

규정 제정 개요

📜 법적 근거
전자정부법 제12조의2 (전자적 대민서비스 제공을 위한 민간서비스 활용) 및 같은 법 시행령에 근거하여 행정안전부장관이 제정·고시
구분내용
고시명민간서비스 활용을 통한 전자적 대민서비스 제공 등에 관한 규정
발행행정안전부 고시 제2023-26호
시행2023년 4월 18일 (최초 제정)
소관 부처행정안전부 (디지털정부국)
적용 대상행정기관·공공기관(개방기관·수요기관) 및 민간기업(수요기관)
제정 목적 = 민관 융합서비스 창출 + 공공서비스 개방 절차 표준화 + 안정성 확보
⚡

제정 배경 및 필요성

  • 문제: 민간기업이 개방된 공공서비스를 자사 서비스와 결합하려 해도 표준화된 절차·방법 부재
  • 현황: 카카오·네이버 등 민간 플랫폼을 통해 공공서비스 이용 수요 급증
  • 목적: 디지털플랫폼정부 구현 — 원사이트토털서비스 제공 기반 마련
  • 기대효과: 민관 융합서비스 창출, 대국민 안정성·신뢰성 향상
🎯

주요 제정 내용 (3가지 축)

  • ① 원칙·체계: 디지털서비스 개방 기본 원칙·추진체계·참여절차 규정
  • ② 개방 절차: 수요조사→서비스 발굴→공개 공모→민간 참여 지원
  • ③ 안전 관리: 사업 단계별 보안성 점검, 복수 민간서비스 선정 의무화, 운영 품질 관리

디지털서비스 개방 원칙

  • 개방성: 다양한 민간기업에 공공서비스 연계 참여 기회 부여
  • 안전성: 민간서비스 적용 후 공공서비스 안정성 확보
  • 중립성: 특정 민간 플랫폼에 대한 독점적 지위 방지
  • 공익성: 국민 편의 증진 및 대국민 서비스 품질 향상
  • 투명성: 공개 공모 방식으로 투명한 참여 절차 보장
📊

민간서비스 선정 기준

  • 안정성: 서비스 가용성 및 장애 복구 능력
  • 보안성: 개인정보 보호, 암호화, 보안 취약점 관리
  • 호환성: 공공시스템과의 API 연계 가능 여부
  • 대중성: 국민 이용 편의성 및 접근성
  • 복수 선정: 동일 기능에 복수의 민간서비스 선정 (단일 의존 방지)
🔑

주요 용어 정의

용어정의
전자적 대민서비스행정기관등이 전자적 방법으로 국민·기업 등에게 제공하는 공공서비스
민간서비스민간기업이 운영하는 플랫폼·앱 등을 통해 제공되는 서비스 (카카오·네이버 등)
개방기관공공서비스를 민간에 개방하는 행정기관 또는 공공기관
수요기관민간에 개방된 공공서비스를 연계·활용하여 대민서비스를 제공하는 기관 또는 민간기업
디지털서비스플랫폼공공서비스와 민간 등을 연계하는 중계 기반 플랫폼 (원사이트토털서비스 구현 핵심)
원사이트토털서비스국민이 하나의 플랫폼에서 필요한 모든 공공·민간 서비스를 이용할 수 있는 개념
민관 융합서비스공공서비스와 민간서비스가 결합되어 새롭게 창출되는 융합 서비스

②
공개 공모 및 민간기업 선정
▾
  • 공모 방식: 공개 공모를 통해 다양한 민간기업에 참여 기회 부여
  • 선정 기준: 안정성·보안성·호환성·대중성 등 종합 평가
  • 복수 선정 의무: 동일 공공서비스에 2개 이상의 민간서비스 의무 선정
  • 목적: 특정 민간 플랫폼 독점 방지, 장애 시 대체 서비스 확보
공모 공고선정 결과 공고참여협약서

API 연계 및 서비스 구축

  • 디지털서비스플랫폼: API 기반 공공서비스와 민간 플랫폼 연계
  • 개방기관: API 표준 규격 제공, 연계 기술 지원
  • 수요기관(민간): 공공서비스 API를 자사 서비스에 통합 개발

  • 현황 관리: 구축 완료 후 정보자원 항목으로 현황 등록
API 연계 명세서보안 검토 결과정보자원 등록

  • 품질 관리: 서비스 응답시간, 오류율, 사용자 만족도 등 품질지표 관리
  • 장애 대응: 민간서비스 장애 시 대체 서비스로 자동 전환 또는 수동 조치

민간서비스 활용 시 보안 요건

보안 영역요건점검 방법
개인정보 보호 개인정보 처리방침 적정성, 수집·이용 목적 명확화, 정보주체 동의 절차 개인정보 영향평가, 개인정보보호법 준수 여부
API 보안 API 인증(OAuth 등) 적용, 전송 구간 암호화(HTTPS), API 키 관리 API 보안 취약점 점검, 침투 테스트
가용성 서비스 가용률 SLA(예: 99.9%), 장애 발생 시 복구 목표시간(RTO/RPO) 가용성 모니터링, 장애 대응 훈련
복수 서비스 동일 기능에 2개 이상 민간서비스 선정 — 특정 서비스 의존 위험 분산 장애 시 자동/수동 전환 테스트
정보자원 관리 민간서비스 활용 현황을 정보자원 목록에 등록·최신화 ITA 시스템 등록 현황 확인
보안 핵심: 개인정보보호 + API 보안 + 가용성(SLA) + 복수선정(리스크 분산) + 정보자원 등록
⚠️

복수 민간서비스 선정 의무 이유

  • 단일 장애점(SPOF) 방지: 특정 민간 플랫폼 장애 시 공공서비스 중단 위험
  • 독점 방지: 특정 민간 플랫폼의 공공서비스 독점 구조 차단
  • 국민 선택권 보장: 다양한 민간 플랫폼에서 동일 서비스 이용 가능
  • 경쟁 촉진: 복수 참여로 서비스 품질 경쟁 유도
  • 협상력 확보: 개방기관이 민간에 대한 협상력 유지
📊

감리 관점 점검 포인트

  • 사업계획: 민간서비스 선정기준(안정성·보안성) 명시 여부
  • 구축 단계: 복수 민간서비스 선정 의무 이행 여부
  • 완료 단계: 정보자원 항목 등록·현행화 여부
  • 운영 단계: 안전성·품질 정기 점검 수행 여부
  • 보안: API 보안·개인정보 처리 적정성
  • SLA: 민간기업과 서비스 수준 협약 체결 여부
