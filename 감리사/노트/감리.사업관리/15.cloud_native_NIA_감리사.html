클라우드 네이티브 정의 (CNCF)

"클라우드 네이티브 기술은 조직이 퍼블릭·프라이빗·하이브리드 클라우드 등 현대적·동적 환경에서 확장 가능한 애플리케이션을 개발·실행할 수 있게 해준다.
컨테이너, 서비스 메쉬, 마이크로서비스, 불변 인프라, 선언형 API가 이러한 접근 방식의 예시이며,
회복성·관리 편의성·가시성을 갖춘 느슨하게 결합된 시스템을 가능하게 한다." — CNCF

컨테이너 개념

애플리케이션 실행에 필요한 코드·런타임·라이브러리·설정 파일을 하나의 이미지로 패키징한 경량 가상화 단위.
Host OS 커널을 공유하므로 VM보다 가볍고 빠름. "한 번 빌드, 어디서나 실행(Write Once, Run Anywhere)".

불변 인프라 (Immutable Infrastructure)

컨테이너 이미지는 불변(Immutable). 배포 후 수정하지 않고, 변경 시 새 이미지를 빌드해 교체. 환경 간 일관성 보장, "내 PC에서는 됐는데" 문제 제거. 감리 시 이미지 태그 관리·레지스트리 보안 확인 필요
한 줄 암기컨테이너 = OS커널 공유·경량·이식성↑(VM 대비). 이미지(불변 템플릿) → 컨테이너(실행 인스턴스). Dockerfile→이미지 빌드→레지스트리 저장→컨테이너 실행

MSA (Microservice Architecture) 정의

애플리케이션을 독립적으로 배포·확장 가능한 작은 서비스 단위로 분리해 개발·운영하는 아키텍처.
각 서비스는 자체 프로세스·데이터베이스·기술 스택을 가지며 API로 통신.
DDD(도메인 주도 설계)의 Bounded Context가 서비스 경계의 기준.

MSA 핵심 설계 원칙

원칙내용
단일 책임 (SRP)하나의 서비스는 하나의 비즈니스 기능만 담당
서비스별 DB각 마이크로서비스는 자체 데이터 저장소 보유. DB 공유 금지
느슨한 결합서비스 간 직접 의존 최소화. API·메시지를 통해 통신
높은 응집도관련 기능·데이터는 같은 서비스 내에 유지
API 우선 (API-First)서비스 인터페이스를 먼저 설계. REST API 또는 gRPC
실패 격리하나의 서비스 실패가 전체로 전파되지 않도록 설계

MSA 주요 패턴

API Gateway
단일 진입점
클라이언트 → API Gateway → 내부 서비스. 인증·라우팅·로드밸런싱·속도제한 담당
Circuit Breaker
장애 전파 차단
연속 실패 감지 시 회로 차단. Fallback 응답 제공. 장애 격리·복구 지원. Hystrix, Resilience4j
Saga
분산 트랜잭션

CQRS
읽기/쓰기 분리
Command(쓰기)와 Query(읽기) 모델 분리. 성능·확장성 최적화
Event Sourcing
이벤트 저장
상태 변경을 이벤트로 저장. 이벤트 재생으로 상태 복원. CQRS와 함께 사용
Service Discovery
서비스 위치 동적 탐색
동적 IP 환경에서 서비스 위치 자동 등록·탐색. Eureka, Consul, Kubernetes Service

MSA 서비스 간 통신 방식

방식특징적합 상황

DevOps 정의 및 문화

Dev(개발) + Ops(운영) 통합 문화·실천 방법론.
개발팀과 운영팀의 사일로를 제거하고, 자동화·협업·빠른 피드백으로 소프트웨어를 더 빠르고 안정적으로 제공.
문화(Culture)·자동화(Automation)·측정(Measurement)·공유(Sharing) — CAMS 원칙.

Kubernetes 개요

컨테이너화된 애플리케이션의 배포·스케일링·관리를 자동화하는 오픈소스 오케스트레이션 플랫폼 (CNCF 졸업 프로젝트).
"클라우드 네이티브의 운영체제"로 불림. 구글에서 시작, CNCF에 기증.

서비스 메시 (Service Mesh) 개요

MSA 환경에서 서비스 간 네트워크 통신을 인프라 레벨에서 투명하게 관리하는 전용 인프라 레이어.
애플리케이션 코드 수정 없이 트래픽 관리·보안·관측 가능성 기능을 제공.
각 서비스 옆에 Sidecar 프록시를 붙여 동작 (대표: Istio + Envoy).

서비스 메시가 해결하는 문제

트래픽 관리
카나리 배포·A/B 테스트·서킷 브레이커·재시도·타임아웃을 코드 없이 설정
보안 (mTLS)
서비스 간 상호 TLS(mTLS) 자동 적용. 인증서 자동 발급·갱신. Zero Trust 네트워크 구현
관측 가능성
분산 추적·메트릭·로그 자동 수집. 서비스 간 의존성 시각화 (Service Graph)

Sidecar 패턴

모든 서비스 Pod에 Envoy 프록시 컨테이너를 자동 삽입(Injection). 인바운드·아웃바운드 트래픽이 모두 Sidecar를 거침. 애플리케이션은 통신 복잡성을 모름. Control Plane(Istiod)이 모든 Sidecar 설정을 중앙 관리

Observability (관측 가능성) 개념

시스템의 외부 출력(로그·메트릭·트레이스)을 통해 내부 상태를 이해·진단할 수 있는 능력.
단순 모니터링(알려진 문제 감지)과 달리, 알 수 없는(unknown) 문제도 탐색 가능.
MSA·분산 시스템에서 단일 서비스 장애를 근본 원인(Root Cause)까지 추적하기 위해 필수.

기출 예상 Q&A

클라우드 네이티브의 정의와 4대 핵심 구성요소를 설명하시오 ▾
CNCF 정의: 퍼블릭·프라이빗·하이브리드 클라우드 등 현대적 동적 환경에서 확장 가능한 애플리케이션을 개발·실행할 수 있게 하는 기술. 회복성·관리 편의성·가시성을 갖춘 느슨하게 결합된 시스템을 가능하게 함. 4대 구성요소: ① 컨테이너 — 앱을 이식 가능한 이미지로 패키징 ② MSA — 독립 배포 가능한 마이크로서비스로 분해 ③ DevOps — 개발·운영 통합 문화와 자동화 ④ CI/CD — 지속적 통합·배포 파이프라인으로 빠른 릴리스 실현.
VM과 컨테이너의 차이를 비교하고 컨테이너의 장점을 설명하시오 ▾
VM은 Hypervisor가 하드웨어를 가상화하고 Guest OS를 포함해 수 GB·분 단위 기동. 컨테이너는 Host OS 커널을 공유하고 앱+런타임만 패키징해 수 MB·초 단위 기동. 컨테이너 장점: 경량성(커널 공유), 이식성(이미지 단위 이동), 밀도(동일 호스트에 다수 실행), 불변성(이미지 기반 일관성), 빠른 배포(초 단위 기동). 단점: VM 대비 격리 수준 약함(커널 공유 취약점 위험).
MSA의 Circuit Breaker 패턴을 설명하고 왜 필요한가? ▾
Circuit Breaker는 서비스 간 호출에서 연속 실패를 감지하면 회로를 차단(Open)하고 더 이상 해당 서비스를 호출하지 않는 패턴. 상태: Closed(정상 호출) → Open(차단·Fallback 응답) → Half-Open(일부 호출 테스트) → Closed(복구) 순환. 필요 이유: MSA에서 하나의 서비스 장애가 연쇄적으로 의존 서비스 전체를 마비시키는 Cascading Failure 방지. 구현: Hystrix, Resilience4j, 서비스 메시(Istio).
CI(지속적 통합)와 CD(지속적 전달·배포)의 차이는? ▾
CI(Continuous Integration): 개발자가 코드를 자주 메인 브랜치에 병합하고, 자동 빌드·테스트를 실행해 통합 오류를 조기에 발견. CD(Continuous Delivery, 지속적 전달): CI를 통과한 코드를 스테이징 환경까지 자동 배포. 운영 배포는 수동 승인. 항상 배포 가능 상태 유지. CD(Continuous Deployment, 지속적 배포): CI 통과 후 사람 개입 없이 운영 환경까지 완전 자동 배포. 블루/그린·카나리 배포 전략과 결합해 무중단 배포 구현.
Kubernetes의 Pod, Deployment, Service의 역할과 관계를 설명하시오 ▾
Pod는 컨테이너 실행의 최소 단위. 1개 이상 컨테이너를 묶어 동일 IP·Volume을 공유. 직접 생성하면 장애 시 재생성 안 됨. Deployment는 Pod의 선언적 관리자. replica 수·롤링 업데이트·롤백을 자동 처리. "이 Pod를 3개 항상 실행하라"고 선언하면 컨트롤러가 유지. Service는 Pod 집합에 안정적 가상 IP와 DNS를 제공하는 로드밸런서. Pod는 IP가 바뀌지만 Service IP는 고정. 관계: Deployment가 Pod를 관리하고, Service가 Pod 집합으로 트래픽을 분산.
Observability의 3대 신호와 모니터링과의 차이는? ▾
3대 신호: ① 메트릭 — CPU·메모리·오류율 등 수치형 시계열 데이터(Prometheus). ② 로그 — 타임스탬프+이벤트 텍스트 기록. 장애 원인 분석용(ELK). ③ 트레이스 — 단일 요청이 여러 서비스를 거치는 전체 경로 추적(Jaeger). 모니터링과 차이: 모니터링은 미리 알려진 문제를 임계값 기반으로 감지하는 반응적 접근. Observability는 알 수 없는 문제도 데이터 탐색으로 근본 원인을 찾는 능동적 접근. MSA의 분산 환경에서 단일 요청이 수십 개 서비스를 거치므로 트레이스가 특히 중요.
서비스 메시와 Sidecar 패턴을 설명하고 mTLS 적용 효과는? ▾
서비스 메시는 MSA에서 서비스 간 통신을 애플리케이션 코드 수정 없이 인프라 레벨에서 관리하는 전용 레이어. Sidecar 패턴: 각 서비스 Pod에 Envoy 프록시 컨테이너를 자동 삽입. 모든 트래픽이 Sidecar를 거쳐 트래픽 관리·보안·관측이 투명하게 이루어짐. mTLS 효과: 서비스 간 통신에서 클라이언트·서버 양쪽이 인증서를 상호 제시(Zero Trust). 내부 네트워크도 암호화·인증. 인증서 자동 발급·갱신. 대표 솔루션: Istio(Envoy 기반, 기능 풍부)·Linkerd(경량).
