클라우드 네이티브 정의 (CNCF)

"클라우드 네이티브 기술은 조직이 퍼블릭·프라이빗·하이브리드 클라우드와 같은 현대적·동적인 환경에서 확장 가능한 애플리케이션을 개발하고 실행할 수 있게 해준다. 컨테이너·서비스 메시·마이크로서비스·불변(Immutable) 인프라·선언형 API가 예시들이다."

— CNCF(Cloud Native Computing Foundation) 정의

  • 핵심 특성: 회복성(Resilience) · 관리 편의성(Manageability) · 가시성(Observability)
  • 목적: 클라우드 이점을 최대한 활용하도록 처음부터 설계된 아키텍처·개발 방식
  • 대비 개념: Lift & Shift(리호스팅) — 기존 시스템을 그대로 클라우드로 이전하는 방식
  • 안내서 발행: 한국지능정보사회진흥원(NIA) · 공공부문 정보화 사업 발주자·기획자 대상
클라우드 네이티브 = 클라우드 이점 최대 활용하도록 처음부터 설계된 방식

📌 발주자 안내서 목적

  • 공공부문 정보화 사업 발주자에게 개념 이해 및 도입 적합성 검토 지원
  • 신규·차세대/ISP/ISMP 프로젝트를 클라우드 네이티브 기반으로 기획·발주 지원
  • 클라우드 네이티브 기술의 필요성·장점 이해
  • 도입 전 사전준비도 및 적합성 체크 지원

🏛️ 공공부문 필요성

  • 대국민 서비스를 안정적·신속하게 제공하기 위한 필수 기반
  • 디지털 전환(DX) 가속화 대응
  • 기존 모놀리식 아키텍처의 유연성·확장성 한계 극복
  • IaaS 단계를 넘어 PaaS 기반 MSA 구축으로 발전

모놀리식 vs 마이크로서비스 아키텍처 비교

🏗️ 모놀리식 (Monolithic)
  • 전체 앱이 단일 패키지로 구성
  • 내부 모듈이 강하게 결합(Tightly Coupled)
  • 일부 수정 시 전체 재배포 필요
  • 확장 시 전체 시스템 수평 확장
  • 단순하나 규모 커지면 유연성↓
⇄
⚙️ MSA (Microservice)
  • 앱을 독립적 소규모 서비스로 분해
  • 서비스 간 느슨한 결합(Loosely Coupled)
  • 특정 서비스만 독립 배포·확장 가능
  • 필요한 서비스만 선택적 확장
  • 복잡하나 민첩성·확장성↑
모놀리식=강결합·전체재배포 vs MSA=느슨한결합·독립배포·서비스단위확장

📦
컨테이너 (Container)
애플리케이션 패키징 기술
앱 실행에 필요한 라이브러리·구성파일 등을 패키지로 묶어 이미지처럼 배포. Host OS 커널 공유로 경량성·이식성 확보. VM 대비 가상화 솔루션 비용 없음.
DockerKubernetes경량성이식성
⚙️
MSA (Micro Service Architecture)
마이크로서비스 아키텍처
하나의 애플리케이션을 독립적으로 배포·확장 가능한 작은 서비스로 분해. API를 통해 서비스 간 통신. 각 서비스 독립 개발·배포·확장 가능.
API GatewayService Mesh느슨한결합
🤝
DevOps
개발+운영 협업 문화·프로세스
Development + Operation의 합성어. 개발·운영 간 협업 강화로 개발부터 배포까지 자동화·속도 향상. 조직 문화와 프로세스의 변화를 포함.
자동화협업문화빠른피드백
🚀
CI/CD
지속적 통합·배포 파이프라인
CI(지속적 통합): 코드 변경사항 자동 빌드·테스트. CD(지속적 배포/전달): 자동화된 배포 파이프라인. 개발에서 운영까지 인간 개입 최소화.
JenkinsGitOps자동배포
4대 요소: 컨테이너(포장) → MSA(설계) → DevOps(문화) → CI/CD(자동화파이프라인)

🚀 CI vs CD 구분

12 Factor App 방법론

클라우드 네이티브 애플리케이션 개발의 모범 방법론 — Heroku가 제창, 클라우드 환경 최적화 12가지 원칙

FACTOR 01
코드베이스 (Codebase)
하나의 코드베이스, 다수 배포. Git 등 버전 관리 시스템으로 단일 저장소 관리
FACTOR 02
의존성 (Dependencies)
의존성 명시적 선언·격리. 시스템 수준 패키지에 암묵적으로 의존하지 않음
FACTOR 03
설정 (Config)
환경변수에 설정 저장. 코드와 설정 분리 — 환경마다 다른 설정값 외부화
FACTOR 04
백엔드 서비스 (Backing Services)
DB·MQ·캐시 등을 부착된 자원으로 취급. 로컬과 서드파티 서비스 구분 없이
FACTOR 05
빌드·릴리즈·실행 (Build/Release/Run)
빌드→릴리즈→실행 단계 엄격히 분리. 각 릴리즈는 고유 ID, 불변(Immutable)
FACTOR 06
프로세스 (Processes)
앱을 하나 이상의 Stateless 프로세스로 실행. 영속 데이터는 DB에 저장
FACTOR 07
포트 바인딩 (Port Binding)
포트 바인딩으로 서비스 공개. 외부 웹서버 의존 없이 자체 HTTP 서버 내장
FACTOR 08
동시성 (Concurrency)
프로세스 모델로 수평 확장. 워크로드 유형에 따라 프로세스 타입 분류
FACTOR 09
폐기 가능성 (Disposability)
빠른 시작·우아한 종료로 견고성 극대화. 프로세스 언제든 시작·중지 가능
FACTOR 10
개발/운영 일치 (Dev/Prod Parity)
개발·스테이징·운영 환경을 최대한 동일하게 유지. 지속적 배포 지원
FACTOR 11
로그 (Logs)
로그를 이벤트 스트림으로 취급. 앱은 로그 파일 관리 않고 stdout으로 출력
FACTOR 12
관리 프로세스 (Admin Processes)
일회성 관리 작업을 별도 프로세스로 실행. DB 마이그레이션, 콘솔 스크립트 등
12 Factor = 클라우드 네이티브 앱의 모범 설계 원칙 / 설정=환경변수, 프로세스=Stateless

🔑 시험 핵심 Factor

  • Factor 3 (설정): 코드와 설정 분리 — 환경변수에 저장
  • Factor 6 (프로세스): Stateless 프로세스 — 세션 등 외부 저장
  • Factor 9 (폐기 가능성): 빠른 시작·우아한 종료
  • Factor 10 (개발/운영 일치): 환경 격차 최소화
  • Factor 11 (로그): 로그 = 이벤트 스트림, stdout 출력

📌 Stateless vs Stateful

  • Stateless: 각 요청이 독립적. 서버에 상태 저장 없음. 수평 확장에 유리
  • Stateful: 서버에 세션·상태 저장. 특정 서버 의존. 확장 어려움
  • 클라우드 네이티브 원칙: 앱 프로세스는 Stateless 설계 권고
  • 상태 저장 위치: DB, Redis, 외부 스토리지 등 Backing Service

단계별 발주자 주요 역할

단계발주자 주요 활동핵심 산출물
①기획·검토 정보시스템 자가진단 체크리스트 활용, 도입 적합성 검토, ISP/ISMP 시 클라우드 네이티브 방향 결정 사업계획서(안), 자가진단 체크리스트
②계획 수립 기술적용계획 수립, 클라우드 서비스 유형(IaaS/PaaS/SaaS) 결정, RFP에 클라우드 네이티브 요건 반영 기술적용계획서, 제안요청서(RFP)
③사업자 선정 기술평가 시행, 보안성 검토, 기술협상 시 클라우드 네이티브 이행사항 구체적 협상 기술평가 결과서, 계약서
④개발·구축 DevOps 환경 구성 확인, CI/CD 파이프라인 구축 현황 점검, 컨테이너 이미지 표준 준수 착수계획서, 진도/품질 관리 보고
⑤감리 클라우드 네이티브 구현 표준 준수 점검, MSA 설계 적정성, 보안 취약점 점검 감리수행결과보고서
⑥검사·종료 클라우드 네이티브 요건 이행 확인, 운영 이관 체크리스트, 컨테이너 운영 교육 검사결과보고서, 운영이관 문서

🔍 클라우드 네이티브 감리 점검 포인트

  • 컨테이너: 이미지 최소화 원칙, 취약점 스캔 수행 여부
  • MSA 설계: 서비스 분해 기준 적정성, API 설계 표준 준수
