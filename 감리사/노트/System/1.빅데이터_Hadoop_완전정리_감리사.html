빅데이터 처리 아키텍처 — Lambda vs Kappa

Lambda 아키텍처 ⭐

  • 배치 레이어: 전체 데이터 배치 처리 (정확성)
  • 스피드 레이어: 실시간 데이터 처리 (최신성)
  • 서빙 레이어: 두 결과를 병합하여 쿼리 응답
  • 장점: 정확성+실시간 모두 보장
  • 단점: 두 레이어 유지 복잡·코드 중복

Kappa 아키텍처

  • 단일 스트림 처리: 배치 레이어 제거 / 모든 처리를 실시간 스트림으로
  • 재처리: 필요 시 스트림 처음부터 재처리
  • 장점: Lambda보다 단순 / 코드 재사용
  • 단점: 스트림 처리 부하 高 / 재처리 비용
  • 적합: Kafka·Flink 기반 환경
Lambda=배치+스피드+서빙 3레이어·정확성+실시간 / Kappa=단일스트림 처리·단순화

Hadoop 에코시스템 개요 ⭐

Apache Hadoop: 대용량 데이터를 상용 서버 클러스터(저가 하드웨어)에 분산 저장·처리하는 오픈소스 프레임워크 (Google GFS·MapReduce 논문 기반, 2006년 Doug Cutting)
핵심 구성요소역할
HDFS ⭐분산 파일 시스템 — 대용량 파일을 블록 단위로 분산 저장 + 복제
MapReduce ⭐분산 데이터 처리 패러다임 — Map(분할처리)+Reduce(집계)
YARN ⭐클러스터 자원 관리자 (Yet Another Resource Negotiator) — CPU·메모리 할당
HiveSQL-like 쿼리(HiveQL)로 HDFS 데이터 처리 — DW 쿼리 도구
HBaseHDFS 위의 NoSQL 컬럼 패밀리 DB — 실시간 읽기·쓰기
Pig데이터 흐름 스크립트 언어(Pig Latin) — MapReduce 추상화
SqoopRDB ↔ HDFS 데이터 임포트·익스포트
Flume로그·이벤트 데이터를 HDFS로 실시간 수집
OozieHadoop 워크플로우 스케줄러
ZooKeeper분산 코디네이션 서비스 — 분산 시스템 동기화·리더 선출
Hadoop 핵심 3요소: HDFS(저장)·MapReduce(처리)·YARN(자원관리) / Hive=SQL쿼리 / HBase=실시간NoSQL

HDFS (Hadoop Distributed File System) ⭐

HDFS 구조 — NameNode & DataNode ⭐
▾

NameNode (마스터) ⭐

  • 역할: 파일 시스템 메타데이터 관리 (파일 이름·블록 위치·권한)
  • 실제 데이터는 저장하지 않음
  • FsImage: 네임스페이스 스냅샷 (디스크)
  • EditLog: 변경 트랜잭션 로그 (디스크)
  • SPOF 위험: NameNode 장애 = 전체 클러스터 중단
  • HA 구성: Active NameNode + Standby NameNode (Quorum JournalManager)
  • Secondary NameNode: HA가 아님! FsImage+EditLog 병합 보조

DataNode (슬레이브)

  • 역할: 실제 데이터 블록 저장·처리
  • NameNode에 주기적으로 Heartbeat 전송
  • Block Report: 보유 블록 목록을 NameNode에 보고
  • 복제(Replication): 기본 복제 계수 = 3 (같은 랙 2개 + 다른 랙 1개)
  • DataNode 장애 시 다른 노드에서 자동 복제 복구
HDFS 블록(Block) & 데이터 특성 ⭐
▾
  • 블록 크기: 기본 128MB (Hadoop 2.x~) / Hadoop 1.x=64MB — 대용량 파일에 최적화
  • Write-Once-Read-Many (WORM): 파일 한 번 쓰면 수정 불가 (Append 제한적 지원) — 불변성
  • 랙 인식(Rack Awareness): 블록 복제 시 다른 랙에 분산하여 랙 장애에도 데이터 보호
  • 스트리밍 데이터 접근: 순차적 대용량 읽기에 최적화 / 임의 접근(Random Access) 성능 낮음
  • 작은 파일 문제(Small Files Problem): 작은 파일 다수 = NameNode 메모리 부하↑ → 합쳐서 저장 권장
  • 적합: 로그 분석·배치 처리·대용량 ETL / 부적합: 실시간 처리·임의 쓰기
NameNode=메타데이터·SPOF위험·HA필요 / DataNode=실제블록저장·Heartbeat / 블록=128MB·복제계수3·WORM

Map ⭐각 레코드를 읽어 중간 키-값 쌍(Key-Value) 생성 — 병렬 실행입력: (key, value) → 출력: (key, value)*
Combiner로컬 Reduce — Map 출력을 로컬에서 먼저 집계하여 네트워크 트래픽 절감 (선택)Reduce 함수와 동일한 로직
Shuffle & Sort ⭐Map 출력을 키 기준으로 정렬·그루핑하여 Reducer로 전달 — 가장 비용이 큰 단계네트워크 전송 포함
Reduce ⭐같은 키의 값들을 집계·요약하여 최종 결과 생성입력: (key, [v1,v2,...]) → 출력: (key, value)
Word Count 예제 — MapReduce 동작 이해
▾
# 입력 텍스트: "hello world hello hadoop" Map 단계: ("hello world") → ("hello",1), ("world",1) ("hello hadoop") → ("hello",1), ("hadoop",1) Shuffle & Sort: "hadoop" → [1] "hello" → [1, 1] "world" → [1] Reduce 단계: ("hadoop", [1]) → ("hadoop", 1) ("hello", [1,1]) → ("hello", 2) ("world", [1]) → ("world", 1)
MapReduce: 입력→분할→Map(키-값)→Shuffle&Sort(정렬·그루핑)→Reduce(집계)→출력 / Shuffle이 가장 비용 큰 단계

YARN (Yet Another Resource Negotiator) ⭐

YARN: Hadoop 2.x에서 도입된 클러스터 자원 관리 레이어 — MapReduce 외에 Spark·Hive 등 다양한 처리 엔진을 동일 클러스터에서 실행 가능
구성요소역할
ResourceManager (RM) ⭐클러스터 전체 자원(CPU·메모리) 관리·할당 — 마스터 역할 / Scheduler + ApplicationsManager
NodeManager (NM)각 노드의 자원 모니터링 및 컨테이너 실행 — RM에게 자원 상태 보고
ApplicationMaster (AM)애플리케이션별 실행 관리 — RM에서 자원 요청 / NM에서 컨테이너 실행·모니터링
ContainerCPU·메모리를 할당받은 실행 단위 — Task(Mapper·Reducer)가 실행되는 환경
YARN: ResourceManager(전체자원관리)·NodeManager(노드자원)·ApplicationMaster(앱관리)·Container(실행단위)

Spark 핵심 개념 — RDD · DataFrame · 변환·액션

RDD (Resilient Distributed Dataset) ⭐
▾
  • 정의: Spark의 기본 추상화 — 불변(Immutable)·분산·내결함성을 가진 데이터 컬렉션
  • 불변성: RDD는 변경 불가 → 변환(Transformation)으로 새 RDD 생성
  • 지연 평가(Lazy Evaluation) ⭐: 변환 연산은 즉시 실행되지 않고 액션 호출 시에만 실제 실행 — DAG 최적화
  • 내결함성(Fault Tolerance): 리니지(Lineage) 그래프로 장애 시 해당 파티션만 재계산
  • 파티션: RDD는 여러 파티션으로 분할되어 병렬 처리
연산 종류특징예시
변환(Transformation)새 RDD 반환 / 지연 실행map, filter, flatMap, groupByKey, reduceByKey
액션(Action)결과 반환 / 즉시 실행 트리거collect, count, reduce, saveAsTextFile, take

NoSQL: "Not Only SQL" — 비정형·반정형 빅데이터를 위해 스키마 유연성·수평 확장을 제공하는 비관계형 데이터베이스

분산 시스템은 다음 3가지 중 최대 2가지만 동시 보장 가능

속성의미포기 시
C (Consistency·일관성)모든 노드가 동시에 같은 데이터를 봄AP 시스템 → 일시적 불일치 허용

BASE vs ACID ⭐
▾

ACID (관계형 DB)

  • Atomicity — 원자성 (전부 or 전무)
  • Consistency — 일관성 (규칙 보장)
  • Isolation — 격리성 (독립 실행)
  • Durability — 지속성 (영구 저장)

BASE (NoSQL)

  • BAsically Available — 기본적 가용성
  • Soft state — 유동적 상태 (일시적 불일치)
  • Eventually consistent — 최종 일관성
NoSQL 4종: 키-값(Redis)·문서(MongoDB)·컬럼패밀리(HBase·Cassandra)·그래프(Neo4j) / CAP=3가지 중 2가지만 / BASE=최종일관성

  • Apache Airflow: 워크플로우 오케스트레이션·DAG 스케줄링
  • Spark: 배치·스트리밍 데이터 처리·변환
  • dbt (data build tool): SQL 기반 데이터 변환 (ELT)
  • Apache NiFi: 데이터 흐름 자동화·시각화
  • Debezium: CDC(Change Data Capture) — DB 변경 이벤트 캡처

OLTP vs OLAP ⭐

항목OLTP (트랜잭션)OLAP (분석)
목적일상 트랜잭션 처리 (삽입·수정·삭제)대용량 데이터 분석·집계 쿼리
데이터 최신성실시간 최신 데이터히스토리·스냅샷 데이터
쿼리단순·짧은 쿼리 / 소수 행복잡·긴 쿼리 / 대용량 스캔
정규화높음 (중복 최소화)낮음 (스타·스노우플레이크 스키마)
응답 시간밀리초 단위초~분 단위
대표 시스템MySQL, PostgreSQL, OracleRedshift, Snowflake, BigQuery, Hive
OLTP=트랜잭션·실시간·정규화·ms응답 / OLAP=분석·히스토리·비정규화·대용량스캔 / ETL vs ELT=변환위치 차이
