IoT (Internet of Things) 정의

IoT란 사람·사물·공간 등 모든 것(Things)이 인터넷으로 연결되어 데이터를 수집·교환·분석하고 자율적으로 제어하는 초연결 지능형 인프라

항목내용
핵심 특성초연결(Hyper-connectivity) / 상황인식(Context-Awareness) / 자율제어(Autonomy)
구성 요소디바이스(센서·액추에이터) → 네트워크(게이트웨이·통신) → 플랫폼(클라우드) → 서비스(앱·분석)
핵심 기술센서·임베디드 / 무선통신(LPWAN·WiFi·BLE) / 클라우드·빅데이터 / AI·머신러닝 / 보안
관련 개념M2M(기계 간 통신)의 확장 / CPS(사이버물리시스템) / 산업용=IIoT
표준 기관ITU-T, ISO/IEC JTC1, IETF, OCF, oneM2M, IEEE
IoT = 사물+인터넷+지능 / 디바이스→네트워크→플랫폼→서비스 4계층 구조

IoT 응용 계층 핵심 프로토콜 ⭐

MQTT (Message Queuing Telemetry Transport)
발행-구독 패턴 / TCP / 브로커 중심 / 포트 1883(TLS:8883)
⭐최중요
▾
  • 아키텍처: 발행자(Publisher) → 브로커(Broker) → 구독자(Subscriber) / 발행-구독(Pub-Sub) 패턴
  • 특징: 경량(헤더 최소 2바이트) / 저대역폭·저전력 / 비동기 메시지 전달 / TCP 기반
  • QoS 레벨: 0(최대 1회, At Most Once) / 1(최소 1회, At Least Once) / 2(정확히 1회, Exactly Once)
  • 보안: TLS/SSL 암호화 / 사용자명·비밀번호 인증 / 인증서 기반 인증
  • 용도: 스마트홈·산업용 IoT·원격 센서 모니터링·메시징 앱
  • 브로커 예시: Mosquitto, HiveMQ, AWS IoT Core, Azure IoT Hub
  • MQTT 5.0: 메시지 만료·사용자 속성·공유 구독 등 기능 강화
CoAP (Constrained Application Protocol)
REST 스타일 / UDP 기반 / 초저사양 디바이스 / 포트 5683(DTLS:5684)
⭐핵심
▾
  • 특징: HTTP를 제약된 환경에 맞게 경량화 / UDP 기반(신뢰성은 선택적 확인 메시지) / RESTful(GET·POST·PUT·DELETE)
  • 설계 목표: 마이크로컨트롤러·배터리 기기 등 초저사양 디바이스용 (코드 크기 수 KB)
  • 관찰(Observe): 리소스 변경 시 자동 알림 — 구독 기능과 유사
  • CoAP vs HTTP: UDP 기반(HTTP는 TCP) / 이진 헤더(HTTP는 텍스트) / 멀티캐스트 지원
  • 보안: DTLS (UDP 기반 TLS) 적용 / 인증서·PSK(Pre-Shared Key)
  • 용도: 스마트 에너지·의료기기·센서 네트워크
  • 표준: IETF RFC 7252
AMQP (Advanced Message Queuing Protocol)
엔터프라이즈 메시징 / TCP / 발행-구독+큐 / 포트 5672(TLS:5671)
엔터프라이즈
▾
  • 특징: 엔터프라이즈급 메시지 큐잉 / 신뢰성 높음 / 트랜잭션 지원 / 복잡한 라우팅
  • 구성: Exchange(라우터) → Binding → Queue → Consumer
  • Exchange 유형: Direct·Fanout·Topic·Headers 라우팅 방식
  • MQTT 비교: AMQP=무거움·엔터프라이즈 / MQTT=경량·IoT 특화
  • 용도: 금융 거래·기업 미들웨어·Azure Service Bus / IoT 백엔드
  • 구현체: RabbitMQ, Apache Qpid, Azure Service Bus
DDS (Data Distribution Service)
실시간 발행-구독 / 브로커리스 / 군사·자율주행·산업
실시간
▾
  • 특징: 브로커 없는 분산 발행-구독 / 실시간 데이터 버스 / 23가지 QoS 정책 / P2P 멀티캐스트
  • 설계 목표: 실시간·고신뢰성·고성능 — 군사·의료·자율주행·로봇(ROS2)
  • 표준: OMG 표준 / ROS2의 기반 통신 미들웨어
  • vs MQTT: DDS=브로커리스·실시간·복잡 / MQTT=브로커 중심·경량·IoT

엣지 컴퓨팅 (Edge Computing)

엣지 컴퓨팅: 데이터를 클라우드로 보내지 않고 데이터가 생성되는 곳(엣지) 근처에서 처리·분석하는 분산 컴퓨팅 패러다임
항목내용
등장 배경IoT 디바이스 폭증 → 클라우드 집중 처리 한계 / 실시간성 요구·대역폭 부족·지연 문제 해결
장점저지연(Low Latency) / 대역폭 절약 / 오프라인 동작 가능 / 개인정보 로컬 처리 / 실시간 제어
단점분산 관리 복잡성 / 엣지 노드 보안 취약 / 스케일아웃 제한 / 중앙화된 분석 어려움
적용 사례자율주행(ms 단위 반응) / 스마트 공장 실시간 제어 / 의료 현장 즉시 분석 / AR/VR
핵심 표준ETSI MEC (Multi-access Edge Computing) / OpenFog / Linux Foundation LF Edge
엣지 컴퓨팅 = 데이터 발생 지점 근처에서 처리 / 저지연·대역폭 절약·오프라인 동작이 핵심 장점

IoT 게이트웨이 (IoT Gateway)

게이트웨이 역할

  • 프로토콜 변환: Zigbee·BLE → MQTT·HTTP 변환
  • 데이터 집계·필터링: 로컬 전처리로 데이터 축소
  • 보안 엔드포인트: TLS 종단·인증·방화벽
  • 오프라인 버퍼링: 네트워크 단절 시 데이터 임시 저장
  • 엣지 처리: 로컬 규칙·알람 처리

디지털 트윈 (Digital Twin)

  • 정의: 물리적 객체·시스템의 가상 복제본 — 실시간 동기화·시뮬레이션
  • 구성: 물리 객체 ↔ 데이터 연결 ↔ 디지털 모델
  • 용도: 예지 정비·시뮬레이션·최적화·설계 검증
  • 플랫폼: Azure Digital Twins, AWS IoT TwinMaker
  • 표준: ISO 23247 (스마트 제조 디지털 트윈)
게이트웨이=프로토콜변환+데이터필터링+보안+버퍼링 / 디지털 트윈=물리 객체의 가상 복제+실시간 동기화

IoT 보안 대응 방안

디바이스 보안

  • 기본 자격증명 변경 강제화 (출고 시 설정)
  • 보안 부팅(Secure Boot): 서명된 펌웨어만 실행
  • OTA(Over-The-Air) 업데이트: 암호화·서명된 펌웨어
  • 하드웨어 보안 모듈(HSM·TPM): 키 저장·암호화
  • 최소 기능 원칙: 불필요한 포트·서비스 비활성화
  • 물리적 보호: 디버그 포트(JTAG·UART) 비활성화

네트워크·플랫폼 보안

  • TLS/DTLS: 모든 통신 암호화
  • IoT VLAN 분리: IoT 기기를 일반 네트워크와 격리
  • 디바이스 인증: X.509 인증서 기반 PKI
  • Zero Trust: 모든 디바이스를 검증 후 최소 권한
  • 이상 탐지(Anomaly Detection): AI 기반 비정상 행동 탐지
  • API 보안: OAuth 2.0·API 키·Rate Limiting
IoT 보안: 기본자격증명변경+보안부팅+TLS+VLAN분리+제로트러스트 / OTA 업데이트 서명 필수
