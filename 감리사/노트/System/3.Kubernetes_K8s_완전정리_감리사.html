K8s 핵심 오브젝트 계층

Pod ⭐ — 최소 배포 단위 K8s 원자 단위
▾
  • 정의: K8s에서 배포·관리하는 가장 작은 단위. 1개 이상의 컨테이너를 포함
  • 공유 자원: 같은 Pod 내 컨테이너는 네트워크 네임스페이스(localhost 통신)와 볼륨(Volume)을 공유
  • 임시성: Pod는 일회성 — 삭제·재시작 시 IP 변경 / 영속적 관리는 Deployment 사용
  • 사이드카 패턴: 주 컨테이너 + 보조 컨테이너(로그 수집·프록시·설정 주입) 함께 실행
  • Pod 상태: Pending → Running → Succeeded/Failed/Unknown
# Pod YAML 예시 apiVersion: v1 kind: Pod metadata: name: my-pod spec: containers: - name: nginx image: nginx:1.25 ports: - containerPort: 80
Deployment ⭐ — Pod 복제·배포 관리 가장 중요
▾
  • 역할: 원하는 수(Replica)의 Pod 유지 / 롤링 업데이트 / 롤백 / 자가 치유
  • 자가 치유: Pod 장애 시 Controller Manager가 새 Pod를 자동 생성
  • ReplicaSet: Deployment가 내부적으로 ReplicaSet을 생성하여 Pod 수 관리 (직접 생성 비권장)
  • 롤링 업데이트: maxSurge(추가 허용 Pod 수)·maxUnavailable(최대 불가 Pod 수) 설정으로 무중단 배포
  • 롤백: kubectl rollout undo deployment/my-app 명령으로 이전 버전으로 즉시 복구
spec: replicas: 3 # Pod 복제본 3개 유지 strategy: type: RollingUpdate rollingUpdate: maxSurge: 1 # 동시에 최대 1개 추가 maxUnavailable: 0 # 동시에 0개 불가 (무중단)
StatefulSet — 상태를 가진 앱 관리 DB·메시지큐
▾
  • vs Deployment: 각 Pod에 고정 이름(stable hostname)과 순서가 보장됨 — pod-0, pod-1, pod-2
  • PVC 연결: 각 Pod가 독립적인 PersistentVolumeClaim을 가짐 → Pod 재시작 후도 같은 데이터
  • 순서 보장: 순서대로 시작·종료 (pod-0 → pod-1 → pod-2) / 역순 종료
  • 용도: MySQL·PostgreSQL·Kafka·ZooKeeper·Elasticsearch 등 상태 있는 앱
DaemonSet · Job · CronJob 특수 워크로드
▾
종류설명용도
DaemonSet모든 노드(또는 선택 노드)에 정확히 1개씩 Pod 실행 / 노드 추가 시 자동 배포로그 수집(Fluentd), 모니터링(Prometheus Node Exporter), 네트워크 플러그인
Job완료 후 종료되는 일회성 작업 / 지정 횟수 성공 시 완료배치 처리, 데이터 마이그레이션, 백업
CronJobJob을 스케줄(cron 표현식)로 반복 실행정기 보고서 생성, 정기 백업, 캐시 워밍업
Pod=최소단위(1개↑컨테이너) / Deployment=복제·롤링업데이트·자가치유 / StatefulSet=고정이름·순서·상태앱 / DaemonSet=모든노드1개씩

Service — Pod에 안정된 네트워크 엔드포인트 ⭐

Service: Pod의 IP는 재시작 시 변경되므로, Service가 안정된 DNS 이름과 IP(ClusterIP)로 Pod 집합에 접근할 수 있게 해줌. kube-proxy가 실제 라우팅 처리.
타입접근 범위용도비고
ClusterIP ⭐클러스터 내부만서비스 간 내부 통신기본값 / 외부 접근 불가
NodePort노드 IP + 지정 포트개발·테스트 외부 접근30000~32767 포트 범위
LoadBalancer ⭐외부 인터넷운영 환경 외부 노출클라우드 제공 LB 프로비저닝
ExternalName외부 DNS 이름외부 서비스 내부 이름으로 접근CNAME 레코드 반환
Service 4종: ClusterIP(내부기본) / NodePort(노드포트) / LoadBalancer(클라우드LB·운영) / ExternalName(CNAME)
🚪

Ingress — 외부 HTTP/HTTPS 라우팅 ⭐

Ingress: 외부 HTTP/HTTPS 요청을 URL 경로·호스트 기반으로 내부 Service로 라우팅하는 규칙 집합. 실제 처리는 Ingress Controller(nginx·traefik 등)가 담당
# Ingress 라우팅 예시 http://myapp.com/api → api-service:8080 http://myapp.com/web → web-service:80 http://admin.myapp.com → admin-service:9090 # TLS 종단 처리 가능 (인증서 관리)
  • vs LoadBalancer: Ingress=여러 서비스를 하나의 진입점으로 통합(L7) / LoadBalancer=서비스마다 별도 LB(L4) → 비용 절감
  • Ingress Controller: nginx Ingress Controller, AWS ALB Controller, Traefik, Istio Gateway
  • TLS/SSL 처리: Secret에 TLS 인증서 저장 후 Ingress에서 참조 → HTTPS 종단
Ingress = L7 HTTP/HTTPS 라우팅 / URL·호스트 기반 / 여러 Service를 단일 진입점으로 통합 / Ingress Controller 필요

K8s 네트워킹 모델 & CNI

K8s 네트워킹 4원칙

  • Pod는 NAT 없이 다른 Pod와 직접 통신 가능
  • 노드는 NAT 없이 모든 Pod와 통신 가능
  • Pod는 자신의 IP를 인식 (= 외부에서 보이는 IP)
  • 각 Pod는 고유한 IP 보유

CNI (Container Network Interface)

  • Calico: BGP 기반 / 네트워크 정책 강력 지원
  • Flannel: 단순 오버레이 네트워크 / 소규모
  • Weave: 암호화 오버레이 네트워크
  • Cilium: eBPF 기반 고성능 / 보안 정책 우수
K8s 네트워크: 각 Pod는 고유 IP·NAT 없이 직접 통신 / CNI 플러그인이 구현 / Calico·Cilium이 보안정책 지원

K8s 스토리지 — PV · PVC · StorageClass ⭐

개념역할핵심 포인트
VolumePod 내 컨테이너가 공유하는 저장소 / Pod 삭제 시 함께 삭제emptyDir(임시), hostPath(노드디스크), configMap, secret
PersistentVolume (PV) ⭐관리자가 사전에 프로비저닝한 클러스터 스토리지 리소스Pod 생명주기와 독립 / NFS·EBS·Ceph 등 백엔드
PersistentVolumeClaim (PVC) ⭐사용자가 PV에 대한 스토리지 요청 — 용량·접근 모드 지정PVC → PV 바인딩 / Pod에서 PVC 참조
StorageClass ⭐PV를 동적으로 프로비저닝하는 템플릿 — 스토리지 유형 정의Dynamic Provisioning / AWS gp3, Azure Premium SSD 등
접근 모드ReadWriteOnce(RWO) / ReadOnlyMany(ROX) / ReadWriteMany(RWX)RWX: 여러 Pod에서 동시 읽기·쓰기 — NFS 필요
# PVC 예시 — 10Gi 스토리지 요청 kind: PersistentVolumeClaim spec: accessModes: [ReadWriteOnce] storageClassName: gp3 # StorageClass 참조 resources: requests: storage: 10Gi
PV=관리자가 스토리지 준비 / PVC=사용자가 스토리지 요청 / StorageClass=동적 프로비저닝 / RWX=여러Pod 동시쓰기(NFS)

ConfigMap & Secret — 설정·민감정보 분리 ⭐

ConfigMap — 일반 설정

  • 환경변수·설정 파일을 컨테이너 이미지와 분리
  • 키-값 쌍 또는 파일 형태로 저장
  • Pod에서 환경변수·볼륨 마운트로 사용
  • 예: DB 호스트명, API 엔드포인트, 설정값
  • 데이터는 평문 저장 — 민감 정보 금지

Secret — 민감 정보 ⭐

  • 비밀번호·API 키·TLS 인증서 등 민감 정보 저장
  • Base64 인코딩 저장 (암호화 아님 — etcd 암호화 별도)
  • Pod에서 환경변수·볼륨·imagePullSecrets으로 사용
  • Secret 타입: Opaque(범용), kubernetes.io/tls, kubernetes.io/dockerconfigjson
  • RBAC + etcd 암호화로 보안 강화 필수
ConfigMap=일반설정·평문 / Secret=민감정보·Base64(암호화아님) / 둘 다 컨테이너 이미지와 설정 분리가 핵심 목적

Pod 스케줄링 제어 — Affinity · Taint · Toleration

개념설명
Node Selector특정 레이블이 있는 노드에만 Pod 배치 (단순 방식)
Node AffinityNode Selector의 확장 — required(필수)/preferred(선호) 조건으로 노드 선택
Pod Affinity특정 Pod와 같은 노드(또는 근처)에 배치 — 성능 최적화
Pod Anti-Affinity특정 Pod와 다른 노드에 배치 — 고가용성(같은 앱의 Pod를 여러 노드에 분산)
Taint노드에 "이 노드는 일반 Pod 배치 거부" 표시 — kubectl taint nodes
TolerationPod가 Taint를 "용인"하여 해당 노드에 배치 가능하게 함
ResourceQuotaNamespace 내 리소스(CPU·메모리·Pod 수) 사용 한도 설정
LimitRangeNamespace 내 Pod·컨테이너의 기본 requests·limits 설정
Affinity=원하는 노드에 배치 / Anti-Affinity=다른 노드에 분산(고가용성) / Taint·Toleration=노드 격리·허용

Namespace & RBAC 구조

Namespace — 논리적 분리

  • 클러스터 내 논리적 격리 공간 — 리소스·정책·쿼터 분리
  • default 네임스페이스: 명시 없으면 기본 사용
  • kube-system: K8s 핵심 컴포넌트 (DNS·프록시)
  • kube-public: 클러스터 공개 정보
  • 개발(dev)·스테이징(staging)·운영(prod) 환경 분리에 활용

RBAC 구성 요소

  • Subject: User·Group·ServiceAccount
  • Role: Namespace 범위 권한 정의 (verbs: get·list·create·delete)
  • ClusterRole: 클러스터 전체 범위 권한
  • RoleBinding: Subject ↔ Role 연결 (Namespace 내)
  • ClusterRoleBinding: Subject ↔ ClusterRole 연결 (전체)
# Role: dev 네임스페이스에서 Pod 조회만 rules: - apiGroups: [""] resources: [pods] verbs: [get, list]
