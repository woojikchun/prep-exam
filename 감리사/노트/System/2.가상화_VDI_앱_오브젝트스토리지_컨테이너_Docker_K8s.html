VDI (Virtual Desktop Infrastructure)

VDI: 서버 VM에서 데스크톱 OS를 구동하고, 사용자는 씬 클라이언트로 네트워크를 통해 접속하는 데스크톱 가상화
구성요소역할대표 제품
하이퍼바이저VM 생성·관리VMware ESXi, Hyper-V
연결 브로커 ⭐인증 + VM 할당 + 부하 분산VMware Horizon CS, Citrix DC
씬 클라이언트사용자 단말 (저사양 가능)HP ThinPro, 일반 PC·모바일
원격 디스플레이 ⭐화면·입력 전송 프로토콜RDP(MS) / PCoIP(VMware) / ICA·HDX(Citrix)

영구(Persistent) VDI

  • 사용자별 전용 VM
  • 로그아웃 후 설정·데이터 유지
  • 스토리지 소비↑ / 관리 복잡
  • 적합: 개발자·디자이너

비영구(Non-Persistent) VDI ⭐

  • 골든 이미지에서 복제
  • 로그아웃 시 초기 상태로 리셋
  • 스토리지 절약 / 보안 강화
  • 적합: 콜센터·창구 업무
VDI 4요소: 하이퍼바이저+연결브로커+씬클라이언트+원격디스플레이 / RDP=MS / PCoIP=VMware / ICA=Citrix
⚡

하이퍼바이저 Type1 vs Type2 ⭐

Type 1 — 베어메탈

VM1 | VM2 | VM3 (Guest OS)
하이퍼바이저 (Type 1)
하드웨어
  • 하드웨어에 직접 설치 — Host OS 없음
  • 성능 우수 / 서버·VDI 환경
  • VMware ESXi, MS Hyper-V, KVM, Xen

Type 2 — 호스트형

VM1 | VM2 (Guest OS)
하이퍼바이저 (Type 2)
Host OS (Windows·Linux·macOS)
하드웨어
  • Host OS 위에 앱처럼 설치
  • 설치 쉬움 / 개발·테스트 환경
  • VMware Workstation, VirtualBox, Parallels
Type1=베어메탈(ESXi·Hyper-V·KVM) 서버용 / Type2=호스트형(VirtualBox·Workstation) 개발용

애플리케이션 가상화 — 3가지 방식

애플리케이션 가상화: 앱을 OS에 직접 설치하지 않고 격리 환경에서 실행하거나 서버에서 실행해 화면만 전송
① 패키징·스트리밍 — 로컬 격리 실행 오프라인 OK
▾
  • 원리: 앱+의존성(레지스트리·DLL)을 하나의 가상화 패키지로 묶어 로컬 격리 레이어(버블)에서 실행 — OS에 설치 없음
  • 장점: 오프라인 실행 / OS 오염 없음 / 앱 충돌 방지 / 빠른 배포·롤백
  • 대표: Microsoft App-V, VMware ThinApp, MSIX 패키지(최신 표준)
② 서버 기반 — 서버 실행 + 화면 전송 중앙관리
▾
  • 원리: 서버에서 앱 실행 → 화면·입력만 원격 프로토콜로 전송 (RDP·ICA·HDX)
  • 장점: 클라이언트 사양 불문 / 중앙 관리 / 라이선스 최적화
  • 대표: Citrix Virtual Apps(구 XenApp), MS RDS, AWS AppStream 2.0
  • VDI와 차이: 앱만 제공(공유 서버) vs VDI=전체 OS 전용 VM
③ 웹·SaaS — 브라우저 기반 설치불필요
▾
  • 브라우저만 있으면 접근 가능 — HTML5·WebAssembly 기반
  • 대표: Microsoft 365, Google Workspace, AWS AppStream
  • Remote Browser Isolation(RBI): 브라우저 자체를 서버에서 격리 실행 → 악성코드 차단
방식실행 위치오프라인대표
패키징·스트리밍클라이언트 로컬✅ 가능Microsoft App-V, ThinApp
서버 기반서버불가Citrix Virtual Apps, RDS
웹·SaaS클라우드불가Microsoft 365, AppStream
App-V=로컬격리·오프라인OK / Citrix Virtual Apps=서버실행·화면전송 / VDI=전체OS / 앱가상화=앱만

오브젝트 스토리지 (Object Storage)

오브젝트 스토리지: 각 오브젝트 = 데이터 + 메타데이터 + 고유 ID(Key) / REST API (S3)로 접근 / 무한 수평 확장
항목내용
오브젝트 구조데이터(payload) + 메타데이터(키-값, 무제한) + 고유 ID(Key/URL)
접근 방식 ⭐REST API (HTTP GET·PUT·DELETE) — AWS S3 API 사실상 표준
계층 구조버킷(Bucket) > 오브젝트 / 폴더 계층 없음 (플랫 네임스페이스)
수정 방식 ⭐전체 교체(덮어쓰기)만 가능 — 부분 수정 불가 / WORM 지원
내구성복제·이레이저코딩 → 99.999999999% (11 Nine)
주요 솔루션AWS S3 / Azure Blob Storage / Google Cloud Storage / MinIO·Ceph(오픈소스)
주요 활용미디어·이미지 / AI ML 학습 데이터 / 백업·아카이브 / 정적 웹 / 데이터 레이크
오브젝트=데이터+메타데이터+Key / REST API(S3) / 수정=전체교체만 / 11Nine 내구성 / 무한수평확장

Kubernetes (K8s) — 컨테이너 오케스트레이션

Kubernetes: 컨테이너의 배포·확장·자가치유·부하분산을 자동화하는 오픈소스 컨테이너 오케스트레이션 플랫폼 (Google 설계 → CNCF)

핵심 오브젝트

객체역할
Pod ⭐최소 배포 단위 / 1개↑ 컨테이너 / 네트워크·스토리지 공유
Deployment ⭐Pod 복제본 관리 / 롤링업데이트·롤백·자가치유
ServicePod에 안정된 네트워크 엔드포인트 / LB·ClusterIP·NodePort
Ingress외부 HTTP/HTTPS → Service 라우팅 / URL 경로 규칙
ConfigMap·Secret설정·자격증명을 코드와 분리 관리
HPA ⭐CPU·메모리 기반 Pod 수 자동 수평 확장
PersistentVolumePod 외부 영구 스토리지 / PVC로 요청
Namespace클러스터 내 논리적 분리 (개발·운영 환경 분리)

K8s 핵심 기능 — 롤링업데이트 · 자가치유 · 스케일링

기능설명
롤링 업데이트Deployment에서 Pod를 순차적으로 교체 — 서비스 중단 없이 배포 / Rollback 지원
자가 치유 (Self-Healing)Pod 장애 시 자동 재시작 / 노드 장애 시 다른 노드로 Pod 이동
수평 확장 (HPA)CPU·메모리 임계치 초과 → Pod 자동 증가(Scale-out) / 감소(Scale-in)
수직 확장 (VPA)Pod의 CPU·메모리 리소스 자동 조정 (Pod 재시작 필요)
서비스 디스커버리Service 객체가 Pod IP 변경에도 안정된 DNS 이름 제공
설정 분리ConfigMap(일반 설정) / Secret(민감 정보) — 컨테이너 이미지와 분리
멀티 클라우드AWS EKS / Azure AKS / GCP GKE — 클라우드 관리형 K8s
HPA=수평(Pod수 조정)·VPA=수직(리소스조정) / 자가치유=Pod장애→자동재시작 / etcd=클러스터상태저장 핵심
