트랜잭션 (Transaction) 정의 & ACID ⭐

트랜잭션: 데이터베이스에서 하나의 논리적 작업 단위를 구성하는 일련의 연산 집합 — 전부 실행(COMMIT)되거나 전혀 실행되지 않아야(ROLLBACK) 함
속성의미보장 메커니즘위반 시 문제
A — 원자성 (Atomicity) ⭐트랜잭션의 모든 연산은 전부 실행되거나 전혀 실행되지 않음UNDO 로그 / ROLLBACK부분 실행 → 불일치
C — 일관성 (Consistency) ⭐트랜잭션 전후에 DB가 일관된 상태를 유지무결성 제약조건 검사비즈니스 규칙 위반
I — 격리성 (Isolation) ⭐동시 실행되는 트랜잭션은 서로 간섭하지 않음Lock / MVCC / 격리 수준Dirty Read 등 이상 현상
D — 지속성 (Durability) ⭐커밋된 트랜잭션은 장애 후에도 영구 반영됨REDO 로그 / WAL장애 후 데이터 소실
ACID: Atomicity(원자성·UNDO)·Consistency(일관성·제약조건)·Isolation(격리성·Lock)·Durability(지속성·REDO)

즉시 갱신 vs 지연 갱신 ⭐

즉시 갱신 (Immediate Update) ⭐

  • 정의: 트랜잭션 수행 중 변경사항을 즉시 디스크에 반영 (COMMIT 전에도)
  • 로그: Before Image(UNDO용) + After Image(REDO용) 모두 필요
  • 복구: REDO + UNDO 모두 필요
  • 장애 시:
    - COMMIT 전 장애 → UNDO (이전값으로 되돌림)
    - COMMIT 후 장애 → REDO (이후값으로 재적용)
  • 성능: 작업 중 디스크 I/O 발생

지연 갱신 (Deferred Update) ⭐

  • 정의: COMMIT 시점까지 변경사항을 로그에만 기록하고 디스크에는 COMMIT 후 반영
  • 로그: After Image(REDO용)만 필요 — Before 불필요
  • 복구: REDO만 필요 (UNDO 불필요)
  • 장애 시:
    - COMMIT 전 장애 → 그냥 로그 버림 (디스크 미반영)
    - COMMIT 후 장애 → REDO (로그로 재적용)
  • 성능: COMMIT 시 한꺼번에 I/O
UNDO · REDO 동작 상세 ⭐
▾
연산목적방향사용 데이터적용 대상 트랜잭션
UNDO ⭐미완료 트랜잭션 취소 — 원자성 보장로그를 역순으로Before Image (이전값)장애 시 COMMIT되지 않은 트랜잭션
REDO ⭐완료 트랜잭션 재실행 — 지속성 보장로그를 순방향으로After Image (이후값)장애 전 COMMIT된 트랜잭션
-- 시스템 장애 복구 시나리오
T1: START → 작업 → COMMIT ← REDO 필요 (커밋됐지만 디스크 미반영 가능)
T2: START → 작업 → (장애) ← UNDO 필요 (커밋 안 됨 → 원상복구)
T3: START → (장애) ← UNDO 필요 (커밋 안 됨)
즉시갱신=디스크즉시반영·UNDO+REDO / 지연갱신=COMMIT후반영·REDO만 / UNDO=미완료취소·역순 / REDO=완료재실행·순방향

체크포인트 (Checkpoint) ⭐

체크포인트: 주기적으로 버퍼의 변경 내용을 디스크에 강제 기록하고 로그에 체크포인트 레코드를 남기는 기법 — 복구 시 로그 탐색 범위를 줄여 복구 시간 단축
체크포인트 동작 & 복구 결정 로직 ⭐
▾
시간축 ──────────────────────────────────────────────►
[CKPT] [FAILURE]
T1: ───────────────commit───────────────── → REDO 불필요 (CKPT때 이미 디스크 반영)
T2: ──────────────────commit───── → REDO 필요 (CKPT후 COMMIT, 디스크 미반영 가능)
T3: ───────────────────── → UNDO 필요 (COMMIT 안 됨)
T4: ───────────── → UNDO 필요 (COMMIT 안 됨, 무시가능)

체크포인트 기준 복구 결정 규칙

트랜잭션 상황복구 연산이유
CKPT 전에 COMMIT없음CKPT 시 이미 디스크 반영됨
CKPT 후에 COMMIT (장애 전)REDOCOMMIT됐지만 디스크 미반영 가능
COMMIT 없이 장애 (CKPT 전후 무관)UNDO미완료 트랜잭션 → 원상복구
체크포인트 종류 ⭐
▾
종류특징장점단점
Consistent (Sharp) Checkpoint체크포인트 시 모든 트랜잭션 일시 중단 → 버퍼 전체 플러시복구 단순서비스 중단 시간 발생
Fuzzy Checkpoint ⭐트랜잭션 중단 없이 점진적 플러시 / 시작·끝 LSN 기록서비스 무중단복구 로직 복잡
  • Fuzzy Checkpoint 처리: CKPT_BEGIN · CKPT_END 두 레코드를 로그에 기록하여 체크포인트 범위 표시
  • ARIES의 Fuzzy Checkpoint: 진행 중인 트랜잭션 목록(ATT)과 더티 페이지 목록(DPT)을 로그에 기록
LSN (Log Sequence Number) & 로그 관리 ⭐
▾
  • LSN: 각 로그 레코드에 부여되는 단조증가 고유 번호 — 복구 시 순서·위치 식별
  • Page LSN: 페이지에 마지막으로 적용된 로그의 LSN — 해당 페이지가 최신인지 확인
  • Flushed LSN: 로그 버퍼에서 디스크로 플러시된 마지막 LSN
  • WAL 강제 규칙: 데이터 페이지를 디스크에 쓰기 전 Page LSN 이하의 모든 로그를 먼저 플러시
  • COMMIT 규칙: COMMIT 로그가 안정 저장소에 기록된 후에만 COMMIT 완료 처리
체크포인트=주기적 버퍼→디스크 강제기록·복구범위축소 / CKPT전COMMIT=복구불필요 / CKPT후COMMIT=REDO / 미COMMIT=UNDO

그림자 페이지 (Shadow Paging) ⭐

Shadow Paging: 트랜잭션이 페이지를 수정할 때 원본 페이지는 그대로 두고 새로운 페이지(Shadow)에 수정된 내용을 기록. COMMIT 시 포인터를 Shadow로 전환
Shadow Paging 동작 원리 ⭐
▾
[ 트랜잭션 시작 ]
현재 디렉터리 (Current Page Table): Page1 → Block_A
그림자 디렉터리 (Shadow Page Table): Page1 → Block_A (복사본 유지)

[ 페이지 수정 ]
현재 디렉터리: Page1 → Block_B (새 블록에 수정 기록)
그림자 디렉터리: Page1 → Block_A (원본 유지)

[ COMMIT ] : 그림자를 현재로 교체 → Block_A 해제
[ ROLLBACK ]: 현재를 그림자로 교체 → Block_B 해제 (즉시 복구)

Shadow Paging 장점

  • 로그 불필요: 별도의 UNDO 로그 없이 복구 가능
  • 빠른 복구: ROLLBACK 시 포인터만 교체 → 즉시 복구
  • 단순성: REDO도 불필요 (Shadow가 이미 안전)

Shadow Paging 단점

  • 저장 공간 낭비: 페이지 2개 유지 — 원본+Shadow
  • 단편화 발생: 포인터 교체로 연속 블록 배치 불가
  • 동시성 제한: 여러 트랜잭션 동시 처리 어려움
  • 가비지 컬렉션: 구 페이지 청소 필요
Shadow Paging: 원본 유지+Shadow에 수정 / COMMIT=Shadow→현재 포인터 교체 / ROLLBACK=즉시 복구 / 단점: 공간낭비·단편화

미디어 복구 (Media Recovery)

  • 대상: 디스크 물리 손상으로 데이터 파일 자체 손실된 경우
  • 방법: ①최근 풀 백업 복원 → ②아카이브 로그 REDO → ③온라인 로그 REDO
  • 아카이브 로그: 오래된 로그 파일을 별도 보관 — 미디어 복구에 필수
  • Point-in-Time Recovery (PITR): 특정 시점으로 복구 — 실수 INSERT·DELETE 복구
  • 복제(Replication): Primary → Standby 실시간 복제로 빠른 페일오버
미디어복구: 풀백업복원→아카이브로그REDO→온라인로그REDO / PITR=특정시점복구 / 아카이브로그=미디어복구 필수
🔬

ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) ⭐

ARIES: IBM이 개발한 WAL + Fuzzy Checkpoint + LSN 기반의 실용적 복구 알고리즘 — 현재 대부분의 상업용 DBMS의 복구 표준
