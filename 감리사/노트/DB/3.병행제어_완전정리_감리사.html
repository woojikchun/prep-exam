병행제어 (Concurrency Control) 정의 ⭐

병행제어: 다수의 트랜잭션이 동시에 실행될 때 데이터 일관성과 무결성을 유지하기 위한 메커니즘 — ACID의 격리성(Isolation) 보장

병행 처리의 필요성

  • 처리량 향상: 여러 트랜잭션 동시 실행 → CPU·I/O 효율↑
  • 응답 시간 단축: 대기 없이 여러 요청 동시 처리
  • 자원 활용 극대화: 유휴 자원 최소화

병행 처리 시 문제점

  • 갱신 분실 (Lost Update)
  • 오손 읽기 (Dirty Read)
  • 반복 불가 읽기 (Non-Repeatable Read)
  • 유령 읽기 (Phantom Read)
병행제어=격리성 보장 / 목적: 처리량↑·응답↓·일관성 유지
⚠️

병행 처리 이상 현상 4가지 ⭐

① 갱신 분실 (Lost Update) ⭐ — 가장 심각
▾
  • 정의: 두 트랜잭션이 같은 데이터를 동시에 수정할 때, 한 트랜잭션의 갱신이 다른 트랜잭션에 의해 덮어쓰여 사라지는 현상
시간 T1 T2 t1 X 읽기 (X=10) t2 X 읽기 (X=10) t3 X에 +5 → X=15 t4 X에 +3 → X=13 ← T1 갱신 분실! 결과: X=13 (올바른 값은 10+5+3=18)
② 오손 읽기 (Dirty Read) ⭐
▾
  • 정의: 한 트랜잭션이 아직 COMMIT되지 않은 다른 트랜잭션의 변경 데이터를 읽는 현상 — 롤백 시 잘못된 데이터 사용
시간 T1 T2 t1 X = 100 → X = 200 t2 X 읽기 (X=200) ← 미완료 데이터! t3 ROLLBACK (X=100 복귀) T2는 이미 X=200을 기반으로 연산 → 잘못된 결과
READ COMMITTED 이상의 격리 수준으로 방지
③ 반복 불가 읽기 (Non-Repeatable Read) ⭐
▾
  • 정의: 한 트랜잭션 내에서 같은 데이터를 두 번 읽었을 때 다른 값이 나오는 현상 — 중간에 다른 트랜잭션이 UPDATE·DELETE
시간 T1 T2 t1 X 읽기 (X=10) t2 X = 20, COMMIT t3 X 읽기 (X=20) ← 같은 트랜잭션인데 값이 다름!
REPEATABLE READ 이상의 격리 수준으로 방지
④ 유령 읽기 (Phantom Read)
▾
  • 정의: 한 트랜잭션 내에서 같은 범위 쿼리를 두 번 실행했을 때 행의 수가 다른 현상 — 중간에 다른 트랜잭션이 INSERT
시간 T1 T2 t1 SELECT WHERE age>20 → 3행 t2 INSERT age=25, COMMIT t3 SELECT WHERE age>20 → 4행 ← 유령 행 출현!
SERIALIZABLE 격리 수준으로만 완전 방지
격리 수준Dirty ReadNon-Repeatable ReadPhantom Read

Lock (락) 기반 병행제어 ⭐

Lock: 트랜잭션이 데이터에 접근하기 전 락을 획득하고, 사용 후 해제함으로써 다른 트랜잭션의 동시 접근을 제어하는 기본 메커니즘
락 종류 — 공유락 vs 배타락 ⭐
▾

공유 락 (Shared Lock, S-Lock) ⭐

  • 목적: 읽기 전용 (SELECT)
  • 공존: 여러 트랜잭션이 동시에 S-Lock 보유 가능
  • 차단: X-Lock 요청은 차단
  • 표기: lock-S(X) 또는 S(X)

배타 락 (Exclusive Lock, X-Lock) ⭐

  • 목적: 읽기 + 쓰기 (SELECT·UPDATE·DELETE·INSERT)
  • 공존: 다른 어떤 락과도 공존 불가
  • 차단: S-Lock, X-Lock 모두 차단
  • 표기: lock-X(X) 또는 X(X)

다중 세분화 락 (MGL — Multi-Granularity Locking) & 의도 락
▾
  • 목적: 다양한 락 단위를 계층적으로 관리 — 상위 객체에 의도 락을 설정하여 하위 객체의 락 확인 비용 절감
  • IS (Intent Shared): 하위에 S-Lock 설정 예정을 상위에 표시
  • IX (Intent Exclusive): 하위에 X-Lock 설정 예정을 상위에 표시
  • SIX (Shared + Intent Exclusive): 현재 객체는 S-Lock, 하위 일부는 X-Lock
  • 규칙: 하위 객체 락 설정 전 반드시 상위 객체에 의도 락 획득

직렬 가능성 (Serializability) ⭐

직렬 스케줄 (Serial Schedule): 트랜잭션이 하나씩 순차적으로 실행되는 스케줄 — 항상 올바른 결과 보장
직렬 가능 스케줄: 병행 실행 결과가 어떤 직렬 스케줄의 결과와 동등한 스케줄 — 병행 처리하면서도 올바른 결과 보장
충돌 직렬 가능성 (Conflict Serializability) ⭐
▾
  • 충돌 연산: 서로 다른 트랜잭션이 같은 데이터에 접근하고 그 중 하나 이상이 쓰기 — 순서 변경 시 결과가 달라짐
  • 충돌 직렬 가능성 판별: 우선순위 그래프(Precedence Graph)로 사이클 여부 확인
  • 사이클 없음: 충돌 직렬 가능 → 올바른 스케줄
  • 사이클 있음: 충돌 직렬 불가능 → 이상 현상 가능
-- 우선순위 그래프 규칙 Ti → Tj: Ti 연산이 Tj 연산보다 먼저 실행되고 충돌하는 경우 사이클 없음 → 직렬 가능 (올바름) 사이클 있음 → 직렬 불가능 (문제 있음)

2단계 락킹 프로토콜 (2PL — Two-Phase Locking) ⭐

2PL: 트랜잭션의 락 획득과 해제를 두 단계로 분리하는 프로토콜 — 충돌 직렬 가능성을 보장하는 가장 중요한 기법
2PL 두 단계 ⭐
▾

① 확장 단계 (Growing Phase)

  • 락을 획득만 가능 — 해제 불가
  • 필요한 모든 락을 이 단계에서 획득
  • 락 포인트(Lock Point)에서 최대 보유

② 수축 단계 (Shrinking Phase)

  • 락을 해제만 가능 — 새 획득 불가
  • 첫 번째 락 해제 이후 단계 진입
  • 보통 COMMIT·ROLLBACK 시 모두 해제
-- 2PL 올바른 예시 lock-S(A) ← 확장 단계 lock-X(B) ← 확장 단계 Read(A), Write(B) unlock(A) ← 수축 단계 시작 (이후 새 락 획득 불가) unlock(B) ← 수축 단계 -- 2PL 위반 예시 (직렬 가능성 보장 안 됨) lock-S(A) → Read(A) → unlock(A) → lock-X(B) → Write(B) ← unlock 후 새 락 획득 = 2PL 위반!
2PL은 충돌 직렬 가능성을 보장하지만 교착상태(Deadlock)는 방지하지 못함!
2PL 변형 프로토콜 ⭐
▾
프로토콜특징교착상태Cascading Rollback
기본 2PL락 포인트 후 해제 가능발생 가능발생 가능
엄격 2PL (Strict 2PL) ⭐X-Lock을 COMMIT·ROLLBACK까지 유지발생 가능방지
강건 2PL (Rigorous 2PL)S-Lock과 X-Lock 모두 COMMIT까지 유지발생 가능방지
보수적 2PL (Conservative 2PL)트랜잭션 시작 전 모든 락 미리 획득방지방지

  • Strict 2PL: X-Lock을 COMMIT까지 유지 → T1 롤백 전 T2가 T1 데이터를 읽을 수 없음 → 연쇄 롤백 방지

타임스탬프 순서 프로토콜 (Timestamp Ordering) ⭐

타임스탬프 프로토콜: 각 트랜잭션에 시작 시간 기반의 타임스탬프를 부여하고, 이 순서에 따라 읽기·쓰기 허용 여부를 결정하는 락 없는(Lock-Free) 병행제어 기법
타임스탬프 프로토콜 규칙 ⭐
▾
  • TS(Ti): 트랜잭션 Ti의 타임스탬프 (시작 시간 또는 순번)
  • R-TS(X): X를 마지막으로 읽은 트랜잭션의 타임스탬프
  • W-TS(X): X를 마지막으로 쓴 트랜잭션의 타임스탬프
연산조건처리
Ti가 X 읽기TS(Ti) < W-TS(X)
(Ti보다 나중 트랜잭션이 이미 씀)		Ti 롤백 (오래된 값 읽기 방지)
Ti가 X 읽기TS(Ti) ≥ W-TS(X)읽기 허용, R-TS(X) = max(R-TS(X), TS(Ti))
Ti가 X 쓰기TS(Ti) < R-TS(X)
(Ti보다 나중 트랜잭션이 이미 읽음)		Ti 롤백
Ti가 X 쓰기TS(Ti) < W-TS(X)
(Ti보다 나중 트랜잭션이 이미 씀)		Ti 롤백 (또는 Thomas Write Rule로 허용)
Ti가 X 쓰기TS(Ti) ≥ W-TS(X) and TS(Ti) ≥ R-TS(X)쓰기 허용, W-TS(X) = TS(Ti)
  • 장점: 교착상태 발생 안 함 (대기 없음) / 락 오버헤드 없음
  • 단점: 잦은 롤백 가능 / 연쇄 롤백 가능 / 기아(Starvation) 위험

MVCC (Multi-Version Concurrency Control) ⭐

MVCC: 데이터의 여러 버전을 유지하여 읽기 트랜잭션이 쓰기를 차단하지 않고, 쓰기 트랜잭션이 읽기를 차단하지 않도록 하는 병행제어 기법 — 읽기/쓰기 차단 없음!
MVCC 동작 원리 ⭐
▾
  • 버전 관리: 데이터 수정 시 기존 버전을 유지하고 새 버전 생성 (버전 체인)
  • 읽기 트랜잭션: 자신의 시작 시점에 COMMIT된 가장 최신 버전을 읽음 → 읽기 작업에 S-Lock 불필요!
  • 쓰기 트랜잭션: 새 버전 생성 → X-Lock 필요 (쓰기 간 충돌은 Lock으로 처리)
  • Undo 세그먼트: Oracle에서 이전 버전을 Undo 세그먼트에 보관
  • MVTO(MVCC + Timestamp): 타임스탬프로 읽을 버전 결정
-- MVCC 예시 (Snapshot Isolation) T1 시작 (Snapshot 시점: X=10) T2: X → 20, COMMIT T1: SELECT X → 10 (T1 시작 시점 스냅샷 사용) T1은 T2의 변경을 보지 못함 → 일관된 읽기 보장
구분Lock 기반MVCC
읽기/쓰기 충돌읽기가 쓰기 대기읽기가 이전 버전 읽음 (대기 없음)
쓰기/쓰기 충돌X-Lock으로 대기X-Lock으로 대기 (동일)
교착상태발생 가능읽기 관련 교착 없음
공간추가 공간 불필요버전 저장 공간 필요
적용 DBMS전통 Lock 기반PostgreSQL, Oracle, MySQL InnoDB
낙관적 병행제어 (Optimistic Concurrency Control) ⭐
▾
  • 가정: 충돌이 드물게 발생한다는 낙관적 가정 → 실행 중 락 없이 진행
  • 3단계: ①읽기 단계: 로컬 복사본에서 작업 ②검증 단계: COMMIT 전 충돌 확인 ③쓰기 단계: 충돌 없으면 반영, 있으면 롤백
  • 장점: 읽기 위주 워크로드에서 성능 우수 / 교착상태 없음
  • 단점: 충돌 많으면 롤백 빈발 → 성능 저하
타임스탬프=락없이 TS순서로 충돌처리·교착없음·롤백多 / MVCC=다중버전유지·읽기쓰기 무차단·PostgreSQL·Oracle / 낙관=충돌없다가정·검증단계

교착상태 (Deadlock) ⭐

교착상태: 두 개 이상의 트랜잭션이 서로 상대방이 보유한 락을 기다리며 영원히 대기하는 상태
교착상태 발생 조건 (Coffman 4가지) ⭐
▾
-- 교착상태 예시 시간 T1 T2 t1 lock-X(A) 획득 t2 lock-X(B) 획득 t3 lock-X(B) 요청 ← B를 T2가 보유 → 대기 t4 lock-X(A) 요청 ← A를 T1이 보유 → 대기 → T1↔T2 서로 기다리며 교착!
조건설명제거 방법
① 상호 배제자원을 한 트랜잭션만 사용공유 가능 자원으로 변경
② 점유와 대기자원 보유하면서 다른 자원 대기모든 자원 한꺼번에 요청
③ 비선점보유 자원을 강제 회수 불가선점 허용 (타임스탬프 기반)
④ 순환 대기 ⭐T1→T2→T3→T1 순환 구조자원에 순서 부여 (순환 불가)
4가지 중 하나라도 제거하면 교착상태 발생 안 함 / 실용적으로는 ④순환대기 제거(자원 순서화)가 가장 많이 사용

탐지 (Detection) ⭐발생 후 탐지대기 그래프(Wait-for Graph)에서 사이클 탐지 → 희생자 선택·롤백실용적 / 탐지 비용
회복 (Recovery)발생 후 처리희생자(Victim) 선택 기준: 최소 비용·최소 작업량·최소 롤백탐지 이후 진행
-- Wait-Die vs Wound-Wait (타임스탬프 기반 예방) -- TS(Ti) < TS(Tj) : Ti가 더 오래된 트랜잭션 Wait-Die (비선점): Ti(오래됨) → Tj(최신) 대기 원하면: 대기 허용 Ti(최신) → Tj(오래됨) 대기 원하면: Ti 롤백(die) Wound-Wait (선점): Ti(오래됨) → Tj(최신) 대기 원하면: Tj 강제 롤백(wound) Ti(최신) → Tj(오래됨) 대기 원하면: 대기 허용
기아 (Starvation) — 교착상태와 구별!
▾
  • 정의: 특정 트랜잭션이 계속 희생자로 선택되어 영원히 진행하지 못하는 상태 — 교착상태는 아님!
  • 원인: 교착 탐지 시 동일 트랜잭션을 반복 선택 / 타임스탬프 기반 잦은 롤백
  • 해결: 롤백 횟수를 카운트하여 오래 기다린 트랜잭션 우선 처리 (FIFO 방식)
  • 교착상태 vs 기아: 교착=서로 기다림(순환) / 기아=한쪽만 계속 기다림(비순환)
