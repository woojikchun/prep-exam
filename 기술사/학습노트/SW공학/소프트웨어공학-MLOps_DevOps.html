DevOps & MLOps 탄생 배경 및 개념 정의
소프트웨어·AI 시스템의 신뢰성 있는 배포를 위한 엔지니어링 패러다임
Ⅰ. 탄생 배경
구분DevOpsMLOps
탄생 시점2009년 (Patrick Debois, DevOpsDays)2015년 이후 (Google·Uber 등 선도)
해결 문제개발팀(Dev)과 운영팀(Ops) 간의 협업 단절로 인한 배포 지연·품질 저하 ("벽을 넘겨라" 문화)데이터 과학자가 만든 ML 모델이 실제 서비스에 배포·운용되지 못하는 "마지막 마일" 문제. 87%의 ML 모델이 프로덕션 미도달(Gartner)

Ⅱ. 핵심 정의
🔧 DevOps 정의

소프트웨어 개발(Development)과 운영(Operations)의 통합 철학으로, 자동화·협업·지속적 피드백을 통해 소프트웨어의 빌드→테스트→배포→운영→모니터링 주기를 단축하는 문화·프로세스·도구의 집합체.


핵심: CI/CD 파이프라인 자동화 + IaC + 문화 변혁

🤖 MLOps 정의

ML(Machine Learning) 모델의 개발(Dev)·운영(Ops)·데이터(Data)를 통합하여, 모델의 실험→학습→검증→배포→모니터링→재학습을 자동화하는 엔지니어링 실천 방법론. (Google MLOps Whitepaper, 2020)


핵심: 모델 파이프라인 자동화 + 데이터 버전 관리 + 드리프트 감지

Ⅲ. 성숙도 모델 비교
수준DevOps 성숙도MLOps 성숙도 (Google 3단계)
Level 0수동 배포, 팀 간 사일로ML 없음 또는 수동 학습·배포. 스크립트 기반
Level 1일부 자동화, CI 도입ML 파이프라인 자동화. 피처 스토어·실험 추적 도입. 모델 학습 자동화
Level 2완전 CI/CD, 자동 테스트CI/CD + CT(Continuous Training) 완전 자동화. 데이터·모델 버전 관리. 드리프트 탐지·재학습 자동 트리거
Level 3+GitOps, SRE, 완전 자동화A/B·Shadow 배포, 자동 재학습, 다중 모델 서빙, 피드백 루프 완성
🔑 한 줄 암기
DevOps = "개발+운영 통합, CI/CD로 빠른 릴리즈"
MLOps = "DevOps 원칙 + 데이터·모델 관리 + CT(지속적 학습)"
MLOps ⊃ DevOps + 데이터 버전관리 + 피처 스토어 + 모델 레지스트리 + 드리프트 감지

DevOps 심화 — 3 Ways · CALMS · CI/CD · DORA
Ⅰ. DevOps 3 Ways (Gene Kim, The Phoenix Project)
① 흐름 (Flow)

개발→운영→고객으로의 단방향 작업 흐름 최적화.
배치 크기 축소, 작업 가시화(Kanban), 흐름 병목 제거.
핵심: 전체 시스템 관점의 최적화

② 피드백 (Feedback)

운영→개발로의 빠른 피드백 루프 구축.
모니터링·경보·사용자 텔레메트리 기반 품질 문제 조기 감지.
핵심: 다운스트림→업스트림 신호

③ 지속적 실험·학습

실패를 학습 기회로 삼는 실험 문화 조성.
사후 분석(Blameless Post-mortem), 카오스 엔지니어링.
핵심: 심리적 안전감 + 지식 공유

Ⅱ. CALMS 프레임워크
원칙의미실천 사례

CI vs CD 구분
CI (Continuous Integration): 코드 변경 시마다 자동 빌드·테스트. 통합 오류 조기 발견
CD (Continuous Delivery): 스테이징까지 자동 배포. 프로덕션은 승인(버튼) 필요
CD (Continuous Deployment): 프로덕션까지 완전 자동화. 승인 없이 자동 릴리즈
Ⅳ. DORA 4대 지표 (DevOps Research and Assessment)
지표정의Elite 기준의미
배포 빈도
(Deployment Frequency)		프로덕션 배포 빈도하루 여러 번속도 지표. 높을수록 리스크 분산
변경 리드타임
(Lead Time for Changes)		코드 커밋 → 프로덕션 반영 시간1시간 미만흐름 지표. 짧을수록 민첩성 높음
변경 실패율
(Change Failure Rate)		배포 후 장애 유발 변경 비율0~15%품질 지표
서비스 복구 시간
(MTTR)		장애 발생 → 복구까지 시간1시간 미만회복 탄력성 지표
Ⅴ. 배포 전략 비교
전략방식장점단점
Rolling Update기존 인스턴스를 순차적으로 새 버전으로 교체추가 인프라 없음. 점진적 전환배포 중 구·신 버전 혼재. 롤백 복잡
Blue-Green동일한 두 환경(Blue/Green)을 두고 트래픽 전환 (로드밸런서 스위칭)즉각 롤백 가능. 다운타임 없음인프라 비용 2배. DB 마이그레이션 복잡
Canary소수(1~5%) 사용자에게 먼저 배포 후 점진적 확대실제 트래픽으로 검증. 리스크 최소화모니터링 복잡. 피처 플래그 필요
Shadow실제 트래픽을 복제하여 신 버전에 병렬 전송. 응답 미노출프로덕션 부하 테스트. 사용자 영향 없음인프라 비용. 응답 검증 복잡
A/B Test두 버전을 동시 배포하여 사용자 반응(전환율) 비교비즈니스 효과 검증통계적 유의성 확보 기간 필요
Ⅵ. IaC (Infrastructure as Code)
개념 및 특징
  • 인프라를 코드(선언적·명령적)로 정의·관리
  • 버전 관리(Git)·테스트·코드 리뷰 적용
  • 환경 재현성·일관성 보장 (Config Drift 방지)
  • 비교: 선언적=목표 상태 기술 / 명령적=절차 기술
주요 도구
  • Terraform: 멀티클라우드 선언적 IaC (HCL). 사실상 표준
  • Ansible: 에이전트리스 구성 관리. YAML 플레이북
  • CloudFormation: AWS 전용. JSON/YAML
  • Pulumi: 범용 언어(Python·TS)로 IaC 작성

MLOps 심화 — 생명주기 · 구성요소 · 핵심 개념
Ⅰ. ML 시스템의 독특한 복잡성 (ML Technical Debt)
⚠ Hidden Technical Debt in ML Systems (Google, 2015 NIPS)
ML 시스템에서 실제 ML 코드는 전체의 5~10%에 불과. 나머지 90%는 데이터 수집·검증·피처 추출·서빙 인프라·모니터링 등 주변 코드. 일반 소프트웨어와 달리 ML은 코드 + 데이터 + 모델이 모두 변하는 3중 관리 부담.
Ⅱ. MLOps 전체 생명주기 파이프라인
━━━ MLOps 전체 생명주기 ━━━ ① 데이터 수집·저장 Data Sources → Feature Store → Data Lake/Warehouse ② 데이터 검증 Great Expectations / TFDV → 스키마·이상치·드리프트 검사 ③ 피처 엔지니어링 Feature Engineering → Feature Store (재사용·공유) ④ 모델 실험·학습 Jupyter → MLflow·W&B Experiment Tracking ⑤ 모델 평가 Accuracy·F1·AUC + 공정성(Fairness)·강건성(Robustness) 평가 ⑥ 모델 레지스트리 MLflow Registry / Vertex AI → 버전·스테이지 관리 ⑦ 모델 배포(Serving) Real-time(REST API) / Batch / Streaming Inference ⑧ 모니터링 Data Drift·Concept Drift 탐지 → 재학습 트리거 ⑨ CT(Continuous Training) 드리프트/스케줄/이벤트 기반 자동 재학습 파이프라인
Ⅲ. MLOps 핵심 구성요소 상세
구성요소역할대표 도구
피처 스토어
(Feature Store)		피처(특징값)를 중앙 저장·공유·재사용하는 저장소. 온라인(저지연 서빙)·오프라인(학습용) 스토어 분리Feast, Tecton, Vertex AI Feature Store, AWS SageMaker
실험 추적
(Experiment Tracking)		하이퍼파라미터·메트릭·아티팩트·코드 버전을 실험별로 기록·비교. 재현성 보장MLflow Tracking, Weights & Biases (W&B), Neptune.ai
모델 레지스트리
(Model Registry)		학습된 모델의 버전 관리. Staging/Production/Archived 스테이지 관리. 승인 워크플로MLflow Model Registry, Hugging Face Hub, Vertex AI Model Registry
파이프라인 오케스트레이션데이터→피처→학습→평가→배포 단계를 DAG(방향 비순환 그래프)로 자동화·스케줄링Apache Airflow, Kubeflow Pipelines, ZenML, Vertex AI Pipelines
데이터·모델 버전 관리데이터셋·모델 파일의 Git 유사 버전 관리. 코드+데이터+모델 재현성DVC (Data Version Control), LakeFS, Delta Lake
모델 서빙
(Model Serving)		학습된 모델을 API로 노출하여 추론(Inference) 서비스 제공. 배치/실시간/스트리밍TorchServe, TensorFlow Serving, Seldon Core, BentoML, Triton
모니터링프로덕션 모델의 데이터 드리프트·모델 드리프트·성능 열화 탐지 및 경보Evidently AI, Arize, WhyLabs, Prometheus+Grafana
Ⅳ. 드리프트(Drift) 유형 — MLOps의 핵심 과제
드리프트 유형정의탐지 방법대응
데이터 드리프트
(Data/Feature Drift)		입력 데이터의 통계적 분포가 학습 시점과 달라지는 현상 (입력 X의 변화)PSI(Population Stability Index), KS-Test, KL 발산재학습 트리거, 피처 재선택
컨셉 드리프트
(Concept Drift)		입력 X와 타겟 Y의 관계(P(Y|X))가 시간에 따라 변하는 현상 (세상의 변화)모델 예측 정확도 모니터링, 윈도우 기반 성능 추이온라인 학습, 주기적 전체 재학습
레이블 드리프트
(Label Drift)		출력 레이블(Y)의 분포 변화 (클래스 불균형 변화)예측 분포 vs 실제 분포 비교샘플링 전략 조정, 임계값 재설정
인프라 드리프트라이브러리 버전·환경 변경으로 인한 예측 결과 불일치모델 서명(Schema) 검증, 컨테이너 버전 고정Docker 이미지 고정, 의존성 잠금
Ⅴ. CT (Continuous Training) — MLOps 고유 개념
CT vs CI/CD 차이
DevOps의 CI/CD는 코드 변경을 트리거로 하지만, MLOps의 CT는 데이터 변경·드리프트 탐지·성능 저하·스케줄을 트리거로 모델을 자동 재학습·재배포.

CT 트리거 유형: ①시간 기반(Weekly/Monthly) ②성능 기반(정확도 임계값 하락) ③데이터 기반(드리프트 탐지) ④이벤트 기반(규제 변경)
Ⅵ. 모델 서빙 패턴
패턴방식적합 사례지연시간
Real-time ServingREST API / gRPC로 단건 즉시 추론추천·사기 탐지·챗봇밀리초
Batch Inference대량 데이터를 일괄 처리 (Spark·Ray)일간 리포트·대규모 스코어링분~시간
Streaming InferenceKafka 스트림 데이터를 실시간 처리실시간 이상 탐지·동적 가격초 단위
Edge Inference모델을 엣지 디바이스에 배포 (ONNX·TFLite·CoreML)IoT·자율주행·스마트폰 AI마이크로초
🔑 MLOps 핵심 암기
구성요소 = 피처스토어 · 실험추적 · 모델레지스트리 · 파이프라인 · DVC · 서빙 · 모니터링
드리프트 = 데이터드리프트(X분포↔) · 컨셉드리프트(P(Y|X)↔) · 레이블드리프트
CT = 데이터/드리프트/성능 기반 자동 재학습 (DevOps CI/CD 확장)

DevOps vs MLOps 완전 비교
논술형 비교표 완벽 대비
Ⅰ. 핵심 비교표
비교 항목DevOpsMLOps
관리 대상코드(Code)코드 + 데이터 + 모델 (3가지 아티팩트)
버전 관리Git (코드 버전)Git + DVC(데이터) + 모델 레지스트리(모델)
테스트 대상단위·통합·E2E 테스트코드 테스트 + 데이터 검증 + 모델 평가 (정확도·공정성·강건성)

Q1
DevOps의 3 Ways와 CALMS 프레임워크를 설명하시오. 단답형 ▼
[3 Ways] Gene Kim이 정의한 DevOps 핵심 원칙:
① 흐름(Flow): 개발→운영→고객으로의 단방향 가치 흐름 최적화. 배치 크기 축소, 병목 제거
② 피드백(Feedback): 운영→개발로의 빠른 피드백 루프. 모니터링·경보 기반 품질 조기 감지
③ 지속 실험·학습: 실패를 학습 기회로. Blameless Post-mortem, 카오스 엔지니어링

[CALMS]: Culture(협업 문화) · Automation(반복 자동화) · Lean(낭비 제거·소배치) · Measurement(DORA 지표 기반 측정) · Sharing(지식·도구 공유)
Q2
DORA 4대 지표를 설명하고 각 지표의 Elite 기준을 기술하시오. 단답형 ▼
DORA(DevOps Research and Assessment) 4대 지표:
① 배포 빈도(Deployment Frequency): 프로덕션 배포 빈도. Elite = 하루 여러 번
② 변경 리드타임(Lead Time for Changes): 코드 커밋→프로덕션 반영 시간. Elite = 1시간 미만
③ 변경 실패율(Change Failure Rate): 배포 후 장애 유발 비율. Elite = 0~15%
④ 서비스 복구 시간(MTTR): 장애 발생→복구 시간. Elite = 1시간 미만

속도지표(①②)와 안정성지표(③④)의 균형이 핵심. Elite 팀은 속도와 안정성을 동시 달성.
Q3
MLOps에서 드리프트(Drift)의 유형 3가지와 각 탐지 방법을 설명하시오. 단답형 ▼
① 데이터 드리프트(Feature/Data Drift): 입력 데이터 X의 통계적 분포 변화. 탐지: PSI(Population Stability Index), KS-Test, KL Divergence. 예) 코로나 이후 소비 패턴 급변

② 컨셉 드리프트(Concept Drift): 입력 X와 출력 Y의 관계 P(Y|X) 변화(세상의 변화). 탐지: 예측 정확도 모니터링, 윈도우 기반 성능 추이 분석. 예) 사기 패턴 진화로 모델 무력화

③ 레이블 드리프트(Label Drift): 출력 레이블 Y의 분포 변화(클래스 불균형 변화). 탐지: 예측 분포 vs 실제 레이블 분포 비교. 대응: 임계값 재설정, 샘플링 전략 조정
Q4
CI와 CD(Continuous Delivery vs Continuous Deployment)의 차이를 설명하시오. 단답형 ▼
CI(Continuous Integration): 개발자가 코드를 메인 브랜치에 자주 통합(Push)하면 자동으로 빌드·단위테스트·정적 분석이 실행되어 통합 오류를 조기 발견하는 실천. 목적: 통합 지옥(Integration Hell) 방지

CD(Continuous Delivery): CI 이후 스테이징 환경까지 자동 배포. 프로덕션 배포는 사람이 승인(버튼 클릭)하는 단계가 남음. 항상 릴리즈 가능한 상태 유지

CD(Continuous Deployment): 모든 테스트 통과 시 프로덕션까지 자동 배포. 사람 개입 없음. 높은 자동화 성숙도·테스트 신뢰성 전제. Netflix·Amazon이 실천
Q5
MLOps의 피처 스토어(Feature Store)의 개념과 온라인·오프라인 스토어의 차이를 설명하시오. 단답형 ▼
피처 스토어(Feature Store): ML 피처(특징값)를 중앙에서 저장·관리·공유·재사용하는 인프라. 여러 모델이 동일 피처를 학습과 서빙에서 일관되게 사용하도록 보장하여 학습-서빙 불일치(Training-Serving Skew) 방지

온라인 스토어(Online Store): 실시간 추론(Serving)을 위한 저지연(밀리초) 피처 제공. Redis·DynamoDB·Bigtable 등 인메모리·NoSQL 기반. 최신 피처값 보관

오프라인 스토어(Offline Store): 모델 학습을 위한 대용량 히스토리컬 피처 제공. S3·GCS·BigQuery·Hive 등 데이터 레이크/웨어하우스 기반. 시점(Point-in-time) 피처 조회 지원(Feature Backfill)

대표 도구: Feast, Tecton, Vertex AI Feature Store, AWS SageMaker Feature Store
Q6
[논술형] DevOps의 개념, CI/CD 파이프라인 구성, 배포 전략(Blue-Green, Canary)을 비교 설명하고, DORA 4대 지표와의 연계성을 논하시오. 논술형 ▼
[Ⅰ. DevOps 개념] 개발(Dev)과 운영(Ops)의 통합 패러다임. 자동화·협업·지속적 피드백으로 소프트웨어 빌드→배포→운영 주기 단축. 3 Ways(흐름·피드백·지속실험)와 CALMS 원칙 기반.

[Ⅱ. CI/CD 파이프라인] CI: 코드 Push→자동빌드→단위테스트→SAST→SCA→통합테스트. CD: 스테이징 자동배포→승인→프로덕션 배포(Delivery) 또는 완전자동(Deployment). 실패 시 배포 차단(Gate).

[Ⅲ. 배포 전략 비교]
• Blue-Green: 동일 두 환경 유지. 로드밸런서로 즉시 전환. 즉각 롤백 가능. 인프라 비용 2배.
• Canary: 소수(1~5%) 사용자에게 선배포 후 점진 확대. 실제 트래픽으로 검증. 피처 플래그 필요.
• Shadow: 실제 트래픽 복제 전송. 사용자 영향 없이 테스트. 가장 안전하나 비용 최대.

[Ⅳ. DORA 연계] Blue-Green·Canary 배포는 변경 실패율↓+MTTR↓ 직결. CI/CD 자동화는 배포 빈도↑·리드타임↓에 기여. DORA는 DevOps 성과를 정량화하여 지속 개선의 피드백 루프를 형성.

[Ⅴ. 결론] DevOps는 속도와 안정성의 역설을 해소한다. DORA Elite 기업은 배포 빈도를 높이면서도 MTTR을 단축하는데, 이는 자동화된 테스트·배포·모니터링의 삼위일체로 달성된다.
Q7
[논술형] MLOps의 개념, 전체 생명주기 파이프라인, Google MLOps 3단계 성숙도 모델을 설명하시오. 논술형 ▼
[Ⅰ. 개념] MLOps는 ML 모델의 개발·운영을 통합하여 실험→학습→검증→배포→모니터링→재학습을 자동화하는 방법론. DevOps 원칙을 ML 시스템에 확장하되, 데이터·모델이라는 2가지 추가 아티팩트를 관리.

[Ⅱ. 생명주기 파이프라인] ①데이터 수집·검증 → ②피처 엔지니어링(Feature Store) → ③모델 실험(Experiment Tracking) → ④모델 학습·평가 → ⑤모델 레지스트리 등록 → ⑥모델 배포(Serving) → ⑦모니터링(드리프트 탐지) → CT 트리거 시 ①로 자동 복귀(자동 재학습 루프).

[Ⅲ. Google 3단계 성숙도]
• Level 0(수동): ML 없음 또는 스크립트 기반 수동 학습·배포. 재현성·추적 불가.
• Level 1(ML 파이프라인 자동화): 학습 파이프라인 자동화. Feature Store·실험 추적 도입. CT 가능. 모델 배포는 수동.
• Level 2(CI/CD+CT 완전 자동화): 파이프라인 자체를 CI/CD로 관리. 코드·데이터·모델 변경 시 자동 재학습→평가→배포. 드리프트 탐지 자동 트리거. 완전한 MLOps 구현.

[Ⅳ. 결론] 대부분 조직은 Level 0~1에 머물며, Level 2 달성이 진정한 MLOps 목표. 성숙도 향상의 병목은 기술보다 데이터 거버넌스·조직 문화에 있는 경우가 많다.
Q8
[논술형] DevOps와 MLOps를 비교하고, MLOps에서 추가로 요구되는 CT(Continuous Training)와 드리프트 관리 체계를 설명하시오. 논술형 ▼
[Ⅰ. DevOps vs MLOps 핵심 비교]
관리 아티팩트: 코드(DevOps) → 코드+데이터+모델(MLOps)
버전 관리: Git → Git+DVC(데이터)+Model Registry
배포 트리거: 코드 변경 → 코드 변경 OR 데이터 변경 OR 드리프트 OR 스케줄
모니터링: 시스템 메트릭 → 시스템 메트릭+모델 성능+데이터 드리프트
MLOps = DevOps + 피처스토어 + 실험추적 + 모델 레지스트리 + CT + 드리프트 감지

[Ⅱ. CT(Continuous Training)] DevOps의 CI/CD가 코드 변경을 트리거로 한다면, MLOps의 CT는 ①시간 기반(주기적) ②성능 저하(정확도 임계값 하락) ③데이터 드리프트 탐지 ④이벤트(규제·비즈니스 변화)를 트리거로 모델 자동 재학습→검증→배포 수행.

[Ⅲ. 드리프트 관리 체계]
• 데이터 드리프트: PSI/KS-Test로 입력 분포 모니터링 → 임계 초과 시 CT 트리거
• 컨셉 드리프트: 예측 정확도 지속 모니터링 → 성능 하락 시 전체 재학습
• 관리 도구: Evidently AI, Arize, WhyLabs, Prometheus+Grafana

[Ⅳ. 결론] ML 모델은 배포 후 시간이 지남에 따라 필연적으로 성능이 열화(Model Decay)된다. CT와 드리프트 모니터링은 모델의 지속적 가치 유지를 위한 MLOps의 핵심 차별화 요소이다.
Q9
[논술형] 공공기관 AI 서비스 도입 시 MLOps 구축 계획을 수립하고, 감리사 관점에서 점검해야 할 핵심 항목을 설명하시오. 논술형 ▼
[Ⅰ. 공공 AI 서비스의 MLOps 요구사항] 공공기관 AI는 민간 대비 높은 투명성·공정성·설명가능성 요구. EU AI Act 고위험 AI 분류 적용 가능. 개인정보보호법·행정절차법 등 법령 준수 필수.

[Ⅱ. MLOps 구축 계획 (4단계)]
1단계(데이터 거버넌스): 학습 데이터 수집·정제·검증 체계. 개인정보 처리 적법성(동의·목적 제한). 데이터 편향 검사.
2단계(실험·학습 관리): MLflow·Kubeflow로 실험 추적·재현성 확보. DVC로 데이터 버전 관리. 모델 성능+공정성+설명가능성(XAI) 동시 평가.
3단계(배포·서빙): 모델 레지스트리 승인 워크플로(보안·법률 검토 포함). Blue-Green 배포로 무중단 전환. REST API 보안(인증·접근제어·로깅).
4단계(모니터링·재학습): 데이터 드리프트·모델 성능·편향 지속 모니터링. CT 자동화. 재학습 시 동일한 검증·승인 프로세스 적용.

[Ⅲ. 감리사 핵심 점검 항목]
• 데이터 품질: 학습 데이터 편향·결측 검증 로직 존재 여부
• 재현성: 코드+데이터+환경 버전 연계 추적 가능 여부
• 공정성 평가: 성별·연령·지역별 모델 성능 차이(FPR Gap) 측정 여부
• 설명가능성: SHAP·LIME 등 XAI 적용으로 행정 결정 설명 가능 여부
• 모니터링: 드리프트 탐지·경보·재학습 트리거 체계 존재 여부
• 규제 준수: 개인정보보호법 §33 PIA 수행, AI 서비스 영향평가 여부

[Ⅳ. 결론] 공공 AI MLOps는 기술적 자동화 외에 법적 준수성·공정성·설명가능성이 추가 필수 조건. 감리는 성능 지표뿐 아니라 AI 거버넌스 체계 전반을 점검해야 한다.
Q10
[논술형] DevOps와 MLOps 통합 시스템 구축 방안을 논하고, 조직 변화 관리(사람·프로세스·기술) 관점에서 성공 요인과 장애 요인을 설명하시오. 논술형 ▼
[Ⅰ. 통합 시스템 구축 방안]
공통 기반: 쿠버네티스 클러스터 위에 DevOps CI/CD와 MLOps 파이프라인을 통합. 단일 GitOps 저장소(코드+IaC+ML 파이프라인 정의)로 관리. 통합 모니터링 플랫폼(Prometheus·Grafana)에 ML 메트릭 추가.
ML 특화 레이어: Feature Store + Experiment Tracking + Model Registry를 DevOps 인프라 위에 추가 구성. CT 파이프라인을 기존 CI/CD 도구(GitHub Actions·Airflow)로 구현하여 운영 부담 최소화.

[Ⅱ. 사람(People) 관점]
성공 요인: ML Engineer·Data Scientist·DevOps Engineer 간 역할 명확화. 공동 책임(Shared Responsibility) 문화. 데이터사이언티스트의 MLOps 리터러시 교육. 심리적 안전감.
장애 요인: 팀 간 사일로·권한 갈등. 데이터과학팀의 운영 부담 거부감. MLOps 전문 인력 부족.

[Ⅲ. 프로세스(Process) 관점]
성공 요인: 모델 배포를 코드 배포와 동일한 승인 프로세스로 관리. 데이터 품질 게이트(Gate) 설정. Blameless Post-mortem 문화.
장애 요인: 수작업이 남은 프로세스 병목. 재학습 기준·승인 절차 미문서화. 거버넌스 부재로 인한 무분별한 모델 배포.

[Ⅳ. 기술(Technology) 관점]
성공 요인: 표준화된 컨테이너·오케스트레이션 기반. 오픈소스 도구(MLflow·DVC·Feast) 도입으로 벤더 종속 방지. 재현 가능한 파이프라인.
장애 요인: 도구 과잉(Tool Sprawl). 학습-서빙 환경 불일치(Skew). 레거시 시스템 통합 복잡성. 데이터 거버넌스 미비.

[Ⅴ. 결론] DevOps+MLOps 통합의 성패는 기술보다 조직 문화에 달려 있다. 공통 플랫폼 팀(Platform Engineering)을 구성하여 개발·데이터·운영팀이 공유 인프라를 사용하는 내부 개발자 플랫폼(IDP) 전략이 가장 효과적인 접근법이다.
