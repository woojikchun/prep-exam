Ⅰ. SOLID 원칙의 개념
SOLID는 Robert C. Martin(Uncle Bob)이 정립한 객체지향 설계의 5가지 핵심 원칙으로, 유지보수성·확장성·재사용성이 높은 소프트웨어를 개발하기 위한 설계 지침이다.
Ⅱ. 5가지 원칙 상세
원칙Full Name핵심 개념위반 사례적용 효과
SSingle Responsibility
단일 책임 원칙		클래스는 하나의 변경 이유(책임)만 가져야 한다. 클래스는 하나의 역할만 담당User 클래스가 인증·DB저장·이메일 발송을 모두 처리응집도 향상, 변경 영향 범위 최소화
OOpen/Closed
개방·폐쇄 원칙		확장에는 열려 있고, 수정에는 닫혀 있어야 한다. 기존 코드 수정 없이 기능 추가새 결제수단 추가 시 기존 if-else 블록 수정 필요기존 코드 안정성 유지, 기능 추가 용이
LLiskov Substitution
리스코프 치환 원칙		자식 클래스는 부모 클래스를 대체할 수 있어야 한다. IS-A 관계 엄격 준수Rectangle 상속 Square에서 setWidth 오동작다형성 안정적 활용, 상속 구조 신뢰성
IInterface Segregation
인터페이스 분리 원칙		클라이언트는 사용하지 않는 메서드에 의존하지 않아야 한다. 뚱뚱한 인터페이스 분리Printable에 scan(), fax()까지 강제 구현불필요한 의존성 제거, 인터페이스 명확화
DDependency Inversion
의존성 역전 원칙		고수준 모듈은 저수준 모듈에 의존하지 않고, 둘 다 추상화(인터페이스)에 의존해야 한다Service 클래스가 MySQLRepository를 직접 import결합도 감소, 테스트 용이성·유연성 향상
Ⅲ. 원칙 간 상호 관계 및 적용 방법
SOLID 원칙 적용 패턴 S + D → 의존성 주입(DI): 책임을 분리하고 인터페이스에 의존 O + D → 전략 패턴(Strategy): 알고리즘 교체 시 기존 코드 수정 없음 I + D → 인터페이스 기반 설계: 세분화된 인터페이스로 DIP 구현 L → 설계 시 계약(Contract) 명세 작성, 사전·사후 조건 검증 예시(Java/Kotlin): // DIP 적용 전 class OrderService { val repo = MySQLOrderRepository() } // DIP 적용 후 class OrderService(private val repo: OrderRepository) // 인터페이스 주입
📌 한 줄 암기 (SOLID)
S-하나의 책임  |  O-수정없이 확장  |  L-자식은 부모 대체  |  I-작은 인터페이스  |  D-추상화에 의존
Ⅳ. 결론 — 감리 포인트

SOLID 원칙은 DDD, 클린 아키텍처, 마이크로서비스 아키텍처의 설계 기반이 된다. 코드 리뷰 및 정적 분석 도구(SonarQube의 응집도·결합도 메트릭)를 통해 SOLID 위반 여부를 자동 점검할 수 있으며, 감리 시에는 모듈 간 의존성 그래프(패키지 결합도)를 주요 점검 항목으로 확인한다.

Ⅰ. 개요 — 테스트 레벨의 개념
소프트웨어 테스트 레벨은 개발 생명주기의 각 단계에 대응하는 테스트 활동의 계층 구조로, ISTQB(국제 소프트웨어 테스팅 자격위원회) 및 V-모델을 기반으로 단위→통합→시스템→인수 테스트의 4단계로 구성된다.
V-모델 대응 관계 요구사항 분석 ─────────────────────── 인수 테스트(AT) 아키텍처 설계 ─────────────────── 시스템 테스트(ST) 상세 설계 ─────────────────── 통합 테스트(IT) 구현(코딩) ─────────────── 단위 테스트(UT) ↑개발 방향 ↑테스트 방향
Ⅱ. 4가지 테스트 레벨 비교
구분단위 테스트통합 테스트시스템 테스트인수 테스트
대상개별 함수·메서드·클래스모듈 간 인터페이스통합된 전체 시스템최종 사용자 관점의 시스템

담당개발자개발자·테스터테스트팀(독립)고객·사용자·QA
주요 기법화이트박스, JUnit/pytest, 목(Mock) 객체빅뱅, 하향식, 상향식, 샌드위치 통합블랙박스, 성능·보안·호환성 테스트알파/베타 테스트, UAT, 계약 인수 테스트

Ⅰ. SCM의 개념
소프트웨어 형상관리(SCM)는 소프트웨어 제품의 무결성·추적성·가시성을 보장하기 위해 형상 항목(Configuration Item)의 식별·통제·감사·상태 보고를 체계적으로 관리하는 프로세스이다. IEEE 828 표준을 기반으로 하며, 소프트웨어 개발 전 생명주기에 걸쳐 적용된다.
Ⅱ. SCM 4대 주요 활동
  • 1
    형상 식별(Configuration Identification): 관리 대상 형상 항목(CI) 선정 및 고유 식별자 부여. 소스코드·문서·데이터베이스 스키마·테스트 케이스 등 포함
  • 2
    형상 통제(Configuration Control): CI에 대한 변경 요청(CR) 검토·승인·추적. CCB(형상통제위원회)가 변경 승인. 변경 이력 관리
  • 3
    형상 감사(Configuration Audit): CI가 명세·계약·표준에 부합하는지 독립적으로 검증. 기능 형상 감사(FCA)·물리 형상 감사(PCA)
  • 4
    형상 상태 보고(Configuration Status Accounting): CI의 현재 상태·변경 이력·승인 현황을 기록·보고. 관리자·개발자에게 시각화 제공
Ⅲ. 기준선(Baseline)의 역할
기준선은 특정 시점에 공식 검토·승인된 형상의 스냅숏으로, 변경 통제의 기준점이 된다. 기준선 이후의 변경은 반드시 CCB 승인을 거쳐야 한다.
기준선 유형설정 시점포함 항목
기능 기준선요구사항 확정 후SRS(소프트웨어 요구사항 명세서)
할당 기준선아키텍처 설계 완료 후설계 명세서, 인터페이스 정의서
제품 기준선시스템 테스트 완료 후 인도 전소스코드, 실행파일, 테스트 케이스·결과
Ⅳ. 대표 도구 비교
도구유형특징브랜치 전략
Git분산형 DVCS오프라인 작업, 경량 브랜치, 오픈소스 표준Git Flow / GitHub Flow / Trunk-Based

23가지 패턴 분류 암기생성·구조·행위실제 적용 예시 필수
GoF 디자인 패턴 3분류 및 대표 패턴 6종
⏱ 목표 15분 📄 1.5~2페이지 💯 10~15점
Ⅰ. GoF 디자인 패턴 개요
GoF(Gang of Four) 디자인 패턴은 1994년 Gamma 외 4인이 객체지향 설계에서 반복적으로 나타나는 문제를 유형화한 23가지 설계 패턴이다. 생성(Creational) 5종, 구조(Structural) 7종, 행위(Behavioral) 11종으로 분류된다.
Ⅱ. 생성 패턴 (Creational) — 객체 생성 방식 추상화
  • ①
    싱글톤(Singleton): 클래스의 인스턴스를 하나만 생성하고 전역 접근점을 제공하는 패턴
    구조: private 생성자 + static 인스턴스 변수 + getInstance() 메서드
    활용: DB 커넥션 풀, 로그 관리자, 설정(Config) 객체
    주의: 멀티스레드 환경에서 double-checked locking 또는 Eager initialization 필요
  • ②
    팩토리 메서드(Factory Method): 객체 생성 인터페이스를 정의하되, 어떤 클래스를 생성할지는 서브클래스가 결정하는 패턴
    구조: Creator(추상 팩토리 메서드) → ConcreteCreator(구체 구현) → Product
    활용: 로그 핸들러(FileLogger/DBLogger), 결제 수단 생성(KakaoPay/NaverPay)
    OCP 준수: 새 Product 추가 시 기존 코드 수정 불필요
Ⅲ. 구조 패턴 (Structural) — 클래스·객체 조합 방식
  • ①
    어댑터(Adapter): 호환되지 않는 인터페이스를 가진 클래스들이 함께 동작할 수 있도록 변환 래퍼를 제공하는 패턴
    구조: Target 인터페이스 ← Adapter(래퍼) → Adaptee(기존 클래스)
    활용: 구버전 결제 API → 신규 인터페이스 연동, JSON↔XML 변환
  • ②
    데코레이터(Decorator): 객체에 동적으로 책임(기능)을 추가하는 패턴. 상속 대신 구성(Composition)을 통해 기능 확장
    구조: Component ← ConcreteComponent / Decorator → ConcreteDecorator
    활용: Java InputStream → BufferedInputStream → DataInputStream (중첩 래핑)
Ⅳ. 행위 패턴 (Behavioral) — 객체 간 통신·알고리즘
  • ①
    옵저버(Observer): 주체(Subject)의 상태 변화를 여러 관찰자(Observer)에게 자동으로 알리는 발행-구독 패턴
    구조: Subject(attach/detach/notify) ← ConcreteSubject / Observer ← ConcreteObserver
    활용: GUI 이벤트 핸들러, 주식 시세 알림, Spring ApplicationEvent
  • ②
    전략(Strategy): 알고리즘 군을 캡슐화하여 상호 교체 가능하게 하는 패턴. 런타임에 알고리즘 선택

    활용: 정렬 알고리즘 교체, 결제 수단 선택(신용카드/페이팔/암호화폐)

구조(7): Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy

Ⅴ. 결론

GoF 패턴은 팀 간 공통 설계 언어(Vocabulary)를 제공하여 의사소통 효율을 높인다. 현대 프레임워크(Spring의 DI·AOP, React의 상태관리, Kubernetes의 Controller)는 GoF 패턴을 적극 활용하고 있으며, 패턴의 무분별한 적용보다 문제 상황에 맞는 선택이 중요하다.

Ⅰ. 결함 관련 핵심 용어 정의
용어IEEE 표준 정의예시
오류 (Error)개발자가 코드를 잘못 작성하는 인간의 실수. 결점을 유발하는 원인 행위개발자가 배열 인덱스를 0 대신 1부터 시작하도록 잘못 코딩
결점 (Fault)오류로 인해 소프트웨어 내에 존재하는 코드 결함. 고장을 일으킬 수 있는 잠재적 문제코드에 잠재된 배열 범위 초과 구문
고장 (Failure)결점이 실행되어 시스템이 요구사항을 충족하지 못하는 상태. 사용자가 체감하는 문제실행 시 ArrayIndexOutOfBoundsException 발생으로 시스템 중단
결함 발생 흐름 (IEEE 610.12) 사람의 실수(Error) → 코드에 결점 존재(Fault) → 결점 실행 시 고장 발생(Failure) → 사용자 불만·장애
Ⅱ. 결함 밀도 (Defect Density)
📐 결함 밀도 계산식
결함 밀도(DD) = 발견된 결함 수 / 소프트웨어 규모(천 라인 또는 기능점수)

예시: 10만 라인 코드에서 결함 50개 발견 → DD = 50 / 100 = 0.5 결함/KLOC
활용: 업계 벤치마크(1.0 결함/KLOC 이하) 대비 품질 수준 평가. 릴리즈 가능 여부 판단 기준
Ⅲ. 결함 제거율 (DRE: Defect Removal Efficiency)
📐 DRE 계산식
DRE = (인도 전 제거 결함 수) / (인도 전 제거 결함 수 + 인도 후 발견 결함 수) × 100%

예시: 인도 전 90개 제거, 인도 후 10개 발견 → DRE = 90/(90+10) × 100 = 90%
세계적 수준: DRE ≥ 95% (CMM Level 5 조직). 공공 프로젝트 목표: DRE ≥ 85%
DRE 수준품질 평가개선 방향
95% 이상세계적 수준현행 프로세스 유지·벤치마킹
85~95%양호리뷰·인스펙션 강화
85% 미만개선 필요테스트 단계 조기화, 정적 분석 도입
Ⅳ. 결론 — 감리 활용

감리 시 결함 밀도와 DRE는 소프트웨어 품질 수준을 정량적으로 평가하는 핵심 지표이다. 「SW사업 감리 점검기준」에서도 테스트 결과 분석 시 결함 추세 분석, 결함 밀도, 미해결 결함 현황을 필수 점검항목으로 규정하고 있다.

    장점: 독립 배포·확장, 기술 스택 다양화, 장애 격리, 팀 자율성(Conway's Law)

    적합: 대규모 팀, 복잡한 도메인, 높은 확장성·가용성 요구 서비스

    장점: 인프라 관리 불필요, 요청 기반 자동 스케일링, 유휴 비용 없음 (pay-per-use)

Ⅰ. 개요 — 소프트웨어 유지보수의 개념
소프트웨어 유지보수는 인도 후 결함 수정, 성능 향상, 변화하는 환경에 적응하기 위해 소프트웨어를 수정하는 활동이다. Boehm 연구에 따르면 소프트웨어 총 생명주기 비용의 60~80%가 유지보수 단계에서 발생하며, 이를 체계적으로 관리하는 것이 핵심이다.

Ⅲ. 나. Lehman의 소프트웨어 진화 법칙 (핵심 5가지)
  • 1
    지속적 변화 법칙 (Law of Continuing Change): E-type 소프트웨어는 변화에 적응하지 않으면 실용성이 점차 감소한다
    시사점: 유지보수를 선택이 아닌 필수 활동으로 계획에 포함해야 함
  • 2
    증가하는 복잡도 법칙 (Law of Increasing Complexity): 프로그램은 진화할수록 복잡도가 증가한다. 복잡도 감소를 위한 별도 작업 없이는 엔트로피가 증가
    시사점: 지속적 리팩토링, 아키텍처 리뷰를 통한 복잡도 억제 필요
  • 3
    자기 조절 법칙 (Law of Self Regulation): 소프트웨어 진화 과정은 자기 조절적이며, 제품 및 프로세스 측정치는 정규 분포를 따름
    시사점: 측정 기반 관리(CBM)로 이상 징후 조기 탐지
  • 4
    조직적 안정성 법칙 (Law of Conservation of Organizational Stability): 소프트웨어 진화의 평균 속도는 생명주기 전반에 걸쳐 일정하게 유지됨
    시사점: 릴리즈 속도를 무리하게 높이면 품질 저하 발생. 지속 가능한 속도 유지
  • 5
    계속적인 성장 법칙 (Law of Conservation of Familiarity): 사용자 만족도 유지를 위해 시스템이 제공하는 기능적 내용이 지속적으로 증가해야 함

Ⅰ. 가. CI/CD 개념과 차이점
구분개념핵심 활동산출물
CI
(지속적 통합)		개발자가 코드 변경 시 자동으로 빌드·테스트하여 통합하는 관행. 하루에도 여러 번 메인 브랜치에 통합코드 커밋·빌드·단위/통합 테스트·정적 분석빌드 아티팩트, 테스트 리포트
CD
(지속적 인도)		CI를 거친 소프트웨어를 언제든지 스테이징 환경에 배포할 수 있는 상태로 유지. 배포는 사람이 승인CI + 스테이징 배포·E2E 테스트·릴리즈 준비릴리즈 후보(Release Candidate)
CD
(지속적 배포)		CI/CD의 최종 단계. 모든 테스트를 통과하면 자동으로 프로덕션에 배포. 인간 개입 최소화자동 프로덕션 배포·모니터링·롤백 자동화프로덕션 릴리즈
CI/CD 성숙도 모델 레벨 1: Manual → 수동 빌드·테스트·배포 레벨 2: CI → 자동 빌드·단위 테스트 (PR마다) 레벨 3: CD(인도) → 스테이징 자동 배포·E2E 테스트 레벨 4: CD(배포) → 프로덕션 자동 배포 (완전 자동화) 레벨 5: DevOps → 모니터링 피드백 루프로 지속 개선
Ⅱ. 나. CI/CD 파이프라인 단계 및 품질 게이트 설계
CI/CD 파이프라인 전체 구성 코드 커밋(Git Push) │ ▼ [QG-1: 빌드 품질 게이트] ┌─ 소스 코드 빌드 (Maven/Gradle/npm) ├─ 단위 테스트 (JUnit/pytest) — 커버리지 ≥ 80% ├─ 정적 코드 분석 (SonarQube) — 치명 결함 0건 └─ 보안 스캔 (SAST/Dependency Check) │ ▼ [QG-2: 통합 품질 게이트] ┌─ 통합 테스트 (API 계약 테스트) ├─ DAST 동적 보안 테스트 (OWASP ZAP) └─ 컨테이너 이미지 빌드 및 취약점 스캔 (Trivy) │ ▼ [스테이징 환경 자동 배포] [QG-3: 스테이징 품질 게이트] ┌─ E2E 테스트 (Selenium/Cypress) ├─ 성능 테스트 (JMeter) — 응답시간 ≤ 3초 (공공 기준) └─ 인수 테스트 (UAT) │ ▼ [릴리즈 승인 — 공공: 담당자 수동 승인] [프로덕션 배포] └─ 배포 전략 적용 (Blue-Green / Rolling / Canary) │ ▼ [QG-4: 운영 품질 게이트] ├─ 헬스체크·모니터링 (Prometheus/Grafana) └─ 오류율 임계값 초과 시 자동 롤백
품질 게이트(Quality Gate) 기준 예시 (공공 기준)
① 단위 테스트 커버리지 ≥ 80% (구문 기준)
② SonarQube: Critical·Blocker 결함 0건, Technical Debt Ratio ≤ 5%
③ 성능: 응답시간 ≤ 3초(95th percentile), TPS(목표 처리량) 달성
④ 보안: OWASP Top 10 항목 취약점 없음
⑤ 미해결 결함: 심각도 Critical 이상 0건 (릴리즈 차단)

블루-그린
(Blue-Green)		구(Blue)·신(Green) 환경을 동시 운영. 트래픽 일괄 전환 후 구 환경 대기없음즉시 (트래픽 재전환)2배 필요중요 민원 서비스. 즉시 롤백 필요 시
카나리
(Canary)		신버전에 트래픽 일부(1~5%)만 전송. 점진적 확대로 위험 분산없음빠름 (트래픽 조절)약간 추가고사용량 서비스. 점진적 안전 배포
롤링
(Rolling)		서버를 하나씩 순차적으로 신버전으로 교체. 구·신버전 공존 기간 존재없음느림 (순차 롤백)기존과 동일쿠버네티스 기본 전략. 일반 업무 시스템

Ⅳ. 결론

CI/CD 파이프라인은 소프트웨어 인도 속도와 품질을 동시에 향상시키는 DevOps의 핵심 실천이다. 공공 정보시스템에서는 법·제도적 제약을 고려한 하이브리드 접근이 필요하며, 품질 게이트를 파이프라인에 통합함으로써 자동화된 품질 보증 체계를 구축해야 한다. 장기적으로는 GitOps 방식과 정책-as-코드(OPA)를 도입하여 컴플라이언스를 자동화하는 것이 바람직하다.
