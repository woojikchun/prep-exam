Ⅰ. 캐시 메모리의 개념 및 필요성
캐시 메모리(Cache Memory)는 CPU와 주기억장치(DRAM) 사이의 속도 격차를 완화하기 위해 사용하는 소용량·고속의 임시 기억장치이다. CPU는 수 GHz 속도로 동작하지만 DRAM 접근 시간은 수십~수백 ns로 수백 배의 속도 차이가 존재하며, 이 병목(Memory Wall)을 해소하는 것이 캐시의 핵심 역할이다.
📌 캐시 동작의 이론적 근거 — 참조의 지역성(Locality of Reference)
• 시간적 지역성(Temporal Locality): 최근에 참조된 데이터는 가까운 미래에 다시 참조될 가능성이 높다
  → 예: 반복문(loop) 내 변수 i가 반복적으로 접근됨
• 공간적 지역성(Spatial Locality): 참조된 데이터 근처의 데이터도 곧 참조될 가능성이 높다
  → 예: 배열 순차 접근 시 인접 원소 a[0], a[1], a[2]... 순으로 접근됨

Ⅲ. 적중률(Hit Ratio)과 AMAT 계산
📐 핵심 계산식
적중률(h) = 캐시 적중 횟수 / 전체 메모리 접근 횟수

AMAT (평균 메모리 접근 시간)
단일 캐시: AMAT = h × Tc + (1-h) × Tm
다중 캐시: AMAT = Tc + (1-h1)(Tc2 + (1-h2)×Tm)

여기서 Tc = 캐시 접근 시간, Tm = 주기억장치 접근 시간
📝 계산 예시
조건: L1 접근시간 2ns, DRAM 접근시간 100ns, L1 적중률 95%

AMAT = 0.95 × 2ns + 0.05 × 100ns = 1.9 + 5.0 = 6.9 ns

→ 캐시 없을 경우 100ns 대비 약 14.5배 성능 향상

L2 추가 시 (L2 접근시간 10ns, L2 적중률 80% 가정):
Miss penalty = 10ns + (1-0.80)×100ns = 10 + 20 = 30ns
AMAT = 2 + 0.05 × 30 = 2 + 1.5 = 3.5 ns
Ⅳ. 결론 — 감리·시험 포인트

캐시 메모리 설계의 핵심 지표는 적중률이며, 현대 프로세서는 L1 95~98%, L2 80~90%, L3 60~80%의 적중률을 목표로 한다. 임베디드 시스템 설계 및 실시간 시스템 감리 시 캐시 메모리 계층 구성의 적절성과 캐시 일관성(Cache Coherency) 프로토콜 적용 여부가 주요 점검 항목이다.

Ⅰ. 직접 매핑의 개념 및 동작 원리
직접 매핑(Direct Mapping)은 주기억장치의 각 블록이 캐시의 오직 하나의 특정 라인(슬롯)에만 적재될 수 있는 방식이다. 주소의 인덱스(Index) 필드가 캐시 라인 번호를 직접 결정하며, 구현이 단순하고 접근 속도가 빠르다.

Ⅰ. 캐시 미스 3C 분류 개요
Mark Hill(1987)이 제시한 3C 모델은 캐시 미스를 발생 원인에 따라 세 가지로 분류하여 캐시 설계 개선 방향을 체계화한 프레임워크이다. 전체 미스율 = Compulsory미스율 + Capacity미스율 + Conflict미스율로 분해된다.
Ⅱ. 3C 미스 분류 상세
  • C1
    강제 미스 (Compulsory Miss / Cold Miss)
    정의: 최초로 특정 블록에 접근할 때 반드시 발생하는 미스. 캐시가 아무리 크더라도 피할 수 없음
    특징: 전체 미스 중 비중이 가장 낮음. 무한 크기 캐시에서도 동일하게 발생
    감소 기법: 프리패칭(Prefetching) — 미래에 사용될 블록을 미리 캐시에 적재. 하드웨어(Stride Prefetcher) 또는 소프트웨어(컴파일러 프리패치 명령) 방식
    감소 기법: 블록 크기 증가 — 공간적 지역성 활용으로 첫 접근 시 더 많은 인접 데이터 캐시에 적재
  • C2
    용량 미스 (Capacity Miss)
    정의: 캐시 용량이 작아서 워킹 셋(Working Set) 전체를 수용하지 못해 이미 캐시에 있던 블록이 교체된 후 재접근 시 발생
    특징: 완전 연관 캐시(충돌 미스 없음)에서도 발생. 캐시 크기만이 영향 요인
    감소 기법: 캐시 용량 증가 (L1→L2→L3), 루프 타일링(Loop Tiling)으로 워킹 셋 크기를 캐시 크기에 맞게 분할
    감소 기법: 데이터 구조 최적화 — SoA(Structure of Arrays) vs AoS(Array of Structures) 선택으로 캐시 활용도 향상
  • C3
    충돌 미스 (Conflict Miss / Collision Miss)
    정의: 충분한 캐시 용량이 있음에도 동일 인덱스(집합)에 매핑되는 블록이 많아 발생하는 미스. 직접 매핑에서 가장 심각
    특징: 완전 연관 캐시에서는 발생하지 않음. 직접 매핑 > n-way > 완전 연관 순으로 감소
    감소 기법: 연관도(Way) 증가 — 1-way → 4-way → 8-way
    감소 기법: Victim Cache — L1과 L2 사이에 소용량 완전 연관 캐시를 두어 L1 충돌 미스 블록 임시 보관
    감소 기법: Way Prediction — 예측 적중 시 직접 매핑 속도 + 적중 실패 시 집합 연관 정확도
Ⅲ. 3C 미스 비교 및 설계 기법 요약
미스 유형발생 원인캐시 크기 증가 시연관도 증가 시핵심 감소 기법
Compulsory최초 접근변화 없음변화 없음프리패칭, 블록 크기 증가
Capacity캐시 용량 부족감소변화 없음캐시 용량 증가, 루프 타일링
Conflict동일 인덱스 충돌소폭 감소급격히 감소연관도 증가, Victim Cache
📌 추가 미스 유형 — Coherence Miss (4C)
멀티프로세서 환경에서 다른 프로세서의 쓰기로 인해 캐시 일관성 프로토콜(MESI 등)에 의해 라인이 무효화(Invalidate)되어 발생하는 미스. 3C에 추가하여 4C 모델이라 부름.
Ⅳ. 결론

3C 분류는 캐시 설계 시 미스 원인별 최적화 방향을 명확히 제시한다. 실제 성능 분석 도구(Intel VTune, perf)를 통해 각 미스 유형별 비율을 측정하고, 지배적 미스 유형에 맞는 최적화 기법을 적용하는 데이터 기반 접근이 중요하다.

Ⅰ. 교체 알고리즘 개요
교체 알고리즘(Replacement Algorithm)은 캐시가 가득 찬 상태에서 새 블록을 적재할 때 기존 라인 중 어느 것을 교체(evict)할지 결정하는 정책이다. 미스율을 최소화하는 것이 목표이며, 현실적 구현 가능성과 하드웨어 복잡도 사이의 트레이드오프가 존재한다.

Ⅰ. 개요 — 캐시 메모리의 필요성
현대 프로세서의 클럭 속도는 DRAM 메모리 속도보다 수백 배 빠르며(Memory Wall 문제), 이 속도 격차를 완화하기 위해 CPU와 DRAM 사이에 캐시 메모리 계층을 배치한다. 캐시 설계의 핵심은 제한된 용량으로 최대 적중률을 달성하는 것이며, 이는 참조의 지역성(Locality of Reference)에 의해 가능해진다.
Ⅱ. 가. 동작 원리와 참조의 지역성
  • 1
    시간적 지역성(Temporal Locality): 최근 참조된 데이터는 가까운 미래에 다시 참조될 가능성이 높다
    예: for문의 루프 카운터 변수, 자주 호출되는 함수 코드
    캐시 활용: 동일 블록을 교체하지 않고 캐시에 유지 → LRU 교체 정책의 근거
  • 2
    공간적 지역성(Spatial Locality): 참조된 주소의 인근 주소도 곧 참조될 가능성이 높다
    예: 배열의 순차적 접근, 순차 실행되는 명령어 코드
    캐시 활용: 단일 워드 대신 블록(캐시 라인) 단위로 데이터를 가져와 공간적 지역성 활용
  • 3
    지역성이 성능에 미치는 영향: 워킹 셋(Working Set)이 캐시에 수용 가능한 크기이면 높은 적중률 달성. 프로그램 실행 시간의 90%가 전체 코드의 10%에서 발생한다는 90:10 법칙이 시간적 지역성의 증거

Ⅰ. 개요 — 쓰기 정책의 중요성
캐시 쓰기 정책은 CPU가 캐시에 데이터를 쓸 때, 주기억장치와의 일관성(Consistency)을 언제·어떻게 유지할지를 결정한다. 쓰기 동작은 읽기보다 복잡한 이유가, 캐시와 메모리 간의 데이터 불일치 문제를 반드시 해결해야 하기 때문이다.
Ⅱ. 가. 쓰기 적중(Write Hit) 정책
  • 1
    Write-Through (즉시 쓰기)
    동작: 캐시에 쓸 때 즉시 주기억장치에도 동시에 쓴다. 캐시와 메모리의 내용이 항상 일치
    장점: 구현 단순. 데이터 일관성 항상 보장. 캐시 교체 시 Write-back 불필요. 전원 장애 시 데이터 손실 없음
    단점: 쓰기마다 느린 메모리 접근 발생 → 쓰기 성능 저하. 버스 대역폭 소모 큼
    활용: L1 캐시에서 L2 캐시로의 쓰기, 임베디드 시스템, 데이터 신뢰성 중시 환경
  • 2
    Write-Back (나중 쓰기)
    동작: 캐시에만 쓰고 메모리 갱신은 보류. 수정된 라인에 Dirty Bit = 1을 표시. 해당 라인이 교체될 때 비로소 메모리에 write-back
    장점: 쓰기 속도 빠름 (메모리 접근 지연 없음). 연속 쓰기 시 메모리 접근 횟수 대폭 감소. 버스 대역폭 절약
    단점: Dirty Bit 하드웨어 필요. 교체 시 메모리 쓰기로 인한 지연 발생. 시스템 충돌 시 미쓰기 데이터 손실 위험. 멀티프로세서 일관성 유지 복잡
    활용: 현대 프로세서 L2·L3 캐시 표준 방식 (쓰기 성능 중시)

Ⅲ. 나. 쓰기 미스(Write Miss) 정책 및 조합
  • 1
    Write-Allocate (쓰기 할당): 쓰기 미스 시 해당 블록을 캐시에 먼저 적재한 후, 캐시에서 수정. 시간적 지역성 활용 (이후 읽기/쓰기 적중 기대)
    Write-Back과 통상 조합: Write-Back + Write-Allocate (현대 L2·L3 표준)
  • 2
    No-Write-Allocate (쓰기 비할당): 쓰기 미스 시 캐시에 적재하지 않고 메모리에 직접 씀. 캐시 오염 방지
    Write-Through와 통상 조합: Write-Through + No-Write-Allocate (L1 또는 단순 시스템)
4가지 조합 정리
① Write-Back + Write-Allocate: 현대 L2·L3의 표준. 쓰기 성능 최고
② Write-Through + No-Write-Allocate: L1 또는 단순 시스템. 구현 단순, 일관성 보장
③ Write-Through + Write-Allocate: 드물게 사용 (일관성 + 적중률 향상 시도)
④ Write-Back + No-Write-Allocate: 스트리밍 쓰기에 활용 (NT-Store 명령어)
Ⅳ. 다. 쓰기 버퍼(Write Buffer)
Write-Through 방식의 메모리 쓰기 지연을 완화하기 위해 캐시와 메모리 사이에 소용량 고속 FIFO 버퍼를 배치한다. CPU는 쓰기 데이터를 버퍼에 넣고 즉시 다음 명령을 실행하며, 버퍼가 백그라운드에서 메모리에 쓰기를 처리한다.

    쓰기 버퍼의 구조적 한계
    버퍼 포화(Write Buffer Full): 쓰기 속도 > 메모리 처리 속도 시 버퍼 가득 찬 상태에서 CPU 정지(Stall) 발생
    RAW 해저드(Read After Write): 버퍼에 있는 주소로 읽기 요청 시 데이터를 메모리가 아닌 버퍼에서 포워딩해야 함 (Write Buffer Forwarding)

    해결 방안
    버퍼 크기 증가: 엔트리 수 확대 (단, 면적·전력 비용 증가)
    쓰기 합병(Write Merging): 동일 블록에 대한 연속 쓰기를 버퍼 내에서 합산하여 메모리 쓰기 횟수 감소
    Write-Back 방식 채택: 메모리 쓰기 빈도 자체를 줄여 버퍼 포화 방지
Ⅴ. 결론

쓰기 정책 선택은 시스템 용도에 따라 달라진다. 데이터 신뢰성과 단순성이 중요한 임베디드 시스템은 Write-Through + No-Write-Allocate를, 쓰기 성능이 중요한 범용 서버·데스크톱 프로세서는 Write-Back + Write-Allocate를 채택한다. 멀티코어 환경에서는 쓰기 정책이 캐시 일관성 프로토콜(MESI)과 상호작용하므로 더욱 신중한 설계가 필요하다.

Ⅲ. 다. MESI 프로토콜 상태 전이
MESI 4가지 상태 정의
M (Modified): 해당 캐시에만 존재. 메모리와 내용 불일치(Dirty). 이 캐시가 메모리 소유권 보유
E (Exclusive): 해당 캐시에만 존재. 메모리와 내용 일치(Clean). 다른 캐시에는 없음

I (Invalid): 캐시 라인이 유효하지 않음(낡은 데이터 또는 미적재 상태)

Ⅳ. 결론

MESI 프로토콜은 현대 멀티코어 프로세서(Intel, AMD)의 표준 캐시 일관성 메커니즘이다. Intel에서는 MESI를 확장한 MESIF(F=Forward), AMD에서는 MOESI(O=Owned) 프로토콜을 채택하여 캐시 간 직접 전달로 메모리 접근을 줄인다. 클라우드 환경의 NUMA 서버에서는 디렉터리 프로토콜과 함께 Inter-NUMA 일관성까지 관리해야 하므로 설계 복잡도가 더욱 높다.

VIPT의 TLB-캐시 병렬 접근 (성능 최적화 핵심) CPU 가상 주소 VA 발행 │ ├─ [인덱스 비트(VA)] ──────→ 캐시 세트 탐색 ─────┐ │ │ 병렬 수행 └─ [VA 전체] → TLB → 물리 주소(PA) → 태그 추출 ──┘ │ 캐시 태그(물리) == PA태그 ? → Hit/Miss ※ 핵심 조건: 페이지 오프셋 비트 ≥ 캐시 인덱스 비트 + 오프셋 비트 → 인덱스 비트가 페이지 오프셋 내에 있으면 VA=PA이므로 동의 문제 없음
Ⅲ. 나. VIVT의 동명·동의 문제 및 해결
  • 1
    동명 문제(Homonym Problem / Aliasing)
    정의: 서로 다른 프로세스가 동일한 가상 주소를 사용하지만 서로 다른 물리 주소를 가리킬 때, 캐시에서 잘못된 데이터를 읽는 문제
    예시: 프로세스A와 프로세스B 모두 VA=0x1000 사용. A는 PA=0x5000, B는 PA=0x6000. A의 데이터가 캐시에 있을 때 B가 VA=0x1000으로 접근하면 A의 데이터를 잘못 읽음
    해결: 문맥 전환(Context Switch) 시 캐시 전체 플러시(Flush). 또는 태그에 ASID(Address Space Identifier) 추가하여 프로세스 구분
  • 2
    동의 문제(Synonym Problem)
    정의: 서로 다른 가상 주소가 동일한 물리 주소를 가리킬 때 (공유 메모리, mmap), 동일 데이터가 캐시에 두 개의 다른 라인에 적재되어 한쪽 수정이 다른 쪽에 반영되지 않는 문제
    예시: VA=0x1000과 VA=0x9000이 모두 PA=0x5000을 가리킴. VA=0x1000으로 수정 시 VA=0x9000 캐시 라인은 낡은 값 유지
    해결: VIPT에서 페이지 오프셋 비트 내에 인덱스 비트를 제한 (캐시 크기 ≤ 페이지 크기 × way 수). 또는 OS가 공유 페이지를 동일 색상(Cache Color)의 물리 페이지에 매핑
Ⅳ. 다. TLB 구조·역할 및 캐시와의 상호작용
TLB(Translation Lookaside Buffer)는 페이지 테이블 접근 속도를 향상시키기 위한 주소 변환(VA→PA) 전용 소용량 완전 연관 캐시이다.

Ⅴ. 결론

현대 프로세서(ARM Cortex-A, x86)는 L1 데이터 캐시에 VIPT를 채택하여 TLB-캐시 병렬 접근으로 성능을 극대화하면서 동의 문제를 설계적으로 방지한다. L2·L3는 PIPT로 정확성을 보장한다. TLB와 캐시의 계층적 협력이 현대 고성능 메모리 시스템의 핵심이며, Spectre·Meltdown 취약점에서 볼 수 있듯이 캐시 부채널 공격 대응도 중요한 보안 설계 요소이다.

Ⅰ. 개요 — 캐시 성능의 3요소
AMAT = 적중 시간 + 미스율 × 미스 패널티. 캐시 성능 최적화는 이 세 가지 요소를 각각 줄이는 방향으로 이루어지며, 하드웨어 기법과 소프트웨어(컴파일러) 기법이 상호 보완적으로 적용된다.

Ⅱ. 가. 하드웨어 기반 최적화
  • 1
    Non-Blocking 캐시 (비정지 캐시, Out-of-Order 캐시)
    원리: 캐시 미스 발생 후 메모리 응답을 기다리는 동안, 후속 메모리 접근 요청을 계속 처리하는 기능 (Hit-Under-Miss). MSHR(Miss Status Holding Register)로 진행 중인 미스를 추적
    효과: 메모리 레이턴시를 은닉(Latency Hiding). 멀티 아웃스탠딩 미스로 메모리 대역폭 활용도 향상
    MSHR 구조: 각 엔트리가 미스 주소·상태·대기 Load 명령 정보 보관. 동일 블록의 중복 미스(2차 미스)는 기존 MSHR 엔트리에 합산
    효과: 비정지 캐시 적용 시 유효 CPI(Clock Per Instruction) 최대 40% 감소 가능
  • 2
    멀티레벨 캐시의 성능 향상 원리
    전역 미스율(Global Miss Rate) = L1 미스율 × L2 미스율 (×L3 미스율...)
    L1: 소용량·고속 → 낮은 적중 시간 우선. L2/L3: 대용량 → 높은 적중률 우선
    Inclusive vs Exclusive 캐시 정책: Inclusive(상위 레벨 데이터가 하위에도 존재)는 일관성 간소화. Exclusive(데이터 중복 없음)는 유효 용량 극대화
Ⅲ. 나. 소프트웨어·컴파일러 기반 최적화
  • 1
    루프 타일링(Loop Tiling / Loop Blocking)
    원리: 큰 행렬을 캐시 크기에 맞는 소블록(Tile)으로 나누어 처리. 워킹 셋을 캐시에 맞게 제한
    // 타일링 전 (행렬 C = A × B, N×N) for(i=0; i<N; i++) for(j=0; j<N; j++) for(k=0; k<N; k++) C[i][j] += A[i][k] * B[k][j]; // B 접근이 열 방향 → 캐시 미스 다수 // 타일링 후 (타일 크기 T) for(ii=0; ii<N; ii+=T) for(jj=0; jj<N; jj+=T) for(kk=0; kk<N; kk+=T) for(i=ii; i<ii+T; i++) for(j=jj; j<jj+T; j++) for(k=kk; k<kk+T; k++) C[i][j] += A[i][k] * B[k][j]; // T²×3 원소가 캐시에 수용 → 캐시 재사용 극대화
  • 2
    루프 교환(Loop Interchange): 내부 루프를 행 우선(Row-Major) 접근으로 변경하여 공간적 지역성 향상
    // 교환 전 (열 우선 접근 → 캐시 미스) for(j=0; j<N; j++) for(i=0; i<N; i++) A[i][j]++; // 교환 후 (행 우선 접근 → 캐시 친화적) for(i=0; i<N; i++) for(j=0; j<N; j++) A[i][j]++;
  • 3
    루프 퓨전(Loop Fusion): 인접한 루프를 하나로 합쳐 동일 데이터를 캐시에 보유한 채 처리. 루프 분열(Loop Fission)의 반대
    워킹 셋이 캐시에 맞을 때 유효. 캐시 재사용(Reuse)을 높여 Capacity Miss 감소
  • 4
    SoA (Structure of Arrays) 데이터 배치: AoS(Array of Structures) 대신 SoA를 사용하여 SIMD 처리와 캐시 효율성 동시 향상
    // AoS: {x,y,z,w, x,y,z,w, ...} → x만 처리 시 y,z,w도 캐시 적재(낭비) struct Particle { float x,y,z,w; } particles[N]; // SoA: {x,x,x,...}, {y,y,y,...} → x 배열 연속 처리 시 캐시 친화적 float px[N], py[N], pz[N], pw[N];
Ⅳ. 다. 캐시 부채널 공격 및 대응 방안
캐시 부채널 공격은 캐시 타이밍 정보를 이용해 다른 프로세스의 비밀 데이터를 간접적으로 유출하는 공격이다.
  • 1
    Flush+Reload 공격
    ① 공격자가 공유 메모리의 특정 캐시 라인을 flush (clflush 명령)
    ② 피해자 프로세스가 실행되며 비밀값에 따라 특정 메모리 접근
    ③ 공격자가 접근 시간을 측정: 빠르면(캐시 Hit) 피해자가 해당 라인 접근했음을 확인 → 비밀값 추론
    Spectre 취약점: 투기적 실행(Speculative Execution) 중 분기 예측 오류로 실행된 코드가 캐시 상태를 변경 → Flush+Reload로 비밀값 유출
  • 2
    Prime+Probe 공격
    ① 공격자가 캐시 집합(Set)을 자신의 데이터로 채움(Prime)
    ② 피해자가 실행되며 캐시 집합에 접근하면 공격자 데이터 교체
    ③ 공격자가 자신의 데이터 접근 시간 측정(Probe): 느리면 피해자가 해당 집합 접근한 것 → 접근 패턴 유출
    Meltdown 취약점: 커널 메모리를 사용자 프로그램이 투기적 실행으로 읽고 캐시에 흔적 남김 → 사이드채널로 커널 데이터 유출
  • 3
    대응 방안
    하드웨어: 투기적 실행 결과의 캐시 적재 제한. Intel의 IBRS/IBPB/STIBP 마이크로코드 패치
    OS: KPTI(Kernel Page Table Isolation) — 커널 페이지 테이블을 사용자 공간에서 완전 분리 (Meltdown 대응)
    컴파일러: Retpoline — 간접 분기를 캐시 스펙터-안전한 트램폴린으로 대체
    소프트웨어: 타이밍 측정 정밀도 저하(RDTSC 제한), 공유 캐시 집합 격리, 시간 분해능 제한
Ⅴ. 결론

캐시 성능 최적화는 하드웨어(Non-Blocking 캐시, 멀티레벨)와 소프트웨어(루프 변환, 데이터 배치)의 협력으로 달성된다. 최근 Spectre·Meltdown 이후 캐시는 성능 이슈뿐 아니라 보안 설계의 핵심 요소가 되었다. 공공 정보시스템 설계 시 고성능 컴퓨팅과 보안의 균형 잡힌 캐시 정책 수립이 필요하며, CPU 마이크로코드·OS 패치·컴파일러 미티게이션을 종합적으로 적용해야 한다.
