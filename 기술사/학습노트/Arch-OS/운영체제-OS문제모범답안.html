프로세스 vs 스레드 비교 PCB·TCB 구조 매회 빈출 기본
프로세스와 스레드의 개념·구조·차이점
⏱ 목표 13분 📄 1~2페이지 💯 10점
Ⅰ. 개념 정의
프로세스 (Process)
운영체제로부터 독립된 메모리 공간(Code·Data·Heap·Stack)을 할당받아 실행 중인 프로그램의 인스턴스. 자원 할당의 기본 단위이며, PCB(Process Control Block)로 상태 관리.
스레드 (Thread)
프로세스 내에서 실행되는 경량 실행 단위(Lightweight Process). 동일 프로세스 내의 Code·Data·Heap 영역을 공유하고, 각자 독립된 Stack과 레지스터를 보유. CPU 스케줄링의 기본 단위.

CPU 스케줄링 알고리즘의 종류를 분류하고, 각각의 특징과 적합 환경을 설명하시오.
선점형 vs 비선점형 평균 대기시간·반환시간 빈출 비교 유형
CPU 스케줄링 알고리즘의 분류·특징·적합 환경
⏱ 목표 13분 📄 1~2페이지 💯 10점
Ⅰ. 개요 및 평가 기준
CPU 스케줄링은 준비(Ready) 큐에 있는 프로세스 중 다음에 CPU를 할당할 프로세스를 결정하는 OS의 핵심 기능이다. 평가 기준: CPU 이용률·처리율(높을수록 좋음), 대기 시간·응답 시간·반환 시간(낮을수록 좋음).
Ⅱ. 알고리즘 분류 및 특징
구분알고리즘원리특징·문제점적합 환경
비선점형
(Non-preemptive)		FCFS도착 순서대로 CPU 할당단순. 호위 효과(Convoy Effect) 발생배치 처리 시스템
SJFCPU 버스트 시간이 짧은 순평균 대기시간 최소(증명됨). 기아(Starvation), 버스트 시간 예측 어려움장기 스케줄링
Priority우선순위 높은 순기아 발생. 에이징(Aging)으로 해결실시간·임베디드
선점형
(Preemptive)		SRT(SRTF)잔여 버스트 시간 가장 짧은 순. 선점 가능SJF의 선점형 버전. 잦은 컨텍스트 스위치시분할 시스템
Round Robin(RR)각 프로세스에 시간 할당량(q) 균등 배분q 값이 관건: 크면 FCFS, 작으면 오버헤드 증가. 응답 시간 균일시분할·대화형
다단계 큐
(MLQ/MLFQ)		우선순위 별 별도 큐. MLFQ는 큐 간 이동 허용복잡하지만 유연. CPU/IO 바운드 분리 처리범용 OS(Linux CFS 유사)
Ⅲ. Round Robin 타임 퀀텀 영향
q→∞ : FCFS와 동일 (컨텍스트 스위치 최소) q→0 : 프로세서 공유(Processor Sharing)에 수렴 (오버헤드 급증) 최적 q : CPU 버스트의 80%를 q 이내에 처리 가능한 수준 (일반적으로 10~100ms)
📌 한 줄 암기
비선점: FCFS·SJF·Priority | 선점: SRT·RR·MLQ | 현대 OS = CFS(Completely Fair Scheduler, 가중치 기반 RR)

내부·외부 단편화 페이징·세그멘테이션 압축(Compaction)

Ⅰ. 단편화 개념 및 종류
구분내부 단편화 (Internal Fragmentation)외부 단편화 (External Fragmentation)
정의할당된 메모리 블록 내부에 사용되지 않는 낭비 공간 발생할당·해제가 반복되며 사용 가능한 메모리가 불연속 작은 조각으로 분산
발생 원인고정 크기 파티션 또는 페이지 단위 할당 시 요청 크기와 할당 크기 불일치가변 크기 파티션에서 프로세스 적재·제거 반복
예시페이지 크기 4KB, 요청 3KB → 1KB 낭비전체 여유 10KB이지만 3+4+3KB로 분산 → 8KB 연속 요청 실패
발생 환경페이징, 고정 분할세그멘테이션, 가변 분할, 동적 할당
Ⅱ. 해결 방법
  • 1
    페이징(Paging): 물리 메모리를 고정 크기 프레임, 논리 메모리를 동일 크기 페이지로 분할. 외부 단편화 제거. 내부 단편화는 소량 발생. 페이지 테이블로 매핑
  • 2
    세그멘테이션(Segmentation): 논리적 의미 단위(코드·데이터·스택)로 가변 분할. 내부 단편화 없음. 외부 단편화 발생 가능
  • 3
    압축(Compaction): 메모리 내 사용 중인 프로세스들을 한쪽으로 집합시켜 연속된 빈 공간 확보. 실행 중 이동 어려움 → 오버헤드 큼
  • 4
    버디 시스템(Buddy System): 2의 거듭제곱 크기 블록으로 분할·합병. 외부 단편화 최소화. Linux 커널 물리 메모리 관리에 사용
  • 5
    슬랩 할당자(Slab Allocator): 자주 사용되는 커널 오브젝트를 미리 할당·재사용. 내부 단편화 최소화. Linux 커널 캐시 메모리 관리

내부 단편화 → 페이징 할당 단위 미사용 공간 | 외부 단편화 → 가변 분할 후 흩어진 빈 공간 | 해결: 페이징·압축·버디

페이지 교체 알고리즘(OPT, FIFO, LRU, LFU)을 비교하고, Belady의 이상 현상을 설명하시오.
페이지 폴트율 비교 Belady의 이상 현상 계산 문제 연계 출제 가능
페이지 교체 알고리즘 비교 및 Belady의 이상 현상
⏱ 목표 13분 📄 1~2페이지 💯 10점
Ⅰ. 페이지 교체 알고리즘 개요
가상 메모리 환경에서 페이지 폴트(Page Fault) 발생 시 새 페이지를 메모리에 적재하기 위해 기존 페이지 중 하나를 제거해야 하는데, 이 때 어떤 페이지를 교체할지 결정하는 알고리즘이다.
Ⅱ. 알고리즘 비교
알고리즘원리페이지 폴트율특징·한계
OPT (최적)앞으로 가장 오래 사용되지 않을 페이지 교체가장 낮음 (이론적 최적)미래 참조 패턴 예측 불가 → 이론적 기준점(Upper Bound)
FIFO메모리에 가장 오래 머문 페이지 교체높음구현 단순. Belady의 이상 현상 발생. 오래된 = 자주 쓰이는 페이지일 수 있음
LRU (최근 최소 사용)가장 오랫동안 참조되지 않은 페이지 교체OPT에 근접스택 알고리즘 → Belady 이상 없음. 구현 비용 높음(카운터·스택 필요)
LFU (최소 사용 빈도)참조 횟수가 가장 적은 페이지 교체중간초기에 많이 쓰인 오래된 페이지가 오래 남는 문제. 구현 복잡
Clock (NRU)참조 비트(R-bit) 순환 검사, 0인 페이지 교체LRU와 유사LRU 근사 알고리즘. 구현 간단. Linux·Windows에서 실제 사용
Ⅲ. Belady의 이상 현상 (Belady's Anomaly)
정의
FIFO 알고리즘에서 프레임 수를 늘렸음에도 불구하고 페이지 폴트 횟수가 오히려 증가하는 현상. 직관에 반하는 이상 현상으로, 스택 알고리즘(LRU·OPT)에서는 발생하지 않음.
참조 열: 1 2 3 4 1 2 5 1 2 3 4 5 프레임 3개 → 페이지 폴트: 9회 프레임 4개 → 페이지 폴트: 10회 ← 프레임 증가에도 폴트 증가! (Belady 이상)
스택 알고리즘(Stack Algorithm)
n 프레임에서 메모리에 있는 페이지 집합 ⊆ (n+1) 프레임에서 메모리에 있는 페이지 집합. 이 조건을 만족하면 프레임 증가 시 폴트 불증가 → LRU, OPT가 해당
📌 한 줄 암기
OPT(이론최적) > LRU > LFU > Clock > FIFO | Belady 이상: FIFO에서만 발생, LRU·OPT는 스택 알고리즘으로 면역

할당 방법원리장점단점예시
연속 할당
(Contiguous)		파일을 연속된 디스크 블록에 저장순차·직접 접근 빠름. 구현 단순외부 단편화. 파일 크기 사전 지정 필요CD-ROM, DVD
연결 할당
(Linked)		각 블록이 다음 블록 포인터 보유. 체인 구조외부 단편화 없음. 파일 크기 동적 증가직접 접근 느림. 포인터 손상 시 전체 손실FAT(File Allocation Table)
인덱스 할당
(Indexed)		인덱스 블록이 실제 데이터 블록 포인터 모두 보유직접 접근 빠름. 외부 단편화 없음인덱스 블록 저장 오버헤드. 소형 파일 낭비UNIX inode, NTFS MFT
inode 구조 (UNIX 파일시스템) inode ├── 직접 포인터 × 12 → 최대 48KB (4KB×12) ├── 단일 간접 포인터 → 4KB × 1024 = 4MB ├── 이중 간접 포인터 → 4MB × 1024 = 4GB └── 삼중 간접 포인터 → 4GB × 1024 = 4TB
📌 한 줄 암기
연속(빠르나 단편화) → 연결(단편화 없으나 직접접근 불가, FAT) → 인덱스(직접접근+단편화 해결, inode)

Ⅰ. 개요 — 교착상태의 개념
교착상태(Deadlock)란 둘 이상의 프로세스가 서로 상대방이 점유한 자원의 해제를 무한히 대기하여 어떤 프로세스도 진행할 수 없는 영구 블로킹 상태이다. E.W. Dijkstra가 1965년 최초 정의하였으며, OS·DBMS·네트워크 등 자원 공유 환경에서 공통적으로 발생한다.
Ⅱ. 가. 교착상태 4가지 필요조건 (Coffman, 1971)
  • 1
    상호 배제(Mutual Exclusion): 자원은 한 번에 하나의 프로세스만 사용 가능. 비공유 자원에 한해 성립
    예: 프린터, 임계 구역, 잠금 장치
  • 2
    점유와 대기(Hold and Wait): 자원을 최소 하나 점유한 상태에서 타 프로세스가 점유한 자원을 추가 요청하며 대기
    해결책: 프로세스 시작 전 필요 자원 모두 선언(All-or-Nothing)
  • 3
    비선점(No Preemption): 프로세스가 사용 중인 자원을 OS가 강제 회수 불가. 자발적 반납만 허용
    해결책: CPU 레지스터·메모리는 선점 가능하도록 설계
  • 4
    환형 대기(Circular Wait): P0→P1→...→Pn→P0 형태의 순환 대기 체인 형성
    해결책: 자원에 전역 순서 번호 부여, 오름차순 요청만 허용
핵심 원리
4가지 조건이 모두 동시에 충족될 때만 교착상태 발생 가능. 예방(Prevention)은 4가지 조건 중 하나를 원천 차단하는 전략.
Ⅲ. 나. 해결 방법 비교 및 은행원 알고리즘
기법전략장점단점대표 방법

은행원 알고리즘 (Banker's Algorithm, Dijkstra 1965)
안전 상태(Safe State): 모든 프로세스가 순차적으로 자원을 할당받고 완료할 수 있는 안전 순서(Safe Sequence)가 존재하는 상태.
자원 요청 시 가상으로 할당 후 안전 상태인지 검사 → 안전하면 실제 할당, 불안전하면 대기.
은행원 알고리즘 예시 (자원 1종류, 프로세스 3개) Available = 3, Allocation = [1, 2, 2], Max = [4, 6, 3] Need = Max - Allocation = [3, 4, 1] ① P2(Need=1) → Available=3 ≥ 1 → 할당 → 완료 후 반납: Available=5 ② P0(Need=3) → Available=5 ≥ 3 → 할당 → 완료 후 반납: Available=9 ③ P1(Need=4) → Available=9 ≥ 4 → 할당 → 완료 안전 순서: P2 → P0 → P1 ✓ → 안전 상태(Safe State)
Ⅳ. 다. RAG 기반 탐지 및 분산 시스템 교착상태
자원 할당 그래프(RAG) 탐지 ○: 프로세스 □: 자원 인스턴스 ○ → □ : 요청 간선 □ → ○ : 할당 간선 교착 탐지 예: P1 ──요청──→ R1 ──할당──→ P2 P2 ──요청──→ R2 ──할당──→ P1 ← 사이클 형성 = 교착상태 ✓ 단일 인스턴스: 사이클 = 교착상태 확정 다중 인스턴스: Wait-for 그래프 변환 + DFS 사이클 검출
  • 1
    분산 교착상태(Distributed Deadlock): 여러 노드에 걸친 자원 요청 사이클. 단일 OS가 전체 상태 파악 불가
    글로벌 Wait-for 그래프 방식: 코디네이터가 각 노드의 로컬 Wait-for 그래프를 수집·합산하여 사이클 탐지
    에지 체이싱(Edge Chasing): 탐침(Probe) 메시지가 Wait-for 간선을 따라 전파, 발신 노드로 되돌아오면 교착 확정 (Chandy-Misra-Haas 알고리즘)
    타임아웃 기반: 일정 시간 응답 없는 자원 요청을 교착으로 가정. 오탐 가능하나 구현 단순
  • 2
    DBMS 교착상태 처리: 트랜잭션 단위 Wait-for 그래프 유지. 사이클 탐지 시 희생 트랜잭션(가장 적은 비용) 롤백. MySQL·Oracle 등 RDBMS에서 기본 구현
Ⅴ. 결론

교착상태는 OS·DBMS·분산 시스템의 근본적 문제로, 실제 시스템은 대부분 탐지+회복 또는 타임아웃 방식을 채택한다. 마이크로서비스 환경에서는 분산 Lock(Redis Redlock, ZooKeeper)과 Saga 패턴을 통해 교착상태를 설계 단계에서 방지하며, 향후 트랜잭션 메모리(Transactional Memory) 기술이 소프트웨어적 교착상태를 하드웨어 수준에서 해결하는 방향으로 발전하고 있다.

Ⅰ. 개요 — 가상 메모리의 개념
가상 메모리(Virtual Memory)는 프로세스가 실제 물리 메모리보다 큰 주소 공간을 사용할 수 있도록 지원하는 메모리 관리 기법이다. 프로세스의 일부만 물리 메모리에 적재하고 나머지는 보조 기억장치(Swap 영역)에 보관함으로써, 멀티프로그래밍 차수를 높이고 메모리 활용을 극대화한다.
Ⅱ. 가. 페이징 기법 및 페이지 테이블 구조
주소 변환 과정 (논리 주소 → 물리 주소) 논리 주소: [페이지 번호 p | 페이지 오프셋 d] │ 페이지 테이블 p → 프레임 번호 f │ 물리 주소: [프레임 번호 f | 오프셋 d] 예) 페이지 크기 4KB(12비트), 논리 주소 32비트 → 페이지 번호: 상위 20비트 (1M 페이지) → 오프셋: 하위 12비트 (4KB 범위)
  • 1
    다단계 페이지 테이블: 페이지 테이블 자체의 메모리 낭비 문제 해결. 32비트 주소 공간에서 1단계 페이지 테이블은 4MB 소요 → 2단계(10/10/12) 또는 3단계 계층화로 실제 사용되는 부분만 할당
    x86-64: 4단계 페이지 테이블 (PML4·PDPT·PD·PT)
    장점: 희소(Sparse) 주소 공간에서 메모리 절약
  • 2
    TLB(Translation Lookaside Buffer): 페이지 테이블 접근 시 메모리 참조 횟수(2회 이상) 오버헤드 해결을 위한 하드웨어 연관 캐시
    TLB Hit: TLB에서 프레임 번호 즉시 획득 → 메모리 1회 접근
    TLB Miss: 페이지 테이블 참조 후 TLB 갱신 → 메모리 2회+ 접근
    유효 접근 시간(EAT) = TLB Hit율 × 1회 접근 + (1-Hit율) × 2회 접근
    컨텍스트 스위치 시 TLB 플러시 또는 ASID(Address Space ID) 태그로 구분
Ⅲ. 나. 요구 페이징 및 페이지 폴트 처리 과정
요구 페이징(Demand Paging)
프로세스의 모든 페이지를 처음부터 적재하지 않고, 실제 참조가 발생할 때만 물리 메모리에 적재하는 지연 적재(Lazy Loading) 전략. 순수 요구 페이징: 초기 메모리 적재 없이 첫 접근 시 폴트 발생.
페이지 폴트 처리 과정 (6단계) ① 프로세스가 논리 주소 참조 → 페이지 테이블의 Valid Bit = 0 확인 ② OS에 페이지 폴트 트랩(Trap) 발생 → OS 인터럽트 처리 ③ 보조 기억장치(Swap) 내 해당 페이지 위치 확인 ④ 빈 프레임 탐색 → 없으면 페이지 교체 알고리즘 수행 ⑤ Swap → 물리 메모리 프레임에 페이지 적재 (I/O 수행) ⑥ 페이지 테이블 갱신(Valid Bit=1, 프레임 번호) → 프로세스 재시작
페이지 폴트 처리 비용
서비스 인터럽트(1~100μs) + 페이지 읽기(8ms) + 프로세스 재시작(1~100μs). 디스크 I/O(8ms)가 지배적 → 페이지 폴트율 최소화가 성능의 핵심.
Ⅳ. 다. 스레싱(Thrashing) 개념·원인·해결
스레싱(Thrashing) 정의
멀티프로그래밍 차수가 높아져 각 프로세스의 워킹 셋(Working Set)을 수용할 물리 메모리가 부족해지면, 프로세스들이 계속적으로 페이지 폴트와 교체를 반복하여 실제 유용한 작업보다 페이지 교체에 더 많은 시간을 소비하는 현상.

해결 방안원리특징
Working Set 모델
(Peter Denning)		각 프로세스의 최근 △ 시간 내 참조 페이지 집합(WS) 합산이 물리 메모리 초과 시 일부 프로세스 일시 중단지역성(Locality)에 기반. 스레싱 예방에 가장 효과적
페이지 폴트 빈도
(PFF) 기법		페이지 폴트율이 상한(Upper)을 초과하면 프레임 추가, 하한(Lower) 미만이면 프레임 회수적응형 제어. 구현 단순. 반응 지연 가능
멀티프로그래밍
차수 조절		스레싱 탐지 시 일부 프로세스를 Swap-out → 나머지에 프레임 재분배근본적 해결. 처리율 일시 감소
지역성 향상프로그램 구조를 지역성(시간적·공간적)에 맞게 재설계예: 배열 순차 접근, 자주 쓰는 함수 집중화
Ⅴ. 결론

가상 메모리는 현대 OS의 핵심 추상화로서 멀티프로그래밍·프로세스 격리·메모리 보호를 동시에 실현한다. TLB·다단계 페이지 테이블은 주소 변환 성능을 보장하며, 스레싱 방지를 위한 Working Set 모델은 실시간 메모리 요구를 동적으로 파악한다. 최근 대용량 메모리(HBM)·CXL 기반 메모리 풀링 환경에서는 Huge Page(2MB·1GB) 적용으로 TLB 미스를 최소화하는 기술도 주목받고 있다.

Ⅰ. 개요
다중 프로세스·스레드가 공유 자원에 동시 접근할 때 실행 순서에 따라 결과가 달라지는 경쟁 조건(Race Condition)이 발생한다. 프로세스 동기화는 공유 자원에 대한 접근을 조율하여 데이터 일관성을 보장하는 메커니즘이다.
Ⅱ. 가. 임계 구역 문제 및 3가지 요구 조건
임계 구역(Critical Section)
둘 이상의 프로세스가 동시에 접근해서는 안 되는 공유 자원 접근 코드 영역. Entry Section(진입 구역) → Critical Section → Exit Section(출구 구역) → Remainder Section 구조.
  • 1
    상호 배제(Mutual Exclusion): 한 프로세스가 임계 구역을 실행 중이면 다른 프로세스는 진입 불가. 가장 기본 조건
  • 2
    진행(Progress): 임계 구역을 실행 중인 프로세스가 없고 진입하려는 프로세스가 있을 때, 진입 결정이 무기한 연기되어서는 안 됨
  • 3
    한정 대기(Bounded Waiting): 프로세스가 임계 구역 진입 요청 후 허용되기 전까지 다른 프로세스가 임계 구역에 진입할 수 있는 횟수에 상한이 있어야 함 (기아 방지)
Ⅲ. 나. 세마포어와 모니터 비교
세마포어(Semaphore) P/V 연산 P(S) 연산 (Wait): V(S) 연산 (Signal): if S > 0: S = S + 1 S = S - 1 if (대기 프로세스 있음): else: 대기 프로세스 1개 깨움 프로세스 Block 이진 세마포어 (Binary Semaphore, Mutex): S ∈ {0, 1} 계수 세마포어 (Counting Semaphore): S ∈ 정수 전체
비교 항목세마포어 (Semaphore)모니터 (Monitor)

Ⅳ. 다. 생산자-소비자 문제 세마포어 해결
문제 정의
생산자(Producer)는 공유 버퍼에 아이템을 생산하고, 소비자(Consumer)는 아이템을 소비. 버퍼 오버플로우(생산자)·언더플로우(소비자) 방지 및 상호 배제 필요.
세마포어를 이용한 생산자-소비자 해결 세마포어 3개 정의: mutex = 1 (상호 배제: 버퍼 접근 제어) empty = N (빈 버퍼 슬롯 수, N = 버퍼 크기) full = 0 (채워진 버퍼 슬롯 수) 생산자 프로세스: 소비자 프로세스: while(true) { while(true) { item = produce(); P(full); // 아이템 있을 때까지 대기 P(empty); // 빈 슬롯 대기 P(mutex); // 임계 구역 진입 P(mutex); // 임계 구역 진입 item = remove_from_buffer(); add_to_buffer(item); V(mutex); // 임계 구역 퇴출 V(mutex); // 임계 구역 퇴출 V(empty); // 빈 슬롯 1 증가 V(full); // 채운 슬롯 1 증가 consume(item); } } 핵심: P(empty)/P(full) 순서가 P(mutex)보다 반드시 먼저! (순서 역전 시 교착상태 발생)
Ⅴ. 결론

동기화는 멀티코어 환경에서 더욱 중요성이 증가하였으며, 세마포어·모니터 외에도 원자적 연산(Compare-and-Swap, CAS) 기반의 Lock-Free·Wait-Free 알고리즘이 현대 언어 런타임(Java Concurrent 패키지, C++ atomic)에 적용되고 있다. 마이크로서비스 환경에서는 분산 잠금(Redis Redlock, ZooKeeper) 메커니즘이 프로세스 동기화의 확장 형태로 활용된다.

Ⅲ. 다. 캐시 일관성(Cache Coherency) 및 MESI 프로토콜
캐시 일관성 문제
멀티코어에서 각 코어가 독립 L1·L2 캐시를 보유할 때, 공유 데이터를 서로 다른 캐시가 각각 다른 버전으로 보유할 수 있는 문제. 한 코어의 쓰기가 다른 코어의 캐시에 즉시 반영되지 않으면 데이터 불일치 발생.
MESI 프로토콜 4가지 상태 M (Modified) : 캐시에만 존재, 메모리와 다름 (Dirty). 유일한 최신 복사본 E (Exclusive) : 캐시에만 존재, 메모리와 동일 (Clean). 타 캐시 복사본 없음 S (Shared) : 여러 캐시에 동시 존재. 메모리와 동일 (Read-Only) I (Invalid) : 캐시 라인 무효. 접근 시 캐시 미스 발생 상태 전이 예: Core0이 데이터 쓰기 → 자신은 M, 타 코어에 Invalidate 신호 Core1이 같은 주소 읽기 → Core0이 M→S로 전환 + 메모리 갱신 + Core1 E 또는 S
메모리 모델특징예시
Sequential Consistency
(SC)		모든 코어가 모든 메모리 연산을 동일 순서로 관찰. 가장 강한 보장이론적 이상. 성능 저하로 현실에서 미채택
Total Store Order
(TSO)		쓰기는 Write Buffer에 먼저 저장, 다른 코어 관찰 지연 허용x86 아키텍처. 비교적 강한 모델
Release Consistency
(RC)		Acquire/Release 연산으로 동기화 경계 표시. 사이 연산은 재정렬 허용ARM, C++ memory_order, Java volatile
메모리 장벽(Memory Barrier / Fence)
프로그래머가 명시적으로 메모리 순서를 강제하는 명령. Load/Store Fence로 재정렬(Reordering)을 차단하여 멀티코어 동기화 보장. Java synchronized·volatile, C++ std::atomic이 내부적으로 메모리 장벽 삽입.
Ⅴ. 결론

멀티코어 OS 설계는 부하 균형·캐시 친화성·메모리 일관성이 복잡하게 얽힌 문제이다. 현대 Linux 커널은 CFS(Completely Fair Scheduler)와 NUMA 인식 스케줄링으로 이를 해결하며, 하드웨어는 MESI 프로토콜·하드웨어 프리패처로 캐시 효율을 극대화한다. AI 학습 워크로드를 위한 수천 코어 환경(GPU, NPU)에서는 NUMA 토폴로지 인식 태스크 배치와 NVLink·CXL 캐시 코히런시 확장이 중요한 연구 과제로 부상하고 있다.
