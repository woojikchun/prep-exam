Ⅰ. 프로그램 I/O(폴링)의 개념 및 동작 원리
프로그램 I/O(Programmed I/O)는 CPU가 I/O 장치의 상태 레지스터(Status Register)를 반복적으로 읽어 I/O 완료 여부를 직접 확인하는 방식이다. I/O 제어의 모든 과정을 CPU가 담당하므로 가장 단순하지만, CPU 자원을 낭비하는 비효율적인 방식이다.
프로그램 I/O 폴링 루프 동작 흐름 1. CPU가 I/O 컨트롤러의 제어 레지스터에 I/O 명령 기록 (예: WRITE command, 대상 주소, 데이터) 2. [폴링 루프 시작] LOOP: Status ← I/O_CONTROLLER.STATUS_REG ← 상태 레지스터 반복 읽기 IF Status == BUSY THEN GOTO LOOP ← 아직 준비 안 됨 → 재확인 IF Status == READY THEN ← I/O 장치 준비 완료 DATA_REG ← data[i] ← 데이터 전송 (CPU → I/O 컨트롤러) i++ IF (모든 데이터 전송 완료) THEN BREAK ELSE GOTO LOOP 3. I/O 완료 → CPU는 다음 작업 수행
Ⅱ. CPU 오버헤드 계산
📐 폴링 오버헤드 계산식
폴링 오버헤드(%) = (폴링 빈도 × 폴링 1회 소요 CPU 사이클) / CPU 총 사이클 × 100

예시: 마우스 폴링 (100Hz), 폴링 1회 = 400 CPU 사이클, CPU 500MHz
→ 초당 폴링 횟수: 100회
→ 초당 폴링 소요 사이클: 100 × 400 = 40,000 사이클
→ CPU 총 사이클/초: 500,000,000
→ 오버헤드 = 40,000 / 500,000,000 × 100 ≈ 0.008% (무시 가능)

예시 2: 네트워크 폴링 (10,000Hz = 10KHz), 폴링 1회 = 400 사이클
→ 10,000 × 400 / 500,000,000 × 100 = 0.8% (허용 범위)

단, I/O 장치 응답 지연이 길수록 폴링 루프 대기 시간 증가 → CPU 낭비 심각
Ⅲ. Busy-Wait의 문제점
  • 1
    CPU 자원 낭비: I/O 장치 응답 대기 중에도 CPU가 폴링 루프를 계속 실행. 다른 프로세스가 CPU를 활용 불가 → 시스템 처리율 저하
  • 2
    에너지 소비 증가: CPU가 유휴 상태로 진입할 수 없어 불필요한 전력 소비. 배터리 기기(모바일·임베디드)에서 치명적
  • 3
    우선순위 역전 가능성: 폴링 루프 실행 중 높은 우선순위 프로세스가 스케줄링되지 못할 수 있음
  • 4
    멀티태스킹 저해: OS 스케줄러가 폴링 중인 프로세스를 선점하더라도, 시스템 전체 CPU 사이클이 낭비됨

Ⅰ. 인터럽트 구동 I/O의 개념 및 동작 원리
인터럽트 구동 I/O는 CPU가 I/O 명령을 내린 후 다른 작업을 계속 수행하다가, I/O 장치가 준비되면 하드웨어 인터럽트 신호를 발생시켜 CPU에 통보하는 방식이다. 폴링(Busy-Wait)의 CPU 낭비 문제를 해결하고, CPU와 I/O 장치의 병렬 동작을 가능하게 한다.
Ⅱ. 인터럽트 처리 절차 (7단계)
인터럽트 처리 전체 흐름 [1] I/O 명령 발행 CPU → I/O 컨트롤러: I/O 시작 명령 기록 후 CPU는 다른 프로세스 실행 계속 [2] 인터럽트 신호 발생 I/O 장치 완료 → I/O 컨트롤러 → CPU의 INT 핀에 인터럽트 신호 전송 [3] 현재 명령어 완료 CPU는 현재 실행 중인 명령어(Instruction)를 완료 후 인터럽트 수락 [4] 컨텍스트 저장 (Context Save) ← 핵심 단계 PC(Program Counter), PSW(Program Status Word), 범용 레지스터(R0~Rn) → 스택(Stack) 또는 프로세스 제어 블록(PCB)에 저장 (인터럽트된 프로세스의 정확한 재개를 위한 스냅숏) [5] 인터럽트 서비스 루틴(ISR) 실행 인터럽트 벡터 테이블(IVT) 참조 → 해당 ISR 주소로 PC 이동 ISR: I/O 데이터 읽기, 버퍼 이동, 완료 처리, 오류 처리 [6] 컨텍스트 복원 (Context Restore) 저장된 PC, PSW, 레지스터 값 → 복원 → 인터럽트 발생 시점의 정확한 상태로 복귀 [7] 인터럽트된 프로세스 재개 CPU는 인터럽트 이전 명령의 다음 명령부터 실행 재개
Ⅲ. 인터럽트 벡터 테이블 (IVT)
인터럽트 벡터 테이블(Interrupt Vector Table)은 인터럽트 번호(Vector Number)와 해당 ISR(인터럽트 서비스 루틴) 주소를 매핑하는 테이블이다. 메모리의 고정 위치(x86: 물리 주소 0번지)에 위치하며, CPU는 인터럽트 번호를 인덱스로 IVT를 참조하여 ISR 주소를 획득한다.
인터럽트 벡터 테이블 구조 (x86 예시) 메모리 주소 0x00: Vector 0 → ISR 주소 (Divide by Zero 핸들러) 메모리 주소 0x04: Vector 1 → ISR 주소 (Debug 예외 핸들러) ... 메모리 주소 0x21: Vector 33 → ISR 주소 (키보드 인터럽트 핸들러) ← IRQ1 메모리 주소 0x25: Vector 37 → ISR 주소 (HDD 컨트롤러 핸들러) ← IRQ14 ... (256개 엔트리, 각 4바이트(세그먼트:오프셋) = 총 1024바이트) 흐름: 인터럽트 발생(IRQ번호) → PIC(우선순위 인터럽트 컨트롤러) → CPU에 Vector 번호 전달 → IVT[Vector] → ISR 주소 로드 → ISR 실행

Ⅴ. 결론

인터럽트 구동 I/O는 현대 OS(Windows·Linux)의 기본 I/O 처리 방식이다. Context Save/Restore의 오버헤드를 최소화하기 위해 현대 CPU는 하드웨어 레지스터 저장·복원을 일부 자동화하며, 멀티코어 시스템에서는 APIC(Advanced PIC)를 통해 인터럽트를 여러 코어에 분산 처리(Interrupt Affinity)하여 성능을 극대화한다.

DMA(Direct Memory Access)는 전용 DMA 컨트롤러 하드웨어가 CPU의 개입 없이 I/O 장치와 메인 메모리 사이에서 직접 대용량 데이터를 전송하는 방식이다. 인터럽트 구동 방식에서 대용량 전송 시 반복되는 ISR 오버헤드와 CPU 부하를 해결하기 위해 도입되었다.

Ⅳ. 캐시 일관성(Cache Coherency) 문제
문제 발생 원인: DMA는 메인 메모리를 직접 수정하지만, CPU의 캐시는 이 변경을 즉시 반영하지 못한다.
⚠️ DMA 쓰기 시 (Device → Memory)

DMA가 메모리를 새 데이터로 갱신했으나, CPU 캐시에 구 데이터가 남아있음 → CPU가 캐시에서 오래된(Stale) 데이터 읽는 문제

⚠️ DMA 읽기 시 (Memory → Device)

CPU 캐시에만 존재하는 수정된 데이터(Dirty Cache Line)가 메모리에 반영 안 된 채로 DMA가 메모리에서 구 데이터를 읽음 → 오래된 데이터를 장치에 전송

Ⅰ. 채널 I/O의 개념 및 등장 배경
채널 I/O는 IOP(I/O Processor, 입출력 프로세서)라 불리는 독립적인 전용 프로세서가 CPU로부터 I/O 작업을 전담하는 방식이다. IOP는 자체 명령어 집합(채널 프로그램)을 실행하여 복잡한 I/O 시퀀스를 완전히 자율적으로 처리한다. DMA가 단순 블록 전송만 담당하는 데 반해, 채널은 완전한 I/O 서브시스템으로 동작한다.

Ⅰ. 개요 — I/O 제어 방식 발전의 핵심 목표
컴퓨터 시스템에서 CPU(GHz 클럭)와 I/O 장치(μs~ms 응답) 간 처리 속도 불균형은 시스템 전체 성능의 병목이다. I/O 제어 방식의 발전은 "CPU를 I/O 대기로부터 해방시켜 연산에 집중시키는 것"을 목표로 진행되었으며, 이는 CPU 이용률 극대화 → 시스템 처리량(Throughput) 향상으로 이어진다.
Ⅱ. 가. 발전 단계별 등장 배경과 해결한 한계
  • 1
    프로그램 I/O(Programmed I/O) — 1950년대 초기 컴퓨터
    배경: 초기 컴퓨터는 단순한 순차 실행 방식. CPU가 모든 I/O를 직접 처리하는 것이 자연스러운 구조
    동작: CPU가 I/O 상태 레지스터를 반복 폴링(Busy-Wait). 준비 시 CPU가 직접 데이터 전송
    한계: CPU가 I/O 대기 중 100% 점유 → 다른 연산 불가. I/O 속도가 곧 시스템 속도 상한
  • 2
    인터럽트 구동 I/O — 1960년대 멀티프로그래밍 등장
    해결한 한계: 폴링의 Busy-Wait(CPU 낭비) 제거. CPU는 I/O 명령 후 다른 프로세스 실행
    새로운 한계: 대용량 전송 시 수천~수만 회 반복 인터럽트 발생 → ISR 오버헤드 누적. 인터럽트 폭풍(Interrupt Storm)
    정량: 4KB 전송 시 1바이트마다 인터럽트 → 4,096회 ISR 실행. ISR 1회 = 수백 CPU 사이클 → 수십만 사이클 낭비
  • 3
    DMA(Direct Memory Access) — 1970년대 디스크·테이프 보편화
    해결한 한계: 대용량 전송의 반복 인터럽트 제거. DMA 컨트롤러가 블록 전체를 CPU 없이 전송. 전송 완료 시 1회만 인터럽트
    새로운 한계: 단순 블록 전송만 가능. 복잡한 I/O 시퀀스(탐색→읽기→오류처리)는 여전히 CPU 개입 필요. 캐시 일관성 문제
  • 4
    채널 I/O(Channel I/O) — 1960~70년대 IBM 메인프레임
    해결한 한계: DMA의 단순 블록 전송 한계 극복. IOP(I/O Processor)가 완전한 채널 프로그램을 실행. 오류 처리·재시도·복잡한 I/O 시퀀스를 CPU 없이 자율 처리
    효과: CPU는 SIO 명령 하나로 복잡한 I/O 시퀀스 위임 → 완전한 CPU-I/O 병렬 처리 실현

Ⅳ. 다. 현대 DMA 확장 기술
  • 1
    스캐터-개더 DMA (Scatter-Gather DMA)
    개념: 물리적으로 불연속한 여러 메모리 버퍼를 DMA Descriptor 링크 리스트로 기술하여 단일 DMA 오퍼레이션으로 처리하는 기술
    구조: DMA Descriptor = {버퍼 주소, 크기, 다음 Descriptor 포인터}의 연결 리스트. DMA 컨트롤러가 리스트를 순회하며 각 버퍼에 전송
    효과: 가상 메모리 환경에서 페이지가 분산되어도 단일 I/O 오퍼레이션으로 처리. 메모리 복사(Memcpy) 제거 → Zero-Copy I/O 실현
    적용: Linux DMA API(dma_map_sg()), NVMe SG 리스트, TCP/IP 네트워크 스택
  • 2
    IOMMU (I/O Memory Management Unit)
    개념: I/O 장치의 DMA 접근을 가상화·제어하는 하드웨어. I/O 장치가 사용하는 가상 주소(IOVA)를 물리 주소로 변환하고 접근 권한을 검사
    보안 역할: 악성 I/O 장치(DMA 공격)가 임의 메모리 주소에 접근하는 것을 차단 (DMA Remapping). PCIe 장치별 접근 가능 메모리 영역 격리
    가상화 역할: SR-IOV와 결합하여 VM에 물리 I/O 장치 직접 할당(VFIO). 게스트 OS의 DMA를 안전하게 호스트 물리 메모리에 매핑
    대표 구현: Intel VT-d (Virtualization Technology for Directed I/O), AMD-Vi, ARM SMMU
  • 3
    PCIe DMA — 현대 고속 I/O의 표준
    개념: PCIe 장치(NVMe·GPU·NIC)가 PCIe 버스를 통해 호스트 메모리에 직접 DMA하는 방식. CPU를 완전히 배제하고 고대역폭 전송
    RDMA(Remote DMA): 네트워크를 통해 원격 호스트 메모리에 직접 DMA. CPU·OS 개입 없이 메모리 간 전송. InfiniBand·RoCE 프로토콜. 고성능 컴퓨팅(HPC)·AI 클러스터에 필수
    NVMe DMA: NVMe 컨트롤러가 PCIe DMA로 호스트 메모리의 I/O 큐(Submission/Completion Queue) 직접 접근. 64개 큐 병렬 처리 → 수백만 IOPS 달성
Ⅴ. 결론

DMA는 단순한 블록 전송 기술을 넘어, Scatter-Gather로 Zero-Copy I/O를, IOMMU로 보안·가상화를, PCIe DMA·RDMA로 초고속 데이터 전송을 실현하는 현대 컴퓨팅의 핵심 기반 기술로 발전하였다. CXL(Compute Express Link) 등 차세대 인터커넥트는 DMA의 개념을 CPU-GPU-메모리 간으로 확장하여 이기종 컴퓨팅 아키텍처를 지원한다.

📐 I/O 성능 핵심 지표 및 계산
IOPS(I/O Operations Per Second) = 1초 / 평균 I/O 응답시간
HDD 예시: 탐색시간 5ms + 회전지연 4ms + 전송 1ms = 10ms → IOPS = 1/0.01 = 100 IOPS
NVMe SSD: 평균 응답시간 0.1ms → IOPS = 1/0.0001 = 10,000 IOPS (10K IOPS)

처리량(Throughput) = IOPS × 블록 크기
HDD: 100 IOPS × 512KB = 51.2MB/s  |  NVMe: 500K IOPS × 4KB = 2GB/s

큐 깊이(Queue Depth): 동시에 처리 중인 I/O 요청 수. HDD: QD=32 권장. NVMe: QD=1024까지 지원
지연시간(Latency): 요청 발행 → 완료까지 시간. HDD ~10ms, SATA SSD ~0.1ms, NVMe ~0.02ms

방식원리성능구현 복잡도장단점
에뮬레이션
(Emulation)		하이퍼바이저(VMM)가 가상 장치를 소프트웨어로 완전 에뮬레이션. 게스트 OS는 가상 장치용 표준 드라이버 사용(e1000 NIC·IDE 디스크)매우 낮음
(소프트웨어 처리)		낮음기존 OS 수정 불필요. QEMU emulation. 하이퍼바이저 CPU 부하 높음
반가상화
(Para-virtualization)		게스트 OS에 수정된 드라이버(VirtIO) 설치. 하이퍼콜(Hypercall)로 하이퍼바이저와 직접 통신. 에뮬레이션 오버헤드 제거높음
(에뮬레이션 대비 3~5×)		중간게스트 OS 드라이버 수정 필요. VirtIO-blk·VirtIO-net. KVM/QEMU 표준. 오픈 클라우드(OpenStack) 기본
SR-IOV
(Single Root I/O Virtualization)		물리 PCIe 장치를 하드웨어에서 다수의 가상 함수(VF)로 분할. 각 VM이 VF를 직접 사용(하이퍼바이저 우회). IOMMU로 메모리 접근 격리최고
(베어메탈과 동등)		높음하이퍼바이저 개입 없음 → 초저지연. DPDK와 SR-IOV 조합으로 패킷 처리 최적화. 고성능 클라우드(AWS ENA, Azure SR-IOV)

Ⅳ. 다. 차세대 I/O 기술
  • 1
    NVMe-oF (NVMe over Fabrics)
    개념: NVMe 프로토콜을 네트워크 패브릭(InfiniBand·RoCEv2·TCP)을 통해 원격 스토리지에 적용. 로컬 NVMe와 동등한 성능의 원격 스토리지 접근
    원리: 호스트 NVMe 드라이버 → NVMe-oF 초기자 → 네트워크(RDMA/TCP) → NVMe-oF 대상(Target) → 원격 NVMe SSD
    효과: 스토리지 풀링(Storage Pooling). 지연시간 <100μs(RDMA 기반). 분리형 스토리지 아키텍처(Storage Disaggregation). 클라우드 스토리지 기반
  • 2
    CXL (Compute Express Link, 인텔 2019~)
    개념: PCIe 물리 계층 기반의 고속 인터커넥트. CPU와 GPU·FPGA·메모리 확장 장치 간에 캐시 일관성(Cache Coherency)을 하드웨어에서 자동 보장하는 차세대 인터페이스
    핵심 기능: CXL.io(PCIe DMA), CXL.cache(CPU-가속기 캐시 공유), CXL.mem(CPU가 가속기 메모리를 직접 주소 지정)
    효과: 메모리 풀링(Memory Pooling). CPU-GPU 간 데이터 복사 제거. 이기종 컴퓨팅(HPC·AI 추론) 성능 획기적 향상
  • 3
    GPU DMA — CUDA Unified Memory (통합 메모리)
    개념: CPU 메모리와 GPU 메모리를 단일 가상 주소 공간으로 통합. 개발자가 명시적 데이터 복사(cudaMemcpy) 없이 CPU-GPU 공유 데이터 접근
    원리: PCIe DMA로 페이지 폴트 시 자동 마이그레이션. NVIDIA NVLink(600GB/s)·CXL로 CPU-GPU 메모리 통합 가속
    효과: AI/딥러닝 모델 학습 코드 단순화. 대형 LLM(Large Language Model) 가중치를 CPU 메모리에 오프로드하여 GPU VRAM 한계 극복

Ⅴ. 결론

I/O 제어 방식은 폴링→인터럽트→DMA→채널의 발전을 거쳐 현대 OS의 계층화된 I/O 스택에 통합되었다. 가상화 환경에서는 SR-IOV로 하이퍼바이저 오버헤드를 제거하고, 차세대 CXL·NVMe-oF·GPU DMA는 CPU·스토리지·가속기 간의 경계를 허물어 이기종 컴퓨팅의 I/O 병목을 근본적으로 해결하는 방향으로 발전하고 있다.
