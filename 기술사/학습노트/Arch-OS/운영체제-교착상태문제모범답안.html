Ⅰ. 교착상태의 개념
교착상태(Deadlock)는 두 개 이상의 프로세스(또는 스레드)가 각자 서로 상대방이 점유한 자원을 무한정 대기하면서 어떤 프로세스도 실행을 계속할 수 없는 상태를 말한다. 1965년 Dijkstra가 처음 공식화하였으며, 1971년 Coffman 외 3인이 4가지 필요조건을 정의하였다.
Ⅱ. 4가지 필요조건 (Coffman, 1971)
⚠️ 핵심 원칙 — 4가지 조건이 동시에 성립할 때만 교착상태 발생
하나라도 부정되면 교착상태는 발생하지 않는다 → 예방(Prevention)의 이론적 근거

① 상호배제 부정자원 공유읽기 전용·공유 가능 자원으로 전환 (스풀링, 읽기 락)임계 구역 자원(프린터 등)은 원천적 공유 불가. 적용 범위 극히 제한
② 점유와 대기 부정일괄 요청 또는 점유 해제 후 요청① 시작 전 모든 필요 자원을 한 번에 요청
② 새 자원 필요 시 기존 자원 모두 반납 후 재요청		자원 낭비(사용 안 해도 독점), 기아(Starvation) 발생 가능, 프로세스가 필요 자원 사전 예측 어려움
③ 비선점 부정자원 선점 허용높은 우선순위 프로세스가 요청 시 낮은 우선순위 프로세스의 자원 강제 회수CPU·메모리는 가능하나 프린터·DB 트랜잭션 등 중간 상태 저장 불가 자원에 적용 불가. 롤백 비용 발생
④ 환형 대기 부정자원 순서 번호 부여모든 자원에 전역 순서(번호)를 부여. 프로세스는 오름차순으로만 자원 요청 허용자원 번호 관리 복잡. 실제로 높은 번호 자원이 먼저 필요한 프로세스는 낮은 번호 자원을 불필요하게 유지해야 함
Ⅳ. 예방과 회피의 비교
예방(Prevention) vs 회피(Avoidance) — 핵심 차이
예방: 교착상태의 4가지 조건 중 하나를 원천 차단 → 교착상태 자체가 불가능한 시스템 설계. 과도한 자원 제약으로 자원 이용률 저하
회피: 4가지 조건은 허용하되, 자원 할당 전 안전 상태(Safe State) 여부를 검사하여 불안전 상태로의 진입 차단 → 은행원 알고리즘
Ⅴ. 결론 — 감리 포인트

교착상태 예방은 이론적으로 교착상태를 원천 차단하지만, 자원 낭비·기아·시스템 처리량 감소 등 부작용이 크다. 실제 운영체제(Linux, Windows)는 예방 대신 탐지+회복 방식을 채택하고, DBMS는 Wait-for 그래프 기반 탐지와 타임아웃 기반 롤백을 주로 사용한다. 감리 시에는 교착상태 방지 정책이 시스템 설계 문서에 명시되어 있는지 확인한다.

Ⅰ. 은행원 알고리즘의 개념
은행원 알고리즘(Banker's Algorithm)은 1965년 Dijkstra가 제안한 교착상태 회피(Avoidance) 알고리즘이다. 이름의 유래는 은행이 대출 시 전체 고객이 최대 대출을 요청해도 지급 가능한지 사전 검토하는 방식에서 착안하였다. 자원 할당 전에 시스템이 안전 상태(Safe State)를 유지하는지 검사하여, 불안전 상태로 전이될 경우 할당을 거부한다.
안전 상태(Safe State)의 정의
시스템이 교착상태에 빠지지 않고 모든 프로세스를 순차적으로 완료할 수 있는 안전 순서열(Safe Sequence)이 존재하는 상태.
안전 순서열: <P1, P2, ..., Pn>에서 각 Pi가 필요한 최대 자원 = 현재 가용 자원 + P1~Pi-1이 반납한 자원으로 충족 가능
Ⅱ. 알고리즘 구성 요소 (자료구조)
변수의미크기
Available[j]현재 가용 자원량 (자원 유형 j의 남은 개수)1×m (m=자원 유형 수)
Max[i][j]프로세스 i가 요구하는 자원 j의 최대량n×m (n=프로세스 수)
Allocation[i][j]프로세스 i에 현재 할당된 자원 j의 수n×m
Need[i][j]프로세스 i가 앞으로 필요한 자원 j의 수
= Max[i][j] − Allocation[i][j]		n×m

Ⅰ. 자원 할당 그래프(RAG) 개념 및 구성 요소
자원 할당 그래프(Resource Allocation Graph)는 프로세스와 자원 간의 할당·요청 관계를 방향 그래프로 표현한 것으로, 교착상태의 존재 여부를 시각적으로 탐지하는 도구이다.

Ⅲ. Wait-for 그래프 — RAG의 간략 변환
Wait-for 그래프는 RAG에서 자원 노드를 제거하고, 프로세스 간 대기 관계만 남긴 단순화된 그래프이다. 단일 인스턴스 자원 시스템에서만 유효하게 적용된다.
RAG → Wait-for 그래프 변환 [RAG] [Wait-for 그래프] P1 → R1 → P2 P1 → P2 P2 → R2 → P1 →→→ P2 → P1 P3 → R1 (대기) P3 → P2 (R1 보유자) Wait-for 그래프에서 사이클(P1↔P2) 탐지 → P1, P2 교착상태 확정 → P3는 교착상태 프로세스(P2)를 기다리므로 연쇄 영향 탐지 주기: OS는 주기적(수 초 단위)으로 Wait-for 그래프 사이클 검사

(Abort One by One)		교착상태 해소될 때까지 프로세스를 하나씩 선택·종료. 매번 사이클 재탐지최소한의 프로세스만 종료. 피해 최소화사이클 탐지를 반복 수행해야 하므로 오버헤드 발생
Ⅲ. 회복 방법 2 — 자원 선점 (Resource Preemption)
교착상태에 관련된 프로세스로부터 자원을 강제로 빼앗아 다른 프로세스에 할당하는 방법이다. 세 가지 핵심 이슈를 해결해야 한다.
  • 1
    희생자 선택 (Victim Selection)
    비용 최소화 원칙: 아래 기준으로 희생자 프로세스를 선정
    ① 지금까지 사용한 자원의 양 (적을수록 손실 적음)
    ② 프로세스의 우선순위 (낮을수록 희생자 선호)
    ③ 프로세스 실행 시간 (짧을수록 재실행 비용 적음)
    ④ 추가로 필요한 자원의 양
    ⑤ 교착상태에 관련된 프로세스의 수 (선점으로 해소 가능한 수)
  • 2
    롤백 (Rollback): 자원을 선점당한 프로세스는 안전한 이전 상태(Checkpoint)로 롤백. 선점된 자원을 반납하고 후에 재실행
    전체 롤백: 프로세스를 완전히 종료 후 재시작 (단순하나 비용 큼)
    부분 롤백: 교착상태 해소에 필요한 최소한만 롤백 (체크포인트 활용)
  • 3
    기아(Starvation) 방지: 같은 프로세스가 반복적으로 희생자로 선택될 경우 무한히 진행 불가 상태에 빠짐
    Age 기법: 희생자 선택 횟수(Count)를 비용 함수에 포함. 횟수가 많을수록 선택 비용 증가 → 다른 프로세스 선택 유도
    대기 시간 기반 우선순위: 오래 대기한 프로세스일수록 높은 우선순위 부여

Ⅴ. 결론

교착상태 회복은 시스템의 가용성(Availability) 회복과 데이터 무결성 보존 사이의 균형을 요구한다. DBMS(Oracle, MySQL)는 트랜잭션 롤백(자원 선점) 방식을 표준으로 채택하며, 가장 최근에 시작한·가장 변경이 적은 트랜잭션을 희생자로 자동 선택한다. 감리 시에는 교착상태 발생 이력, 롤백 정책, 모니터링 경보 설정 여부를 점검한다.

Ⅱ. 라이브락 상세 설명 및 해결
라이브락은 두 사람이 좁은 복도에서 서로 비켜주려다가 같은 방향으로 움직여 결국 아무도 통과하지 못하는 상황에 비유할 수 있다. 프로세스는 실행 중이나 의미 있는 작업을 수행하지 못한다.
라이브락 발생 예시 (스핀락 환경) P1: 자원A 획득 시도 → P2가 점유 중 → 양보(Release) → 재시도 P2: 자원B 획득 시도 → P1이 점유 중 → 양보(Release) → 재시도 (P1, P2 동시에 양보·재시도를 무한 반복 → 진전 없음) 해결: Exponential Backoff (이더넷 CSMA/CD 충돌 회피) - 1차 충돌: 0~1 랜덤 대기 - 2차 충돌: 0~3 랜덤 대기 - n차 충돌: 0~(2^n-1) 랜덤 대기 → 충돌 확률 지수적 감소
Ⅲ. 기아 상세 설명 및 Age 기법
Age (에이징) 기법 — 기아 방지
프로세스가 대기 중인 시간에 비례하여 우선순위를 점진적으로 높이는 기법.
예: 대기 매 1초마다 우선순위 +1 증가 → 낮은 우선순위 프로세스도 결국 최고 우선순위에 도달하여 자원 획득 보장.
적용: Linux의 CFS(완전 공정 스케줄러)에서 vruntime 기반 에이징 구현

Ⅱ. 가. 4가지 전략의 개념과 동작 원리
  • 1
    예방(Prevention): Coffman 4가지 필요조건 중 하나 이상을 시스템 설계 단계에서 원천 차단
    상호배제 부정: 자원 공유화 (읽기 전용 자원에 한해 적용 가능)
    점유대기 부정: 자원 일괄 요청 또는 보유 자원 해제 후 재요청
    비선점 부정: 자원 강제 선점 허용 (CPU·메모리 등 상태 저장 가능한 자원에 한정)
    환형대기 부정: 자원에 전역 번호 부여, 오름차순 요청 강제
  • 2
    회피(Avoidance): 4가지 조건 허용, 자원 할당 전 안전 상태(Safe State) 검사. 불안전 상태 유발 요청 거부
    은행원 알고리즘: 프로세스별 최대 자원 요구량 사전 신고 필요
    안전 순서열이 존재하면 안전 상태 → 할당 승인
    안전 순서열이 없으면 불안전 상태 → 할당 거부, 자원 해제까지 대기
  • 3
    탐지(Detection): 교착상태 발생 허용 후 주기적으로 Wait-for 그래프 사이클 검사
    단일 인스턴스: Wait-for 그래프에서 DFS로 O(n+e) 사이클 탐지
    다중 인스턴스: 자원 요청·할당 행렬 기반 탐지 알고리즘 O(n²m)
    탐지 주기: 너무 잦으면 오버헤드, 너무 드물면 교착상태 지속
  • 4
    회복(Recovery): 탐지 후 교착상태 해소 조치
    프로세스 종료: 교착 관련 전부 또는 순차적 종료 (작업 손실 발생)
    자원 선점: 희생자 선택 → 롤백 → 기아 방지(Age 기법)

Ⅰ. 개요 — 은행원 알고리즘의 위치
은행원 알고리즘은 교착상태 회피(Avoidance) 전략의 대표 알고리즘으로, 자원 할당 전 시스템이 안전 상태(Safe State)를 유지하는지 사전 검증한다. 두 개의 하위 알고리즘으로 구성된다: ①안전성 알고리즘(시스템 상태 검사)과 ②자원 요청 알고리즘(개별 요청 처리).

Ⅱ. 가. 안전성 알고리즘 (Safety Algorithm)
안전성 알고리즘 (의사코드) 입력: Available, Max, Allocation, Need 출력: 안전(Safe) / 불안전(Unsafe) 1. Work = Available.copy() Finish[i] = false for all i 2. 다음 조건을 만족하는 i를 탐색: 조건1: Finish[i] == false 조건2: Need[i] ≤ Work (i번 프로세스의 모든 자원 유형에 대해) 3. 조건을 만족하는 i가 존재하면: Work = Work + Allocation[i] ← i가 완료 후 자원 반납 Finish[i] = true 2단계로 돌아감 4. 모든 i에 대해 Finish[i] == true → 시스템 안전(Safe State) 일부 Finish[i] == false → 시스템 불안전(Unsafe State)

Ⅲ. 나. 자원 요청 알고리즘 (Resource-Request Algorithm)
자원 요청 알고리즘 (의사코드) 입력: 프로세스 Pi의 자원 요청 Request[i] 1. [요청 유효성 검사] if Request[i] > Need[i]: Error! 최대 요구량 초과 요청 → 오류 처리 2. [가용 자원 확인] if Request[i] > Available: Pi를 대기 상태로 전환 (자원 부족) 3. [가상 할당 수행 — 시뮬레이션] Available = Available − Request[i] Allocation[i] = Allocation[i] + Request[i] Need[i] = Need[i] − Request[i] 4. [안전성 검사] if 안전성 알고리즘 결과 == Safe: 실제로 자원 할당 확정 else: 가상 할당 취소 (원상복구) → Pi 대기 Available, Allocation, Need 복원 핵심: 가상 할당 후 안전 상태 검사 → 불안전 시 롤백

은행원 알고리즘은 교착상태 회피의 이론적 완성도가 높지만, 실용적 제약으로 범용 OS에서 채택되지 않는다. 현대 고성능 시스템에서는 MVCC 기반 낙관적 병행 제어(PostgreSQL, MySQL InnoDB)와 타임아웃+롤백 조합이 사실상 표준이다. 분산 환경에서는 Saga 패턴, 분산 락 관리자(Redis Redlock), Circuit Breaker 패턴이 교착상태 회피의 실용적 대안으로 사용된다.

분산 교착상태(Distributed Deadlock)는 분산 시스템에서 복수의 노드에 걸쳐 있는 프로세스·트랜잭션들이 서로 다른 노드의 자원을 순환 대기하여 발생하는 교착상태이다. 단일 노드 교착상태와 달리, 전역 상태를 한 번에 파악하기 어렵고 네트워크 지연·파티션이 탐지를 복잡하게 만든다.

Ⅱ. 나. 분산 교착상태 탐지 알고리즘 3방식 비교
  • 1
    집중화 방식 (Centralized)
    중앙 코디네이터가 모든 노드에서 로컬 Wait-for 그래프를 수집하여 전역 Wait-for 그래프 구성
    전역 그래프에서 사이클 탐지 → 교착상태 판정
    장점: 구현 단순, 정확한 탐지
    단점: 코디네이터 단일 장애점(SPOF), 대규모 시스템에서 병목
  • 2
    분산화 방식 (Distributed) — Chandy-Misra-Haas 알고리즘
    각 노드가 Probe 메시지를 자신이 대기 중인 노드로 전파. 발신 노드로 Probe 메시지가 돌아오면 사이클 = 교착상태 탐지
    단계: ① Pi가 Pj를 대기 시 Probe(i,i,j) 발송 → ② Pj가 Pk를 대기 시 Probe(i,j,k) 전달 → ③ Pi에게 Probe 귀환 시 교착상태
    장점: 단일 장애점 없음, 노드 자율 탐지
    단점: 메시지 복잡도 높음, 유령 교착상태 가능성
  • 3
    계층적 방식 (Hierarchical)
    노드를 계층적 클러스터로 구성. 클러스터 내부는 집중화, 클러스터 간은 분산화 방식 혼용
    장점: 확장성과 정확성의 균형. 대규모 분산 시스템에 적합
    단점: 클러스터 구성·관리 복잡

  • 1
    Saga 패턴 — 분산 트랜잭션 교착 회피
    2PC 대신 각 서비스의 로컬 트랜잭션 시퀀스로 분산 트랜잭션 분해
    락을 장시간 보유하지 않아 분산 교착상태 가능성 제거

    Choreography Saga: 이벤트 기반 (Kafka/RabbitMQ). Orchestration Saga: 오케스트레이터가 조율
  • 2
    Circuit Breaker 패턴 — 연쇄 장애 및 교착 방지
    서비스 호출 실패율이 임계값 초과 시 회로 차단(Open) → 즉시 실패 반환으로 대기 누적 차단
    상태: Closed(정상) → Open(차단) → Half-Open(복구 시도)
    Resilience4j, Hystrix, Istio의 DestinationRule로 구현
    교착상태 유발 서비스 호출 차단으로 전파 방지

    분산 락 (Distributed Lock) — Redis Redlock

    Redlock 알고리즘: N개 Redis 노드 중 과반수(N/2+1) 노드에서 락 획득 성공 시만 유효

Ⅳ. 결론

분산 교착상태는 MSA 환경에서 가장 다루기 어려운 동시성 문제이다. 설계 단계에서 Saga 패턴으로 장기 분산 트랜잭션을 회피하고, Circuit Breaker로 연쇄 장애를 방지하며, Redis 분산 락의 TTL로 교착상태 자동 해소 기전을 마련하는 것이 현대적 모범 사례이다. 감리 시에는 MSA 간 분산 락 설계, TTL 설정, Saga 보상 트랜잭션 로직 존재 여부를 핵심 점검항목으로 확인해야 한다.

Linux 커널 lockdep 메커니즘:

Ⅳ. 결론

현대 클라우드·컨테이너 환경에서의 교착상태는 단순 OS 개념을 넘어 분산 락, 서비스 의존성, 자원 스케줄링 등 다층적 문제로 확장되었다. Linux lockdep으로 커널 락 의존성을 검증하고, Java tryLock·ThreadMXBean으로 애플리케이션 교착상태를 탐지하며, K8s의 readinessProbe·자원 할당량·eBPF 모니터링으로 컨테이너 환경 교착상태를 체계적으로 관리해야 한다. 감리 시에는 멀티스레드 설계 표준, 락 타임아웃 정책, 컨테이너 의존성 그래프 검증 절차를 중점 점검한다.
