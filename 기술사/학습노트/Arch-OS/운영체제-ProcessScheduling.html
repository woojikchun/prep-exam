① FCFS (First-Come First-Served) — 비선점

동작: 도착 순서대로 CPU를 배정. 가장 단순하나 긴 프로세스 뒤에 짧은 프로세스가 오래 기다리는 호위 효과(Convoy Effect) 발생 가능.

② SJF (Shortest Job First) — 비선점

동작: 현재 Ready 큐에서 버스트 시간이 가장 짧은 프로세스를 먼저 선택. 평균 대기 시간 최소화 (이론적 최적). 단, 버스트 시간 사전 예측 어려움, 긴 프로세스의 기아(Starvation) 위험.

③ SRTF (Shortest Remaining Time First) — 선점형 SJF

④ Round Robin (RR) — 선점형, q = 2

동작: 각 프로세스에 동일한 타임 퀀텀(q)을 순환 배정. 공정성 보장·응답 시간 예측 가능으로 대화형 OS의 기본 방식. 퀀텀 크기가 성능을 결정하며, 너무 작으면 문맥 교환 오버헤드↑, 너무 크면 FCFS에 근접.

Ⅳ. 현대 운영체제의 고급 스케줄링 기법 및 실무 고려사항
1. 다단계 큐 스케줄링 (Multilevel Queue)

Ready 큐를 여러 개의 별도 큐로 분리하고 각 큐에 서로 다른 알고리즘을 적용한다. 프로세스는 영구적으로 특정 큐에 배정되며 큐 간 이동이 없다.

【그림 3】 다단계 큐 스케줄링 구조 — 큐 간 우선순위 고정
2. 다단계 피드백 큐 (Multilevel Feedback Queue, MLFQ)

다단계 큐와 달리 프로세스가 큐 간 이동(Feedback)을 허용한다. CPU를 많이 사용한 프로세스(CPU-Bound)는 하위 큐로 강등, I/O 위주 프로세스(I/O-Bound)는 상위 큐로 승격하는 자동 우선순위 조정 메커니즘이다. Linux CFS, Windows Thread Scheduler 등 현대 범용 OS의 기반 방식.

4. 실무 설계 시 고려사항

① 기아(Starvation) 방지: 우선순위 스케줄링 사용 시 반드시 에이징(Aging) 메커니즘 구현 — 오래 대기한 프로세스의 우선순위를 점진적으로 상향 조정

② 우선순위 역전(Priority Inversion) 방지: RTOS 환경에서 낮은 우선순위 프로세스가 공유 자원을 보유해 높은 우선순위를 블록하는 문제 → 우선순위 상속(Priority Inheritance) 또는 우선순위 천장(Priority Ceiling) 프로토콜 적용

③ 타임 퀀텀 튜닝: RR 퀀텀은 문맥 교환 비용의 약 10배로 설정하는 것이 경험적 최적값. 워크로드 특성(Interactive vs Batch)에 따라 동적 조정 가능

④ 멀티코어 확장성: 코어별 런큐(Per-core Run Queue) 구성으로 락 경쟁 최소화, NUMA 아키텍처에서는 메모리 지역성(Locality) 고려한 스케줄링 필수

⑤ 에너지 효율(Green Scheduling): 모바일·IoT 환경에서 유휴 CPU 코어 파워 게이팅(Power Gating), 빅-리틀(big.LITTLE) 아키텍처의 이기종 코어 배분 전략 고려

Ⅴ. 결론

프로세스 스케줄링은 CPU 이용률·공정성·응답성 간의 트레이드오프를 최적 균형으로 조정하는 운영체제의 핵심 정책이다. 단일 최적 알고리즘은 존재하지 않으며, 시스템의 워크로드 특성(배치·대화형·실시간)에 맞는 알고리즘 선택과 파라미터 튜닝이 중요하다. 현대 OS는 MLFQ 기반의 동적 우선순위 조정과 멀티코어 부하 분산을 결합하여 다양한 워크로드에 적응적으로 대응한다.

★ 기술사 핵심 코멘트
스케줄링의 본질은 CPU 이용률과 공정성·응답성 간 트레이드오프의 정책적 결정이며, 시스템 감리 시 다음을 중점 검토해야 한다.
① 기아 방지 메커니즘: 우선순위 기반 시스템에서 에이징 정책 구현 여부 및 에이징 증가율의 적정성 확인
② 실시간 시스템 결정론적 응답 보장: RTOS에서 최악 실행 시간(WCET) 분석 및 우선순위 역전 방지 프로토콜 적용 여부
③ 멀티코어 부하 불균형: 코어 간 런큐 길이 모니터링, 주기적 부하 분산(Load Balancing) 정책 설계 적정성
④ 퀀텀 크기 설계: RR 퀀텀 크기가 문맥 교환 오버헤드 대비 적정 비율인지, 워크로드 변화 시 동적 조정 가능 여부 점검
