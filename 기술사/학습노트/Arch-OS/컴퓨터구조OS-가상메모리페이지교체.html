Ⅰ. 가상 메모리의 개념
가상 메모리(Virtual Memory)는 프로세스가 실제 물리 메모리(RAM) 크기를 초과하는 주소 공간을 사용할 수 있도록 보조 기억장치(디스크)를 메모리처럼 활용하는 기법이다. 프로세스는 연속된 가상 주소 공간을 갖지만, 실제로는 일부만 물리 메모리에 적재되며 나머지는 디스크(스왑 영역)에 존재한다. 다중 프로그래밍 정도(Degree of Multiprogramming)를 높이고 메모리 활용 효율을 극대화한다.
Ⅱ. 페이징(Paging) — 고정 크기 분할
원리: 가상 주소 공간과 물리 메모리를 동일한 고정 크기의 페이지(Page) / 프레임(Frame)으로 분할한다. 가상 주소는 <페이지 번호(p), 페이지 오프셋(d)>로 구성되며, 페이지 테이블을 통해 물리 프레임 번호로 변환된다.
가상 주소 → 물리 주소 변환 (페이징) 가상 주소 [ 페이지 번호(p) | 오프셋(d) ] │ [페이지 테이블] (프로세스마다 존재) p → 프레임 번호(f) │ 물리 주소 [ 프레임 번호(f) | 오프셋(d) ] 예) 페이지 크기 4KB, 가상 주소 0x3FFE → p = 0x3FFE / 4096 = 0 번 페이지 → d = 0x3FFE % 4096 = 0x3FFE (오프셋) → 페이지 테이블[0] = 프레임 5 → 물리 주소 = 5×4096 + 0x3FFE

연속성비연속 메모리 할당 가능. 공유 페이지(Copy-on-Write) 지원

Ⅲ. 세그멘테이션(Segmentation) — 가변 크기 분할
원리: 프로그램을 논리적 의미 단위(코드·데이터·스택·힙 등)인 세그먼트로 분할한다. 가상 주소는 <세그먼트 번호(s), 오프셋(d)>로 구성되며, 세그먼트 테이블(기준 주소+한계 값)을 통해 변환된다.
가상 주소 → 물리 주소 변환 (세그멘테이션) 가상 주소 [ 세그먼트 번호(s) | 오프셋(d) ] │ [세그먼트 테이블] s → (Base, Limit) │ d < Limit 검증 (한계 초과 시 → 세그멘테이션 폴트) │ 물리 주소 = Base + d

현대 OS(Linux·Windows)는 세그멘테이션의 논리적 보호 특성과 페이징의 물리적 유연성을 결합한 세그멘테이션+페이징 혼합 방식을 사용한다. x86 아키텍처에서는 세그먼트 → 선형 주소 → 페이지 테이블 → 물리 주소의 2단계 변환으로 두 기법의 장점을 모두 활용한다.

스래싱(Thrashing)은 프로세스가 실제 실행보다 페이지 교체 작업에 대부분의 시간을 소비하여 CPU 이용률이 급격히 저하되는 현상이다. 다중 프로그래밍 정도를 높여 CPU 이용률을 높이려 할 때, 임계점을 초과하면 역설적으로 CPU 이용률이 폭락하는 현상이다.

Ⅰ. 요구 페이징의 개념
요구 페이징(Demand Paging)은 프로세스 실행에 필요한 페이지를 실제로 요구할 때(접근 시점에)만 물리 메모리에 적재하는 지연 로딩(Lazy Loading) 기법이다. 초기에 프로세스 전체를 메모리에 적재하지 않아 메모리 효율을 극대화하며, 가상 메모리 구현의 핵심 기법이다.
요구 페이징의 장점
① 프로세스 시작 시간 단축 (전체 로딩 불필요)
② 물리 메모리 요구량 감소 → 더 많은 프로세스 동시 실행
③ 실제 사용되지 않는 코드·데이터는 메모리 미점유 (예: 오류 처리 루틴)
Ⅱ. 유효-무효 비트(Valid-Invalid Bit)
비트 상태의미동작
Valid (v)해당 페이지가 현재 물리 메모리(프레임)에 존재정상적 메모리 접근 → 주소 변환 완료
Invalid (i)해당 페이지가 메모리에 없음 (디스크 스왑 영역 또는 미할당)하드웨어 트랩 발생 → OS 페이지 폴트 핸들러 호출
Ⅲ. 페이지 폴트 처리 과정 (9단계)
페이지 폴트 처리 흐름 CPU가 가상 주소 접근 → 페이지 테이블 참조 │ 유효(v) ──────────── 물리 주소 변환 후 접근 완료 │ 무효(i) → [1] 트랩 발생 (하드웨어 → OS 전환) [2] OS: 유효한 참조인지 검증 무효 참조 → 프로세스 종료(Segfault) 유효 참조 → 다음 단계 [3] 빈 프레임(Free Frame) 탐색 없으면 → 페이지 교체 알고리즘으로 희생 프레임 선택 [4] 희생 페이지가 변경됨(Dirty) → 디스크에 기록(Write-back) [5] 필요한 페이지를 디스크에서 빈 프레임으로 읽기(Swap-in) [6] 페이지 테이블 갱신: Invalid → Valid, 프레임 번호 기록 [7] 트랩 발생 시점의 명령어 재시작(Restart) [8] CPU가 동일 가상 주소 재접근 → 이번엔 Valid [9] 물리 주소 변환 완료 → 정상 실행
Ⅳ. 유효 접근 시간(EAT) 계산
📐 EAT(Effective Access Time) 계산식
EAT = (1 - p) × ma + p × 페이지 폴트 처리 시간

• p: 페이지 폴트 발생 확률 (0 ≤ p ≤ 1)
• ma: 메모리 접근 시간 (예: 200ns)
• 페이지 폴트 처리 시간: 디스크 I/O 시간 (예: 8ms = 8,000,000ns)

계산 예시: p = 0.001 (폴트율 0.1%), ma = 200ns, 폴트 처리 = 8,000,000ns
EAT = (1 - 0.001) × 200 + 0.001 × 8,000,000
    = 0.999 × 200 + 8,000
    = 199.8 + 8,000 = 약 8,200ns (메모리만 접근 시 대비 41배 느림)

→ p를 최소화(지역성 활용·충분한 프레임 확보)하는 것이 성능의 핵심
Ⅴ. 결론

요구 페이징은 가상 메모리의 핵심 구현 기법으로, 페이지 폴트 처리 비용(디스크 I/O)이 성능을 좌우한다. SSD 보급으로 폴트 처리 시간이 HDD 대비 약 100배 단축되었으나, 여전히 메모리 접근 대비 수천 배 느려 페이지 폴트 최소화가 OS 성능의 핵심이다.

Ⅰ. 지역성(Locality)의 개념
지역성 원리(Principle of Locality)는 프로세스가 실행 중 특정 시간·공간 범위 내의 일부 메모리 영역에 집중적으로 접근하는 경향을 의미한다. 이 원리는 캐시·TLB·가상 메모리 시스템의 성능 근거가 된다.
Ⅱ. 지역성 3가지 유형
유형정의예시활용
시간적 지역성
(Temporal Locality)		최근에 참조된 항목은 가까운 미래에 다시 참조될 가능성이 높다루프 변수(i), 자주 호출되는 함수, 스택 포인터LRU 알고리즘의 이론적 근거. 캐시 유효성의 핵심
공간적 지역성
(Spatial Locality)		참조된 항목 근처의 항목도 곧 참조될 가능성이 높다배열 순차 접근, 명령어 순차 실행, 인접 데이터 구조페이지 단위 로딩(인접 데이터 함께 적재), 캐시 라인
순차적 지역성
(Sequential Locality)		명령어나 데이터가 순서대로 접근되는 경향프로그램 코드의 순차 실행, 파일 순차 읽기프리페칭(Prefetching), 스트림 버퍼
Ⅲ. 지역성과 워킹셋의 관계
워킹셋(Working Set)은 지역성 원리의 실증적 구현이다. 프로세스는 특정 실행 단계에서 일정한 페이지 집합(지역 = Locality)에 집중적으로 접근하며, 이 집합이 워킹셋이다.
워킹셋과 지역성의 관계 프로그램 실행 타임라인: 시간 → [루프 A 실행] [함수 B 호출] [배열 C 처리] 지역: {p1, p2, p3} → {p3, p4, p5} → {p5, p6, p7} 각 구간에서 집중 참조되는 페이지 집합 = 해당 구간의 워킹셋 지역 전환 시 → 워킹셋 변화 → 페이지 폴트 일시 증가(지역 전환 비용) ∴ 워킹셋을 메모리에 완전히 유지 = 해당 지역의 페이지 폴트 거의 없음 ∴ 워킹셋 일부 누락 = 해당 지역 내에서도 빈번한 폴트 = 스래싱 전조
Ⅳ. 지역성과 TLB(Translation Lookaside Buffer)의 관계
TLB는 페이지 테이블의 최근 참조 항목을 저장하는 고속 하드웨어 캐시(연관 메모리)이다. 지역성 덕분에 TLB는 소용량(64~1024 엔트리)으로도 높은 적중률을 달성한다.
📐 TLB를 포함한 유효 접근 시간
EAT = α × (t_TLB + ma) + (1 - α) × (t_TLB + 2×ma)

• α: TLB 적중률(Hit Ratio), 시간적·공간적 지역성이 높을수록 α → 1
• t_TLB: TLB 탐색 시간 (약 20ns)
• ma: 메모리 접근 시간 (약 200ns)

α = 0.99 (99% 적중) 시:
EAT = 0.99 × (20+200) + 0.01 × (20+400) = 217.8 + 4.2 = 222ns
→ TLB 없이 두 번 메모리 접근(페이지 테이블+데이터) = 400ns 대비 44% 단축
Ⅴ. 결론

지역성 원리는 가상 메모리·TLB·CPU 캐시 등 컴퓨터 시스템 전반의 계층적 메모리 구조가 실용적으로 동작할 수 있는 이론적 근거이다. 지역성이 좋은 프로그램(배열 순차 접근·루프 집중 실행)은 TLB 적중률이 높고 페이지 폴트가 적어 성능이 우수하며, 따라서 알고리즘·자료구조 설계 시 지역성을 고려하는 것이 중요하다.

Ⅰ. 개요 — 가상 메모리 시스템의 개념
가상 메모리는 프로세스에게 실제 물리 메모리보다 큰 연속된 주소 공간의 환상(Illusion)을 제공하는 OS의 핵심 메모리 관리 기법이다. 물리 메모리(RAM) + 스왑 공간(디스크)을 계층적으로 활용하며, 페이지 테이블과 TLB를 통한 주소 변환이 핵심이다.
Ⅱ. 가. 가상 주소 공간의 구조
프로세스 가상 주소 공간 구조 (32비트 기준, 4GB) 0xFFFFFFFF ┌─────────────────────────┐ │ 커널 공간 (1GB) │ ← OS 독점 (사용자 접근 불가) 0xC0000000 ├─────────────────────────┤ │ 스택(Stack) │ ← 함수 호출, 지역 변수 (높은 주소→낮은 주소 성장) │ ↓ (성장 방향) │ │ (가드 페이지) │ │ ↑ (성장 방향) │ │ 힙(Heap) │ ← 동적 할당(malloc/new) (낮은→높은 주소 성장) ├─────────────────────────┤ │ BSS 세그먼트 │ ← 초기화되지 않은 전역·정적 변수 │ 데이터 세그먼트 │ ← 초기화된 전역·정적 변수 │ 코드(텍스트) 세그먼트 │ ← 프로그램 코드 (읽기 전용) 0x00000000 └─────────────────────────┘ 물리 메모리 (RAM): 프레임 단위로 분할. 현재 사용 중인 페이지만 적재 스왑 공간 (Disk): 메모리에서 제거된 페이지 임시 보관 (페이지 파일)

Ⅲ. 나. 페이지 테이블 3종 구조 및 비교
  • 1
    단일 레벨 페이지 테이블: 모든 페이지 번호를 하나의 선형 배열로 관리
    구조: PT[p] → 프레임 번호(f). 32비트 주소·4KB 페이지 시 테이블 크기 = 2²⁰ × 4B = 4MB (프로세스당)
    단점: 프로세스당 테이블이 물리 메모리에 연속 존재해야 함. 희소(Sparse) 주소 공간 낭비 극심
  • 2
    다단계(2단계) 페이지 테이블: 가상 주소를 <p1|p2|d>로 분할. 외부 PT → 내부 PT → 프레임
    구조: CR3 → 외부 PT[p1] → 내부 PT 기주소 + p2 → 프레임 번호(f)
    장점: 사용하지 않는 주소 영역의 내부 PT 미생성 → 메모리 절약 (희소 주소 공간에 유리)
    단점: 메모리 접근 횟수 증가 (2회 + TLB 미스 시 추가). 64비트 주소 → 4단계(x86-64: PML4·PDP·PD·PT)
  • 3
    역(Inverted) 페이지 테이블: 물리 프레임 수를 기준으로 하나의 전역 테이블 유지. 각 엔트리 = (PID, 페이지 번호)
    구조: IPT[f] = (PID, p). 가상 주소 (PID, p)로 IPT를 해시 검색 → 프레임 번호 f
    장점: 테이블 크기 = 물리 메모리 크기 기준 (고정). 64비트 대용량 가상 주소 공간에 유리
    단점: 탐색 시간 O(n). 해시 충돌 처리 필요. 공유 메모리 구현 어려움. IBM RISC System/6000 적용

TLB 성능 최적화 방안
  • 1
    TLB 크기 증가: 엔트리 수 확대(128→1024)로 적중률 향상. 단, 하드웨어 비용·탐색 시간 증가 트레이드오프
  • 2
    대형 페이지(Huge Page): 4KB → 2MB·1GB 페이지 사용. 동일 TLB 엔트리로 더 넓은 메모리 커버 → 적중률 대폭 향상. Linux: mmap(MAP_HUGETLB), Windows: Large Pages
  • 3
    ASID(Address Space Identifier): 컨텍스트 스위치 시 TLB 전체 플러시 대신 ASID로 프로세스 구분 → TLB 재사용률 향상
  • 4
    멀티레벨 TLB: L1 TLB(소형·초고속) + L2 TLB(대형·고속)의 2단 구조. Intel x86의 경우 L1 ITLB/DTLB + 공유 L2 STLB
Ⅴ. 결론

가상 메모리 주소 변환의 성능은 TLB 적중률에 의해 결정된다. 현대 x86-64 프로세서는 4단계 페이지 테이블(PML4)과 계층적 TLB(L1+L2)를 결합하여 64비트 대용량 가상 주소 공간을 효율적으로 관리한다. 대형 페이지 활용과 ASID 적용은 고성능 서버·데이터베이스 시스템에서 메모리 접근 성능을 획기적으로 향상시키는 핵심 최적화 기법이다.

Ⅰ. 개요 — 페이지 교체의 필요성
페이지 교체는 물리 메모리의 빈 프레임이 없을 때 기존 페이지를 디스크로 내보내고 새 페이지를 적재하는 과정이다. 어떤 페이지를 교체하느냐(교체 정책)가 페이지 폴트 횟수와 시스템 성능에 직접 영향을 미친다.
Ⅱ. 가. 페이지 교체 알고리즘 구현 방법 상세
  • 1
    FIFO: 큐(Queue) 자료구조 사용. 메모리 적재 순서대로 큐에 삽입. 교체 시 큐 앞(가장 오래된 것) 제거. 구현 단순. 하드웨어 지원 불필요
  • 2
    OPT: 미래 참조 열을 스캔하여 다음 참조까지 거리가 가장 먼 페이지 교체. 실제 구현 불가 (미래 정보 필요). 성능 비교 기준선(Benchmark)으로만 사용
  • 3
    LRU: 두 가지 구현 방법
    카운터 기반: 각 페이지에 마지막 접근 타임스탬프 기록. 교체 시 가장 오래된 카운터 값 탐색 O(n). 하드웨어 타임스탬프 필요
    스택 기반: 이중 연결 리스트로 스택 구현. 접근 시 해당 노드를 스택 최상위로 이동. 교체 시 스택 최하위 제거. 교체 O(1), 이동 O(1). 하드웨어 지원 없이도 OS에서 구현 가능
  • 4
    Clock (2차 기회 알고리즘): 원형 큐(Ring Buffer) + 참조 비트(R-bit) 활용. 시계 바늘이 순환하며 R=0인 페이지 교체, R=1이면 0으로 초기화 후 다음 페이지로. LRU 근사를 가장 낮은 오버헤드로 구현. Linux·BSD의 기본 교체 알고리즘
  • 5
    LFU: 각 페이지의 참조 횟수(Frequency Counter) 유지. 최솟값 탐색을 위한 힙(Min-Heap) 또는 버킷 정렬 구조 사용. 초기 빈번 접근 후 방치된 페이지가 오래 잔류하는 노화(Aging) 문제 발생 → 주기적 카운터 절반 감소(Aging 기법)로 해결
Ⅲ. 나. Belady의 이상현상과 스택 알고리즘
Belady의 이상현상(Belady's Anomaly): FIFO 알고리즘에서 프레임 수를 늘렸음에도 불구하고 페이지 폴트 횟수가 오히려 증가하는 반직관적 현상이다.
Belady의 이상현상 예시 (참조 열: 1,2,3,4,1,2,5,1,2,3,4,5) 프레임 3: 페이지 폴트 9회 (FIFO) 프레임 4: 페이지 폴트 10회 (FIFO) ← 프레임 증가했는데 폴트 증가! 원인: FIFO는 교체할 페이지 선택 시 '얼마나 최근에 사용됐는가'를 고려하지 않아, 프레임 수가 늘면 적재 순서가 달라져 오히려 더 자주 사용되는 페이지가 교체될 수 있음
스택 알고리즘 (Stack Algorithm) — Belady 이상현상 없음
정의: n개 프레임에서 메모리에 있는 페이지 집합이 n+1개 프레임에서의 집합의 부분집합이 되는 성질을 가진 알고리즘
→ S(n) ⊆ S(n+1) 항상 성립 → 프레임 증가 시 페이지 폴트 단조 감소 보장

스택 알고리즘 해당: LRU, OPT, LFU(변형)
비해당 (Belady 가능): FIFO, Clock (FIFO 유사 계열)
Ⅳ. 다. 전역/지역 교체 정책 및 프레임 할당 전략
비교 항목전역 교체 (Global Replacement)지역 교체 (Local Replacement)
교체 범위전체 프로세스의 프레임 중 교체 대상 선정해당 프로세스 자신의 프레임에서만 교체
프레임 수실행 중 프레임 수 동적 변화할당된 프레임 수 고정
장점메모리 이용률 높음. 고우선순위 프로세스가 더 많은 프레임 확보 가능다른 프로세스의 성능에 영향 미치지 않음. 예측 가능한 성능
단점다른 프로세스의 페이지를 교체 → 의도치 않은 성능 저하 유발메모리 이용률 낮을 수 있음. 비효율적 프레임 고정
적용Linux, Windows (기본값)일부 실시간 시스템, 멀티테넌트 환경
프레임 할당 전략 3가지:
  • 1
    균등 할당(Equal Allocation): 전체 프레임 수 / 프로세스 수. 모든 프로세스에 동일 프레임 배분. 단순하나 프로세스 크기·중요도 무시
  • 2
    비례 할당(Proportional Allocation): 프로세스 크기(가상 주소 공간)에 비례하여 프레임 할당. f_i = (s_i / Σs) × m (m: 전체 프레임). 크기가 큰 프로세스에 더 많은 프레임
  • 3
    우선순위 기반 할당(Priority Allocation): 프로세스 우선순위에 따라 프레임 할당. 높은 우선순위 프로세스가 더 많은 프레임 확보. 전역 교체 정책과 결합하여 낮은 우선순위 프레임을 높은 우선순위 프로세스가 가져갈 수 있음
Ⅴ. 결론

페이지 교체 알고리즘의 선택은 시스템 성능에 직접적 영향을 미친다. 실제 OS는 OPT의 근사인 LRU를 Clock 알고리즘으로 구현하여 낮은 오버헤드로 높은 성능을 달성한다. 전역 교체와 비례 할당을 결합하고, 워킹셋 기반 프레임 관리를 통해 스래싱을 예방하는 것이 현대 OS 메모리 관리의 핵심 원칙이다.

Ⅰ. 가. 스래싱 발생 메커니즘
스래싱은 프로세스의 활성 지역(Locality)을 수용할 프레임이 부족할 때 발생한다. 각 프로세스는 특정 실행 단계에서 집중적으로 참조하는 페이지 집합(워킹셋)을 가지며, 이 집합이 메모리에 수용되지 못하면 지속적인 페이지 교체가 발생한다.
CPU 이용률과 다중 프로그래밍 정도의 관계 CPU 100% │ ╭──────╮ 이용률 │ ╱ ╲ │ ╱ ╲ ← 스래싱 구간 │ ╱ ╲_______ │ ╱ 0% └─────────────────────────────────────── │ 임계점(Knee) ← 정상 구간 │ 스래싱 구간 → 다중 프로그래밍 정도(MPD) 임계점 이전: MPD 증가 → CPU 이용률 증가 (정상 동작) 임계점 이후: MPD 증가 → 페이지 폴트 폭발적 증가 → CPU는 I/O 대기 → CPU 이용률 급락 → OS가 MPD를 더 높임 → 악순환
지역성과 스래싱의 연관성
프로세스는 실행 단계마다 다른 지역(Locality)을 가짐
지역 A → B → C 전환 시 → 새 지역의 페이지 적재 필요 → 폴트 일시 증가 (정상)
스래싱 = 지역 크기 > 할당 프레임 수 → 지역 내에서도 끊임없는 폴트 발생
Ⅱ. 나. 워킹셋 모델 — 수학적 정의 및 스래싱 방지
워킹셋 수학적 정의 (Peter Denning, 1968)
WS(t, Δ) = {p | p는 (t-Δ, t] 구간에서 1회 이상 참조된 페이지}

• t: 현재 시각
• Δ: 워킹셋 윈도우 크기 (Working Set Window)
• WS(t, Δ): 시각 t에서 프로세스 i의 워킹셋 (t-Δ 이후 참조된 페이지 집합)

예) 참조 열: 2,6,1,5,7,7,7,5,1,6   Δ=5   t=10
WS(10, 5) = {5,1,6,7} ← 마지막 5회 참조에서 등장한 페이지 집합
워킹셋 윈도우(Δ) 크기 결정:
  • 1
    Δ가 너무 작으면: 현재 지역을 완전히 포함하지 못함 → 워킹셋 과소 추정 → 폴트 증가
  • 2
    Δ가 너무 크면: 여러 지역에 걸친 페이지 포함 → 워킹셋 과대 추정 → 메모리 낭비
  • 3
    최적 Δ: 두 지역 전환 사이의 평균 실행 시간. 실제로는 샘플링과 실험적 측정으로 결정
워킹셋 기반 스래싱 방지 메커니즘 1. 각 프로세스 i의 워킹셋 크기 |WS_i(t, Δ)| 주기적 측정 2. 전체 필요 프레임 = D = Σ|WS_i(t, Δ)| (모든 프로세스 합산) 3. 판단: D ≤ m (물리 프레임 총수) → 정상. 새 프로세스 추가 가능 D > m → 스래싱 위험! → 우선순위 낮은 프로세스 선택 → 해당 프로세스 일시 중단 + 스왑 아웃 → 해방된 프레임을 나머지 프로세스 재분배 → D ≤ m 될 때까지 반복

  • 1
    프로세스 일시 중단(Suspend): 우선순위 가장 낮은 프로세스를 선택하여 스왑 아웃. 해방된 프레임을 나머지 프로세스에 재분배. 가장 일반적인 회복 기법
  • 2

  • 3

현대 시스템에서의 스래싱 관리 방안
① 메모리 오버커밋(Memory Overcommit): Linux의 경우 물리 메모리 초과 가상 메모리 할당 허용. OOM Killer가 스래싱 진입 전 메모리 부족 프로세스를 강제 종료
② cgroups 메모리 제한: 컨테이너·프로세스 그룹별 메모리 상한 설정으로 스래싱 전파 차단. Kubernetes의 Resource Limits 연계
③ NUMA 인식 메모리 관리: 다중 소켓 서버에서 NUMA 노드별 로컬 메모리 우선 할당. 원격 메모리 접근(NUMA Miss) 최소화
④ 대형 페이지(Transparent Huge Pages): 2MB 페이지로 TLB 부하 감소. 워킹셋 크기 대비 TLB 커버리지 확대 → 폴트 감소
⑤ 메모리 압축(zRAM/zswap): 스왑 아웃 대신 메모리 내 압축으로 유효 메모리 용량 확대. 디스크 I/O 없이 스래싱 완화
Ⅳ. 결론

스래싱은 메모리 과부하의 가장 심각한 증상으로, 예방이 회복보다 중요하다. 워킹셋 기반 관리는 이론적으로 가장 우수한 스래싱 방지 기법이나 구현 오버헤드가 크므로, 현대 OS는 PFF 기반 프레임 조절·OOM Killer·cgroups 제한·메모리 압축을 복합적으로 적용한다. 클라우드·컨테이너 환경에서는 Resource Limits 설정이 스래싱 예방의 핵심 수단이다.

결합 이유: 세그멘테이션은 논리적 보호·공유에 유리하지만 외부 단편화를 유발한다. 페이징은 외부 단편화가 없지만 논리적 보호·공유가 비직관적이다. 두 기법을 결합하면 세그먼트별 논리적 보호 + 페이지별 물리 배치 유연성을 모두 확보한다.
Intel x86 보호 모드 — 3단계 주소 변환 ① 논리 주소(Logical Address) = <세그먼트 셀렉터, 오프셋> │ ▼ 세그멘테이션 (GDT/LDT 참조) ② 선형 주소(Linear Address) = 세그먼트 기주소 + 오프셋 │ ▼ 페이징 (CR3 레지스터 → 페이지 디렉터리 → 페이지 테이블) ③ 물리 주소(Physical Address) = 프레임 기주소 + 페이지 오프셋 [세그멘트 셀렉터] → GDT(전역 서술자 테이블) → 세그먼트 서술자 {Base, Limit, 접근권한(DPL, Type)} [선형 주소 = Base + Offset] │ ├→ TLB 탐색 → 히트 → 물리 주소 └→ TLB 미스 → 2단계 페이지 테이블 순회 현대 Linux/x86-64: 세그멘테이션 최소화 (CS·DS·SS의 Base=0, Limit=최대 → 사실상 페이징만 사용) → 64비트 모드에서 세그먼트 레지스터 역할 대폭 축소
Ⅱ. 나. 메모리 단편화 문제와 해결 기법
단편화 유형발생 원인발생 기법해결 기법
내부 단편화
(Internal Fragmentation)		할당 단위(페이지)보다 작은 요청으로 내부 남은 공간 낭비페이징페이지 크기 축소 (단, 페이지 테이블 증가). 슬랩 할당자
외부 단편화
(External Fragmentation)		가변 크기 할당·해제 반복으로 메모리 구멍 발생세그멘테이션, 동적 메모리 할당압축(Compaction), 통합(Coalescing), 버디 시스템
  • 1
    압축(Compaction): 사용 중인 메모리 블록을 한쪽으로 이동시켜 연속된 빈 공간 생성. 단점: 실행 중 메모리 이동 → 높은 오버헤드. 가상 메모리 환경에서는 페이지 테이블 갱신 필요
  • 2
    통합(Coalescing): 인접한 빈 메모리 블록을 하나로 합병. 해제 시 즉시 인접 블록 확인. 단편화 방지에 효과적. 연결 리스트 기반 free list에서 구현
  • 3
    버디 시스템(Buddy System): 메모리를 2의 거듭제곱 크기 블록으로 관리. 요청 크기에 맞는 최소 2^k 블록 할당. 해제 시 동일 크기의 '버디' 블록과 합병
    분할: 2^n → 2^(n-1) + 2^(n-1) (요청보다 큰 블록을 반복 분할)
    합병: 인접 버디가 모두 비어있으면 2^(n+1)로 합병
    장점: 합병 빠름 O(log n). 외부 단편화 최소화
    단점: 내부 단편화 존재 (요청 크기가 2^k가 아닐 때). Linux 커널 페이지 할당자의 기반 알고리즘
Ⅲ. 다. 메모리 보호 기법
보호 기법개념보안적 의의
한계 레지스터
(Limit Register)		프로세스의 주소 범위 상한 설정. 범위 초과 접근 → 세그멘테이션 폴트프로세스 간 메모리 침범 원천 차단. OS 커널 보호
접근 권한 비트
(Protection Bit)		페이지·세그먼트별 읽기(R)·쓰기(W)·실행(X) 권한 설정. 위반 시 보호 폴트코드 섹션 읽기 전용 → 자기 수정 코드 방지. 데이터 실행 방지
NX 비트
(No-Execute Bit)		데이터 페이지에 실행 불가 표시. 스택·힙 등 비코드 영역 실행 차단버퍼 오버플로 공격에서 셸코드 실행 방지. DEP(Data Execution Prevention)의 하드웨어 기반
ASLR
(Address Space Layout Randomization)		프로세스 주소 공간(스택·힙·라이브러리)의 시작 주소를 실행 시마다 무작위화ROP(Return-Oriented Programming)·코드 재사용 공격 방어. 공격자가 대상 주소 예측 불가
현대 메모리 보호 계층 (Linux 기준)
ASLR(주소 무작위화) + NX/DEP(비코드 영역 실행 차단) + Stack Canary(스택 버퍼 오버플로 탐지) + PIE(Position Independent Executable) 조합으로 다층 메모리 보호 구현
Ⅳ. 결론

세그멘테이션+페이징 결합은 현대 x86 아키텍처의 메모리 관리 기반이며, 두 기법의 장점을 활용하여 유연한 물리 배치와 강력한 논리적 보호를 동시에 달성한다. 버디 시스템은 Linux 커널의 핵심 메모리 할당 알고리즘이며, ASLR+NX 비트는 현대 보안의 필수 방어선으로 공공 정보시스템의 OS 보안 설정 감리 시 필수 점검항목이다.

Ⅰ. 개요 — 현대 OS 가상 메모리 관리의 특성
현대 OS의 가상 메모리 관리는 단순한 페이지 교체를 넘어 다양한 최적화 기법을 복합적으로 적용한다. Linux를 중심으로 CoW·mmap·슬랩 할당자·Huge Page·NUMA·KSM 등이 계층적으로 동작하여 성능과 메모리 효율을 극대화한다.
Ⅱ. 가. CoW·mmap·파일 백업 페이지
  • 1
    Copy-on-Write (CoW, 쓰기 시 복사)
    개념: 부모 프로세스가 fork()로 자식 생성 시, 페이지를 즉시 복사하지 않고 동일 페이지를 읽기 전용으로 공유. 쓰기 시도 시 보호 폴트 발생 → OS가 해당 페이지를 그때 복사
    효과: fork() 비용 O(1)으로 단축. 자식이 exec()로 바로 교체되면(shell의 fork-exec 패턴) 복사 불필요 → 메모리 낭비 없음
    적용: Linux fork(), Docker 컨테이너 이미지 레이어, Redis BGSAVE
  • 2
    메모리 맵(mmap)
    개념: 파일 또는 익명 메모리를 프로세스의 가상 주소 공간에 직접 매핑. 파일 접근이 일반 메모리 접근처럼 동작 → 시스템 콜 오버헤드 최소화
    파일 백업 페이지(File-backed Page): mmap된 파일 페이지는 물리 프레임이 부족할 때 디스크(원본 파일)로 단순히 버릴 수 있음 (스왑 불필요). 재접근 시 파일에서 다시 로드
    익명 페이지(Anonymous Page): 스택·힙처럼 파일 비연계. 교체 시 스왑 영역에 저장 필요
파일 백업 페이지 vs 익명 페이지 교체 차이 파일 백업 페이지: 메모리→디스크 기록 불필요 (원본 파일이 이미 존재) 교체 시: 페이지 테이블에서 제거만 하면 됨 재접근 시: 파일에서 read-ahead로 재로드 익명 페이지: 메모리→스왑 영역에 기록 후 교체 (별도 I/O 필요) 재접근 시: 스왑 영역에서 read-in
Ⅲ. 나. Linux 메모리 관리 핵심 구성 요소
  • 1
    버디 시스템(Buddy System): Linux 커널의 물리 페이지 할당자. 2의 거듭제곱 크기(4KB~4MB)로 페이지 블록 관리. 효율적 합병·분할로 외부 단편화 최소화. /proc/buddyinfo로 현황 확인
  • 2
    슬랩 할당자(Slab Allocator): 커널 객체(inode·task_struct·dentry 등) 전용 소형 객체 할당자
    구조: Cache → Slab(1~수 페이지) → Object(동일 크기 구조체들)
    장점: 객체 초기화 비용 절약(미리 초기화된 객체 재사용). 내부 단편화 최소화. CPU 캐시 지역성 향상(같은 크기 객체 연속 배치)
    현대 Linux: SLUB 할당자(Slab 개선판) 사용. /proc/slabinfo로 현황 확인
  • 3
    OOM Killer: 물리 메모리 완전 소진 시 프로세스를 강제 종료하여 메모리 회수. OOM Score(메모리 사용량·우선순위·실행 시간 기반 점수)가 가장 높은 프로세스 선택 종료
    /proc/PID/oom_score_adj로 OOM 점수 조정(-1000: 종료 면제, +1000: 최우선 종료)
    컨테이너 환경: cgroups 메모리 상한 초과 시 컨테이너 OOM Killer 동작
  • 4
    스왑 정책(vm.swappiness): 0~100 값으로 스왑 공격성 조절. 100: 적극 스왑. 0: 가능한 스왑 안 함. 데이터베이스 서버는 vm.swappiness=1 권장 (갑작스런 성능 저하 방지)
Ⅳ. 다. 가상 메모리 성능 최적화 기법
기법개념적용 효과적합 환경
대형 페이지
(Huge Page / THP)		4KB 대신 2MB·1GB 페이지 사용. Transparent Huge Pages(THP)는 자동 관리TLB 미스 감소(동일 TLB로 더 넓은 메모리 커버). 페이지 테이블 크기 감소대용량 메모리 사용 앱 (DB·JVM·HPC)
NUMA 인식
(NUMA-aware)		다중 소켓 서버에서 CPU와 가까운 메모리(로컬 NUMA 노드) 우선 할당메모리 접근 지연 감소 (로컬 ~60ns vs 원격 ~100ns). 처리량 향상멀티소켓 서버, 고성능 데이터베이스
zRAM / zswap
(메모리 압축)		스왑 아웃 대신 RAM 내에서 압축 보관. LZ4·ZSTD 압축 알고리즘 사용디스크 I/O 없이 효과적 메모리 용량 확장. 저장장치 없는 임베디드·모바일에서 효과적모바일(Android), 임베디드, 램 제약 환경
KSM
(Kernel Samepage Merging)		내용이 동일한 페이지를 탐지하여 하나의 CoW 읽기 전용 페이지로 합병. ksmd 백그라운드 데몬동일 콘텐츠 VM·컨테이너에서 메모리 중복 제거. 수십% 메모리 절약 가능KVM 가상화, 컨테이너 플랫폼(Kubernetes)
클라우드·컨테이너 환경 적용 효과
Kubernetes Pod 환경: KSM으로 동일 이미지 컨테이너의 코드 페이지 공유 → 노드당 수십~수백 MB 절약
VM 오버커밋: KSM + 메모리 발룬(Balloon Driver) + zswap 결합으로 물리 메모리 150~200% 오버커밋 가능
대형 페이지: PostgreSQL·Oracle DB에서 THP 적용 시 TLB 히트율 향상으로 5~15% 처리 성능 향상
Ⅴ. 결론

현대 OS의 가상 메모리 관리는 단순한 페이징·교체를 넘어 CoW·mmap·슬랩·KSM·Huge Page·NUMA·zRAM 등 다양한 최적화 기법이 계층적으로 조합된 정교한 시스템이다. 클라우드·컨테이너 환경에서는 cgroups 메모리 제한·KSM·Huge Page를 적절히 조합하여 밀도(Density)와 성능을 동시에 최적화하는 것이 핵심이며, 공공 정보시스템의 서버 튜닝·감리 시에도 이러한 메모리 관리 파라미터의 적절한 설정 여부를 확인해야 한다.
