파이프라이닝(Pipelining)의 개념과 해저드(Hazard)의 유형 및 해결 방법을 설명하시오.

파이프라이닝의 개념, 해저드 유형 및 해결 방법
⏱ 목표 12분 📄 1.5~2페이지 💯 10점
Ⅰ. 파이프라이닝의 개념
파이프라이닝(Pipelining)은 CPU에서 명령어 실행을 여러 단계(Stage)로 분할하고, 각 단계를 병렬로 겹쳐 수행하여 단위 시간당 명령어 처리량(Throughput)을 향상시키는 기법이다.
전통적 순차 실행: IF─ID─EX─MEM─WB │ IF─ID─EX─MEM─WB (직렬) 파이프라인 실행: 클럭 → 1 2 3 4 5 6 7 8 9 명령1 : [IF][ID][EX][M ][WB] 명령2 : [IF][ID][EX][M ][WB] 명령3 : [IF][ID][EX][M ][WB] 명령4 : [IF][ID][EX][M ][WB] 명령5 : [IF][ID][EX][M ][WB] 5-Stage Pipeline: IF(명령어인출) → ID(해석) → EX(실행) → MEM(메모리접근) → WB(결과저장)
성능 이론치
파이프라인 속도향상 ≈ 스테이지 수(k). k-단계 파이프라인에서 n개 명령 실행 시간 = (k + n − 1) × 클럭 사이클. 스테이지 간 레지스터(Latch) 추가 오버헤드 존재.
Ⅱ. 해저드(Hazard) 유형 및 해결 방법
유형정의해결 방법
구조적 해저드
(Structural)		두 명령어가 동일 하드웨어 자원(메모리·ALU 등)을 동시에 요구할 때 발생• 자원 중복화 (별도 명령어 캐시·데이터 캐시 분리)
• 파이프라인 스톨(버블 삽입)
데이터 해저드
(Data)		후행 명령어가 선행 명령어의 결과를 필요로 할 때 의존성 충돌 발생 (RAW·WAR·WAW)• 포워딩(Forwarding/Bypassing): EX 단계 결과를 다음 EX 단계로 직접 전달
• NOP 삽입(Stall)
• 컴파일러 명령어 재배치(Out-of-Order)
제어 해저드
(Control)		분기(Branch) 명령어로 인해 다음 실행 명령어를 확정할 수 없을 때 발생• 분기 예측(Branch Prediction): 정적(항상 taken/not-taken)·동적(BTB, 2비트 예측기)
• 지연 분기(Delayed Branch): 분기 후 슬롯 명령 유효 실행
• 투기적 실행(Speculative Execution)
데이터 해저드 세부 유형
RAW(Read After Write): 선행 명령 쓰기 전에 후행 명령이 읽음 → 가장 빈번
WAR(Write After Read): 선행 명령 읽기 전에 후행 명령이 덮어씀 → Out-of-order 시 발생
WAW(Write After Write): 같은 위치를 두 명령이 순서 없이 씀 → 레지스터 재명명으로 해결
Ⅲ. 슈퍼스칼라·슈퍼파이프라인과의 연계
기법특징비고
슈퍼파이프라인스테이지를 더 세분화(8~16단계)하여 클럭 속도 향상해저드 증가 가능성
슈퍼스칼라동일 클럭에 다수의 파이프라인으로 복수 명령 동시 발행(IPC>1)의존성 검사 로직 복잡
VLIW컴파일러가 병렬 가능한 명령을 하나의 긴 명령어로 묶어 발행하드웨어 단순화
Ⅳ. 결론

파이프라이닝은 현대 CPU의 핵심 성능 기법이나, 해저드로 인해 CPI(Clocks Per Instruction)가 1 이상으로 증가한다. 이를 극복하기 위해 포워딩·분기예측·비순서 실행(OoOE) 등 하드웨어적 기법과 컴파일러 최적화를 복합 적용하며, 최신 Intel/ARM 프로세서는 15~30단계 파이프라인을 운용한다.

단답형 ② — 캐시 메모리(Cache Memory)
1교시 단답형 · 10점 · 목표 10~12분 · 1.5~2페이지
단② 단답형 / 10점 목표 1.5~2p
캐시 메모리의 개념과 매핑 방식(Direct·Associative·Set Associative), 교체 정책 및 일관성 유지 방법을 설명하시오.
매핑 3종 비교표빈출
캐시 메모리의 개념·매핑 방식·교체 정책·일관성 유지
⏱ 목표 12분 📄 1.5~2페이지 💯 10점
Ⅰ. 캐시 메모리의 개념
캐시 메모리(Cache Memory)는 CPU와 주기억장치(DRAM) 사이의 속도 차이를 완화하기 위해 CPU 내부 또는 근방에 위치하는 고속 소용량 메모리이다. 참조 지역성(Locality of Reference) 원리를 기반으로 자주 사용되는 데이터를 임시 저장한다.
참조 지역성(Locality)
시간적 지역성: 최근 사용된 데이터는 곧 다시 사용될 가능성이 높다 (루프 변수)
공간적 지역성: 최근 사용된 데이터 근처의 데이터가 사용될 가능성이 높다 (배열 순회)
Ⅱ. 캐시 매핑(Mapping) 방식 비교
방식개념장점단점충돌
직접 매핑
(Direct)		주소의 인덱스 필드로 캐시 라인 위치 고정 결정. 1:1 대응하드웨어 단순, 빠른 탐색충돌 미스(Conflict Miss) 빈번높음
완전 연관
(Full Assoc)		어느 캐시 라인에도 적재 가능. 태그 전체 비교(병렬)충돌 미스 없음비교 회로 복잡·비용 높음없음
집합 연관
(Set Assoc)		캐시를 집합으로 분할(k-way). 집합 내 위치 자유, 집합 선택은 인덱스로 결정충돌 미스 감소, 비교적 단순k증가 시 비교기 증가낮음
주소 구조 (Set Associative)
TAG | INDEX | BLOCK OFFSET
• INDEX: 집합(Set) 선택
• TAG: 집합 내 어느 라인인지 식별
• BLOCK OFFSET: 블록 내 바이트 위치
Ⅲ. 교체 정책(Replacement Policy)
정책내용특징
LRU가장 오랫동안 사용되지 않은 블록 교체시간적 지역성 반영. 2-way부터 효율적
FIFO가장 먼저 적재된 블록 교체구현 단순. LRU 대비 적중률 낮을 수 있음
LFU참조 횟수가 가장 적은 블록 교체참조 빈도 카운터 필요. 오래된 블록 잔류 가능
Random임의 블록 교체하드웨어 구현 최단순. 성능 예측 어려움
Ⅳ. 캐시 일관성(Cache Coherence) 유지
기법Write-ThroughWrite-Back
동작 방식쓰기 시 캐시와 메인 메모리에 동시 기록캐시에만 우선 기록. Dirty Bit 설정 후 교체 시 메모리 반영
장점일관성 항상 유지, 구현 단순메모리 버스 트래픽 최소화, 성능 우수
단점모든 쓰기마다 메모리 접근 → 버스 병목교체 시 쓰기 지연(Write Penalty). 복잡한 설계
멀티코어 캐시 일관성 프로토콜
MESI 프로토콜: Modified·Exclusive·Shared·Invalid 4가지 상태로 각 코어의 캐시 라인 상태를 버스를 통해 추적. 스누핑(Snooping) 또는 디렉터리(Directory) 방식으로 구현.
Ⅴ. 결론

현대 CPU는 L1(32KB~)·L2(256KB~)·L3(수~수십 MB) 다단계 캐시 계층을 구성하며, 적중률(Hit Rate) 향상을 위해 프리패치(Prefetch)·하드웨어 프리페처를 추가한다. 캐시 미스는 Cold/Capacity/Conflict/Coherence 4C 미스로 분류되며, 시스템 성능 병목의 주요 원인이다.

    x86의 RISC 내부 변환: Intel Core·AMD Zen 시리즈는 CISC 명령을 내부에서 RISC 마이크로 옵(μop)으로 변환하여 RISC 파이프라인으로 실행 → 두 아키텍처 장점 융합
  • 2

Ⅳ. 결론

RISC vs CISC의 경계는 현대 CPU에서 사실상 희석되었다. 외부로는 CISC ISA 호환성을 유지하면서 내부적으로 RISC 파이프라인을 채택하는 하이브리드 설계가 표준이 되었으며, 오픈 RISC-V 생태계의 성장이 향후 CPU 시장의 핵심 변수가 될 것이다.

단답형 ④ — 가상 메모리(Virtual Memory)
1교시 단답형 · 10점 · 목표 10~12분 · 1.5~2페이지
단④ 단답형 / 10점 목표 1.5~2p
가상 메모리(Virtual Memory)의 개념과 페이지 교체 알고리즘(OPT·FIFO·LRU·LFU), 스래싱(Thrashing) 현상 및 해결 방법을 설명하시오.
페이지 교체 비교스래싱 필수
가상 메모리 개념·페이지 교체 알고리즘·스래싱 해결
⏱ 목표 12분 📄 1.5~2페이지 💯 10점
Ⅰ. 가상 메모리의 개념
가상 메모리(Virtual Memory)는 실제 물리 메모리(RAM)보다 큰 주소 공간을 프로세스에 제공하는 기법으로, 디스크(스왑 영역)를 보조 저장소로 활용한다. MMU(Memory Management Unit)가 가상 주소를 물리 주소로 변환하며, 페이징(Paging) 또는 세그멘테이션(Segmentation)으로 구현한다.
가상 주소 변환 과정: 프로세스 → [가상 주소] → MMU(TLB 조회) → [물리 주소] → RAM TLB Miss 시: 페이지 테이블(메모리) → 물리 주소 획득 → TLB 갱신 Page Fault 시: 디스크에서 페이지 로드 → 물리 프레임에 적재 → 재실행
Ⅱ. 페이지 교체 알고리즘 비교
알고리즘교체 기준특징현실성
OPT (최적)미래에 가장 오랫동안 사용되지 않을 페이지최소 페이지 부재율 (이론적 최솟값 기준)미래 참조열 필요 → 구현 불가
FIFO가장 먼저 적재된 페이지구현 단순. 벨라디의 이상 현상(프레임 증가 시 부재 증가) 발생 가능구현 가능
LRU가장 오랫동안 참조되지 않은 페이지시간적 지역성 반영. 참조 시간 기록 필요 (카운터·스택)구현 가능 (우수)
LFU참조 횟수가 가장 적은 페이지참조 빈도 반영. 오래전 자주 쓰인 페이지 잔류 가능구현 가능
Clock (NRU)참조 비트 0인 페이지 순환 교체LRU 근사. 하드웨어 지원으로 효율적. Linux에서 사용실용적
Ⅲ. 스래싱(Thrashing) 현상 및 해결
스래싱 발생 원인
다중 프로그래밍 정도가 너무 높아 각 프로세스에 할당된 프레임 수가 충분하지 않을 때, 페이지 부재가 연속 발생하여 CPU 이용률이 급감하고 페이지 교체만 반복되는 현상.
해결 방법내용
작업 집합(Working Set) 모델프로세스가 최근 Δ 시간 동안 참조한 페이지 집합을 작업 집합으로 정의. 작업 집합 크기 이상의 프레임 보장
페이지 부재 빈도(PFF) 제어PFF가 상한선 초과 시 프레임 추가 할당, 하한선 미만 시 프레임 반환
다중 프로그래밍 수준 조절스래싱 감지 시 일부 프로세스를 일시 중단(Suspend)하여 나머지 프로세스 프레임 확보
Ⅳ. 결론

가상 메모리는 현대 OS의 핵심 메모리 관리 기법으로, TLB 적중률이 전체 성능에 결정적 영향을 미친다. 스래싱 방지를 위한 작업 집합 모델과 PFF 기반 동적 프레임 할당이 실제 OS(Linux, Windows)에 적용되며, 최근 대용량 페이지(Huge Page)를 통해 TLB 미스를 줄이는 최적화 기법도 광범위하게 사용된다.

Ⅰ. 암달의 법칙 (Amdahl's Law, 1967)
병렬화할 수 없는 순차(직렬) 부분이 존재할 때, 프로세서 수(N) 증가에 따른 고정된 문제 크기에서의 이론적 최대 속도 향상(Speedup)의 한계를 표현한 법칙.
Speedup(N) = 1 / (S + (1-S)/N) S = 순차 실행 비율 (병렬화 불가 부분) N = 프로세서(코어) 수 (1-S) = 병렬화 가능 비율 예시: S=0.2, N=∞ → Speedup = 1/0.2 = 5배 (이론적 최대) → 순차 비율이 20%이면 아무리 코어를 늘려도 5배 이상 빠르게 할 수 없다
Ⅱ. 구스타프슨의 법칙 (Gustafson's Law, 1988)
암달의 법칙과 달리, 문제 크기를 프로세서 수에 비례하여 함께 확장할 경우의 속도 향상. 실제 대규모 과학 계산에서 더 현실적인 모델.
Speedup(N) = N - S × (N - 1) S = 순차 실행 비율 N = 프로세서 수 예시: S=0.1, N=16 → Speedup = 16 - 0.1×15 = 14.5배 → 같은 시간 동안 N배 더 큰 문제 해결 가능 = 확장성 중심 관점
Ⅲ. 두 법칙 비교
구분암달의 법칙구스타프슨의 법칙
관점고정된 문제 크기. 실행 시간 단축 중심확장 가능한 문제 크기. 처리량 향상 중심
핵심 메시지순차 비율이 속도 향상의 상한을 결정프로세서 증가 시 같은 시간 내 더 큰 문제 해결 가능
병렬화 비관론비관적 (상한 명확)낙관적 (문제 확장 시 효과 큼)
적용 분야소규모 응답 시간 중심 시스템대규모 HPC, 빅데이터, AI 학습

  • 2

Ⅴ. 결론

암달의 법칙은 병렬화의 한계를 명확히 제시하며, 순차 비율을 줄이는 알고리즘 개선이 근본적 해결책임을 시사한다. 구스타프슨의 법칙은 AI·HPC처럼 데이터 규모를 확장할 수 있는 분야에서 멀티코어의 실질적 가치를 설명하며, 현대 서버 CPU(AMD EPYC, Intel Xeon)의 방향성과 일치한다.

Ⅰ. 개요
인터럽트(Interrupt)는 CPU가 현재 실행 중인 프로그램을 일시 중단하고, 긴급한 이벤트를 처리하기 위해 미리 정의된 처리 루틴(ISR)으로 제어를 전환하는 메커니즘이다. 폴링(Polling) 방식 대비 CPU 낭비를 최소화하며, 비동기 이벤트를 효율적으로 처리한다.
Ⅱ. 가. 인터럽트 유형 분류
분류유형발생 원인예시
하드웨어
인터럽트		외부 인터럽트
(Maskable)		외부 장치의 I/O 완료, 키보드·마우스 입력IRQ0(타이머), IRQ1(키보드), NIC 수신
NMI
(Non-Maskable)		하드웨어 오류, 전원 이상. INTR 핀 무시 불가메모리 패리티 오류, 버스 오류
소프트웨어
인터럽트		트랩(Trap)의도적인 인터럽트. OS 서비스 요청INT 0x80 (Linux 시스템 콜), SYSCALL 명령
소프트 IRQ소프트웨어가 명시적으로 발생Linux SoftIRQ, Tasklet
예외

Ⅲ. 나. 인터럽트 처리 절차 및 IVT
인터럽트 처리 절차 ① 인터럽트 신호 감지 (INTR 핀 활성화 또는 예외 발생) ② 현재 명령 완료 대기 (원자적 실행 보장) ③ 현재 CPU 상태 저장 → PC(Program Counter), PSW(Program Status Word), 범용 레지스터 → 스택(Stack) 또는 PCB에 Push ④ 인터럽트 벡터 테이블(IVT) 참조 → 인터럽트 번호(IRQ#)로 IVT 인덱싱 → ISR 주소 획득 ⑤ ISR(인터럽트 서비스 루틴) 실행 → 인터럽트 원인 처리 (I/O 완료 통지, 오류 처리 등) ⑥ CPU 상태 복원 → IRET(인터럽트 반환 명령) ⑦ 중단된 프로그램 실행 재개
인터럽트 벡터 테이블(IVT) / IDT
x86에서는 IDT(Interrupt Descriptor Table)를 사용. IDTR 레지스터가 IDT 시작 주소를 가리키며, 256개 항목(각 8~16바이트)으로 구성. 각 항목에는 ISR의 세그먼트:오프셋 주소, 권한(DPL), 게이트 유형(인터럽트/트랩 게이트)이 기록됨. OS 부팅 시 커널이 IDT를 초기화(LIDT 명령).
Ⅳ. 다. 다중 인터럽트 처리 방식 및 DMA 연계
처리 방식개념장점단점
순차 처리
(Disable All)		ISR 실행 중 모든 인터럽트 비활성화 (플래그로 마스킹)구현 단순, 우선순위 충돌 없음긴급 인터럽트 지연(인터럽트 레이턴시 증가)
우선순위 기반
(PIC/APIC)		현재 처리 중인 인터럽트보다 높은 우선순위 인터럽트는 선점 허용 (중첩 처리)긴급 이벤트 즉시 처리스택 오버플로 위험, 복잡한 설계
폴링(Polling)CPU가 주기적으로 장치 상태 레지스터를 확인단순 구현CPU 낭비, 실시간성 부족
DMA(Direct Memory Access)와 인터럽트 연계
DMA 없는 I/O: CPU가 I/O 장치와 메모리 간 데이터를 1Byte씩 직접 전송 (PIO) → CPU 점유 높음

DMA 방식:
① CPU가 DMA 컨트롤러에 전송 파라미터(원본·목적 주소, 크기) 설정
② DMA 컨트롤러가 CPU 개입 없이 메모리↔I/O 간 블록 전송 수행 (사이클 스틸링)
③ 전송 완료 후 DMA 컨트롤러가 CPU에 완료 인터럽트 발생
→ CPU는 전송 완료 통보만 받으면 되므로 다른 작업 가능 (비동기 I/O)
Ⅴ. 결론

인터럽트는 현대 컴퓨터의 비동기 이벤트 처리와 멀티태스킹의 근간이다. APIC(Advanced PIC)는 멀티코어 환경에서 인터럽트를 코어별로 분배하는 x2APIC로 발전하였으며, MSI(Message Signaled Interrupt)는 PCIe 장치의 인터럽트를 메모리 쓰기로 처리하여 핀 기반 IRQ의 한계를 극복한다. 인터럽트 레이턴시와 처리 효율은 실시간 임베디드 시스템의 핵심 설계 지표이다.

Ⅲ. 다. 부동소수점 오차 유형 및 AI 학습 영향·대책
  • 1
    반올림 오차(Rounding Error): 무한 소수(1/3, π)를 유한 비트로 표현 시 필연적 오차. IEEE 754는 4가지 반올림 모드(RN·RP·RM·RZ) 지원
  • 2
    취소(Catastrophic Cancellation): 거의 같은 두 수의 차(a-b, |a|≈|b|)를 구할 때 유효 비트가 급감. 예: a=1.23457, b=1.23456 → a-b=0.00001 (유효숫자 1개만 남음)
  • 3
    흡수(Absorption/Swamping): 크기 차이가 매우 큰 두 수의 덧셈 시 작은 수가 무시됨. 예: 1e20 + 1.0 ≈ 1e20
  • 4
    AI 학습에서의 영향
    그래디언트 소실(Vanishing Gradient): 역전파 시 FP32 언더플로우로 그래디언트가 0이 되어 학습 중단
    그래디언트 폭발(Exploding Gradient): FP16 학습 시 Loss Scale 없이 사용 시 오버플로우
    수치 불안정성: Softmax·LogSumExp 연산에서 취소·오버플로우 발생
  • 5
    AI 학습 대책
    혼합 정밀도 학습(Mixed Precision Training): 순전파·역전파는 BF16/FP16, 가중치 업데이트는 FP32 유지 → NVIDIA Tensor Core 활용
    Loss Scaling: FP16 그래디언트에 큰 스케일 인수를 곱해 언더플로우 방지 후 업데이트 전 역스케일
    수치 안정화 기법: Log-Sum-Exp 트릭, Stable Softmax (최댓값 빼기) 적용
    FP8 양자화: H100 GPU의 FP8 Transformer Engine으로 메모리·대역폭 50% 절감
Ⅳ. 결론

부동소수점 표현과 오차 이해는 수치 해석·AI·금융 시스템 설계에서 필수적이다. AI 학습의 효율화를 위해 FP32→BF16→FP8로 정밀도를 낮추는 추세가 이어지고 있으며, 이에 따른 수치 안정성 확보가 새로운 핵심 설계 과제로 부각되고 있다. 감리사는 AI 시스템의 학습 정밀도 설정과 Loss Scaling 적용 여부를 품질 점검 항목으로 관리해야 한다.

Ⅲ. 나. CUDA 프로그래밍 모델 — 스레드·메모리 계층
CUDA 스레드 계층 구조 Grid (전체 커널 실행 공간) └─ Block (CTA: Cooperative Thread Array, SM 하나에 스케줄) └─ Warp (32스레드, 하드웨어 스케줄 단위) └─ Thread (개별 실행 단위) 메모리 계층: Thread → 레지스터(Register File) : ~수 ns, 가장 빠름 Block → 공유 메모리(Shared Memory) : ~수십 ns, L1 캐시 수준 Grid → 글로벌 메모리(Global/HBM) : ~수백 ns, 모든 스레드 접근 (상수·텍스처 메모리: 읽기 전용, 캐시 최적화)
메모리 종류범위지연특징
레지스터Thread 전용~1 클럭가장 빠름. 오버플로우 시 Local Memory(느림)로 Spill
공유 메모리Block 공유~수 클럭프로그래머가 명시적 관리. Bank Conflict 주의
L2 캐시GPU 전체~수십 클럭자동 관리. Global 메모리 접근 완화
글로벌 메모리(HBM)모든 스레드~수백 클럭접합 접근(Coalesced Access)으로 대역폭 최대화

이종 컴퓨팅 시스템 설계 고려사항
① 데이터 이동 최소화: PCIe/NVLink/CXL로 CPU↔GPU 간 대역폭 확보. 데이터 전송이 계산보다 느리면 성능 병목
② 메모리 통합(Unified Memory): CPU-GPU 주소 공간 통합(CUDA UVM, Apple M 시리즈 Unified Memory)으로 명시적 복사 제거
③ 워크로드 분배 최적화: 제어 흐름 많은 부분은 CPU, 대규모 병렬 부분은 GPU/NPU로 분리
④ 전력 예산(Power Budget): GPU는 TDP 300~700W. 데이터센터 전력·냉각 설계 필수
⑤ 소프트웨어 스택: CUDA, ROCm, OpenCL, oneAPI 등 에코시스템 이식성 고려
Ⅴ. 결론

AI 시대의 컴퓨팅 아키텍처는 단일 CPU에서 CPU+GPU+NPU+PIM으로 이루어진 이종 가속 생태계로 진화하고 있다. NVIDIA Blackwell(B200), Google TPU v5, 삼성 HBM-PIM 등이 LLM·생성형 AI의 연산 수요를 충족시키고 있으며, CXL 3.0 기반 메모리 풀링과 칩렛(Chiplet) 패키징이 차세대 이종 컴퓨팅 플랫폼의 핵심 기술로 부상하고 있다. 시스템 감리 시에는 GPU 활용률, 메모리 대역폭, 데이터 이동 오버헤드를 핵심 성능 지표로 점검해야 한다.
