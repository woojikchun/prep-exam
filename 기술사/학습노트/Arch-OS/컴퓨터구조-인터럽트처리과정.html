Ⅰ. 인터럽트의 개념
인터럽트(Interrupt)란 CPU가 현재 실행 중인 프로그램을 일시 중단(Suspend)하고, 보다 긴급하거나 우선순위가 높은 사건(Event)을 처리한 후 원래 프로그램으로 복귀하는 메커니즘이다. CPU와 주변장치 간의 비동기적 통신 수단으로, 폴링(Polling) 방식의 자원 낭비를 해결하며 시스템 효율을 극대화한다.

Ⅰ. 인터럽트 벡터 테이블의 개념
인터럽트 벡터 테이블(IVT: Interrupt Vector Table)은 각 인터럽트 번호(벡터 번호)에 대응하는 ISR(인터럽트 서비스 루틴)의 시작 주소를 저장하는 메모리 내 테이블이다. CPU는 인터럽트 발생 시 이 테이블을 인덱싱하여 대응 ISR로 분기한다.

Ⅴ. 결론

IVT/IDT는 인터럽트 기반 I/O의 핵심 자료구조로, CPU 아키텍처·동작 모드에 따라 구현 방식이 달라진다. 보호 모드 IDT는 DPL을 통해 특권 수준을 제어함으로써 사용자 프로그램이 임의로 시스템 인터럽트를 호출하는 것을 방지한다. 운영체제의 안정성은 IDT의 올바른 초기화와 특권 레벨 설정에 크게 의존한다.

Ⅲ. 결론 — 현대 시스템의 APIC

멀티코어 시스템에서는 PIC 대신 APIC(Advanced PIC)가 사용된다. 각 CPU 코어마다 Local APIC가 있고, I/O APIC가 장치 IRQ를 관리하며 특정 코어로 인터럽트를 라우팅한다. 이를 통해 인터럽트 부하를 여러 코어에 분산하는 Interrupt Affinity 설정이 가능하며, 고성능 서버 환경에서의 인터럽트 처리 성능을 크게 향상시킨다.

Ⅰ. 개요 — 인터럽트의 필요성과 개념
인터럽트는 CPU와 주변 장치·소프트웨어 간의 비동기적 이벤트 통신 메커니즘으로, CPU가 각 장치의 상태를 지속 확인하는 폴링 방식의 비효율을 해결하기 위해 고안되었다. 인터럽트 기반 설계는 CPU가 실제 처리가 필요한 순간에만 장치를 서비스함으로써 시스템 처리량과 응답성을 동시에 향상시킨다.

우선순위 상속
(Priority Inheritance, PI)		L이 H가 필요한 자원 보유 시 L의 우선순위를 H 수준으로 임시 상승. 자원 반환 시 원복구현 직관적. 무한 대기 방지상속 체인이 길어지면 복잡. 분석 어려움

(Priority Ceiling Protocol, PCP)		각 자원에 최고 우선순위 상한값(Ceiling) 사전 정의. 자원 획득 시 상한값으로 즉시 상승교착상태 방지도 동시에 해결. 분석 용이자원별 상한값 사전 계산 필요

Ⅴ. 결론

RTOS에서 인터럽트 처리의 결정론적 응답 시간 보장은 시스템 안전성의 핵심이다. 인터럽트 지연을 최소화하기 위해 ISR은 Top-Half 최소 처리·Bottom-Half 지연 처리 구조를 따르고, ISR-태스크 간 동기화는 Non-blocking 메커니즘을 사용해야 한다. 우선순위 역전은 PI 또는 PCP 프로토콜로 반드시 해결해야 하며, 이는 항공·자동차·의료 등 안전 필수(Safety-Critical) 시스템의 IEC 61508·DO-178C 인증 요건이기도 하다.

Ⅴ. 결론

인터럽트와 프로세스 스케줄링은 현대 OS 멀티태스킹의 두 기둥이다. 타이머 인터럽트는 선점 스케줄링을 가능하게 하고, I/O 완료 인터럽트는 차단된 프로세스를 깨워 CPU 활용률을 극대화한다. 컨텍스트 스위칭의 TLB 비용과 캐시 오염을 최소화하는 것이 고성능 서버 시스템 설계의 핵심 과제이며, PCID·스레드 풀·비동기 I/O(io_uring) 등이 이를 위한 현대적 해법이다.

Ⅰ. 개요
단일 코어·단일 시스템 시대의 PIC 기반 인터럽트 아키텍처는 멀티코어·가상화 환경의 요구사항을 충족하지 못한다. APIC는 멀티코어 환경에서 인터럽트를 여러 코어에 분산하고, 가상화 환경에서는 VMM이 인터럽트 가상화를 담당하여 게스트 OS에 투명한 인터럽트 처리를 제공한다.
Ⅱ. 가. APIC 아키텍처 — Local APIC·I/O APIC·IRQ Affinity

IRQ Affinity 최적화 사례 — 고성능 네트워크

Ⅳ. 다. 인터럽트 처리 성능 최적화 기법
  • 1
    인터럽트 코얼레싱(Interrupt Coalescing): 여러 인터럽트를 묶어서 한 번에 처리. NIC이 여러 패킷을 수신해도 일정 시간/패킷 수 기준으로 인터럽트 1회만 발생. Intel ethtool: ethtool -C eth0 rx-usecs 50. 처리량 향상 vs 지연 트레이드오프
  • 2
    NAPI(New API) — 네트워크 폴링 하이브리드: 고속 네트워크에서 인터럽트 과부하 문제 해결. 첫 패킷 인터럽트로 NAPI 활성화 → 이후 인터럽트 비활성화 + 소프트IRQ에서 폴링으로 남은 패킷 일괄 처리 → 큐 소진 시 인터럽트 재활성화. 100Gbps 이상 네트워크에서 필수
  • 3
    인터럽트 스레딩(Threaded IRQ): 기존 ISR Top-Half/Bottom-Half 대신 ISR을 커널 스레드로 분리 실행. request_threaded_irq(). ISR 내 슬립 가능. PREEMPT_RT(실시간 패치)에서 완전 선점형 인터럽트 처리 가능. 지연 예측성(Determinism) 향상
  • 4
    io_uring (Linux 5.1+) — 인터럽트 의존도 감소: 사용자 공간과 커널 공간 간의 공유 링 버퍼로 I/O 요청·완료를 시스템 콜·인터럽트 없이 처리. 제출 큐(SQ)·완료 큐(CQ)를 통한 비동기 배치 처리. I/O 집약적 워크로드에서 인터럽트 오버헤드를 획기적으로 감소
io_uring vs 전통 인터럽트 기반 I/O 비교 [전통 인터럽트 기반 read()] 사용자: read() 시스템 콜 → 커널 진입 (시스템 콜 오버헤드) 커널: I/O 요청 → 프로세스 SLEEP 디스크 ISR: 완료 인터럽트 → 프로세스 WAKE 커널: 데이터 복사 → 사용자 공간 복귀 (시스템 콜 반환) 오버헤드: 시스템 콜 2회 + 인터럽트 1회 + 컨텍스트 스위칭 [io_uring 비동기 I/O] 사용자: SQ(제출 큐)에 I/O 요청 기록 (시스템 콜 없음) 커널: SQ 폴링 → I/O 실행 (io_uring 커널 스레드) 완료: CQ(완료 큐)에 결과 기록 (인터럽트 최소) 사용자: CQ 폴링으로 결과 확인 (시스템 콜 없음) 오버헤드: 사실상 0 (공유 메모리 링 버퍼만 사용)
Ⅴ. 결론

멀티코어·가상화 환경에서 인터럽트 아키텍처는 APIC 기반 분산 처리, 하드웨어 지원 가상화(APIC-v·AVIC), NAPI·io_uring 등 소프트웨어 최적화 기법이 총동원되어 진화하고 있다. 클라우드 데이터센터에서는 SR-IOV로 물리 장치를 게스트에 직접 할당하여 인터럽트 가상화 오버헤드를 제거하며, io_uring의 폴링 기반 I/O는 100만 IOPS 이상의 NVMe 스토리지 환경에서 인터럽트 기반 한계를 뛰어넘는 핵심 기술로 자리잡고 있다.
