Ⅰ. 파이프라인의 개념
파이프라인(Pipeline)은 하나의 명령어 처리를 여러 독립적인 단계로 분할하여, 각 단계를 병렬로 중첩 실행함으로써 CPU의 처리량(Throughput)을 향상시키는 기법이다. 공장의 조립 라인(Assembly Line)에서 착안한 개념으로, 단일 명령어의 지연(Latency)은 줄이지 못하지만 단위 시간당 완료 명령어 수(처리량)를 크게 향상시킨다.

Ⅲ. 파이프라인 성능 지표 계산
📐 핵심 성능 공식
비파이프라인 실행 시간 = n × k × Δt
파이프라인 실행 시간 = (k + n - 1) × Δt
속도 향상(Speedup) = n×k×Δt / (k+n-1)×Δt = nk / (k+n-1)
→ n → ∞ 시 Speedup → k (이상적 최대 속도 향상 = 단계 수)

여기서 n = 명령어 수, k = 파이프라인 단계 수, Δt = 각 단계 수행 시간
예시 계산: n=100개 명령어, k=5단계, Δt=2ns일 때
비파이프라인: 100 × 5 × 2ns = 1,000ns 파이프라인: (5 + 100 - 1) × 2ns = 208ns 속도 향상: 1000 / 208 ≈ 4.81배 (이상적 최대: 5배) CPI(파이프): 이상적 = 1.0 (해저드 없는 경우) 처리량: 1개/2ns = 500 MIPS (클럭=500MHz 기준)
📌 파이프라인 효율 저하 요인
단계별 수행 시간 불균형 → 가장 느린 단계가 클럭 주기 결정 (병목)
해저드(Hazard) 발생 → 스톨(Stall) 삽입으로 CPI > 1.0
실제 CPI = 이상 CPI(1.0) + 해저드로 인한 평균 스톨 사이클
Ⅳ. 결론 — 감리·시험 포인트

파이프라인은 현대 CPU의 근간 기술로, ARM Cortex-A/X 계열은 10~13단계, Intel Core 계열은 14~19단계 파이프라인을 사용한다. 기술사 시험에서는 타임 다이어그램 작성, Speedup 계산, 해저드 유형별 해결 방안이 핵심 출제 포인트이다.

Ⅰ. 파이프라인 해저드 개요
파이프라인 해저드(Hazard)는 다음 명령어를 예정된 클럭 사이클에 실행하지 못하게 하는 상황으로, 성능 저하(CPI 증가)의 직접적 원인이다. 크게 구조적·데이터·제어 해저드의 3가지로 분류된다.
Ⅱ. 구조적 해저드 (Structural Hazard)
개념·원인: 여러 명령어가 동일한 하드웨어 자원(메모리, ALU, 레지스터 파일 등)을 동시에 사용하려 할 때 발생. 하드웨어 자원 부족이 원인
예시: 단일 메모리 구조에서 IF(I4)와 MEM(I1)이 동시에 메모리 접근 시도 CC1 CC2 CC3 CC4 CC5 CC6 I1: IF - ID - EX - MEM- WB ← 메모리 쓰기 I4: IF ← 충돌! 동시에 메모리 읽기 시도 ↑ 구조적 해저드: 메모리가 하나뿐임
해결 기법
① 자원 복제(Resource Duplication): 명령어 메모리(I-Cache)와 데이터 메모리(D-Cache) 분리 → Harvard 아키텍처 채택 (MIPS, ARM 등 대부분의 현대 CPU)
② 스톨(Stall/버블) 삽입: 충돌 회피를 위해 파이프라인을 일시 정지 (NOP 삽입)
③ 파이프라인 자원 분리: EX 단계에 ALU 다중 배치
Ⅲ. 데이터 해저드 (Data Hazard)
개념·원인: 명령어 간 데이터 의존성으로 인해 이전 명령어의 결과가 준비되기 전에 후속 명령어가 해당 데이터를 필요로 할 때 발생
유형정의예시위험도
RAW
(Read After Write)		쓰기 전에 읽기: 이전 명령이 쓰기 완료 전에 후속 명령이 같은 레지스터를 읽음. 진성 의존성ADD R1,R2,R3 → SUB R4,R1,R5
(R1 의존)		가장 심각 (실제 값 오류)
WAR
(Write After Read)		읽기 전에 쓰기: 이전 명령이 읽기 완료 전에 후속 명령이 같은 위치에 쓰기. 반의존성순차 파이프라인에서는 발생 안함비순차 실행(OOO)에서 발생
WAW
(Write After Write)		두 명령이 같은 목적지에 쓰기. 최종 값이 잘못될 수 있음. 출력 의존성순차 파이프라인에서는 발생 안함비순차 실행·다중 이슈에서 발생
RAW 해결 기법 (핵심)
① 포워딩/바이패싱(Forwarding/Bypassing): EX 단계 결과를 WB 완료 전에 다음 명령의 EX 단계 입력으로 직접 전달하는 전용 경로 추가. 대부분의 RAW 해결
② 스톨(Stall) 삽입: 포워딩으로도 해결 안 되는 Load-Use 해저드 시 1 사이클 버블 삽입
③ 코드 재배열(Code Scheduling): 컴파일러가 독립 명령어를 의존 명령어 사이에 재배치하여 스톨 회피
Ⅳ. 제어 해저드 (Control Hazard)
개념·원인: 분기(Branch)·점프(Jump) 명령어 실행 시, 다음에 실행할 명령어 주소를 파이프라인이 아직 모르는 상태에서 IF 단계가 진행될 때 발생. 분기 결과가 MEM 또는 EX 단계에서 결정되므로 그 이전에 인출된 명령어들이 무효가 될 수 있음
해결 기법
① 파이프라인 플러시(Pipeline Flush): 분기 결과 확정 후 잘못 인출된 명령어 취소. 분기 페널티 = 결과 확정 단계 - 1 사이클
② 지연 분기(Delayed Branch): 분기 명령 직후 슬롯(Branch Delay Slot)에 항상 유용한 명령 배치. MIPS/SPARC 채택
③ 분기 예측(Branch Prediction): 정적(항상 분기 안 함/함) 또는 동적(BHT·BTB·2비트 예측기) 방식으로 분기 방향 예측. 현대 CPU의 표준 기법 (별도 문제로 상세 설명)
Ⅴ. 결론 — 3가지 해저드 비교 요약

구조적 해저드는 하드웨어 복제(I/D-Cache 분리)로, 데이터 해저드(RAW)는 포워딩+컴파일러 재배열로, 제어 해저드는 동적 분기 예측으로 대부분 해결한다. 현대 고성능 CPU(OOO 프로세서)는 레지스터 리네이밍(Register Renaming)으로 WAR·WAW를 제거하고, 투기적 실행(Speculative Execution)으로 제어 해저드를 숨긴다.

Ⅰ. 동적 분기 예측의 개념
동적 분기 예측은 프로그램 실행 중 분기 명령어의 실제 이력(History)을 하드웨어에 기록·분석하여 다음 분기 방향을 예측하는 기법이다. 정적 예측(컴파일 시 결정, 정확도 ~65%)보다 높은 정확도(현대 CPU ~97%)를 달성한다.
Ⅱ. 2비트 포화 카운터 동작 원리
2비트(4상태) 포화 카운터는 분기 이력을 2비트로 표현하여 한 번의 예측 실패로 예측 방향이 바뀌지 않도록 이력 관성(Hysteresis)을 제공한다.
2비트 포화 카운터 상태 천이도 분기함(T) 분기함(T) ┌──────────────────────────────┐ │ ↓ [강 분기안함] → [약 분기안함] → [약 분기함] → [강 분기함] (SN=00) (WN=01) (WT=10) (ST=11) │ ↑ │ ↑ └──────────────────┘ └──────────────────┘ 분기안함(NT) 분기함(T) 예측 규칙: 상태 00(SN), 01(WN) → "분기 안 함" 예측 (Not Taken) 상태 10(WT), 11(ST) → "분기 함" 예측 (Taken) 포화(Saturate): 00에서 NT → 00 유지 / 11에서 T → 11 유지
2비트 vs 1비트 예측기 비교
1비트: 마지막 분기 결과 그대로 예측. 루프 최초·최후 1회씩 오예측 (루프 내 정확도 = (n-2)/n)
2비트: 상태 변화에 이력 관성 부여. 루프 최초 1회만 오예측 (루프 내 정확도 = (n-1)/n)
→ 2비트가 루프 구조에서 현저히 우수

Ⅰ. ILP(명령어 수준 병렬성)의 개념
ILP(Instruction-Level Parallelism)는 단일 프로세서 내에서 독립적인 명령어들을 동시에 실행함으로써 처리량을 높이는 기법이다. CPI(Cycles Per Instruction)를 1.0 미만으로 낮추는 것이 목표이며, 슈퍼스칼라·VLIW·슈퍼파이프라인이 대표적 구현 방식이다.
Ⅱ. 3가지 ILP 아키텍처 상세
  • 1
    슈퍼스칼라(Superscalar): 여러 개의 독립적인 실행 유닛(ALU·FPU·Load/Store)을 갖추고, 런타임에 하드웨어(동적 스케줄러)가 매 클럭 다수의 명령어를 동시 발행(Issue)·실행하는 아키텍처
    구조: 다중 디코더 → 발행 큐(Issue Queue/Reservation Station) → 복수 실행 유닛 → 재순서 버퍼(ROB)
    스케줄링: 동적(하드웨어). Tomasulo 알고리즘, 레지스터 리네이밍으로 WAR·WAW 제거
    장점: 바이너리 호환성 유지. 하드웨어가 자동 병렬화. 일반 순차 코드에서도 ILP 추출
    단점: 복잡한 하드웨어(높은 전력·면적). 발행 폭(Issue Width) 증가에 따른 수확 체감
    사례: Intel Core i9(6-wide), Apple M4(슈퍼스칼라 OOO), ARM Cortex-A77
  • 2
    VLIW(Very Long Instruction Word): 컴파일러가 정적으로 병렬 실행 가능한 명령어들을 하나의 긴 명령어 워드(수백 비트)로 패키징하면, 하드웨어는 단순히 이를 각 실행 유닛에 분배하는 아키텍처
    구조: 폭넓은 명령어 워드 = ALU op + FPU op + Load op + Store op + Branch op 묶음
    스케줄링: 정적(컴파일러). 컴파일 시 의존성 분석·재배열·루프 언롤링·소프트웨어 파이프라이닝
    장점: 단순한 하드웨어(낮은 전력). 컴파일러 최적화가 잘 되는 DSP·임베디드에 적합
    단점: 바이너리 비호환(하드웨어 변경 시 재컴파일 필요). 불규칙 코드(분기 多)에서 빈 슬롯 낭비. 컴파일러 복잡도 매우 높음
    사례: Intel Itanium(IA-64), TI C6000 DSP, EPIC(Explicitly Parallel Instruction Computing)
  • 3
    슈퍼파이프라인(Superpipelined): 파이프라인 단계를 더 세분화(깊은 파이프라인)하여 각 단계 수행 시간을 줄이고 클럭 주파수를 높임으로써 처리량을 향상시키는 아키텍처
    원리: 5단계 × 2ns = 500MHz → 10단계 × 1ns = 1GHz. 같은 작업을 더 높은 주파수로 처리
    장점: 단순 구조. 높은 클럭 주파수 달성 가능
    단점: 파이프라인이 깊을수록 분기 예측 실패 페널티 증가. 데이터 해저드 심화. 전력 소모 증가
    사례: Intel Pentium 4 Prescott(31단계 NetBurst), MIPS R4000(8단계)

Ⅳ. 결론

현대 고성능 CPU는 슈퍼스칼라 + OOO(비순차 실행) + 슈퍼파이프라인을 결합하여 ILP를 극대화한다. 반면 저전력 임베디드·IoT 영역에서는 VLIW 기반 DSP가 전력 효율로 우위를 점한다. Amdahl 법칙에 따라 ILP만으로는 성능 향상에 한계가 있으므로, 현대는 TLP(스레드 수준 병렬성, 멀티코어)와 DLP(데이터 수준 병렬성, SIMD)를 병행하는 방향으로 발전하고 있다.

Ⅰ. 캐시 메모리와 파이프라인의 관계
캐시 메모리는 CPU와 주메모리 사이에 위치하는 고속 소용량 메모리로, 파이프라인의 IF·MEM 단계에서 발생하는 메모리 접근 지연(Memory Access Latency)을 최소화하는 핵심 요소이다. 캐시 미스(Cache Miss) 발생 시 파이프라인이 스톨되어 성능이 저하된다.

Ⅲ. 캐시 일관성(Cache Coherence)과 MESI 프로토콜
문제 정의: 멀티코어 프로세서에서 각 코어가 독립적인 L1/L2 캐시를 보유할 때, 여러 코어가 동일 메모리 주소의 복사본을 각자의 캐시에 가지고 있으면 일관성 문제 발생.
MESI 프로토콜 — 4가지 캐시 라인 상태 M (Modified) : 자신의 캐시에만 존재. 주메모리와 다른 값(더티). 독점 소유 → 다른 코어 Read 요청 시: 주메모리 업데이트(Write-back) 후 S로 전환 E (Exclusive) : 자신의 캐시에만 존재. 주메모리와 동일(클린). 독점 소유 → 다른 코어 Read 요청 시: S로 전환 S (Shared) : 여러 코어가 동일 내용 보유. 주메모리와 동일(클린). 읽기 전용 → 쓰기 발생 시: 다른 코어 캐시 무효화(Invalidate) 후 M으로 전환 I (Invalid) : 해당 캐시 라인이 유효하지 않음. 미스로 처리 상태 천이 요약: 처음 읽기: I → E (독점) 또는 I → S (공유) 쓰기(독점): E → M 쓰기(공유): S → M (다른 코어 S→I 무효화, Invalidation) 다른 코어 읽기: M → S (Write-back 후)
MESI 구현 방식
스누핑(Snooping) 방식: 각 캐시 컨트롤러가 공유 버스를 모니터링(Snoop)하여 자신의 캐시 상태를 갱신. 소규모 멀티코어(2~8코어) 적합. 버스 대역폭이 병목
디렉터리(Directory) 방식: 중앙 디렉터리가 각 메모리 블록의 상태·공유 정보 관리. 대규모 멀티코어(NUMA, 수십~수백 코어) 적합
Ⅳ. 결론

캐시 메모리는 파이프라인 성능을 결정하는 핵심 요소로, 현대 CPU는 L1(4-way, ~32KB)/L2(8-way, ~256KB)/L3(16-way, ~수십MB)의 계층형 캐시 구조를 채택한다. 멀티코어 시대에는 MESI(또는 MOESI, MESIF 확장) 프로토콜을 통한 캐시 일관성 유지가 필수이며, 이는 동시성 프로그래밍의 메모리 모델(volatile, memory barrier)과도 직결된다.

Ⅰ. 개요 — 파이프라인 해저드의 영향
파이프라인 해저드는 이상적인 CPI(=1.0)를 증가시켜 실질 성능을 저하시키는 요인으로, 실제 CPI = 1.0 + 데이터 해저드 스톨 + 제어 해저드 스톨로 표현된다. 해저드를 하드웨어(포워딩·분기 예측)와 소프트웨어(컴파일러 스케줄링)의 협력으로 최소화하는 것이 고성능 프로세서 설계의 핵심이다.
Ⅱ. 가. 데이터 해저드와 포워딩 기법
RAW(Read After Write) 포워딩 동작 원리: 이전 명령어의 ALU 결과를 WB 완료 전에 다음 명령어의 EX 단계 입력으로 직접 연결하는 전용 데이터 경로(Bypass Path)를 추가한다.
포워딩 타임 다이어그램 명령어 순서: I1: ADD R1, R2, R3 ← R1에 결과 저장 I2: SUB R4, R1, R5 ← R1 사용 (RAW 의존) I3: AND R6, R1, R7 ← R1 사용 (RAW 의존) 포워딩 없는 경우 (스톨 발생): CC1 CC2 CC3 CC4 CC5 CC6 CC7 CC8 I1: IF - ID - EX - MEM- WB I2: IF - ID -[스톨][스톨]EX - MEM- WB ← 2사이클 스톨 I3: IF -[스톨][스톨][스톨]... 포워딩 적용 후 (스톨 제거): CC1 CC2 CC3 CC4 CC5 CC6 CC7 I1: IF - ID - EX ─►MEM- WB │포워딩(EX→EX) I2: IF - ID - EX - MEM- WB ← 스톨 없음! │포워딩(MEM→EX) I3: IF - ID - EX - MEM- WB ← 스톨 없음! 포워딩 경로: EX/MEM 레지스터 → 다음 명령 EX 입력 (EX→EX 포워딩, 1사이클 간격) MEM/WB 레지스터 → 다다음 명령 EX 입력 (MEM→EX 포워딩, 2사이클 간격)
WAR·WAW 해저드 처리
WAR(Write After Read): 순차 파이프라인에서는 ID(읽기)가 WB(쓰기)보다 먼저 실행되므로 발생하지 않음. 비순차(OOO) 실행에서만 문제
WAW(Write After Write): 순차 파이프라인에서는 발생 안 함. 멀티사이클 FPU 등에서 발생 가능 → 레지스터 리네이밍으로 해결
Ⅲ. 나. Load-Use 해저드와 컴파일러 스케줄링

컴파일러 루프 최적화 기법
루프 언롤링(Loop Unrolling): 루프 본체를 N배 복제하여 독립 명령어 수 증가 → Load Delay Slot 채우기 용이

Ⅰ. 개요 — 비순차 실행의 개념과 필요성
비순차 실행(OOO: Out-of-Order Execution)은 프로그램의 원래 순서(프로그램 순서, Program Order)에 관계없이, 데이터 의존성이 없는 명령어들을 준비되는 즉시 실행함으로써 파이프라인 스톨을 최소화하고 ILP를 극대화하는 기법이다. IBM 360/91 (1966년)에서 최초 구현되었으며, Tomasulo의 알고리즘이 이론적 기반이다.
OOO 필요성 — 순차 실행의 한계
순차 실행: 앞 명령어가 스톨되면 뒤의 독립적 명령어도 모두 대기
예: I1(LW 10사이클 Cache Miss) → I2~I10이 모두 대기(I2~I10이 I1에 무관해도)
OOO: I1이 스톨 중에도 I2~I10 중 독립 명령어는 즉시 실행 → 스톨 시간을 유용한 실행으로 대체
Ⅱ. 가. Tomasulo 알고리즘 핵심 구성요소

    예약 스테이션(Reservation Station, RS): 발행된 명령어와 피연산자를 보관하는 큐. 피연산자가 아직 준비 안 된 경우 생성 명령어의 RS 태그를 저장. 피연산자 준비 즉시 실행 유닛으로 전송
    RS 항목: {Op, Vj, Vk(피연산자 값), Qj, Qk(대기 중인 RS 태그), Busy 비트}

  • 3
    재순서 버퍼(Reorder Buffer, ROB): 투기적으로 실행된 명령어의 결과를 프로그램 순서대로 커밋(Commit)하기 위한 순환 큐. 오예측·예외 발생 시 ROB를 비워 상태 롤백. 명확하게 커밋된 결과만 아키텍처 상태에 반영하여 정확한 예외(Precise Exception) 보장
Ⅲ. 나. 레지스터 리네이밍 — WAR·WAW 제거
개념: 아키텍처 레지스터(ISA 레지스터, 예: x86의 32개)를 더 많은 물리 레지스터(Physical Register, 예: Intel Core의 280개)로 동적 매핑하여, WAR·WAW 이름 의존성(Name Dependency)을 제거하는 기법이다.
레지스터 리네이밍 예시 (WAW 제거) 원래 코드: I1: MUL R1, R2, R3 → R1 쓰기 (WAW) I2: ADD R4, R1, R5 → R1 읽기 (I1에 RAW) I3: SUB R1, R6, R7 → R1 쓰기 (WAW: I1과 같은 R1에 쓰기) I4: AND R8, R1, R9 → R1 읽기 (I3에 RAW) 리네이밍 후 (물리 레지스터 P10~P13 사용): I1: MUL P10, R2, R3 → P10에 저장 (R1→P10 매핑) I2: ADD P11, P10, R5 → P10 사용 (진성 RAW 유지) I3: SUB P12, R6, R7 → P12에 저장 (R1→P12로 재매핑!) I4: AND P13, P12, R9 → P12 사용 (WAW 해결: I1과 독립) 결과: I1과 I3의 WAW 제거 → I1·I3 병렬 실행 가능!
리네이밍 구현 방식
ROB 기반: ROB 항목 자체가 물리 레지스터 역할. 커밋 시 아키텍처 파일에 복사
