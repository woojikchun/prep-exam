Ⅰ. 교착상태의 개념, 4가지 필요충분 조건 및 자원 할당 그래프
1. 교착상태(Deadlock)의 정의

교착상태(Deadlock)란 둘 이상의 프로세스 집합이 서로 상대방이 점유한 자원을 기다리며 어떤 프로세스도 영원히 진행할 수 없는 무한 대기 상태를 의미한다. 1965년 E. W. Dijkstra가 최초 기술하였으며, 운영체제·데이터베이스·분산 시스템에서 공통으로 발생하는 핵심 동시성 문제이다.

교착상태는 기아(Starvation)와 구별해야 한다. 기아는 자원을 얻을 가능성이 있으나 계속 지연되는 상태인 반면, 교착상태는 관련된 모든 프로세스가 영원히 자원을 얻을 수 없다는 점에서 본질적으로 다르다.

2. 교착상태 발생의 4가지 필요충분 조건 (Coffman, 1971)

아래 4가지 조건이 동시에 성립할 때만 교착상태가 발생한다. 따라서 하나라도 제거하면 교착상태를 예방할 수 있다.

①
상호 배제 (Mutual Exclusion)
한 번에 하나의 프로세스만 자원 사용
자원은 공유 불가능하며, 하나의 프로세스만이 독점적으로 사용 가능. 공유 가능 자원(읽기 전용 파일 등)은 해당 없음
②
점유와 대기 (Hold and Wait)
자원 보유 중 추가 자원 대기
최소 1개의 자원을 보유한 채로 다른 프로세스가 점유 중인 추가 자원을 획득하기 위해 대기하는 상태
③
비선점 (Non-preemption)
자원을 강제로 빼앗을 수 없음
프로세스가 보유한 자원은 해당 프로세스가 자발적으로 반환하기 전까지 OS가 강제로 회수 불가
④
순환 대기 (Circular Wait)
프로세스 간 순환 형태의 대기
{P0→P1→P2→…→Pn→P0} 형태로 각 프로세스가 다음 프로세스가 보유한 자원을 기다리는 순환 구조 형성
【표 1】 교착상태 4가지 필요충분 조건 (Coffman Conditions, 1971)
3. 자원 할당 그래프 (Resource Allocation Graph, RAG)

RAG는 프로세스(원)와 자원(사각형)을 노드로, 할당(자원→프로세스)과 요청(프로세스→자원)을 방향 간선으로 표현한 그래프이다. 사이클(Cycle) 존재 = 교착상태 가능성을 의미하며, 자원 인스턴스가 1개인 경우 사이클 = 교착상태가 확정된다.

【그림 1】 자원 할당 그래프(RAG) — 좌: 교착상태(사이클), 우: 교착상태 없음(잔여 인스턴스)

Ⅲ. 교착상태 예방(Prevention)과 회피(Avoidance)
1. 교착상태 예방 — 4대 조건 각각의 제거
제거 대상 조건구현 방법문제점
① 상호 배제 제거 공유 가능 자원(읽기 전용 파일, 프린터 스풀링)으로 전환 본질적으로 공유 불가한 자원(뮤텍스 등)은 제거 불가
② 점유·대기 제거 ① 모든 자원 한꺼번에 요청·할당(All-or-Nothing)
② 자원 요청 시 보유 자원 전부 반환 후 재요청		 자원 이용률 극히 낮음, 기아 가능성, 실용성 부족
③ 비선점 제거 추가 자원 획득 불가 시 보유 자원 강제 선점·반환. CPU·레지스터처럼 상태 저장 가능한 자원에 한해 적용 프린터·테이프 등 중간 상태 저장 불가 자원에는 적용 불가
④ 순환 대기 제거 자원에 전역 순서(Ordering) 부여 — 프로세스는 반드시 오름차순으로만 자원 요청 가능 자원 순서 설계 복잡, 번호 체계 변경 시 전체 재설계 필요
【표 3】 교착상태 예방 — 4대 조건 제거 방법
2. 교착상태 회피 — 안전 상태(Safe State)의 개념

안전 상태(Safe State)란 모든 프로세스가 요구하는 최대 자원을 순차적으로 할당·완료할 수 있는 안전 순서열(Safe Sequence)이 존재하는 상태이다. 시스템이 항상 안전 상태를 유지하도록 자원을 할당하면 교착상태가 발생하지 않는다.

안전 상태 ⊂ 교착상태 없음    불안전 상태 ⊃ 교착상태 가능
3. 은행원 알고리즘 (Banker's Algorithm — Dijkstra, 1965)

자원 할당 요청 시 가상으로 자원을 할당한 뒤 안전 상태 여부를 검사하고, 안전 상태일 때만 실제 할당하는 회피 알고리즘이다. 마치 은행이 대출 후 파산 여부를 시뮬레이션하는 방식과 유사하다.

가. 알고리즘 핵심 데이터 구조
자료구조의미크기
Allocation[i][j]프로세스 i에 현재 할당된 자원 j의 수n × m
Max[i][j]프로세스 i가 요구할 수 있는 자원 j의 최대량n × m
Need[i][j]= Max[i][j] − Allocation[i][j] (잔여 필요량)n × m
Available[j]현재 시스템에서 가용한 자원 j의 수1 × m

라. 은행원 알고리즘의 한계

① 최대 자원 요구량 사전 선언 필요 → 동적 환경에서 현실적 적용 어려움

② 프로세스·자원 수 고정 가정 → 동적 생성·소멸 환경 부적합

③ O(n² × m) 시간 복잡도 → 프로세스·자원 수 증가 시 오버헤드

Ⅳ. 교착상태 탐지(Detection)와 복구(Recovery) 및 실무 고려사항
1. 교착상태 탐지 알고리즘
구분방법시간 복잡도
단일 인스턴스 자원 자원 할당 그래프(RAG)에서 사이클 탐지 (DFS). 사이클 = 교착상태 O(V + E)
복수 인스턴스 자원 은행원 알고리즘 변형 — Wait-for Graph 또는 Allocation·Request 행렬로 완료 가능 프로세스 탐색. 완료 불가 프로세스 = 교착상태 O(n² × m)
【표 7】 교착상태 탐지 알고리즘
2. 탐지 주기 설계 트레이드오프

탐지 알고리즘을 너무 자주 실행하면 오버헤드 증가, 너무 드물게 실행하면 교착 상태로 묶인 시간 증가. 일반적으로 자원 요청이 즉시 승인되지 않는 시점에 트리거하거나, CPU 이용률이 일정 임계값 이하로 떨어질 때 실행하는 전략이 유효하다.

Ⅴ. 결론

교착상태는 상호 배제·점유 대기·비선점·순환 대기 4가지 조건이 동시에 충족될 때 발생하며, 이를 처리하는 방법은 예방·회피·탐지·복구의 네 전략으로 분류된다. 실무에서는 단일 전략보다 워크로드 특성에 맞는 복합 전략이 적합하다. DBMS는 탐지·복구, RTOS는 예방 중심, 범용 OS는 무시에 가까운 접근을 택한다. 분산 환경에서는 글로벌 상태 수집의 어려움으로 타임아웃 기반 복구가 현실적 대안으로 활용된다.

★ 기술사 핵심 코멘트
교착상태의 본질은 자원의 순환 의존(Circular Dependency)에 의한 무한 대기이며, 시스템 감리 시 다음을 중점 검토해야 한다.
① 자원 할당 정책 적정성: 복수 자원 동시 획득 로직에서 자원 요청 순서의 전역 일관성(Ordering) 준수 여부 확인 — 순환 대기 예방의 핵심
② DBMS 트랜잭션 격리: 교착 탐지 주기·희생자 롤백 정책 설계 적정성, MVCC 적용으로 교착 빈도 최소화 여부
③ 분산 시스템 타임아웃 설계: 타임아웃 값이 지나치게 짧으면 정상 지연을 교착으로 오인, 지나치게 길면 복구 지연 — 워크로드 기반 튜닝 필요
④ 멀티스레드 애플리케이션: 락 획득 순서 코딩 표준 수립·정적 분석 도구(lockdep, ThreadSanitizer) 적용 여부 감리
