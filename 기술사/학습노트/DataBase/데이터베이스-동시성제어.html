Ⅰ. 동시성 제어의 필요성과 두 전략의 기본 철학
1. 동시성 제어(Concurrency Control)의 필요성

다수의 트랜잭션이 동시에 동일 데이터에 접근·수정할 때 갱신 손실(Lost Update)·더티 읽기(Dirty Read)·반복 불가 읽기(Non-Repeatable Read) 등의 이상 현상이 발생한다. 동시성 제어는 이러한 이상 현상을 방지하면서 동시성(Throughput)과 일관성(Correctness)을 균형 있게 유지하는 핵심 메커니즘이다.

2. 두 전략의 기본 철학 비교
낙관적 동시성 제어 (OCC)
철학: "충돌은 드물다 — 일단 실행하고 커밋 직전에만 충돌을 검사한다."

작업 중 아무 잠금도 획득하지 않고 자유롭게 읽기·수정을 수행한다. 커밋 시점에 다른 트랜잭션이 동일 데이터를 변경했는지 검증하고, 충돌이 감지되면 롤백·재시도한다.

적합 환경: 읽기 집약, 충돌 빈도 낮음
비관적 동시성 제어 (PCC)
철학: "충돌은 언제든 발생한다 — 데이터 접근 전부터 잠금으로 미리 방지한다."

데이터를 읽거나 수정하기 전에 잠금(Lock)을 획득하여 다른 트랜잭션의 접근을 차단한다. 트랜잭션 종료 시 잠금을 해제한다.

적합 환경: 쓰기 집약, 충돌 빈도 높음
【그림 1】 충돌 빈도에 따른 OCC·MVCC·PCC 적합 구간
Ⅱ. 낙관적 동시성 제어(OCC)의 동작 원리
1. OCC 3단계 절차
① 읽기 단계 (Read Phase) 데이터를 읽어 프라이빗 작업 공간(Private Workspace)에 복사·수정.
잠금 없음. 타임스탬프/버전 번호 기록
→
② 검증 단계 (Validation Phase) 커밋 직전 충돌 검사.
"내가 읽는 동안 다른 T가 같은 데이터를 수정했는가?"
충돌 시 → 롤백
→
③ 쓰기 단계 (Write Phase) 검증 통과 시 작업 공간의 변경 내용을 실제 DB에 반영.
검증 실패 시 → Abort + 재시도
【그림 2】 OCC 3단계 절차 — Read → Validate → Write
2. 충돌 탐지 메커니즘 — 버전 번호(Version Number) 방식

각 행(Row)에 버전 번호(Version) 또는 타임스탬프를 유지한다. 트랜잭션은 데이터 읽기 시 버전을 기록하고, 쓰기 시 현재 DB의 버전과 비교한다.

읽은 버전 == 현재 DB 버전 → 검증 통과 → 커밋 (버전 +1)
읽은 버전 != 현재 DB 버전 → 충돌 감지 → 롤백·재시도
3. OCC 타임라인 예제 — 재고 차감 시나리오
시각 T1 (주문 A: 재고 -3) T2 (주문 B: 재고 -5) DB 재고 (버전)
t1재고=10, v=1 읽기 (로컬 복사)—10 (v=1)
t2—재고=10, v=1 읽기 (로컬 복사)10 (v=1)
t3로컬: 재고 = 10-3 = 7 계산—10 (v=1)
t4—로컬: 재고 = 10-5 = 5 계산10 (v=1)
t5검증: DB v=1 == 읽은 v=1 ✔
커밋: 재고=7, v=2		—7 (v=2)
t6—검증: DB v=2 ≠ 읽은 v=1 ✗
충돌! → ROLLBACK → 재시도		7 (v=2)
t7—재시도: 재고=7, v=2 재읽기
→ 7-5=2 계산 → 커밋 (v=3)		2 (v=3)
【표 1】 OCC 버전 번호 기반 충돌 탐지 타임라인
4. OCC 구현 방식 비교
구현 방식메커니즘사용 예
버전 번호 (Version) 행마다 version 컬럼 유지. UPDATE 시 WHERE version=? 조건 추가. 영향 행 수=0이면 충돌 JPA @Version, Hibernate
타임스탬프 (Timestamp) 읽기·쓰기 타임스탬프 비교. Thomas Write Rule 적용으로 일부 충돌 허용 분산 DB, Google Spanner
ETag (HTTP) REST API에서 응답 ETag를 If-Match 헤더로 전송. 불일치 시 409 Conflict REST API 낙관적 갱신
CAS (Compare-And-Swap) 원자적 비교 교환 연산. 기대값과 현재값 일치 시만 갱신 Redis, 캐시 계층
【표 2】 OCC 구현 방식 비교
5. JPA @Version 코드 예시
// JPA 낙관적 잠금 — @Version 어노테이션
@Entity
public class Product {
    @Id private Long id;
    private int stock;
    @Version              // ← 버전 컬럼 자동 관리
    private int version;
}

// 서비스 레이어 — 충돌 시 OptimisticLockException 처리
try {
    Product p = em.find(Product.class, id);  // version=1 읽기
    p.setStock(p.getStock() - 3);           // 로컬 수정
    em.flush();  // UPDATE SET stock=7, version=2 WHERE id=? AND version=1
} catch (OptimisticLockException e) {
    // 충돌 → 재시도 로직 (Retry with Exponential Backoff)
}
6. OCC의 한계 — Starvation(기아) 문제

충돌이 빈번한 환경에서는 트랜잭션이 반복적으로 롤백·재시도되는 라이브락(Livelock) 또는 기아(Starvation)가 발생할 수 있다. 지수 백오프(Exponential Backoff) + 최대 재시도 제한으로 완화한다.

6. PCC의 한계 — 교착상태(Deadlock)

두 트랜잭션이 서로 상대방의 잠금을 기다리면 교착상태(Deadlock)가 발생한다. DBMS는 Wait-for Graph로 사이클을 탐지하고 희생자를 선택하여 롤백·복구한다. 예방을 위해 자원 획득 순서를 전역적으로 일관되게 강제하거나 타임아웃(Timeout)을 적용한다.

Ⅴ. 결론

낙관적 동시성 제어는 잠금 없이 높은 처리량을 달성하므로 읽기 집약·분산 환경에 적합하며, 비관적 동시성 제어는 사전 잠금으로 충돌 자체를 차단하므로 쓰기 집약·완전 정합성이 요구되는 환경에 적합하다. 현대 시스템에서는 핵심 자원(재고·잔액)에는 PCC, 일반 읽기·갱신에는 OCC 또는 MVCC를 혼합하는 전략이 가장 현실적이다. 전략 선택의 핵심 지표는 워크로드의 읽기/쓰기 비율과 충돌 빈도이며, 이를 측정·모니터링한 뒤 결정해야 한다.

★ 기술사 핵심 코멘트
동시성 제어의 본질은 충돌 가능성과 처리량 간의 트레이드오프를 워크로드 특성에 맞게 최적화하는 것이며, 시스템 감리 시 다음을 중점 검토해야 한다.
① OCC 재시도 전략 설계 여부: 버전 충돌 시 무한 재시도(Infinite Retry)는 CPU 폭증·Livelock 위험 — 지수 백오프(Exponential Backoff)와 최대 재시도 횟수 제한이 구현되어 있는지 확인
② PCC 잠금 타임아웃 설정: innodb_lock_wait_timeout(MySQL) 등 잠금 대기 상한이 설정되지 않으면 교착 미탐지 상황에서 무한 대기 발생 — 타임아웃·데드락 탐지 주기 적정성 점검
③ PCC 잠금 에스컬레이션: SQL Server의 행 잠금 → 페이지 → 테이블 자동 에스컬레이션은 동시성을 급격히 저하시킴 — 에스컬레이션 임계값(LOCK_ESCALATION) 설정 적정성 확인
④ 분산 트랜잭션에서의 OCC: Saga 패턴·이벤트 소싱 기반 마이크로서비스에서는 OCC + 보상 트랜잭션(Compensating Transaction)의 조합이 표준 — 보상 로직의 멱등성(Idempotency) 구현 여부 감리
