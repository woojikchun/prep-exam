관계형 데이터베이스의 동시성 제어와 관련하여 트랜잭션 격리 수준(Transaction Isolation Level)에 대하여 다음을 설명하시오.

 (1) 격리성(Isolation)의 필요성과 발생 가능한 이상 현상의 종류 및 원인  (7점)
 (2) SQL 표준 격리 수준 4단계의 정의, 허용 이상 현상 및 구현 메커니즘  (12점)
 (3) 격리 수준별 이상 현상 발생 관계 요약 및 실무 선택 기준  (6점)
Ⅰ. 격리성의 필요성과 이상 현상의 종류 및 원인
1. 격리성(Isolation)의 필요성

다수의 트랜잭션이 동시에 실행될 때 각 트랜잭션은 다른 트랜잭션의 중간 상태에 영향을 받지 않고 독립적으로 실행된 것처럼 보여야 한다. 이를 보장하지 않으면 데이터 일관성이 훼손되는 다양한 이상 현상이 발생한다. 완전한 격리(SERIALIZABLE)는 직렬 실행과 동일한 결과를 보장하지만, 동시성(Concurrency) ↓ — 성능 저하를 초래하므로 실무에서는 이상 현상 허용 범위에 따라 격리 수준을 선택한다.

2. 이상 현상(Concurrency Anomalies)의 종류
①
Dirty Read (더티 읽기)
커밋되지 않은 데이터 읽기
정의: T1이 아직 COMMIT하지 않은 변경 데이터를 T2가 읽고, 이후 T1이 ROLLBACK될 경우 T2는 실제로 존재하지 않는 데이터를 기반으로 처리를 완료한 상태가 된다.
시각 T1 (잔액 갱신) T2 (잔액 조회) 잔액 값
t1BEGIN100만
t2UPDATE 잔액 = 50만 (미커밋)50만*
t3SELECT 잔액 → 50만 읽음50만*
t4ROLLBACK (50만→100만 복원)100만
t5T2는 존재하지 않는 50만 기준으로 처리 완료 → 오류불일치!
②
Non-Repeatable Read (반복 불가 읽기)
동일 행(Row) 재조회 시 값이 달라짐
정의: T1이 동일 행을 두 번 조회하는 사이에 T2가 해당 행을 UPDATE 후 COMMIT하여, T1의 두 번째 조회 결과가 첫 번째와 다른 값을 반환한다.
시각 T1 (두 번 조회) T2 (잔액 수정) 잔액 값
t1SELECT 잔액 → 100만100만
t2UPDATE 잔액 = 80만80만
t3COMMIT80만 ✔
t4SELECT 잔액 → 80만 ← 첫 조회(100만)와 다름!불일치!
③
Phantom Read (유령 읽기)
동일 조건 재조회 시 행(Row) 집합이 달라짐
정의: T1이 동일 조건(WHERE절)의 쿼리를 두 번 실행하는 사이에 T2가 해당 조건을 만족하는 행을 INSERT 또는 DELETE 후 COMMIT하여, 결과 행 집합의 수가 달라지는 현상이다.
시각 T1 (집합 조회) T2 (행 삽입) 결과 행 수
t1SELECT * WHERE 잔액>50만 → 3건3건
t2INSERT 잔액=70만 행 추가4건
t3COMMIT4건 ✔
t4SELECT * WHERE 잔액>50만 → 4건 ← 유령 행 출현!불일치!
④
Lost Update (갱신 손실)
두 트랜잭션의 동시 갱신 시 한쪽 갱신 소실
정의: T1과 T2가 동일 데이터를 동시에 읽은 후 각각 갱신하면, 나중에 커밋된 트랜잭션의 갱신이 앞선 트랜잭션의 갱신을 덮어씌워 앞선 변경이 소실된다.
시각 T1 (재고 -5) T2 (재고 -3) 재고 값
t1SELECT 재고 → 100100
t2SELECT 재고 → 100100
t3UPDATE 재고 = 100-5 = 95 → COMMIT95
t4UPDATE 재고 = 100-3 = 97 → COMMIT
(T1 갱신 무시, 97로 덮어씀)		97 (정상: 92)

Ⅱ. SQL 표준 격리 수준 4단계 — 정의, 허용 이상 현상 및 구현 메커니즘

SQL-92 표준은 격리 수준을 4단계로 정의하며, 낮을수록 동시성↑·일관성↓, 높을수록 일관성↑·동시성↓의 트레이드오프를 갖는다.

① READ UNCOMMITTED — 격리 수준 최저

정의: 다른 트랜잭션이 커밋하지 않은 변경 데이터도 읽을 수 있다. 사실상 잠금 없이 데이터를 조회하므로 Dirty Read를 허용한다.

구현: 읽기 시 공유 잠금(Shared Lock) 미획득. 쓰기 시 배타 잠금(Exclusive Lock)만 적용.

허용 이상 현상: Dirty Read Non-Repeatable Read Phantom Read

적합 상황: 데이터 정확도보다 속도가 절대 우선인 대용량 집계·근사치 보고(허용 오차 있음). 실무에서 거의 미사용.

-- Oracle / PostgreSQL에서 설정
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
-- T2가 미커밋 데이터도 읽음 → Dirty Read 발생 가능
SELECT balance FROM account WHERE id = 1; -- T1 ROLLBACK 전 값 반환 가능
② READ COMMITTED — 대부분 DBMS 기본값

정의: 커밋된 데이터만 읽을 수 있다. 읽기 완료 직후 공유 잠금을 해제하므로 동일 트랜잭션 내에서 같은 행을 두 번 읽으면 다른 값이 반환될 수 있다.

구현(Lock 방식): 읽기 시 공유 잠금 획득 → 읽기 완료 즉시 해제 (Short Duration Lock).
구현(MVCC 방식): 문장(Statement) 실행 시점의 스냅샷을 사용하여 커밋된 최신 버전만 반환. Oracle·PostgreSQL 기본.

방지: Dirty Read ✔   허용: Non-Repeatable Read Phantom Read

적합 상황: 일반 OLTP 서비스 — Oracle, PostgreSQL, SQL Server의 기본값. 대부분의 웹 애플리케이션이 이 수준 사용.

SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- T1 내 두 번 조회 사이 T2가 커밋하면 두 번째 조회 결과가 달라질 수 있음
SELECT balance FROM account WHERE id = 1; -- 100만 반환
-- (T2가 80만으로 UPDATE + COMMIT)
SELECT balance FROM account WHERE id = 1; -- 80만 반환 ← Non-Repeatable Read
③ REPEATABLE READ — 행 수준 반복 읽기 보장

정의: 트랜잭션 내에서 한 번 읽은 행은 트랜잭션이 끝날 때까지 동일한 값을 반환한다. 단, 범위 조건(Range Query)에 의한 신규 행 삽입은 막지 못한다.

구현(Lock 방식): 읽기 시 공유 잠금을 트랜잭션 종료 시까지 유지 (Long Duration Shared Lock).
구현(MVCC 방식): 트랜잭션 시작 시점의 스냅샷을 고정하여 전체 트랜잭션 동안 동일 버전 반환. MySQL InnoDB 기본.

방지: Dirty Read ✔ Non-Repeatable Read ✔   허용: Phantom Read

특이사항 — MySQL InnoDB: MVCC + Next-Key Lock(Gap Lock)으로 Phantom Read도 실질적으로 방지. SQL 표준보다 강한 보장을 제공.

SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- 동일 행 재조회 결과 동일 보장 (Non-Repeatable Read 방지)
SELECT balance FROM account WHERE id = 1; -- 100만
-- (T2가 80만으로 UPDATE + COMMIT)
SELECT balance FROM account WHERE id = 1; -- 여전히 100만 ← 반복 읽기 보장
-- 단, 범위 재조회 시 Phantom Read는 발생 가능 (Lock 방식의 경우)
④ SERIALIZABLE — 최고 격리, 완전한 직렬성 보장

정의: 동시 실행되는 모든 트랜잭션의 결과가 어떤 순서로 직렬 실행한 결과와 동일함을 보장한다. 모든 이상 현상이 방지된다.

구현(Lock 방식): 범위 잠금(Range Lock) 또는 서술 잠금(Predicate Lock)으로 Phantom 방지. SELECT 범위에 공유 잠금 적용.
구현(SSI 방식): Serializable Snapshot Isolation — MVCC + 충돌 탐지(Conflict Detection)로 성능 개선. PostgreSQL 9.1+, CockroachDB 지원.

방지: Dirty Read ✔ Non-Repeatable Read ✔ Phantom Read ✔

단점: 동시성 최저, 교착상태 빈발 → 처리량 급감. 금융 원장, 의료 기록 등 정합성 최우선 시스템에만 선택적 사용.

SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- 범위 쿼리에도 잠금 적용 → Phantom Read 방지
SELECT * FROM account WHERE balance > 500000;
-- 다른 T2가 balance=700000 행 INSERT를 시도하면 잠금 대기 발생
-- → T1 COMMIT 전까지 T2 대기 → 완전한 직렬 실행 보장

Ⅳ. 결론

트랜잭션 격리 수준은 이상 현상 허용 범위와 동시성 성능 간의 정책적 트레이드오프를 결정한다. 실무에서는 대부분의 시스템이 READ COMMITTED를 기반으로 애플리케이션 레이어의 낙관적/비관적 잠금 전략을 조합하여 특정 이상 현상을 추가 방지하는 방식을 택한다. 완전한 직렬성이 요구되는 시스템에는 Serializable Snapshot Isolation(SSI) 방식이 전통적인 잠금보다 높은 성능을 제공하는 현실적 대안이다.

★ 기술사 핵심 코멘트
격리 수준 설계의 핵심은 허용 가능한 이상 현상의 범위를 업무 요건에 맞게 명확히 정의하는 것이며, DB 시스템 감리 시 다음을 중점 검토해야 한다.
① 격리 수준 설정 근거 문서화: 기본값(READ COMMITTED)에서 상향·하향 조정 시 반드시 업무 위험도 분석 결과와 설정 근거를 설계 문서에 명시
② Lost Update 방지 누락: READ COMMITTED에서 선점적 잠금(SELECT … FOR UPDATE) 없이 Read-Modify-Write 패턴 사용 시 갱신 손실 발생 가능 — 코드 레벨 감리 필수
③ 장시간 트랜잭션과 MVCC 공간 낭비: REPEATABLE READ + 장시간 트랜잭션은 구버전 데이터(Undo 레코드) 과다 누적 → PostgreSQL Autovacuum 정책, MySQL InnoDB Purge 스레드 설정 적정성 확인
④ SERIALIZABLE 선택 시 교착상태 대응: 범위 잠금에 의한 교착상태 빈발 가능 — 탐지 주기, 희생자 선택 정책, 재시도(Retry) 로직 구현 여부 점검
