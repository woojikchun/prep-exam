Ⅰ. B+ 트리의 정의, B 트리와의 비교, 구조적 특성 및 복잡도
1. B+ 트리(B+ Tree)의 정의

B+ 트리(B+ Tree)는 차수(Order) m의 균형 다방향 탐색 트리(Balanced Multiway Search Tree)로, 모든 리프 노드가 동일한 깊이에 위치하며 실제 데이터(레코드 포인터)는 리프 노드에만 저장하고 내부 노드(Internal Node)는 오직 라우팅(경로 안내) 키만 보유한다. 리프 노드들은 이중 연결 리스트(Doubly Linked List)로 연결되어 범위 탐색(Range Scan)을 효율적으로 지원한다.

Ⅱ. 검색(Search) 연산 — 등치·범위 검색
1. 등치 검색 (Point Query)

알고리즘: ① 루트에서 시작 ② 현재 노드에서 탐색 키 k와 비교하여 적절한 자식 포인터 선택 ③ 리프 노드에 도달할 때까지 반복 ④ 리프 노드에서 k를 선형 탐색하여 레코드 포인터 반환

【그림 2】 등치 검색 k=25 — 루트→내부→리프 경로 추적 (주황 화살표)
2. 범위 검색 (Range Query) — 20 ≤ k ≤ 40

알고리즘: ① 하한(20)으로 등치 검색 → 시작 리프 도달 ② 리프의 이중 연결 리스트를 오른쪽으로 순회하며 상한(40) 이하인 키를 순서대로 수집 ③ 상한 초과 키를 만나면 종료

결과: 20, 25, 30, 35, 40 — 트리 재탐색 없이 연결 리스트만 순회하므로 O(log_m N + k)의 효율을 달성한다.

Ⅲ. 삽입(Insert) 연산 — 노드 분할(Split)
1. 삽입 알고리즘 개요

① 삽입할 키의 올바른 리프 노드를 탐색
② 리프 노드에 공간이 있으면 키를 정렬된 위치에 삽입 후 종료
③ 리프 노드가 가득 찬 경우(overflow) → 노드 분할(Split) 수행
 · 중간 키(median)를 기준으로 두 노드로 분리
 · 중간 키의 복사본(Copy-Up)을 부모 노드로 올림
 · 부모도 가득 찼으면 재귀적으로 분할 → 루트까지 전파 가능

2. 삽입 예제 — 차수 m=4, 키 45 삽입

삽입 전: 리프 노드 [40 | 50]에 45를 삽입 → 최대 3키 초과(오버플로우) → 분할

【그림 3】 삽입 연산 — 오버플로우 시 리프 노드 분할 및 Copy-Up
3. 루트 분할

분할이 루트까지 전파되어 루트가 오버플로우되면 새 루트를 생성하고 트리 높이가 1 증가한다. 이것이 B+ 트리가 항상 균형을 유지하는 핵심 메커니즘이다.

Ⅳ. 삭제(Delete) 연산 — 재분배 및 합병
1. 삭제 알고리즘 개요

① 삭제할 키를 리프 노드에서 탐색 후 제거
② 리프 노드 키 수가 최솟값(⌊(m-1)/2⌋) 이상이면 종료 — 단순 삭제
③ 최솟값 미만(Underflow) 발생 시 두 가지 방법 중 하나 선택:
 재분배(Redistribution): 형제 노드에 여유 키가 있으면 빌려옴
 합병(Merge): 형제 노드도 최솟값이면 두 노드를 병합 → 부모의 키 삭제 → 연쇄 전파 가능

2. 삭제 경우 ① — 단순 삭제 (여유 있음)

리프 [20 | 25]에서 20 삭제 → [25] — 최솟값(1) 충족, 종료.

3. 삭제 경우 ② — 재분배(Redistribution)

오른쪽 형제 노드에서 가장 작은 키를 빌려와 부모의 구분 키를 갱신한다.

【그림 4】 재분배 삭제 — 오른쪽 형제에서 40을 빌려오고 부모 키 갱신
4. 삭제 경우 ③ — 합병(Merge)

양쪽 형제 모두 최솟값이어서 빌릴 수 없을 때, 두 리프 노드를 합치고 부모의 구분 키를 삭제한다. 부모에서도 underflow 발생 시 재귀적으로 합병 전파.

【그림 5】 합병 삭제 — 양쪽 형제 모두 최솟값 → 병합 후 부모 구분 키 삭제
5. 삭제 연산 의사결정 요약
삭제 후 상황처리 방법부모 영향
키 수 ≥ 최솟값단순 삭제 후 종료없음
키 수 < 최솟값
형제에 여유 있음		재분배 — 형제에서 키 1개 가져옴구분 키(Separator) 갱신
키 수 < 최솟값
형제도 최솟값		합병 — 형제와 병합부모 구분 키 삭제 → 연쇄 전파 가능
루트 합병 후 자식 1개루트 제거, 자식이 새 루트가 됨트리 높이 1 감소
【표 4】 삭제 연산 의사결정 요약
Ⅴ. 결론

B+ 트리는 균형 유지·높은 팬아웃·리프 연결 리스트의 세 가지 특성으로 RDBMS 인덱스의 표준 자료구조 지위를 확립하였다. 검색·삽입·삭제 모두 O(log_m N)의 균일한 복잡도를 보장하며, 디스크 I/O 최소화와 범위 탐색 효율성이 결합되어 대용량 데이터베이스 환경에 최적화된 구조이다. 삽입 시 분할(Split)과 삭제 시 재분배·합병(Merge)은 트리의 균형을 자동으로 유지하는 자기 조정(Self-Balancing) 메커니즘이다.

★ 기술사 핵심 코멘트
B+ 트리의 본질은 균형 유지 + 고팬아웃으로 디스크 접근 횟수를 로그 수준(h≈3~4)으로 억제하는 것이며, DB 시스템 설계·감리 시 다음을 중점 검토해야 한다.
① 인덱스 설계 적정성: 선택도(Selectivity) 낮은 컬럼(성별 등)에 B+ 트리 인덱스는 비효율 — 전체 테이블 스캔이 오히려 빠를 수 있음. 쿼리 실행 계획(EXPLAIN) 분석 여부 확인
② 페이지 분할(Page Split) 과다: 순차 삽입이 아닌 무작위 삽입 시 리프 노드 분할이 빈발하여 페이지 단편화 발생 → FILLFACTOR 설정 및 주기적 인덱스 재구성(REBUILD) 정책 수립 여부
③ 클러스터드 vs. 논클러스터드 인덱스: InnoDB 프라이머리 키는 클러스터드(리프에 실제 데이터 저장) — 보조 인덱스는 리프에 PK값 저장 후 2차 탐색 발생. 설계 시 PK 선택 전략 검토
④ 범위 스캔 최적화: 리프 연결 리스트를 활용한 범위 탐색의 장점을 최대한 활용하기 위해 복합 인덱스의 컬럼 순서(범위 조건 컬럼을 후순위에 배치)를 올바르게 설계했는지 감리
