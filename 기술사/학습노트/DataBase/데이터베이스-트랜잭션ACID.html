Ⅰ. 트랜잭션의 개념, 상태 전이 및 ACID 4대 특성
1. 트랜잭션(Transaction)의 정의

트랜잭션(Transaction)이란 데이터베이스에서 하나의 논리적 작업 단위(Logical Unit of Work)를 구성하는 연산의 집합이다. 트랜잭션 내 모든 연산은 완전히 실행되거나(COMMIT), 전혀 실행되지 않은 것처럼 취소(ROLLBACK)되어야 한다.

예: 계좌 이체 — "A계좌 잔액 감소 + B계좌 잔액 증가"는 두 연산이 반드시 함께 성공하거나 함께 실패해야 하는 하나의 트랜잭션이다.

2. 트랜잭션 상태 전이
【그림 1】 트랜잭션 상태 전이도 — Active→Partially Committed→Committed / Failed→Aborted
3. ACID 4대 특성
A
원자성 (Atomicity)
All or Nothing
트랜잭션 내 연산은 전부 실행되거나 전혀 실행되지 않아야 한다. 중간 상태의 부분 완료는 허용되지 않는다.
▶ 보장 메커니즘: WAL(Write-Ahead Log), Undo Log, ROLLBACK
C
일관성 (Consistency)
Valid State Transition
트랜잭션 실행 전·후 데이터베이스는 항상 정의된 무결성 제약을 만족하는 일관된 상태여야 한다.
▶ 보장 메커니즘: 무결성 제약(PK·FK·Check), Trigger, 애플리케이션 로직
I
격리성 (Isolation)
Concurrent Execution
동시 실행 중인 트랜잭션은 서로의 중간 상태를 볼 수 없으며, 직렬(Serial) 실행과 동일한 결과를 보장해야 한다.
▶ 보장 메커니즘: 2PL(잠금), MVCC, 직렬화 가능성(Serializability)
D
지속성 (Durability)
Permanent Persistence
COMMIT된 트랜잭션의 결과는 시스템 장애(전원 차단·충돌)가 발생해도 영구적으로 보존되어야 한다.
▶ 보장 메커니즘: WAL Redo Log, Checkpoint, 이중화·백업
【표 1】 ACID 4대 특성 — 정의 및 보장 메커니즘 요약
Ⅱ. 원자성(Atomicity) 보장 메커니즘 — WAL과 회복 기법
1. WAL(Write-Ahead Logging) 원칙

WAL(Write-Ahead Logging)이란 데이터베이스 페이지를 디스크에 기록하기 전에 반드시 로그 레코드를 먼저 안정 저장소(Stable Storage)에 기록해야 한다는 프로토콜이다. 이 원칙으로 장애 발생 시 로그를 이용한 완전한 회복이 가능해진다.

WAL 원칙내용
Undo 규칙데이터 페이지를 디스크에 쓰기 전에 해당 변경의 Undo 로그 레코드를 먼저 안정 저장소에 기록
Redo 규칙COMMIT 처리 전에 해당 트랜잭션의 모든 Redo 로그 레코드를 안정 저장소에 기록
【표 2】 WAL 두 가지 규칙
2. 로그 레코드 구조
# 로그 레코드 구조
[LSN]  [TransactionID]  [PageID]  [Before Image (Undo)]  [After Image (Redo)]  [PrevLSN]

# 계좌 이체 트랜잭션 로그 예시 (T1: A계좌 -10만원, B계좌 +10만원)
LSN:001  T1  BEGIN
LSN:002  T1  Page:A  Before:100만  After:90만   PrevLSN:001
LSN:003  T1  Page:B  Before:50만   After:60만   PrevLSN:002
LSN:004  T1  COMMIT                                    PrevLSN:003
# → COMMIT 전 LSN:002~003이 반드시 디스크 로그에 기록되어야 함 (WAL)
3. 회복 기법 — ARIES 알고리즘

현대 DBMS(Oracle, PostgreSQL, MySQL InnoDB)는 ARIES(Algorithm for Recovery and Isolation Exploiting Semantics) 방식으로 회복을 수행한다.

회복 단계내용대상 트랜잭션
① 분석(Analysis) 마지막 체크포인트부터 로그 순방향 스캔 — 장애 시점의 활성 트랜잭션 목록(ATT)과 더티 페이지 목록(DPT) 파악 모든 트랜잭션
② Redo(재실행) DPT 기준으로 필요한 로그부터 순방향 재실행. COMMIT 여부와 무관하게 모든 변경 재적용 (Repeating History) 모든 트랜잭션 (COMMIT 포함)
③ Undo(취소) ATT에 남은 미완료 트랜잭션을 역방향으로 Undo. PrevLSN 체인을 따라 변경 전 값(Before Image)으로 복원 미COMMIT 트랜잭션만
【표 3】 ARIES 회복 3단계
【그림 2】 ARIES 회복 타임라인 — T1(Redo만), T2·T3(Redo 후 Undo)
Ⅲ. 일관성(Consistency) 및 지속성(Durability) 보장 메커니즘
1. 일관성(Consistency) 보장

일관성은 트랜잭션 실행 전후 데이터베이스의 무결성 제약(Integrity Constraints)이 항상 만족되어야 하는 속성이다. DBMS와 애플리케이션 양측에서 공동으로 보장 책임을 진다.

보장 계층메커니즘예시
DBMS — 선언적 제약 PRIMARY KEY, FOREIGN KEY, UNIQUE, CHECK, NOT NULL 제약 조건 자동 검증 잔액 CHECK (잔액 ≥ 0) 위반 시 트랜잭션 자동 ABORT
DBMS — 트리거(Trigger) 데이터 변경 이벤트에 자동 반응하는 프로시저 실행 주문 INSERT 시 재고 자동 감소 트리거
애플리케이션 로직 비즈니스 규칙을 코드·서비스 레이어에서 검증 이체 한도 초과 방지, 음수 잔액 검사
직렬화 가능성
(Serializability)		 동시 트랜잭션 실행 결과가 어떤 직렬 순서와 동일해야 함 — 격리성과 연계 2PL, MVCC로 보장
【표 4】 일관성 보장 메커니즘
2. 지속성(Durability) 보장

지속성은 COMMIT된 트랜잭션 결과가 시스템 충돌·전원 장애·미디어 오류에도 소실되지 않음을 보장한다.

메커니즘내용보장 수준
WAL Redo 로그 COMMIT 전 모든 Redo 로그를 동기적 디스크 기록(fsync) → 장애 후 Redo로 복원 단일 노드 장애
체크포인트(Checkpoint) 주기적으로 더티 페이지를 디스크에 플러시 → 회복 시 Redo 범위 축소 회복 시간 단축
이중화·복제
(Replication)		 동기 복제: COMMIT 전 팔로워 확인 → 미디어 장애 대응
비동기 복제: 성능 우선, RPO 존재		 디스크·서버 장애
백업·아카이브 전체 백업(Full) + 증분 백업(Incremental) + 아카이브 로그 재해 복구(DR)
【표 5】 지속성 보장 메커니즘
Ⅳ. 격리성(Isolation) 보장 메커니즘 — Lock, MVCC, 격리 수준
1. 격리성 위반 현상 (동시성 문제)
현상정의발생 조건
Dirty Read 다른 트랜잭션이 아직 COMMIT하지 않은 데이터를 읽음 T1 갱신 중 T2가 읽기 → T1이 ROLLBACK
Non-Repeatable Read 같은 쿼리를 두 번 실행 시 다른 행(Row)의 값이 달라짐 T1 첫 읽기 후 T2가 UPDATE+COMMIT → T1 두 번째 읽기
Phantom Read 같은 조건의 쿼리 재실행 시 행(Row)의 집합이 달라짐 T1 집합 읽기 후 T2가 INSERT+COMMIT → T1 재실행
Lost Update 두 트랜잭션이 동일 데이터 갱신 시 한쪽 갱신이 소실 T1·T2 동시 읽기 후 각각 갱신 — 마지막 쓰기만 반영
【표 6】 격리성 위반 현상(동시성 문제)
2. 2단계 잠금 프로토콜 (2PL, Two-Phase Locking)

2PL(Two-Phase Locking)은 모든 잠금 획득이 완료된 후에야 잠금 해제를 시작하는 프로토콜로, 직렬 가능성(Serializability)을 보장한다.

① 확장 단계(Growing Phase): 필요한 잠금 획득만 가능, 해제 불가
   ② 축소 단계(Shrinking Phase): 잠금 해제만 가능, 신규 획득 불가

2PL 변형특징교착상태
기본 2PL직렬 가능성 보장, COMMIT 전 잠금 해제 가능가능
엄격 2PL (Strict 2PL)COMMIT/ABORT 시까지 Write Lock 유지 → Cascading Rollback 방지가능
강한 엄격 2PL (SS2PL)COMMIT/ABORT 시까지 모든 잠금 유지 → 가장 안전, 동시성↓가능
【표 7】 2PL 변형 비교
3. MVCC (Multi-Version Concurrency Control)

MVCC(다중 버전 동시성 제어)는 데이터의 여러 버전을 동시에 유지하여, 읽기 연산이 잠금 없이 일관된 스냅샷을 조회할 수 있도록 한다. 읽기-쓰기 간 잠금 경합이 없어 동시성이 크게 향상된다.

【그림 3】 MVCC 동작 원리 — 버전 스냅샷으로 읽기-쓰기 잠금 경합 제거
4. SQL 표준 격리 수준 (Isolation Level)
격리 수준 Dirty Read Non-Repeatable Read Phantom Read 구현 방식
READ UNCOMMITTED 발생발생발생 잠금 없음 (최고 성능)
READ COMMITTED 방지발생발생 문장 레벨 스냅샷 (Oracle·PostgreSQL 기본)

Ⅴ. 결론

ACID의 네 속성은 각각 독립된 DBMS 메커니즘으로 보장되지만 상호 의존적이다. 원자성·지속성은 WAL과 회복 알고리즘(ARIES)으로, 일관성은 무결성 제약과 트랜잭션 설계로, 격리성은 2PL 또는 MVCC와 격리 수준 정책으로 보장된다. 실무에서는 완전한 직렬화(SERIALIZABLE)보다 READ COMMITTED 또는 REPEATABLE READ 수준에서 성능과 일관성의 균형을 취하는 것이 일반적이며, NoSQL이나 분산 DB에서는 BASE(Basically Available, Soft state, Eventually consistent) 모델로 ACID의 일부를 완화하기도 한다.

★ 기술사 핵심 코멘트
ACID의 본질은 신뢰할 수 있는 데이터 처리의 4가지 불변 조건이며, DB 시스템 감리 시 다음을 중점 검토해야 한다.
① 원자성 — WAL 설정 적정성: fsync 비활성화(성능 목적)로 인한 지속성 훼손 여부 확인. PostgreSQL의 synchronous_commit=off 등 위험 설정 검토
② 일관성 — 제약 조건 완전성: FK·Check 제약이 비활성화(DISABLE NOVALIDATE)된 테이블의 무결성 보장 방안 별도 수립 여부
③ 격리성 — 격리 수준 설계 적정성: 금융·의료 등 고정합성 요구 시스템에서 READ COMMITTED 이하 사용 시 Non-Repeatable Read 허용 위험성 검토
④ 지속성 — 동기 복제 여부: 비동기 복제(Async Replication) 사용 시 RPO(목표 복구 시점) 위반 가능성 및 장애 시나리오별 데이터 손실 허용 범위 명시 여부 확인
