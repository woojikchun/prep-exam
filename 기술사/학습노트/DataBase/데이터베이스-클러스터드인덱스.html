Ⅰ. 두 인덱스의 개념, 물리적 구조 및 탐색 원리
클러스터드 인덱스 (Clustered Index)
정의: 테이블의 실제 데이터 행(Row)이 인덱스 키 순서대로 물리적으로 정렬·저장되는 인덱스. B+ 트리의 리프 노드 = 실제 데이터 페이지이므로 인덱스와 데이터가 분리되지 않는다.

핵심 제약: 테이블당 1개만 생성 가능 (데이터를 두 가지 순서로 동시에 저장 불가)
논클러스터드 인덱스 (Non-Clustered Index)
정의: 인덱스 구조가 실제 데이터와 별도의 저장 공간에 위치하며, B+ 트리의 리프 노드에는 키 값 + 행 위치 포인터(RID 또는 클러스터드 키)만 저장한다.

핵심 특징: 테이블당 다수 생성 가능 (SQL Server 최대 999개, MySQL InnoDB 64개)

【표 2】 클러스터드 vs. 논클러스터드 인덱스 종합 비교
커버링 인덱스(Covering Index) — 논클러스터드의 최적화 기법

커버링 인덱스(Covering Index)란 쿼리에서 요구하는 모든 컬럼이 논클러스터드 인덱스 내에 포함되어 있어 데이터 페이지 접근(2차 탐색)이 불필요한 상태를 말한다. INCLUDE 컬럼을 인덱스에 추가하거나 복합 인덱스로 구성하면 달성 가능하다.

예: CREATE INDEX idx ON orders(cust_id) INCLUDE (order_date, amount)
→ SELECT order_date, amount FROM orders WHERE cust_id = 100 — 데이터 페이지 접근 0회

Ⅲ. DBMS별 구현 차이
【그림 2】 DBMS별 클러스터드·논클러스터드 인덱스 구현 차이
DBMS클러스터드 구현논클러스터드 구현특이사항
MySQL InnoDB PK 자동 클러스터드. PK 없으면 UNIQUE NOT NULL 첫 컬럼, 없으면 내부 RowID 보조 인덱스 리프에 PK 값 저장 → PK를 통해 클러스터드 2차 탐색 PK 크기가 모든 보조 인덱스 크기에 영향. PK는 짧게 설계 필수
SQL Server PK 생성 시 기본 클러스터드. 클러스터드 없는 테이블 = 힙(Heap) 클러스터드 없으면 RID 포인터, 있으면 클러스터드 키 저장 INCLUDE 컬럼으로 커버링 인덱스 구성 용이. 필터 인덱스 지원
Oracle 기본은 힙 테이블. IOT(Index-Organized Table) 선택 시 클러스터드 효과 B-Tree 인덱스(기본), Bitmap 인덱스(저선택도), 함수 기반 인덱스 Bitmap 인덱스: 성별·상태 등 카디널리티 낮은 컬럼에 DW 환경 특화
PostgreSQL 클러스터드 인덱스 개념 미지원. CLUSTER 명령으로 일시 물리 정렬 가능 B-Tree(기본), Hash, GiST, GIN, BRIN 등 다양한 인덱스 유형 지원 힙 기반 구조. 모든 인덱스는 논클러스터드 성격. INCLUDE 컬럼 지원
【표 3】 DBMS별 구현 차이 비교
Ⅳ. 실무 선택 기준 및 복합 인덱스 설계 고려사항
1. 클러스터드 인덱스 선택 기준

① 범위 검색이 빈번한 컬럼: ORDER_DATE, CREATED_AT 등 날짜·순번 컬럼 — 물리 정렬로 범위 스캔 최적화

② 단조 증가(Monotonically Increasing) 키: AUTO_INCREMENT, SEQUENCE — 페이지 분할 최소화. UUID처럼 무작위 키는 피해야 함

③ 가장 자주 사용되는 조회 기준 컬럼: 전체 쿼리 중 가장 빈번한 WHERE 조건 컬럼에 배정하여 최대 효과

④ 짧은 컬럼: INT(4 bytes) > BIGINT(8 bytes) > VARCHAR — 모든 보조 인덱스가 클러스터드 키를 복사 저장하므로 크기가 작을수록 전체 인덱스 공간 절약

2. 논클러스터드 인덱스 선택 기준

① 등치 검색이 주 패턴인 컬럼: 이메일·고객ID 조회 — 단일 행 검색, 2차 탐색 비용 미미

② 고선택도(High Cardinality) 컬럼: 주민번호·이메일처럼 중복값이 적을수록 효율↑. 저선택도(성별·상태)는 효과 미미

③ 복수 접근 패턴 지원: 클러스터드 1개 한계를 보완하여 다양한 조회 조건에 대응

④ 커버링 인덱스 구성 가능 시: SELECT 컬럼 + WHERE 컬럼을 모두 포함하여 2차 탐색 제거

3. 선택 기준 매트릭스
상황·요건 클러스터드 논클러스터드 비고
범위 검색 (BETWEEN, ≥) 강력 추천 비효율 물리 정렬로 순차 I/O
단건 등치 검색 (=) 적합 적합 차이 미미, 둘 다 O(log n)
ORDER BY + LIMIT 정렬 강력 추천 정렬 추가 비용 이미 정렬된 상태 활용
다수 쿼리 패턴 지원 필요 1개 한계 다수 생성 각 쿼리에 맞는 논클러스터드
INSERT 집약 워크로드 주의 (페이지 분할) 상대적 유리 무작위 PK(UUID) 회피 권장
SELECT 컬럼 수 적고 고정 — 커버링 인덱스 INCLUDE 컬럼으로 2차 탐색 제거
저선택도 컬럼 (성별·상태) 부적합 B-Tree 비효율 Oracle Bitmap, 파티셔닝 고려
DW·분석 집계 쿼리 IOT / 날짜 정렬 Bitmap (Oracle) 컬럼스토어 인덱스 병행 검토
【표 4】 상황별 인덱스 선택 기준 매트릭스
4. 복합 인덱스(Composite Index) 설계 원칙

① 선두 컬럼 원칙(Left-Most Prefix): 복합 인덱스 (A, B, C)는 A, (A,B), (A,B,C) 조건에만 유효. B 단독 조건은 인덱스 미사용

② 등치 조건 선행, 범위 조건 후행: (=) 컬럼을 앞에, (BETWEEN, >) 컬럼을 뒤에 배치. 범위 조건 이후 컬럼은 인덱스 활용 불가

③ 선택도 높은 컬럼 우선: 카디널리티가 높은 컬럼을 앞에 배치하여 초기 필터링 효율 극대화

④ 인덱스 수 최소화: 논클러스터드 인덱스 과다 생성은 INSERT/UPDATE/DELETE 성능 저하. 실제 사용되는 쿼리 기반으로 최적화

Ⅴ. 결론

클러스터드 인덱스는 물리적 정렬로 범위 탐색과 정렬 쿼리에 최적이며, 논클러스터드 인덱스는 다양한 접근 패턴 지원과 커버링 인덱스로 2차 탐색 비용을 제거할 수 있다. 실무에서는 클러스터드 인덱스 키를 단조 증가·짧은 정수형으로 설계하고, 논클러스터드는 고선택도 컬럼과 쿼리 패턴 기반의 커버링 인덱스로 보완하는 것이 최적 전략이다. 과도한 인덱스는 쓰기 성능을 저하시키므로 실제 쿼리 실행 계획(EXPLAIN) 분석을 통한 선택적 설계가 필수이다.

★ 기술사 핵심 코멘트
인덱스 설계의 본질은 읽기 성능 향상과 쓰기 오버헤드 간의 트레이드오프를 워크로드에 맞게 최적화하는 것이며, DB 설계·감리 시 다음을 중점 검토해야 한다.
① 클러스터드 키 선택 적정성: UUID·GUID처럼 무작위 값을 PK로 설계한 경우 페이지 분할 빈발 및 단편화 위험 — 순번(AUTO_INCREMENT, SEQUENCE) 또는 시간 순 정렬 키로 교체 권고
② InnoDB 보조 인덱스 크기: PK가 길수록(VARCHAR 등) 모든 보조 인덱스 크기 배증 — PK 크기를 INT 또는 BIGINT로 제한했는지 확인
③ 커버링 인덱스 활용 여부: 빈번한 조회 쿼리의 실행 계획에서 'Index Scan'이 아닌 'Index Only Scan'(PostgreSQL) 또는 'Covering Index'(MySQL)가 활성화되는지 점검
④ 인덱스 수 과다 여부: 논클러스터드 인덱스 10개 이상 보유 테이블은 INSERT/UPDATE 시 다수 인덱스 동시 갱신으로 잠금 경합·응답 지연 발생 가능 — 미사용 인덱스 식별 후 삭제 권고
