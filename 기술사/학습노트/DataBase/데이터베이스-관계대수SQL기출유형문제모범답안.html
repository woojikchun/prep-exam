Ⅰ. 관계대수의 개념
관계대수(Relational Algebra)는 관계형 데이터베이스에서 릴레이션(테이블)을 입력으로 받아 새로운 릴레이션을 출력하는 절차적 질의 언어이다. 모든 연산의 결과가 다시 릴레이션이 되는 닫힘 성질(Closure Property)을 가지며, SQL의 이론적 기반이 된다. E.F. Codd(1970)가 정의하였다.

Ⅴ. 결론 — 감리 포인트

관계대수는 SQL 쿼리 최적화기(Query Optimizer)의 이론적 기반으로, 동치 변환 법칙(σ를 먼저 적용하여 데이터 크기 감소)을 통해 실행 계획을 최적화한다. 감리 시에는 디비전 연산을 잘못 구현한 이중 NOT EXISTS 로직의 오류 여부, 카티션 곱의 불필요한 사용으로 인한 성능 문제를 점검한다.

정규화 — 이상현상·함수적 종속성·1NF~BCNF 조건 및 위반 예시
⏱ 15분📄 1.5~2페이지💯 10~15점
Ⅰ. 정규화의 목적 및 이상현상
정규화(Normalization)는 릴레이션에서 데이터 중복을 최소화하고 삽입·삭제·수정 이상현상을 제거하기 위해 릴레이션을 더 작은 릴레이션으로 분해하는 과정이다.
이상현상정의예시
삽입 이상
(Insertion)		불필요한 데이터를 함께 삽입해야 하거나, 삽입이 불가능한 현상학생-강좌 테이블에서 수강 강좌 없는 신입생 정보를 삽입 불가
삭제 이상
(Deletion)		일부 데이터 삭제 시 의도치 않게 다른 정보까지 손실되는 현상마지막 수강생 삭제 시 강좌 정보까지 함께 삭제됨
수정 이상
(Update)		중복된 데이터 중 일부만 수정되어 데이터 불일치가 발생하는 현상교수 이메일 변경 시 해당 교수의 모든 강좌 행을 수정해야 하며, 일부 누락 시 불일치 발생
Ⅱ. 함수적 종속성 (Functional Dependency)
함수적 종속성 정의
릴레이션 R에서 속성 X의 값이 결정되면 속성 Y의 값도 유일하게 결정될 때, "Y는 X에 함수적으로 종속된다"고 하며 X → Y 로 표기한다.

• 완전 함수 종속: Y가 복합 키 X 전체에 종속 (X의 부분집합에는 종속되지 않음)
• 부분 함수 종속: Y가 복합 키 X의 일부 속성에만 종속
• 이행 함수 종속: X→Y, Y→Z 이면 X→Z (Y가 키가 아닐 때)
Ⅲ. 1NF ~ BCNF 정규형 조건 및 위반 예시
정규형조건제거 대상위반 예시
1NF
(제1 정규형)		모든 속성값이 원자값(Atomic)이어야 함. 반복 그룹, 다중값 속성 불허비원자 속성, 반복 그룹취미 속성에 "독서, 등산, 요리" 처럼 다중값 저장
2NF
(제2 정규형)		1NF 만족 + 기본 키에 완전 함수 종속 (부분 함수 종속 제거)부분 함수 종속(학번, 과목코드)→성적 ✓ 但 학번→학생명 (부분 종속, 과목코드 불필요)
3NF
(제3 정규형)		2NF 만족 + 이행 함수 종속 제거. 기본 키 이외의 속성이 다른 비키 속성을 결정하지 않아야 함이행 함수 종속학번→학과코드→학과명 (학번→학과명은 이행 종속)
BCNF
(Boyce-Codd NF)		3NF 만족 + 모든 결정자(Determinant)가 후보 키이어야 함. 3NF보다 강한 조건후보 키가 아닌 결정자후보키가 여러 개이고 그 중 하나가 다른 후보키의 속성을 결정할 때
정규화 단계 도식 비정규형 릴레이션 R │ 원자값 보장 (반복그룹 제거) ▼ 1NF — 원자값 만족 │ 부분 함수 종속 제거 ▼ 2NF — 완전 함수 종속 │ 이행 함수 종속 제거 ▼ 3NF — 이행 종속 제거 │ 모든 결정자 = 후보키 보장 ▼ BCNF ← 대부분의 실무 목표 정규형 │ 다치 종속(MVD) 제거 ▼ 4NF │ 조인 종속(JD) 제거 ▼ 5NF (PJNF)
📌 정규화 vs 반정규화 (역정규화)
정규화는 이상현상 제거에 유리하나 조인 증가로 성능 저하 가능.
반정규화(Denormalization): 성능 향상을 위해 의도적으로 중복 허용. 자주 조회되는 집계 컬럼 추가, 테이블 분할·통합.
감리 시: 반정규화 적용 사유·영향도 분석 문서 존재 여부 확인
Ⅳ. 결론

정규화는 데이터 무결성과 일관성을 보장하는 DB 설계의 핵심 원칙이다. 실무에서는 대부분 3NF 또는 BCNF를 목표로 설계하며, 성능과의 트레이드오프를 고려하여 선택적으로 반정규화를 적용한다. 공공 정보화사업 DB 설계 감리 시에는 정규화 준수 여부와 반정규화 근거의 명시를 주요 점검항목으로 확인한다.

Ⅲ. 상관 서브쿼리 vs 비상관 서브쿼리 및 EXISTS·IN 비교
상관 서브쿼리 (Correlated Subquery)
외부 쿼리의 각 행마다 서브쿼리가 반복 실행됨. 외부 쿼리의 컬럼을 서브쿼리에서 참조.
비상관 서브쿼리: 서브쿼리가 독립적으로 1회만 실행. 결과를 캐싱하여 외부 쿼리에 전달.
비교 항목EXISTSIN
동작 방식서브쿼리 결과의 존재 여부 확인. 첫 매칭 발견 시 즉시 중단 (Short-circuit)서브쿼리 결과 목록 전체를 메모리에 올린 후 비교
NULL 처리NULL에 안전 (존재 여부만 확인)서브쿼리 결과에 NULL 포함 시 NOT IN이 항상 FALSE 반환 → 위험
성능 우위서브쿼리 결과가 대용량일 때 유리 (Short-circuit으로 조기 종료)서브쿼리 결과가 소용량일 때 유리 (IN 리스트 캐싱)
상관 여부주로 상관 서브쿼리로 사용비상관 서브쿼리에 주로 사용
-- EXISTS 사용 (부서 10에 속한 직원이 있는 부서 조회) SELECT 부서명 FROM 부서 d WHERE EXISTS ( SELECT 1 FROM 직원 e WHERE e.부서번호 = d.부서번호 -- 외부 쿼리 참조 → 상관 서브쿼리 ); -- NOT IN의 NULL 함정 → NOT EXISTS 권장 SELECT 이름 FROM 직원 WHERE 부서번호 NOT IN (SELECT 부서번호 FROM 부서); -- ↑ 부서.부서번호에 NULL이 있으면 결과 0건 (의도치 않은 결과!)
Ⅳ. 결론 — 감리 포인트

NOT IN 서브쿼리에 NULL이 포함될 경우 항상 결과가 0건이 되는 치명적 버그가 발생한다. 따라서 NOT IN보다 NOT EXISTS 사용을 권장한다. UNION과 UNION ALL의 차이(중복 제거 여부)를 명확히 인식하여 불필요한 SORT+DISTINCT 연산을 제거해야 한다.

Ⅰ. 인덱스의 개념
인덱스는 테이블의 특정 컬럼(들)에 대한 검색 속도를 향상시키기 위한 별도의 데이터 구조이다. 전체 테이블 스캔(Full Table Scan) 대신 인덱스 스캔(Index Scan)을 수행하여 I/O를 최소화한다. 책의 찾아보기(색인)에 비유된다.
Ⅱ. B-Tree 인덱스 vs Hash 인덱스

Ⅳ. 인덱스 최적 설계 원칙
  • 1
    선택성(Selectivity) 높은 컬럼에 생성: 유니크한 값이 많은 컬럼(주민번호, 이메일)이 효과적. 성별처럼 카디널리티 낮은 컬럼은 비효율
  • 2
    복합 인덱스(Composite Index) 컬럼 순서: 선택성 높은 컬럼을 앞에, WHERE 조건에 자주 함께 사용되는 컬럼 구성. 좌측 전위(Leftmost Prefix) 규칙 준수
  • 3
    인덱스 남용 금지: DML(INSERT/UPDATE/DELETE) 성능 저하 유발. 테이블당 5~6개 이내 권장
  • 4
    커버링 인덱스(Covering Index): 쿼리에 필요한 모든 컬럼을 인덱스에 포함하여 테이블 접근 없이 인덱스만으로 결과 반환
Ⅴ. 결론

인덱스는 읽기 성능과 쓰기 성능 간의 트레이드오프가 존재한다. 실행 계획(EXPLAIN)을 통해 인덱스 활용 여부를 확인하고, 미사용 인덱스는 제거하여 DML 오버헤드를 최소화해야 한다. 감리 시에는 인덱스 설계서 존재 여부, 실행계획 분석 결과(Full Scan 존재 여부), 인덱스 단편화(Fragmentation) 관리 계획을 점검한다.

Ⅰ. 개요 — 조인의 개념
조인(JOIN)은 두 개 이상의 테이블에서 연관된 컬럼을 기준으로 데이터를 결합하는 연산으로, 관계형 DB의 핵심 기능이다. 관계대수의 카티션 곱(×)에 셀렉트(σ) 조건을 결합한 형태이다.
Ⅱ. 가. 조인 종류와 결과 집합 비교
조인 종류정의NULL 행 포함관계대수
INNER JOIN두 테이블에서 조인 조건을 만족하는 행만 반환 (교집합)없음R ⋈ S
LEFT OUTER JOIN왼쪽 테이블의 모든 행 + 오른쪽 매칭 행. 불일치 시 NULL오른쪽에 NULLR ⟕ S
RIGHT OUTER JOIN오른쪽 테이블의 모든 행 + 왼쪽 매칭 행. 불일치 시 NULL왼쪽에 NULLR ⟖ S
FULL OUTER JOIN양쪽 모든 행 반환. 불일치 시 양쪽에 NULL (합집합 효과)양쪽에 NULLR ⟗ S
CROSS JOIN카티션 곱. 모든 가능한 행 조합. |R| × |S| 행 생성없음R × S
SELF JOIN동일 테이블을 서로 다른 별칭으로 두 번 조인. 계층/자기 참조 구조조건에 따라R ⋈ R
-- LEFT OUTER JOIN: 부서 없는 직원도 포함 SELECT e.이름, d.부서명 FROM 직원 e LEFT OUTER JOIN 부서 d ON e.부서번호 = d.부서번호; -- SELF JOIN: 직원-상관 관계 (관리자 이름 조회) SELECT e.이름 AS 직원명, m.이름 AS 관리자명 FROM 직원 e LEFT JOIN 직원 m ON e.관리자번호 = m.사번;
Ⅲ. 나. 조인 알고리즘 비교
알고리즘원리시간복잡도인덱스적합 상황
Nested Loop Join
(NL Join)		외부 테이블(Driving) 행마다 내부 테이블(Driven)을 순차 탐색. 이중 for문과 동일O(m × n)
인덱스 시 O(m × log n)		내부 테이블 인덱스 필수소용량 테이블, OLTP 단건 조회, 드라이빙 테이블 행 수 적을 때
Sort-Merge Join
(SM Join)		두 테이블을 조인 키로 각각 정렬 후 순차 병합. 병합 단계는 O(m+n)O(m log m + n log n)불필요 (정렬로 대체)대용량 테이블, 이미 정렬된 데이터(클러스터드 인덱스), 등호(=) 조인

Ⅴ. 결론

조인은 관계형 DB의 핵심 연산으로, 조인 종류 선택(INNER vs OUTER)과 알고리즘 선택(NL/SM/Hash)이 성능에 결정적 영향을 미친다. 실행계획(EXPLAIN)을 통해 풀 테이블 스캔·불필요한 정렬·임시 테이블 생성을 탐지하고, 인덱스·힌트·통계 정보 갱신으로 최적화해야 한다. 감리 시에는 성능 테스트 결과에서 응답시간 초과 쿼리의 실행계획 분석 여부를 점검한다.

① 비정규형 → 1NF (원자값 확보)
위 예시 데이터는 이미 원자값이므로 1NF 만족. (만약 과목번호에 "C101,C102"처럼 다중값이 있었다면 행으로 분리) 1NF 릴레이션: 수강1NF(학번, 학생명, 학과코드, 학과명, 과목번호, 과목명, 성적, 교수번호, 교수명) 기본 키: (학번, 과목번호)
② 1NF → 2NF (부분 함수 종속 제거)
복합 키 (학번, 과목번호)에서 부분 종속 탐지:
부분 종속 발견: 학번 → 학생명, 학과코드, 학과명 (과목번호 없이도 결정) 과목번호 → 과목명, 교수번호, 교수명 (학번 없이도 결정) 분해 → 2NF 릴레이션: 학생(학번, 학생명, 학과코드, 학과명) ← 학번이 결정자 과목(과목번호, 과목명, 교수번호, 교수명) ← 과목번호가 결정자 수강2NF(학번, 과목번호, 성적) ← 기본 키 (학번, 과목번호)
③ 2NF → 3NF (이행 함수 종속 제거)
이행 종속 발견: 학생 릴레이션: 학번 → 학과코드 → 학과명 (이행 종속) 과목 릴레이션: 과목번호 → 교수번호 → 교수명 (이행 종속) 분해 → 3NF 릴레이션: 학생3NF(학번, 학생명, 학과코드) 학과(학과코드, 학과명) 과목3NF(과목번호, 과목명, 교수번호) 교수(교수번호, 교수명) 수강3NF(학번, 과목번호, 성적)
④ 3NF → BCNF (모든 결정자 = 후보 키 확인)
BCNF 검증: 과목3NF(과목번호, 과목명, 교수번호) 만약 교수번호 → 과목번호 (한 교수가 한 과목만 담당) 이면: 후보 키: {과목번호}, {교수번호} 교수번호 → 과목명 (교수번호가 결정자이나 후보 키 ✅) → 이 경우 BCNF 만족 최종 BCNF 정규화 릴레이션: 학생(학번, 학생명, 학과코드) PK: 학번 학과(학과코드, 학과명) PK: 학과코드 교수(교수번호, 교수명) PK: 교수번호 과목(과목번호, 과목명, 교수번호) PK: 과목번호, FK: 교수번호 수강(학번, 과목번호, 성적) PK: (학번, 과목번호)
Ⅲ. 나. FD 다이어그램 및 최소 커버
함수적 종속성 다이어그램 (FD Diagram) 학번 ──────────────────→ 학생명 학번 ─────────→ 학과코드 ──→ 학과명 (이행: 학번→학과명 제거 대상) 과목번호 ──────→ 과목명 과목번호 ─────→ 교수번호 ──→ 교수명 (이행: 과목번호→교수명 제거 대상) (학번, 과목번호) ────────→ 성적 (완전 함수 종속) 최소 커버(Minimal Cover) 도출: 1. 우변 단일화: 이미 단일 속성 2. 중복 FD 제거: 이행 종속 경로에서 직접 FD 제거 3. 불필요한 좌변 속성 제거 결과 최소 커버: 학번 → {학생명, 학과코드} 학과코드 → 학과명 과목번호 → {과목명, 교수번호} 교수번호 → 교수명 (학번, 과목번호) → 성적
Ⅳ. 다. 무손실 분해 및 종속성 보존 검증
속성무손실 분해 (Lossless Decomposition)종속성 보존 (Dependency Preservation)
정의분해된 릴레이션을 다시 자연 조인(Natural Join)했을 때 원래 릴레이션과 동일한 결과 반환원래 릴레이션의 모든 FD가 분해된 릴레이션 중 하나에 포함
중요성정보 손실 없는 분해 보장. 정규화의 필수 조건각 릴레이션에서 독립적으로 무결성 검사 가능
검증 방법R1 ∩ R2 → R1 또는 R1 ∩ R2 → R2 (교집합이 슈퍼키이면 무손실)모든 FD F가 ∪F_i에 포함되는지 확인. BCNF는 종속성 보존을 보장하지 않을 수 있음
주의모든 정규화 분해는 무손실이어야 함BCNF 분해 시 종속성 보존이 불가능한 경우 → 3NF로 타협
📌 BCNF vs 3NF 선택 기준
BCNF: 이상현상 완전 제거. 단, 종속성 보존 불가능할 수 있음
3NF: 종속성 보존 + 무손실 분해 모두 보장. 일부 이상현상 가능성 존재
→ 실무 권장: 3NF까지 정규화 + 필요 시 BCNF 적용 (종속성 보존 영향 분석 후)
Ⅴ. 결론

정규화는 데이터 무결성 보장의 이론적 기반으로, 공공 정보화사업의 DB 설계 단계에서 반드시 수행해야 한다. 감리 시에는 ①정규화 수준(3NF/BCNF 준수 여부), ②반정규화 적용 근거 문서, ③무손실 분해 검증 기록을 주요 점검항목으로 확인한다. 지나친 정규화로 인한 조인 증가는 실행계획 분석을 통해 모니터링하고, 필요 시 전략적 반정규화를 적용한다.

Ⅱ. 나. Lock 기반 제어 및 MVCC 비교
Lock 유형
S-Lock (공유 잠금, Shared Lock): 읽기 시 획득. 다른 T의 S-Lock 동시 허용. X-Lock 불허
X-Lock (배타 잠금, eXclusive Lock): 쓰기 시 획득. 다른 모든 Lock 불허
호환성 행렬: S+S=허용, S+X=불허, X+X=불허
2PL (2-Phase Locking) 프로토콜:
2단계 잠금 프로토콜 (직렬 가능성 보장) Growing Phase (확장 단계): Lock 획득만 가능, 해제 불가 ↕ (잠금 점, Lock Point) Shrinking Phase (수축 단계): Lock 해제만 가능, 획득 불가 직렬 가능성(Serializability) 보장: 2PL을 준수하면 트랜잭션 스케줄이 직렬 스케줄(Serial Schedule)과 동등함을 보장 단점: Deadlock 발생 가능, Cascading Rollback 위험
비교 항목Lock 기반 (Pessimistic)MVCC (Optimistic/Multi-Version)
기본 철학충돌이 자주 발생한다고 가정. 사전 차단충돌이 드물다고 가정. 버전으로 분리
원리읽기/쓰기 시 Lock 획득. 충돌 시 대기쓰기 시 새 버전 생성. 읽기는 스냅숏 조회 (Lock 불필요)
읽기 성능낮음 (S-Lock 경쟁)높음 (Lock 없는 읽기, 스냅숏)
쓰기 성능중간 (단순 Lock)버전 관리 오버헤드 존재
Phantom ReadGap Lock으로 방지스냅숏 격리로 방지
주요 DBMSDB2, SQL Server (기본)PostgreSQL (항상), MySQL InnoDB (기본), Oracle

Ⅲ. 다. 교착상태(Deadlock) 탐지 및 예방
Deadlock 발생 시나리오 (Wait-for 그래프) T1: X-Lock(A) 보유 → X-Lock(B) 대기 T2: X-Lock(B) 보유 → X-Lock(A) 대기 Wait-for 그래프: T1 → T2 → T1 (사이클 = 교착상태)
구분방법내용장단점
탐지
(Detection)		Wait-for 그래프주기적으로 그래프의 사이클 탐지. 사이클 발견 시 희생자(Victim) 선택·Rollback탐지 오버헤드. 희생자 재실행 필요
타임아웃일정 시간 이상 대기 시 자동 롤백 (MySQL innodb_lock_wait_timeout=50s)구현 단순. 정상 대기도 종료 위험
DBMS 자동 탐지Oracle·MySQL·PostgreSQL 모두 Deadlock 자동 탐지 후 하나의 트랜잭션 롤백실무 권장 (자동 처리)
예방
(Prevention)		Lock 순서 통일모든 T가 동일한 순서로 Lock 획득 (예: 항상 작은 ID 테이블 먼저)프로그래밍 규칙으로 방지 가능
Lock 범위 최소화트랜잭션 짧게 유지, 필요한 행만 잠금, 작업 완료 즉시 Commit가장 실용적인 방지 방법
Ⅳ. 결론

동시성 제어는 데이터 무결성과 시스템 성능의 균형점을 찾는 핵심 DB 관리 영역이다. 현대 DBMS는 MVCC를 기본 채택하여 읽기 성능을 극대화하고, 쓰기 충돌은 Lock으로 보완한다. 공공 정보시스템에서는 트랜잭션 격리 수준을 업무 특성에 따라 명시적으로 설정하고, Deadlock 탐지 로그를 모니터링하여 주기적으로 쿼리·인덱스를 최적화해야 한다.

    OR 조건의 인덱스 무력화: OR로 연결된 컬럼이 각각 다른 인덱스 사용 시 병합 비효율
    -- 🚫 OR 조건: 인덱스 비효율 WHERE 사번 = 'S001' OR 이메일 = 'hong@company.com' -- ✅ UNION ALL로 분리 (각각 인덱스 활용) SELECT * FROM 직원 WHERE 사번 = 'S001' UNION ALL SELECT * FROM 직원 WHERE 이메일 = 'hong@company.com' AND 사번 <> 'S001';

Ⅰ. 가. 윈도우 함수 개념 및 OVER절
윈도우 함수(Window Function, 분석 함수)는 현재 행과 관련된 행의 집합(Window)에 대해 집계 연산을 수행하되, 행을 그룹으로 합치지 않고 각 행의 결과를 개별로 반환하는 함수이다. SQL:2003 표준에 포함되었다.
OVER절 구성 문법
함수명() OVER (
  [PARTITION BY 그룹컬럼] ← 논리적 그룹 분할 (GROUP BY와 달리 행 유지)
  [ORDER BY 정렬컬럼 ASC/DESC] ← 윈도우 내 정렬 순서
  [ROWS|RANGE BETWEEN 시작 AND 끝] ← 프레임 범위 지정
)

프레임 기준: UNBOUNDED PRECEDING(처음) / CURRENT ROW / UNBOUNDED FOLLOWING(끝)
-- 부서별 급여 순위 + 전월 대비 + 누적합 SELECT 사번, 이름, 부서번호, 급여, -- 순위 함수 ROW_NUMBER() OVER (PARTITION BY 부서번호 ORDER BY 급여 DESC) AS 행번호, RANK() OVER (PARTITION BY 부서번호 ORDER BY 급여 DESC) AS 순위, DENSE_RANK() OVER (PARTITION BY 부서번호 ORDER BY 급여 DESC) AS 밀집순위, -- 이전/다음 행 참조 LAG(급여, 1, 0) OVER (PARTITION BY 부서번호 ORDER BY 입사일) AS 전행급여, LEAD(급여, 1, 0) OVER (PARTITION BY 부서번호 ORDER BY 입사일) AS 다음행급여, -- 누적합 (처음 행 ~ 현재 행) SUM(급여) OVER ( PARTITION BY 부서번호 ORDER BY 입사일 ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS 누적급여합, -- 이동 평균 (최근 3행) AVG(급여) OVER ( ORDER BY 입사일 ROWS BETWEEN 2 PRECEDING AND CURRENT ROW ) AS 이동평균급여 FROM 직원;
함수동점 처리예 (급여 500,500,400) ROW_NUMBER동점도 고유 번호1, 2, 3 RANK동점 같은 순위, 다음 건너뜀1, 1, 3 DENSE_RANK동점 같은 순위, 건너뜀 없음1, 1, 2
Ⅱ. 나. 집계 함수 vs 윈도우 함수 및 PIVOT·UNPIVOT
구분집계 함수 (GROUP BY)윈도우 함수 (OVER)
결과 행그룹당 1행으로 압축원본 행 수 유지
원본 컬럼그룹 키 + 집계 컬럼만 가능모든 원본 컬럼 + 집계 결과 동시 반환
WHERE 사용HAVING으로 집계 필터WHERE로 일반 필터 후 윈도우 적용
적합 상황요약 통계, 집계 리포트순위, 누적합, 이동 평균, 전후 행 비교
-- PIVOT: 행 → 열 변환 (분기별 매출을 열로) SELECT * FROM 매출 PIVOT ( SUM(금액) FOR 분기 IN ('Q1' AS Q1, 'Q2' AS Q2, 'Q3' AS Q3, 'Q4' AS Q4) ); -- Oracle 문법 -- MySQL 방식 (CASE WHEN으로 PIVOT 구현) SELECT 제품코드, SUM(CASE WHEN 분기='Q1' THEN 금액 ELSE 0 END) Q1매출, SUM(CASE WHEN 분기='Q2' THEN 금액 ELSE 0 END) Q2매출 FROM 매출 GROUP BY 제품코드;
Ⅲ. 다. 저장 프로시저·트리거·커서
객체개념주요 용도주의사항
저장 프로시저
(Stored Procedure)		DB 서버에 컴파일·저장된 SQL 문장의 집합. 파라미터 IN/OUT 지원복잡한 비즈니스 로직 캡슐화, 반복 쿼리 성능 향상, 보안 레이어DBMS 종속성 높음, 버전 관리 어려움, 과도한 로직 집중 회피 (Application Layer 권장)
트리거
(Trigger)		특정 이벤트(INSERT/UPDATE/DELETE) 발생 시 자동 실행되는 프로시저. BEFORE/AFTER/INSTEAD OF감사 로그(Audit Trail) 자동 기록, 참조 무결성 보완, 파생 데이터 자동 갱신숨겨진 로직으로 디버깅 어려움, 연쇄 트리거(Cascading) 주의, DML 성능 영향, 트랜잭션 범위 확대
커서
(Cursor)		SELECT 결과 집합을 한 행씩 순차적으로 처리하는 포인터 메커니즘행 단위 복잡한 처리, 집합 연산으로 해결 불가한 순차 처리집합 기반 SQL 대비 매우 느림 (행 단위 왕복). 가능하면 집합 연산(JOIN, 윈도우 함수)으로 대체 권장
-- 감사 트리거 예시: 급여 변경 이력 자동 기록 CREATE TRIGGER 급여변경감사 AFTER UPDATE ON 직원 FOR EACH ROW WHEN (OLD.급여 <> NEW.급여) BEGIN INSERT INTO 급여변경이력(사번, 변경전, 변경후, 변경일시, 변경자) VALUES(OLD.사번, OLD.급여, NEW.급여, SYSDATE, USER); END;
Ⅳ. 결론

윈도우 함수는 복잡한 분석 쿼리(순위, 누적, 이동 평균)를 단일 SQL로 우아하게 표현할 수 있는 강력한 도구이다. 저장 프로시저·트리거는 DB 내 로직 캡슐화에 유용하나, 관리 복잡성과 DBMS 종속성을 고려하여 적절히 활용해야 한다. 감리 시에는 트리거의 연쇄 실행 가능성, 커서의 성능 영향 여부, 저장 프로시저의 버전 관리 체계(형상관리 포함 여부)를 점검한다.
