Ⅰ. 개요 — 자율주행의 위치 인식 중요성
자율주행 차량은 GPS의 수 미터 오차로는 부족하고 차선 수준(10~20cm 이내) 정밀 위치 추정이 필요하다. HD맵 기반 사전 정보 활용과 SLAM 기반 실시간 환경 인식을 결합하는 것이 현재 최선의 접근이다.
Ⅱ. HD맵 (고정밀지도, High Definition Map)
HD맵은 자율주행 전용으로 제작된 cm 단위 정밀도의 3D 디지털 지도로, 사전에 수집된 정적 환경 정보를 담고 있다.

SLAM은 사전 지도 없이 센서 데이터만으로 동시에 자신의 위치를 추정하고 환경 지도를 생성하는 알고리즘이다.

Ⅰ. 개요
자율주행 차량은 수억 라인의 소프트웨어, 다수의 외부 통신 인터페이스(V2X·OTA·텔레매틱스·OBD-II), 50~150개 ECU로 구성된 복잡한 사이버-물리 시스템이다. 사이버 공격은 단순 서비스 장애를 넘어 탑승자·보행자의 생명과 직결되는 물리적 위해를 초래할 수 있어 기능안전(ISO 26262)과 동등한 수준의 보안 체계가 요구된다.
Ⅱ. 가. 자율주행 차량 공격 표면 및 공격 유형
자율주행 차량 공격 표면 (Attack Surface) 외부 연결: ├─ V2X (DSRC/C-V2X) → 위조 메시지로 가상 장애물·신호 삽입 ├─ 텔레매틱스 (LTE/5G TCU) → 원격 차량 제어 탈취 ├─ Wi-Fi/Bluetooth → 근거리 페어링 공격, 펌웨어 변조 ├─ OTA 업데이트 서버 → 악성 SW 배포 └─ GPS 수신기 → 스푸핑으로 위치 오도 내부 네트워크: ├─ CAN 버스 (비암호화) → 악성 프레임 주입 (Replay·Injection) ├─ CAN-FD / Ethernet → 중간자 공격(MitM) └─ OBD-II 포트 → 물리 접근 후 ECU 플래싱 센서 공격: ├─ LiDAR: 레이저 펄스 재밍·스푸핑 → 유령 객체 생성 ├─ Camera: 적대적 패치, 강한 빛으로 인식 방해 └─ GPS: 스푸핑으로 위치 오도 (오염된 경로 탐색)

Ⅲ. 나. ISO/SAE 21434 표준 주요 내용
ISO/SAE 21434(2021)는 자동차 사이버보안 엔지니어링 국제표준으로, 차량 개발 전 생명주기에 걸친 사이버보안 활동을 정의한다.
  • 1
    TARA (Threat Analysis and Risk Assessment, 위협 분석 및 리스크 평가)
    자산 식별: 보호 대상 사이버 자산(조향 ECU, 개인정보, 암호화 키) 식별
    위협 시나리오: STRIDE 방법론(Spoofing·Tampering·Repudiation·Info Disclosure·DoS·Elevation) 적용
    영향 분석: 안전·재정·운영·프라이버시 측면 피해 영향 평가 (Safety, Financial, Operational, Privacy)
    CAL 결정: 공격 실현 가능성 × 피해 영향 → CAL 1~4 등급 결정 (ISO 26262 ASIL과 유사)
  • 2
    CAL (Cybersecurity Assurance Level): CAL 1(최저)~CAL 4(최고). 등급에 따라 요구되는 보안 통제·검증 활동 수준 결정. CAL 4는 원격 차량 제어 ECU 등 안전 직결 기능에 적용
  • 3
    공급망 보안(Supply Chain): OEM이 Tier-1·Tier-2 부품 공급업체의 사이버보안 역량을 계약·감사로 관리. SBOM(소프트웨어 부품 명세서) 요구. 각 ECU 공급업체가 CSMS(사이버보안 관리 시스템) 운영 의무
ISO 26262 vs ISO/SAE 21434 비교
ISO 26262: 무작위적 HW 고장 + 설계 오류 (비의도적) 대응. ASIL 등급
ISO/SAE 21434: 악의적 사이버 공격 (의도적) 대응. CAL 등급
→ 두 표준은 서로 보완적이며, 현대 자율주행 차량은 두 표준 동시 준수 필요
Ⅳ. 다. 차량 사이버보안 아키텍처 설계 및 OTA 보안
  • 1
    보안 아키텍처 설계 원칙
    네트워크 분리(Segmentation): 파워트레인·ADAS·인포테인먼트 도메인 방화벽으로 격리. ADAS 도메인 침해가 엔진 ECU로 전파 차단
    최소 권한 원칙(Least Privilege): ECU 간 통신 권한을 필요 최소로 제한. CAN ID 화이트리스트 기반 필터링
    HSM (Hardware Security Module): ECU 내 암호화 키 안전 보관. 안전 부트(Secure Boot)·메모리 암호화
    IDS/IPS (침입 탐지·방지): 차량용 IDS(AUTOSAR SecOC)로 CAN 버스 이상 메시지 탐지. 행위 기반 이상 탐지
    VSOC (Vehicle Security Operations Center): 클라우드 기반 차량 보안 이벤트 중앙 모니터링·분석·대응
  • 2
    OTA 소프트웨어 업데이트 보안 (UNECE WP.29 규정 R156 준수)
    코드 서명(Code Signing): 업데이트 패키지를 OEM 개인키로 서명 → ECU에서 공개키 검증 후 적용
    전송 보안: TLS 1.3으로 업데이트 서버-차량 간 암호화 채널. 인증서 고정(Certificate Pinning)
    롤백 방지(Anti-Rollback): 이전 취약 버전으로의 다운그레이드 방지. 버전 카운터를 HSM에 보관
    원자적 업데이트(Atomic Update): A/B 파티션 방식으로 업데이트 실패 시 이전 버전 자동 복구. 주행 중 업데이트 금지 (안전 정차 시만 적용)
    무결성 검증: SHA-256 해시로 다운로드 패키지 변조 여부 확인. 부분 업데이트 지원으로 대역폭 최소화
Ⅴ. 결론

자율주행 차량의 사이버보안은 생명 안전과 직결된 최고 우선순위 과제이다. ISO/SAE 21434 TARA 기반 위험 식별, 심층 방어(Defense-in-Depth) 아키텍처, VSOC 기반 실시간 모니터링, 그리고 UNECE WP.29 기반 안전한 OTA 체계를 갖추는 것이 자율주행 상용화의 필수 조건이다. 국내에서는 「자동차관리법」 개정으로 CSMS(사이버보안 관리 시스템) 인증 의무화가 진행되고 있으며, 제조사-공급망-인프라 전반의 보안 생태계 구축이 요구된다.
