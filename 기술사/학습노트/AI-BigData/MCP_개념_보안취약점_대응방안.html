MCP(Model Context Protocol)란 AI 어시스턴트(LLM)와 외부 데이터 소스·도구를 표준화된 방식으로 연결하는 오픈 프로토콜이다. 2024년 11월 Anthropic이 공개하였으며, USB-C가 다양한 기기를 단일 인터페이스로 연결하듯 LLM이 어떤 외부 시스템과도 일관된 방법으로 상호작용할 수 있도록 설계되었다.

등장 배경 — 기존 통합 방식의 한계

LLM을 기업 시스템(DB, API, 파일 서버 등)과 연동하려면 각 시스템별 커스텀 플러그인이나 별도 어댑터를 개발해야 했다. 시스템 수만큼 N×M의 통합 복잡도가 증가하여 유지보수 비용이 기하급수적으로 늘어나는 문제가 있었다.
기존 방식 문제점
  • LLM마다 별도 통합 코드 필요
  • 보안 정책·권한 관리 비표준
  • 컨텍스트 전달 방식 불일치
  • 신규 도구 추가 시 재개발 필요
MCP 해결 방식
  • 단일 표준 프로토콜로 M:N 통합 해소
  • 표준화된 인증·권한 위임 체계
  • JSON-RPC 2.0 기반 일관된 메시지
  • 새 서버 추가 시 클라이언트 수정 불필요

핵심 개념 3요소

Resources (리소스)파일·DB·웹 콘텐츠 등 LLM이 읽을 수 있는 데이터. URI 기반 식별·접근
Tools (도구)LLM이 호출 가능한 함수·API. 서버가 등록하고 클라이언트가 승인 후 실행
Prompts (프롬프트)재사용 가능한 프롬프트 템플릿. 서버가 제공·클라이언트가 UI로 노출
Sampling (샘플링)서버가 LLM에 추론을 요청. Human-in-the-loop 원칙 적용
Roots (루트)클라이언트가 서버에 허용한 작업 범위 경계. 접근 가능 URI 목록
Transportstdio / HTTP+SSE 두 가지 전송 방식. JSON-RPC 2.0 메시지 포맷

MCP 3계층 아키텍처

MCP 통신 흐름 — 도구 호출 시퀀스

#단계방향내용
1InitializeClient → Server프로토콜 버전·기능 협상
2Tool DiscoveryClient ← Server사용 가능한 Tools·Resources 목록 수신
3사용자 입력User → Host자연어 쿼리 입력
4LLM 추론Host 내부LLM이 적절한 Tool 선택·파라미터 생성
5사용자 승인Host → User도구 호출 전 사용자 확인 (Human-in-loop)
6Tool 실행Client → ServerJSON-RPC tools/call 요청
7외부 시스템 호출Server → External실제 DB 쿼리·API 호출·파일 접근
8결과 반환Server → ClientJSON 포맷 결과 반환
9최종 응답 생성LLM → User결과를 바탕으로 자연어 응답 생성
핵심 원칙: Human-in-the-Loop — 모든 도구 실행은 사용자 승인을 거친 후에만 수행. LLM이 자율적으로 외부 시스템을 호출하지 않음.
MCP는 LLM에게 외부 시스템 접근 권한을 부여하므로, 기존 애플리케이션 보안과는 다른 신규 공격 벡터가 존재한다. 특히 AI 에이전트가 자율적으로 행동하는 특성상 공격 파급 범위가 매우 크다.

대응 전략 핵심: MCP 보안은 ① 최소 권한 원칙 ② Human-in-the-Loop ③ 입력 검증 ④ 공급망 신뢰 검증의 4대 축으로 구성된다.

① 프롬프트 인젝션 대응

입력 경계 구분시스템 지시·사용자 입력·외부 데이터를 명확히 분리하여 LLM에 전달
외부 데이터 샌드박스웹/파일 내용은 신뢰할 수 없는 데이터로 태그하여 별도 처리
출력 검증 필터LLM 생성 Tool 호출 파라미터를 실행 전 정규식·화이트리스트로 검증
지시 계층 적용시스템 프롬프트 > 사용자 입력 > 외부 데이터 순으로 신뢰 계층 설정

② 최소 권한 원칙 (Least Privilege)

MCP 서버는 업무 수행에 필요한 최소 권한만 LLM에 제공해야 한다. 과도한 Tool 노출은 공격 성공 시 피해 범위를 극대화한다.
항목권장 사항
Roots 설정클라이언트가 접근 허용할 URI 범위를 명시적으로 화이트리스트로 정의
Tool 권한읽기 전용 Tool과 쓰기 Tool을 분리하여 기본은 읽기 전용 노출
외부 시스템 계정MCP 서버가 사용하는 DB·API 계정에 최소 권한 부여 (READ-ONLY 우선)
네트워크 접근MCP 서버의 아웃바운드 연결을 허용 도메인 화이트리스트로 제한
토큰 수명OAuth 2.1 토큰에 짧은 만료 시간(TTL) 설정 + 갱신 시 재인증

③ Human-in-the-Loop 강화

필수 승인 대상
  • 데이터 쓰기/삭제 Tool 실행 전
  • 외부 네트워크 호출 (이메일·API 전송)
  • 민감 정보(PII) 포함 데이터 접근
  • 금전 거래·계약 관련 작업
자동화 허용 가능
  • 읽기 전용 데이터 조회
  • 내부 분석·계산 작업
  • 비민감 공개 정보 검색
  • 단순 포맷 변환·요약

④ 공급망 보안 — 악성 서버 방지

MCP 서버는 npm/PyPI 패키지처럼 제3자가 배포하므로, 신뢰할 수 없는 서버를 연결하면 데이터 유출·원격 코드 실행 위험이 있다.
서버 레지스트리 검증공식 인증된 MCP 서버 레지스트리에서만 설치. 비공식 서버 차단
코드 서명 검증MCP 서버 패키지에 디지털 서명·해시 검증 의무화 (SBOM 관리)
샌드박스 격리MCP 서버를 컨테이너(Docker)로 격리. 호스트 파일시스템 직접 접근 차단
동작 모니터링서버의 Tool 호출 로그·API 요청을 SIEM으로 실시간 이상 탐지
Pin 버전 관리MCP 서버 버전을 고정(Pin)하여 의존성 혼란 공격(Confusion Attack) 방지
주기적 감사설치된 MCP 서버 목록 및 권한을 분기별 감사. 미사용 서버 즉시 제거

⑤ 통신 및 인증 보안

보안 영역대응 방안적용 기준
전송 암호화HTTP+SSE 방식에 TLS 1.3 이상 강제. stdio는 프로세스 격리로 보호모든 원격 MCP 서버
인증·권한 위임OAuth 2.1 + PKCE 사용. API 키 방식 지양. 토큰 스코프 최소화외부 노출 서버
메시지 무결성JSON-RPC 요청·응답에 HMAC 서명 추가하여 변조 탐지고위험 환경
세션 관리짧은 TTL 토큰 + 갱신 시 재인증. 비활성 세션 자동 종료모든 환경
로깅·감사모든 Tool 호출·Resource 접근을 불변 감사 로그(Immutable Audit Log)에 기록엔터프라이즈 환경
