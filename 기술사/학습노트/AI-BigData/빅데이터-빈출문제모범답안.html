Ⅰ. MapReduce 개념
MapReduce는 Google이 2004년 발표한 대용량 데이터 병렬 처리 프레임워크로, 분산 클러스터에서 Map과 Reduce 두 단계의 함수형 프로그래밍 모델을 통해 페타바이트급 데이터를 처리한다. Hadoop의 핵심 처리 엔진으로 구현되었다.
Ⅱ. 처리 단계 및 원리
입력 데이터 (HDFS) │ ├─ [Input Split] : 데이터를 블록(128MB) 단위로 분할 │ ├─ [Map 단계] │ 각 Split에 Map 함수 병렬 적용 │ Input : (key1, value1) │ Output : list(key2, value2) ← 중간 키-값 쌍 생성 │ ├─ [Shuffle & Sort] │ 중간 결과를 키(key2) 기준으로 정렬·그룹핑 │ 동일한 키를 가진 값들을 같은 Reducer로 전송 │ ├─ [Reduce 단계] │ Reduce 함수로 같은 키의 값 목록을 집계 │ Input : (key2, list(value2)) │ Output : list(key3, value3) ← 최종 결과 │ └─ 결과 저장 (HDFS)
단계역할처리 주체
Input Split입력 데이터를 블록 단위로 분할하여 Mapper에 할당JobTracker / ResourceManager
Map각 레코드에 사용자 정의 함수 적용 → 중간 키-값 쌍 생성Mapper (각 DataNode)
Shuffle & Sort중간 키 기준 정렬·그룹핑 후 네트워크 전송MapReduce 프레임워크 자동 수행
Reduce같은 키의 값 목록을 집계·변환하여 최종 결과 산출Reducer
Ⅲ. Word Count 예시
입력: "Hello World Hello Hadoop World" [Map 결과] (Hello,1) (World,1) (Hello,1) (Hadoop,1) (World,1) [Shuffle & Sort 후] Hello → [1, 1] Hadoop → [1] World → [1, 1] [Reduce 결과] (Hello, 2) (Hadoop, 1) (World, 2)
📌 MapReduce 한계 및 발전
① 디스크 I/O 병목: Map·Reduce 단계마다 HDFS에 중간 결과 저장 → Spark의 인메모리(RDD) 처리로 극복
② 실시간 처리 불가: 배치 처리 전용 → Flink·Spark Streaming으로 보완
③ 반복 알고리즘 비효율: ML 학습 등 반복 작업에 취약

Ⅰ. 개념
Lambda Architecture는 Nathan Marz(2011)가 제안한 빅데이터 처리 아키텍처로, 배치 처리(정확성)와 실시간 처리(신속성)를 병렬로 운영하여 대용량 데이터에 대한 임의 쿼리를 낮은 지연시간으로 처리하는 분산 데이터 처리 설계 패턴이다.
Ⅱ. 3계층 구성 및 데이터 흐름

계층역할특성기술 스택
Batch Layer전체 데이터셋 저장 및 주기적 배치 뷰 재계산고정확도, 고지연(수 시간~수 일)HDFS, Hadoop, Spark
Speed Layer최근 미처리 데이터의 실시간 증분 뷰 생성저지연, 근사 정확도Spark Streaming, Flink, Storm
Serving Layer배치 뷰 + 실시간 뷰 병합하여 쿼리 응답읽기 최적화, 색인HBase, Cassandra, Druid

📌 최근 트렌드
스트리밍 처리 기술(Apache Flink, Kafka Streams)의 발전으로 Kappa Architecture 채택이 증가하고 있으며, 배치·스트리밍을 통합하는 통합 처리 플랫폼(Apache Beam, Spark Structured Streaming)이 주목받고 있다.

Ⅰ. 개요 — Hadoop의 개념
Hadoop은 Apache 재단이 개발한 오픈소스 분산 컴퓨팅 프레임워크로, Google의 GFS·MapReduce 논문(2003~2004)을 기반으로 구현되었다. 수백~수천 대의 범용 서버(Commodity Hardware)에 데이터를 분산 저장·처리하여 페타바이트급 빅데이터를 처리하는 사실상 표준 플랫폼이다.
Ⅱ. 가. Hadoop 핵심 구성요소
① HDFS (Hadoop Distributed File System)
블록 단위(기본 128MB)로 데이터를 분할하여 3개 노드에 복제(Replication Factor=3) 저장.
NameNode: 메타데이터(파일명·블록 위치·권한) 관리. SPOF 해결을 위해 HA(Active-Standby) 구성.
DataNode: 실제 블록 저장. 주기적으로 NameNode에 Heartbeat·Block Report 전송.
Secondary NameNode: EditLog와 FsImage를 주기적으로 병합(Checkpoint). HA 환경에서는 Standby NameNode가 대체.

② YARN (Yet Another Resource Negotiator)

ApplicationMaster: 애플리케이션별 자원 요청·태스크 스케줄링 담당.
③ MapReduce
분산 병렬 처리 프레임워크. Map → Shuffle&Sort → Reduce 3단계 처리. 디스크 기반 처리로 높은 내결함성을 제공하나 반복 처리에 비효율적. → Apache Spark(인메모리)으로 대체 추세.

Ⅰ. 개요
Apache Spark는 UC Berkeley AMPLab이 개발(2009)한 인메모리 기반 범용 분산 데이터 처리 엔진으로, Hadoop MapReduce의 디스크 I/O 병목을 극복하고 배치·스트리밍·ML·그래프 처리를 단일 플랫폼에서 지원한다. MapReduce 대비 인메모리 처리 시 최대 100배, 디스크 기반 처리 시 최대 10배 빠른 성능을 제공한다.
Ⅱ. 가. Spark 아키텍처 및 실행 모델
Spark 클러스터 아키텍처 사용자 코드(spark-submit) │ [Driver Program] SparkContext 생성 DAG 생성 → Logical Plan → Physical Plan Task 스케줄링 → Executor 배포 │ 자원 요청 [Cluster Manager] (YARN / Kubernetes / Standalone) │ ├─ [Executor A (Worker Node 1)] │ Task1-1 / Task1-2 (각 Task = 1개 파티션 처리) │ 인메모리 캐시(RDD Cache) │ └─ [Executor B (Worker Node 2)] Task2-1 / Task2-2
실행 단위설명특징
Job하나의 Action(collect, save 등) 호출로 생성되는 최상위 실행 단위사용자 Action당 1개 Job 생성
StageShuffle 경계로 분리된 Task 집합. Narrow/Wide Dependency로 구분Wide Transformation(groupBy, join)마다 Stage 경계 생성
Task하나의 파티션을 처리하는 최소 실행 단위. Executor의 스레드에서 실행파티션 수 = Task 수 (기본값 200개)
Ⅲ. 나. RDD·DataFrame·Dataset 비교 및 Catalyst Optimizer
추상화도입 버전특징최적화타입 안전성

Catalyst Optimizer 동작 원리 (Spark SQL)

Ⅳ. 다. Spark 성능 최적화 기법
  • 1
    파티셔닝(Partitioning): 데이터를 처리 목적에 맞게 분할하여 병렬성 극대화.
    파티션 수 = 코어 수 × 2~3배 권장 (기본값 spark.sql.shuffle.partitions=200)
    repartition(): 균등 분산 (Shuffle 발생) / coalesce(): 축소 (Shuffle 없음)
  • 2
    캐싱(Caching): 반복 사용 데이터를 메모리에 캐시하여 재계산 방지.
    persist(MEMORY_AND_DISK): 메모리 부족 시 디스크 Spill
    cache() = persist(MEMORY_ONLY). 반복 ML 학습 루프에 필수
  • 3
    셔플 최소화: Wide Transformation(groupByKey, join) 줄이기.
    groupByKey → reduceByKey/aggregateByKey (Map 측 사전 집계로 네트워크 전송량 감소)
    Adaptive Query Execution(AQE, Spark 3.0): 런타임 통계 기반 동적 파티션 병합
  • 4
    브로드캐스트 조인(Broadcast Join): 소규모 테이블(기본 10MB 이하)을 모든 Executor에 브로드캐스트하여 Shuffle 없는 조인.
    spark.broadcast.threshold 조정으로 임계값 변경 가능 (권장: 50~200MB)

Ⅴ. 결론

Apache Spark는 인메모리 처리, Catalyst Optimizer, Tungsten 실행 엔진을 통해 빅데이터 처리의 속도와 범용성을 획기적으로 향상시켰다. Spark 3.x의 Adaptive Query Execution(AQE), Dynamic Partition Pruning, GPU 가속(Rapids) 기능으로 지속 발전 중이며, Kubernetes 위에서 실행되는 클라우드 네이티브 Spark가 현대적 빅데이터 아키텍처의 표준으로 자리잡고 있다.

Ⅰ. 개요 — Apache Flink의 등장 배경
Apache Flink는 독일 베를린 공대(TU Berlin)에서 개발(2014)한 상태 기반(Stateful) 분산 스트림 처리 엔진으로, 진정한(True) 스트리밍 처리를 지원한다. Spark Streaming(마이크로 배치)과 달리 이벤트 단위로 처리하는 네이티브 스트리밍 아키텍처로, 밀리초 수준의 극저지연(Ultra-Low Latency)과 정확히 한 번(Exactly-Once) 처리를 보장한다.
Ⅱ. 가. Flink 아키텍처 및 스트림 처리 모델
Apache Flink 클러스터 아키텍처 사용자 프로그램 (DataStream API / Table API) │ 제출 [JobManager] ├─ Dispatcher: REST 요청 수신, Job 제출 처리 ├─ ResourceManager: TaskManager 자원 관리 └─ JobMaster: Job 실행 그래프(DAG) 관리 Checkpoint 조율 장애 복구 결정 │ Task 배포 [TaskManager × N] ├─ Task Slot: 병렬 처리 단위 (1 Slot = 1 Thread) ├─ Operator: Source → Transform → Sink └─ State Backend: 상태 저장 (RocksDB / Heap)
Checkpoint 메커니즘 (Chandy-Lamport 알고리즘 기반)
① JobManager가 주기적으로 Barrier(체크포인트 마커)를 스트림에 삽입
② 각 Operator는 Barrier 수신 시 현재 상태를 State Backend에 비동기 스냅샷 저장
③ 장애 발생 시 가장 최근 완료된 체크포인트로 롤백 → Exactly-Once 보장
④ Savepoint: 수동 체크포인트. 버전 업그레이드·클러스터 마이그레이션에 활용
Ⅲ. 나. 이벤트 시간·처리 시간·Watermark
시간 개념정의특성활용
이벤트 시간 (Event Time)이벤트가 실제로 발생한 시간 (소스 장치의 타임스탬프)정확하나 지연 데이터(Late Event) 처리 필요정확한 시간 기반 분석 (금융 거래, 클릭스트림)
처리 시간 (Processing Time)스트림 처리 엔진이 레코드를 처리하는 시스템 시간단순하나 이벤트 실제 발생 순서와 다를 수 있음지연에 덜 민감한 실시간 모니터링
수집 시간 (Ingestion Time)데이터가 Flink 소스에 도착한 시간이벤트 시간과 처리 시간의 중간소스 타임스탬프 없는 데이터
Watermark를 이용한 지연 데이터 처리 시간축: ──────────────────────────────────────────▶ 이벤트: E(t=1) E(t=3) E(t=2) E(t=5) E(t=4) E(t=2) ← 지연 도착 (Late Event) Watermark(t=3) 발행 → "t≤3인 모든 이벤트가 도착했다고 선언" → [1,3] 윈도우 트리거 (t=2 이벤트는 포함) Watermark = max(이벤트 시간) - 허용 지연 시간(ε) 예) max=5, ε=2 → Watermark=3 t=2인 지연 이벤트: Watermark(3) 발행 전 도착 → 정상 처리 (해당 윈도우에 포함) Watermark(3) 발행 후 도착 → 지연 데이터 (allowedLateness로 추가 허용 또는 Side Output)
Flink 윈도우(Window) 유형
Tumbling Window: 고정 크기, 겹치지 않는 시간 윈도우 (예: 매 5분)
Sliding Window: 겹치는 윈도우. Slide < Size (예: 10분 윈도우, 5분마다 슬라이드)
Session Window: 이벤트 간 Gap 기반 동적 윈도우. 사용자 세션 분석에 적합

Ⅴ. 결론

Apache Flink는 이벤트 시간 기반 처리, Watermark를 통한 지연 데이터 처리, Exactly-Once 보장, 풍부한 상태 관리로 금융 거래 이상 탐지·실시간 모니터링 등 극저지연이 요구되는 분야의 사실상 표준으로 자리잡고 있다. Spark Streaming은 ML/배치 워크로드와의 통합 편의성이 강점이므로, 요구사항(지연시간·ML 통합·생태계)에 따라 적합한 플랫폼을 선택하는 것이 중요하다.
