IP 스푸핑의 개념·동작 원리·공격 절차 및 대응 방안
⏱ 목표 15분 📄 1.5~2페이지 💯 10~15점
Ⅰ. 개념
IP 스푸핑(IP Spoofing)은 공격자가 IP 패킷의 출발지 IP 주소(Source IP)를 신뢰받는 다른 호스트의 주소로 위조하여, 피해자 시스템이 공격자를 신뢰할 수 있는 호스트로 착각하게 만드는 네트워크 계층(L3) 공격 기법이다. TCP의 3-Way Handshake 신뢰 관계를 악용하며, DoS·세션 하이재킹·반사 공격의 기반이 된다.
Ⅱ. 동작 원리 및 공격 절차
IP 스푸핑 공격 절차 (Blind Spoofing 기준) [사전 단계] ① 공격자가 피해자(B)와 신뢰 관계에 있는 신뢰 호스트(A) 식별 ② 신뢰 호스트(A)를 SYN Flood 등으로 무력화 (응답 불가 상태) [공격 단계] ③ 공격자가 출발지 IP를 A의 IP로 위조한 SYN 패킷 → 피해자(B) 전송 ④ 피해자(B)가 SYN-ACK를 A(위조된 IP)로 응답 → 공격자는 못 받음 ⑤ 공격자가 B의 ISN(초기 순서번호)을 예측하여 ACK 패킷 전송 (Blind Spoofing: 응답 패킷 미수신 상태에서 ISN 추측) ⑥ B는 신뢰 호스트 A로부터의 연결로 인식 → 공격자의 명령 수행
Ⅲ. 공격 유형 분류
유형설명활용 공격
Blind Spoofing공격자가 응답 패킷을 받지 못하는 상태에서 ISN 추측으로 공격. 동일 네트워크 외부에서 수행신뢰 관계 우회, DoS
Non-Blind Spoofing공격자가 피해자와 같은 서브넷에 있어 응답 패킷 스니핑 가능. ISN 추측 불필요세션 하이재킹 전 단계
반사 공격
(Reflection)		출발지 IP를 피해자 IP로 위조하여 반사 서버로 대량 요청. 반사 서버가 피해자에게 응답 전송DRDoS, Smurf Attack
Ⅳ. 대응 방안
대응 기법내용적용 계층
Ingress Filtering
(RFC 2827)		라우터·방화벽에서 내부 네트워크 주소를 출발지로 하는 외부 유입 패킷 차단. 위조 IP 패킷 입구 차단네트워크(L3)
Egress Filtering내부에서 출발지 IP가 자사 IP 대역이 아닌 패킷의 외부 발신 차단네트워크(L3)
uRPF
(Unicast RPF)		라우팅 테이블을 역방향으로 검사하여 출발지 IP의 경로가 수신 인터페이스와 일치하지 않으면 패킷 폐기라우터
ISN 랜덤화TCP 초기 순서번호를 예측 불가능하게 랜덤 생성. Blind Spoofing의 ISN 추측 무력화전송(L4)
IPsecIP 계층에서 패킷 출발지 인증(AH 헤더). 위조 IP 패킷 인증 실패로 폐기네트워크(L3)
Ⅴ. 결론 — 감리 포인트

IP 스푸핑은 단독 공격보다 DDoS·세션 하이재킹·ARP 스푸핑의 선행 단계로 활용된다. 네트워크 장비 감리 시 Ingress/Egress Filtering 설정 여부, uRPF 활성화, 경계 라우터의 ACL 정책을 필수 점검항목으로 확인해야 한다.

ARP 스푸핑의 개념·공격 원리·ARP 캐시 포이즈닝 및 대응 방안
⏱ 목표 15분 📄 1.5~2페이지 💯 10~15점
Ⅰ. ARP 프로토콜 개념 및 취약점
ARP(Address Resolution Protocol)는 IP 주소로부터 MAC 주소를 얻기 위한 데이터링크 계층(L2) 프로토콜이다. ARP는 설계상 비인증(Stateless) 방식으로, 요청 없이 수신한 ARP Reply도 무조건 캐시에 저장하는 취약점(Gratuitous ARP 악용)이 있다. ARP 스푸핑은 이 취약점을 이용해 피해자의 ARP 캐시를 위조하는 공격이다.
Ⅱ. 공격 원리 및 ARP 캐시 포이즈닝
ARP 스푸핑 / ARP 캐시 포이즈닝 공격 흐름 [정상 환경] PC-A (IP:192.168.1.2) ←→ 게이트웨이 (IP:192.168.1.1, MAC:AA:AA) [공격 수행 — 공격자 MAC: CC:CC] ① 공격자가 PC-A에게 위조 ARP Reply 전송: "192.168.1.1 (게이트웨이)의 MAC은 CC:CC(공격자)입니다" ② 공격자가 게이트웨이에게 위조 ARP Reply 전송: "192.168.1.2 (PC-A)의 MAC은 CC:CC(공격자)입니다" [결과 — MITM(중간자 공격) 성립] PC-A → (CC:CC 경유) → 게이트웨이 : 공격자가 패킷 도청·변조 ARP 캐시 포이즈닝: PC-A와 게이트웨이의 ARP 캐시 테이블이 위조된 상태
ARP 스푸핑 vs ARP 캐시 포이즈닝
• ARP 스푸핑: 위조된 ARP 응답 패킷을 전송하는 공격 행위 자체
• ARP 캐시 포이즈닝: ARP 스푸핑의 결과로 피해자의 ARP 캐시 테이블이 오염된 상태
→ 원인(스푸핑) → 결과(포이즈닝)의 관계. 혼용하여 사용되기도 함

Ⅴ. 결론

ARP 스푸핑은 내부 네트워크에서 가장 빈번히 발생하는 MITM 공격의 기반 기술이다. 스위치 보안 설정(DAI + DHCP Snooping + 포트 보안)의 3중 적용과 전 구간 암호화(Zero Trust 네트워크)가 핵심 대응 전략이다.

⏱ 목표 15분 📄 1.5~2페이지 💯 10~15점
Ⅰ. 개념 및 DNS 취약점
DNS 스푸핑은 공격자가 위조된 DNS 응답을 피해자에게 먼저 전송하여 사용자를 악성 IP로 유도하는 공격이다. DNS 프로토콜은 UDP 기반(포트 53)으로 설계상 응답 인증 메커니즘이 없어, 먼저 도착한 응답을 정답으로 처리하는 취약점을 가진다.
Ⅱ. 공격 절차
DNS 스푸핑 공격 흐름 (로컬 네트워크 기준) ① 공격자가 ARP 스푸핑으로 MITM 위치 선점 ② 피해자 PC → DNS Query: "www.bank.co.kr의 IP는?" (UDP/53) ③ 공격자가 실제 DNS 서버보다 먼저 위조 DNS 응답 전송: "www.bank.co.kr → 공격자 IP (1.2.3.4)" ④ 피해자 PC가 위조 응답을 캐시에 저장 (DNS 캐시 포이즈닝) ⑤ 피해자가 피싱 사이트(공격자 서버)에 접속 → 자격증명 탈취 DNS 캐시 포이즈닝 (Kaminsky 공격, 2008): ① 공격자가 재귀 DNS 서버에 존재하지 않는 하위 도메인 대량 쿼리 ex) random1234.bank.co.kr ② 재귀 서버가 권한 서버에 질의하는 동안, 공격자가 트랜잭션 ID(16bit = 65536 경우의 수)를 무차별 대입 위조 응답 + NS 레코드 조작: bank.co.kr → 공격자 DNS 서버 ③ 성공 시 전체 도메인 영역(zone) 하이재킹
Ⅲ. 대응 방안 — DNSSEC 중심
대응 기법내용효과
DNSSEC
(DNS Security Extension)		DNS 응답에 디지털 서명(RSA/ECDSA) 추가. 공개키 기반으로 응답 무결성·인증 검증. RRSIG·DNSKEY·DS·NSEC 레코드 활용위조 응답 탐지·차단 (근본 해결책)
DNS over HTTPS
(DoH, RFC 8484)		DNS 질의·응답을 HTTPS(443) 암호화 터널로 전송. 도청·스푸핑 방지전송 구간 암호화 (Cloudflare 1.1.1.1)
DNS over TLS
(DoT, RFC 7858)		DNS를 TLS(853) 암호화 채널로 전송전송 구간 암호화
Source Port 랜덤화DNS 쿼리 시 출발지 포트를 랜덤 선택. 트랜잭션 ID(16bit) + 포트(16bit) 조합으로 예측 공간 확장 (2^32)Kaminsky 공격 완화
TTL 관리짧은 TTL로 포이즈닝된 캐시의 영향 기간 최소화피해 범위 축소

Ⅳ. 결론

DNS 스푸핑은 피싱·파밍 공격의 핵심 수단으로, DNSSEC 도입이 근본적 해결책이나 전 세계 도입률이 낮다는 한계가 있다. 단기적으로는 Source Port 랜덤화·DoH/DoT를 적용하고, 장기적으로 DNSSEC 전면 도입과 DANE(DNS-based Authentication of Named Entities)을 통한 TLS 인증서 검증 연계가 권장된다.

Ⅰ. DoS 공격 개요
서비스 거부 공격(DoS: Denial of Service)은 공격자가 대량의 패킷이나 비정상 패킷을 전송하여 피해자 시스템의 자원(CPU·메모리·대역폭·연결)을 고갈시켜 정상 사용자의 서비스 이용을 불가능하게 하는 공격이다.
Ⅱ. 유형별 공격 원리 및 대응 비교
공격 유형악용 취약점공격 원리대응 방안
SYN FloodingTCP 3-Way Handshake의 Half-Open 연결 자원 소모위조 출발지 IP로 대량 SYN 전송 → 서버가 SYN-ACK 후 대기 (Backlog 큐 고갈, 정상 연결 불가)SYN Cookie, SYN Backlog 크기 증가, SYN Rate Limiting, 방화벽 임계값 설정
Smurf AttackICMP + 브로드캐스트 + IP 스푸핑 조합출발지 IP를 피해자 IP로 위조 → 브로드캐스트 주소로 ICMP Echo 전송 → 네트워크 내 모든 호스트가 피해자에게 ICMP Reply 폭주 (증폭 공격)브로드캐스트 ICMP 비활성화(directed broadcast 차단), Ingress Filtering, 반사 네트워크 차단
Ping of DeathIP 단편화 재조합 시 버퍼 오버플로IP 최대 크기(65,535 byte) 초과 ICMP 패킷을 단편화하여 전송. 재조합 시 버퍼 오버플로 → 시스템 충돌·재부팅ICMP 패킷 크기 필터링(방화벽), OS 패치(재조합 버퍼 크기 검증), 대형 ICMP 패킷 차단
Teardrop AttackIP 단편화 오프셋 값 조작IP 단편 패킷의 오프셋 값을 중복·겹치도록 위조하여 전송. 재조합 과정에서 오버랩 처리 오류 → 커널 크래시·시스템 다운IP 단편 오프셋 검증 (OS 패치), 단편 패킷 재조합 타임아웃 설정, 방화벽에서 비정상 단편 차단
Ⅲ. SYN Cookie 동작 원리 (SYN Flooding 핵심 대응)
SYN Cookie 동작 방식 정상 TCP: Client SYN → Server: Backlog 큐에 저장 후 SYN-ACK SYN Cookie: Client SYN → Server: 큐 미사용, SYN-ACK의 ISN에 해시값(IP·포트·타임스탬프) 암호화 삽입 Client ACK 수신 시 ISN 검증 → 정상이면 연결 수립 → Backlog 큐를 사용하지 않아 SYN Flooding 무력화

Ⅳ. 결론

Smurf·Ping of Death·Teardrop은 현대 OS 패치로 대부분 완화되었으나, SYN Flooding은 여전히 DDoS의 핵심 공격으로 사용된다. SYN Cookie + Rate Limiting + Anti-DDoS 장비의 조합이 현실적 방어 전략이다.

(Spear Phishing)		특정 대상 맞춤형 이메일. 실존 인물 사칭, 업무 관련 첨부파일(악성 매크로·LNK 파일)이메일 샌드박스 분석, DMARC/SPF/DKIM, 사용자 보안 인식 교육, 첨부파일 실행 차단 정책
워터링홀
(Watering Hole)		표적이 자주 방문하는 웹사이트를 사전 감염. 방문 시 드라이브바이 다운로드로 악성코드 설치취약점 패치 최신화, 브라우저 샌드박스, NoScript/광고차단, 웹 게이트웨이 URL 필터링
제로데이 공격
(Zero-Day)		아직 패치되지 않은 취약점 악용. 탐지 어려움. CVE 미공개 상태에서 사용행위 기반 탐지(EDR), 가상화·샌드박스 격리, 최소 권한 원칙, 네트워크 마이크로세그멘테이션
측면 이동
(Lateral Movement)		초기 거점에서 내부 네트워크로 확산. Pass-the-Hash·PsExec·WMI 활용. 정상 관리 도구(LoL) 사용네트워크 세그멘테이션, 계정 권한 최소화, 특권 계정 관리(PAM), EDR 행위 탐지, 내부 동서 트래픽 모니터링
C&C 은닉DNS 터널링·HTTPS 채널·SNS·클라우드 서비스로 C&C 통신 은닉. 정상 트래픽과 혼재DNS 이상 쿼리 탐지, TLS 인스펙션, UEBA(사용자·개체 행위 분석), 국가별 IP 차단, IoC(침해지표) 기반 차단
MITRE ATT&CK 프레임워크 활용
ATT&CK(Adversarial Tactics, Techniques & Common Knowledge)는 14개 전술(Tactic)·수백 개 기술(Technique)로 APT 공격 행위를 체계화한 지식 베이스.
활용: ① 자사 방어 커버리지 매핑 ② Red Team 시나리오 설계 ③ 탐지 룰셋(Sigma·YARA) 작성 ④ 사고 대응 시 공격자 TTP 분류
Ⅳ. 결론

APT는 단일 보안 솔루션으로 방어가 불가능하며, 가시성(Visibility)·탐지(Detection)·대응(Response)의 순환 체계가 필수이다. 「국가 사이버위기 관리 매뉴얼」에 따라 침해 발생 즉시 KISA·NCSC에 신고하고, XDR(eXtended Detection and Response)과 위협 인텔리전스(CTI) 공유로 능동적 APT 방어 체계를 구축해야 한다.

Ⅰ. 가. 랜섬웨어 개념 및 발전 역사·최신 유형
랜섬웨어(Ransomware)는 피해자의 파일·시스템을 암호화하거나 접근 차단한 후 복호화 키 제공 대가로 금전(주로 비트코인)을 요구하는 악성코드이다. 1989년 AIDS Trojan을 시초로, 2017년 WannaCry·NotPetya의 글로벌 대유행 이후 RaaS 형태로 진화하였다.
유형설명대표 사례
단순 암호화
(1세대)		파일 암호화 후 복호화 키 요구. 단일 공격자CryptoLocker(2013), CTB-Locker
RaaS
(Ransomware as a Service)		랜섬웨어 개발자가 코드를 서비스로 제공. 운영자(Operator)가 공격, 수익 분배 (개발자 20~30%)LockBit, REvil(Sodinokibi), BlackCat
이중 협박
(Double Extortion)		암호화 전 데이터 먼저 탈취. 복호화 + 데이터 공개 차단의 이중 협박Maze(2019), Cl0p, ALPHV
트리플 협박
(Triple Extortion)		이중 협박 + 피해자 고객·파트너에게 직접 연락하여 추가 압박LockBit 3.0, Dark Side
Wiper 랜섬웨어금전 목적 없이 복호화 불가능하게 파괴 목적으로 설계. 국가 사이버 전쟁 수단NotPetya(2017), Shamoon
Ⅱ. 나. 랜섬웨어 동작 단계 및 암호화 메커니즘
랜섬웨어 동작 단계 1단계 [초기 침투]: 스피어피싱 이메일·취약점 악용·RDP 무차별 대입 2단계 [내부 정찰]: 네트워크 매핑, 백업 서버·도메인 컨트롤러 위치 파악 3단계 [권한 상승]: 로컬 관리자 → 도메인 관리자 권한 탈취 4단계 [측면 이동]: PsExec·WMI로 내부 시스템 전파 5단계 [데이터 탈취]: 민감 데이터 압축·외부 C&C 전송 (이중 협박 준비) 6단계 [방어 무력화]: 백신·EDR 종료, 볼륨 섀도우 복사본(VSS) 삭제 7단계 [암호화 실행]: 파일 암호화 실행 8단계 [협박 메시지]: 복호화 키 요구 메시지 표시, 다크웹 게시 협박
암호화 메커니즘 — 대칭 + 비대칭 조합
① 랜섬웨어 공격자: RSA 키쌍(공개키PK/개인키SK) 사전 생성. PK만 악성코드에 내장
② 감염 시: 악성코드가 AES 세션키(256bit) 랜덤 생성 → AES로 피해자 파일 대량 암호화
③ 키 보호: AES 세션키를 공격자 RSA 공개키(PK)로 암호화하여 저장
④ 복호화 조건: 피해자가 몸값 지불 후 공격자의 RSA 개인키(SK)로 AES 키 복호화 → 파일 복구

※ 설계 이유: AES만 사용 시 코드 분석으로 키 노출 위험. RSA로 AES 키를 보호하여 공격자만 복호화 가능하게 함
Ⅲ. 다. 랜섬웨어 대응 방안 (예방·탐지·복구)
  • 1
    예방 (Prevent)
    취약점 패치: 운영체제·소프트웨어 보안 패치 즉시 적용 (WannaCry는 EternalBlue SMB 취약점 악용 — MS17-010 패치로 방어)
    이메일 보안: 샌드박스 분석, 악성 첨부파일 차단, DMARC·SPF·DKIM 적용
    RDP 보안: RDP(3389) 방화벽 차단 또는 VPN 뒤에만 허용, MFA 적용, 계정 잠금 정책
    최소 권한 원칙: 도메인 관리자 계정 일상 업무 사용 금지, PAM(특권접근관리) 도입
    네트워크 세그멘테이션: 백업 서버를 업무망과 완전 분리. 측면 이동 차단
  • 2
    탐지 (Detect)
    EDR(Endpoint Detection and Response): 파일 대량 암호화 행위, VSS 삭제 명령(vssadmin), 이상 프로세스 생성 실시간 탐지·차단
    SIEM 연계: 비정상 파일 접근 패턴, C&C 통신 IoC(침해지표) 탐지
    허니팟·데코이 파일: 접근 시 즉시 경보 발생하는 미끼 파일로 초기 감지
  • 3
    복구 (Recover) — 3-2-1 백업 원칙
    3-2-1 백업 원칙 (NIST 권장)
    3: 데이터 사본을 최소 3개 유지 (원본 + 2개 백업)
    2: 서로 다른 2가지 미디어에 저장 (HDD·테이프·클라우드)
    1: 백업 사본 중 1개는 오프사이트(오프라인·격리 네트워크) 보관

    확장: 3-2-1-1-0 원칙: 추가로 1개는 변경 불가(Immutable) 백업, 복구 테스트 오류 0건 보장
    몸값 지불 금지 원칙: 지불해도 복호화 보장 없음(40%만 완전 복구). 추가 공격 유발
    사고 대응: 감염 시스템 즉시 네트워크 격리 → 포렌식 보존 → KISA·수사기관 신고 → 백업으로 복구
Ⅳ. 결론

랜섬웨어는 RaaS 생태계로 진입 장벽이 낮아져 공격 빈도가 급증하고 있다. 공공기관은 「사이버 위기 대응 매뉴얼」에 따라 ①백업 체계 검증, ②EDR 전 단말 배포, ③MFA 전면 적용을 우선 이행해야 한다. 사전 예방이 사후 복구보다 비용 효율이 100배 이상 높으며, 정기적인 백업 복구 훈련(Ransomware Drill)이 필수이다.

Ⅰ. 가. 사이버킬체인 7단계 및 방어자 대응
사이버킬체인(Cyber Kill Chain)은 Lockheed Martin이 2011년 군사적 킬체인 개념을 사이버보안에 적용한 APT 공격 분석·방어 프레임워크이다. 7단계 중 어느 한 단계라도 차단하면 공격이 실패한다는 원칙에 기반한다.
단계공격자 활동방어자 대응 (Detect·Deny·Disrupt·Degrade·Deceive·Destroy)
1. 정찰
Reconnaissance		OSINT·능동 스캔으로 표적 정보 수집Detect: 외부 스캔 탐지 (IDS), 웹 노출 정보 최소화, 직원 SNS 보안 교육
2. 무기화
Weaponization		취약점 익스플로잇·악성코드 제작Deny: 위협 인텔리전스 수집으로 TTPs 사전 파악. 공격자 인프라 사전 모니터링
3. 전달
Delivery		스피어피싱·워터링홀·USBDeny: 이메일 샌드박스, 웹 게이트웨이 URL 필터링, 외부 매체 통제 정책
4. 익스플로잇
Exploitation		취약점 코드 실행Disrupt: 패치 관리, DEP·ASLR 활성화, 애플리케이션 화이트리스트, EDR
5. 설치
Installation		백도어·루트킷 설치, 지속성 확보Disrupt: EDR 행위 탐지, 파일 무결성 검사(FIM), 레지스트리 변경 모니터링
6. C&C 통신
Command & Control		원격 제어 채널 수립Disrupt: DNS 싱크홀, TLS 인스펙션, 이상 DNS 쿼리 탐지, C&C IoC 차단
7. 목표 달성
Actions on Objectives		데이터 탈취·파괴·측면 이동Deceive: DLP(데이터유출방지), 허니팟, 마이크로세그멘테이션, SOAR 자동 대응
Ⅱ. 나. MITRE ATT&CK 프레임워크 및 사이버킬체인 비교
MITRE ATT&CK(Adversarial Tactics, Techniques & Common Knowledge)는 실제 관측된 공격자의 행위를 14개 전술(Tactic)·수백 개 기술(Technique)·절차(Procedure)로 체계화한 지식 베이스이다(2013년 MITRE 공개).
비교 항목사이버킬체인MITRE ATT&CK
관점선형 순서 (공격 흐름)전술별 기술 매트릭스 (세부 행위)
단계/항목 수7단계 (고수준)14개 전술, 200+ 기술, 400+ 절차 (세부)
적용 목적공격 단계 조기 차단 전략탐지 룰 개발, 레드팀 시나리오, 방어 커버리지 평가
커버 범위외부 침투 중심 (침투 전~실행)침투 후 내부 행위(측면이동·권한상승 등) 상세 커버
갱신 주기정적 (변경 없음)분기별 갱신 (새 TTP 지속 추가)
ATT&CK TTP 구조
전술(Tactic): 공격자의 목적(예: TA0001 Initial Access)
기술(Technique): 목적 달성 방법(예: T1566 Phishing)
절차(Procedure): 특정 그룹의 구체적 구현(예: Lazarus가 사용한 특정 피싱 이메일 도구)

활용: Navigator 도구로 방어 커버리지 히트맵 시각화, Sigma 룰로 SIEM 탐지 자동화
Ⅲ. 다. Diamond 모델 및 위협 인텔리전스(CTI)
Diamond Model of Intrusion Analysis (Caltagirone, 2013) [적대자(Adversary)] / \ / \ [능력(Capability)] ── [인프라(Infrastructure)] \ / \ / [피해자(Victim)] 4개 꼭짓점 관계: • 적대자가 능력(악성코드·TTPs)을 사용해 • 인프라(C&C 서버·도메인)를 통해 • 피해자를 공격 → 하나의 사건이 다이아몬드 하나. 사건 연결로 캠페인 분석
CTI 활용 방안내용
IoC(침해지표) 공유악성 IP·도메인·해시값을 STIX/TAXII 표준으로 자동화 공유. KISA·ISAC·MISP 플랫폼 활용
위협 그룹 프로파일링ATT&CK 그룹 페이지에서 Lazarus·Kimsuky 등 주요 APT 그룹의 TTPs 분석
사전 방어 (Proactive)CTI 기반 방어 취약점 선제 파악. 공격자가 사용할 인프라 사전 차단
사고 대응 가속화침해 발생 시 CTI와 Diamond 모델로 공격자 귀속(Attribution) 및 추가 공격 경로 예측
Ⅳ. 결론

사이버킬체인은 공격 흐름을 이해하는 전략적 프레임워크, ATT&CK는 세부 탐지·대응을 위한 전술적 도구, Diamond 모델은 사건 분석의 구조화 도구로 상호 보완적이다. 국내 KISA·국정원과의 CTI 공유 체계를 구축하고, MITRE ATT&CK 기반 보안 운영 성숙도(SOC Maturity)를 정기 평가하여 방어 커버리지를 지속 향상시켜야 한다.

Ⅰ. 가. IDS·IPS 개념·탐지 기법·오탐/미탐 문제
IDS(침입탐지시스템)는 네트워크 또는 호스트에서 공격을 탐지하고 경보를 발생시키는 수동형 보안 시스템이고, IPS(침입방지시스템)는 탐지와 동시에 공격 트래픽을 차단하는 능동형 보안 시스템이다.
탐지 기법원리장점단점
시그니처 기반
(Signature)		알려진 공격 패턴(시그니처) DB와 패킷 비교. 오탐 낮음탐지 정확도 높음, 처리 속도 빠름제로데이·변형 공격 탐지 불가. DB 최신화 필수
이상 탐지
(Anomaly)		정상 트래픽 기준선(Baseline) 학습 후 편차 탐지. 통계·ML 기반제로데이·내부자 공격 탐지 가능오탐율 높음. 기준선 구축·유지 어려움
행위 기반
(Behavior)		프로세스 행위 패턴 분석. API 호출·파일 접근·네트워크 행위 모니터링우회 공격·APT 탐지 효과적연산 자원 소모 높음. 정상 행위와 구분 어려움
오탐(FP) vs 미탐(FN) 트레이드오프
• False Positive(오탐): 정상 트래픽을 공격으로 판단 → 업무 장애, 분석팀 피로
• False Negative(미탐): 실제 공격을 탐지 못함 → 보안 사고 발생
• 임계값 조정: 낮추면 → FP 증가·FN 감소 (민감도↑). 높이면 → FP 감소·FN 증가 (특이도↑)
• 최적화: ROC 곡선 분석으로 임계값 결정. 화이트리스트 튜닝으로 FP 최소화
Ⅱ. 나. 보안 솔루션 역할 비교 및 통합 SOC 체계
솔루션계층역할특징
방화벽
(Firewall)		L3~L4IP·포트 기반 접근 제어. 허용/차단 정책 적용상태 기반(Stateful) 패킷 검사. 애플리케이션 공격 탐지 불가
WAF
(Web Application Firewall)		L7HTTP/HTTPS 트래픽 검사. SQL Injection·XSS·CSRF 등 웹 공격 차단OWASP Top 10 기반 규칙. 역방향 프록시로 배치. 방화벽 보완
IDS/IPSL3~L7패턴 기반 공격 탐지(IDS)·차단(IPS). 시그니처·이상 탐지IDS: 수동(경보만) / IPS: 능동(인라인 차단). 최신 시그니처 갱신 필수
SIEM
(Security Information and Event Management)		분석 계층전사 로그·이벤트 수집·상관 분석·대시보드. 위협 탐지·규정 준수 리포팅Splunk·IBM QRadar·LogRhythm. 수동 대응 한계. SOAR 연계 자동화
SOAR
(Security Orchestration, Automation and Response)		대응 계층보안 이벤트에 대한 자동화 대응 플레이북 실행. 평균 대응시간(MTTR) 단축Palo Alto XSOAR, IBM Resilient. SIEM 경보 → SOAR 자동 격리·차단
EDR
(Endpoint Detection and Response)		엔드포인트단말 행위 실시간 모니터링·위협 탐지·격리. 포렌식 아티팩트 수집CrowdStrike·SentinelOne·MS Defender. 랜섬웨어·파일리스 공격 탐지
통합 SOC(보안운영센터) 아키텍처 [탐지 레이어] SIEM ← (로그) ← 방화벽·IDS/IPS·WAF·EDR·서버·네트워크 [분석 레이어] SIEM 상관 분석 → 위협 경보 생성 → SOC 분석가 (Tier 1/2/3) CTI 피드 연계 (STIX/TAXII) + UEBA (사용자행위분석) [대응 레이어] SOAR 플레이북 자동화: 경보 수신 → 자동 IoC 조회 → 위협 확인 → 자동 격리/차단 → 티켓 생성 → 사고 대응 → 사후 분석(Post-mortem)
Ⅲ. 다. 제로 트러스트 아키텍처 및 공공 적용
제로 트러스트(Zero Trust)는 "절대 신뢰하지 말고, 항상 검증하라(Never Trust, Always Verify)"는 원칙에 기반한 보안 아키텍처이다. 기존의 경계 기반(Perimeter) 보안이 내부 트래픽을 신뢰하는 한계를 극복하기 위해 John Kindervag(Forrester, 2010)가 제안하였다.
  • 1
    핵심 원칙
    최소 권한 접근(Least Privilege): 사용자·기기가 업무에 필요한 최소 자원만 접근. 과도한 권한 제거
    명시적 검증(Explicit Verification): 모든 접근에 대해 신원(ID)·기기 상태·위치·행위를 동적으로 검증
    침해 가정(Assume Breach): 내부 네트워크도 이미 침해되었다고 가정. 측면 이동 차단을 위한 마이크로세그멘테이션
  • 2
    제로 트러스트 구성 요소
    PEP(정책 집행 포인트): 모든 접근 요청의 허용·차단을 집행
    PDP(정책 결정 포인트): ID·기기·컨텍스트 기반으로 접근 여부 동적 결정 (NIST SP 800-207)
    ID 관리: IDAM + MFA + 조건부 접근 (사용자·서비스 계정 포함)
    마이크로세그멘테이션: 네트워크를 소규모 구역으로 분할, 구역 간 트래픽 최소화
  • 3
    공공 정보시스템 적용 방안 (NIST SP 800-207 기준)
    단계 1: ID 기반 보안 강화 (MFA 전면 도입, 특권계정 PAM)
    단계 2: 기기 신뢰도 검증 (단말 보안 상태 확인, 미준수 단말 접근 차단)
    단계 3: 마이크로세그멘테이션 도입 (업무 중요도별 네트워크 분리)
    단계 4: 지속적 모니터링 (SIEM·UEBA 연계, 이상 행위 실시간 탐지)
    국내 근거: 「국가 정보보안 기본지침」 제로트러스트 도입 권고(2023), 행안부 「제로트러스트 가이드라인 1.0(2023)」
Ⅳ. 결론

현대 사이버 위협은 경계 보안만으로 방어가 불가능하며, IDS/IPS·SIEM·SOAR·EDR의 통합 SOC 체계와 제로 트러스트 아키텍처의 결합이 필수적이다. 공공 부문에서는 행안부의 「제로트러스트 가이드라인」에 따라 단계적 전환 로드맵을 수립하고, ML 기반 이상 탐지(UEBA)와 위협 인텔리전스(CTI) 공유를 통해 능동적 사이버 방어 체계를 구현해야 한다.
