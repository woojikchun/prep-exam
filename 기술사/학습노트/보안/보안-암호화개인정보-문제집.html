⏱ 10분 📄 1.5~2페이지 💯 10점
Ⅰ. 블록암호 운용 모드의 개념
블록암호(AES, ARIA 등)는 고정 크기 블록(128bit) 단위로 암호화한다. 운용 모드(Mode of Operation)는 평문이 블록 크기를 초과할 때 블록 간 연산 방식을 정의하는 알고리즘 체계이다.
Ⅱ. 4가지 운용 모드 비교
모드동작 원리특징취약점/단점
ECB
(Electronic CodeBook)		각 블록을 독립적으로 암호화 (키만 사용)병렬 처리 가능. 구현 단순동일 평문 블록 → 동일 암호문 블록. 패턴 노출 (펭귄 이미지 문제). 사용 금지 권고
CBC
(Cipher Block Chaining)		이전 암호문 블록과 현재 평문 XOR 후 암호화. IV(초기화 벡터) 필요패턴 은닉. 순차 처리패딩 오라클 공격(POODLE) 취약. 직렬 처리로 성능 저하. 무결성 미보장
CTR
(CounTeR)		증가하는 카운터를 암호화한 키스트림과 평문 XOR병렬 처리 가능. 스트림 암호처럼 동작. 랜덤 접근 가능무결성(인증) 미보장. Nonce 재사용 시 치명적 취약
GCM
(Galois/Counter Mode)		CTR 모드 + GHASH(갈루아 필드) 인증 태그 생성. 암호화+인증 동시 제공AEAD(인증 암호화). 병렬 처리. 하드웨어 가속(AES-NI). TLS 1.3 필수Nonce 재사용 시 키 복구 위험. 구현 복잡도 높음
Ⅲ. GCM이 현대 보안 환경에서 권장되는 이유
  • 1
    AEAD(Authenticated Encryption with Associated Data): 암호화(기밀성)와 무결성·인증(Authentication Tag, 128bit)을 단일 연산으로 동시 보장. 별도 MAC이 불필요하여 취약점 감소
  • 2
    성능 우위: CTR 기반 병렬 암호화 + 갈루아 필드 인증의 하드웨어 가속(Intel AES-NI, ARMv8). CBC 대비 처리 속도 2~5배 향상
  • 3
    표준 채택: TLS 1.3의 필수 암호 스위트(TLS_AES_256_GCM_SHA384), NIST SP 800-38D 표준, IETF RFC 5288 지정
  • 4
    패딩 오라클 공격 면역: CTR 기반으로 패딩이 불필요하여 POODLE·Lucky13 등 CBC의 패딩 오라클 공격 원천 방지
⚠ GCM 사용 시 주의사항
Nonce(96bit)는 절대 재사용 금지. 동일 키+Nonce 재사용 시 두 암호문 XOR로 키스트림 복구 가능. 실무에서는 랜덤 Nonce(CSPRNG) 또는 카운터 기반 Nonce로 관리.
🔑 핵심 암기 (1줄 요약)
ECB=패턴노출(사용금지) / CBC=패딩오라클(레거시) / CTR=병렬·무결성없음 / GCM=AEAD+병렬+TLS1.3 표준 (현행 최선)

Ⅰ. Zero-Trust의 개념
정의 (NIST SP 800-207, 2020)

Zero-Trust(ZT)란 "절대 신뢰하지 말고, 항상 검증하라(Never Trust, Always Verify)"는 원칙 하에, 네트워크 내·외부 모든 사용자·기기·트래픽을 지속적으로 인증·인가·모니터링하는 보안 패러다임이다. 내부망도 안전하지 않다는 전제를 기반으로 한다.

Ⅱ. 경계 보안 vs Zero-Trust 비교

인증 범위입구(방화벽·VPN) 1회 인증리소스 접근 시마다 연속 인증
측면 이동내부 이동 무제한 허용마이크로세그멘테이션으로 차단
원격·클라우드VPN 집중, 성능 병목SDP·ZTNA로 직접 접근 제어
내부자 위협탐지 어려움지속 모니터링으로 조기 탐지
Ⅲ. NIST 7가지 핵심 원칙

    모든 데이터·서비스를 리소스로 간주
  • ②

  • ③
    개별 세션 단위로 기업 리소스 접근 허가
  • ④
    클라이언트 신원·앱·자산 상태·기타 행위 속성으로 동적 접근 제어
  • ⑤
    모든 자산의 무결성·보안 상태 모니터링·측정
  • ⑥
    모든 리소스 인증·인가는 동적 엄격 수행
  • ⑦
    자산·네트워크 인프라 현황 수집·개선

Ⅰ. 개인정보 영향평가(PIA)의 개념 및 법적 근거
정의

PIA(Privacy Impact Assessment)란 개인정보를 포함하는 신규 정보시스템 구축·기존 시스템 변경 시, 해당 사업이 개인정보에 미치는 영향을 사전에 조사·분석·평가하여 개인정보 침해 위험 요인을 발굴·개선하는 제도이다.

법적 근거개인정보보호법 제33조, 동법 시행령 제35조~제38조
대상 기관공공기관 중 5만 명 이상의 민감정보·고유식별정보 처리, 또는 50만 명 이상 개인정보 처리 시스템 구축·변경 시 의무
평가 기관개인정보보호위원회 지정 영향평가 전문기관 (KISA 등)
Ⅱ. PIA 수행 절차
①대상선정 → ②현황분석 → ③위험도평가 → ④위험요인개선 → ⑤결과보고서 ①대상선정: 의무 대상 해당 여부 판단, 영향평가 전문기관 선정 ②현황분석: 개인정보 수집·처리 항목, 시스템 구성, 관련 법령 분석 ③위험도평가: 개인정보 처리 단계별 침해 위험 도출, 위험도 산정(가능성×영향도) ④위험요인개선: 기술적·관리적·물리적 보호조치 수립, 잔여 위험 수용 여부 결정 ⑤결과보고서: 개인정보보호위원회 제출, 시스템 설계·개발에 반영

SPF·DKIM·DMARC 동작 원리 및 상호 보완 관계
⏱ 10분 📄 1.5~2페이지 💯 10점
Ⅰ. 이메일 스푸핑 위협
이메일 스푸핑이란 공격자가 From 헤더의 발신자 도메인을 위조하여 피싱·사기 메일을 발송하는 공격이다. SMTP는 발신자 인증 메커니즘이 없어 위조가 용이하며, SPF·DKIM·DMARC는 이를 방어하는 3계층 방어 체계이다.
Ⅱ. SPF·DKIM·DMARC 동작 원리
기술방어 원리DNS 레코드한계
SPF
(Sender Policy Framework)		도메인 소유자가 DNS TXT 레코드에 허가된 발신 IP 목록 등록. 수신 서버는 실제 발신 IP와 대조하여 유효성 검증TXT: "v=spf1 ip4:1.2.3.0/24 include:sendgrid.net ~all"IP만 검증. 헤더 From 위조는 차단 불가. 포워딩 시 실패
DKIM
(DomainKeys Identified Mail)		발신 서버가 메일 헤더+본문에 도메인 개인키로 서명 첨부. 수신 서버는 DNS 공개키로 서명 검증하여 무결성 확인TXT: selector._domainkey.example.com → 공개키발신 IP 검증 없음. 서명 헤더만 보호 (일부 헤더 위조 가능)
DMARC
(Domain-based Message Authentication)		SPF·DKIM 결과와 헤더 From 도메인의 일치 여부(Alignment) 검사. 정책(none/quarantine/reject)과 리포트 수신 지정TXT: _dmarc.example.com → "v=DMARC1; p=reject; rua=mailto:..."SPF 또는 DKIM 중 하나라도 통과하면 인정. 양쪽 모두 취약 시 우회 가능
Ⅲ. 3기술 상호 보완 관계
이메일 수신 시 검증 흐름 [수신 서버] 1. SPF 검사: 발신 MTA IP ∈ DNS SPF 레코드? (발신 서버 IP 인증) 2. DKIM 검사: 메일 서명 → DNS 공개키로 서명 검증 (메시지 무결성) 3. DMARC 검사: SPF/DKIM 결과 + Header-From 도메인 Alignment 확인 ├─ Aligned Pass → 수신함 전달 ├─ Fail + p=quarantine → 스팸함 분류 └─ Fail + p=reject → 메일 반려
🔑 핵심 암기
SPF=발신IP인증 / DKIM=서명무결성 / DMARC=정책·Alignment·리포팅.
세 기술의 조합으로 발신 서버 인증 + 메시지 무결성 + 도메인 위조 방지의 3중 보호 달성.

Ⅰ. 디지털 포렌식의 개념
정의

디지털 포렌식(Digital Forensics)이란 컴퓨터·모바일·네트워크 등 디지털 저장매체에서 법적 효력이 있는 디지털 증거를 수집·보존·분석·제출하는 과학적 절차와 기술의 총체이다. 법정에서의 증거 능력 확보가 핵심 목표이다.

Ⅱ. 디지털 포렌식 4대 원칙
  • 1
    정당성(Legality): 적법한 절차와 방법으로 증거 수집·분석. 위법 수집 증거는 법정 효력 없음
  • 2
    재현성(Reproducibility): 동일 조건에서 동일 결과 재현 가능. 분석 절차와 도구를 문서화
  • 3
    무결성(Integrity): 수집 이후 원본 데이터가 변조·훼손되지 않음. 해시값(SHA-256)으로 증명
  • 4
    연계보관성(Chain of Custody): 증거 수집~법정 제출까지 모든 취급자·취급 행위를 빠짐없이 기록·유지
Ⅲ. 수행 절차 및 연계보관성의 중요성

연계보관성(Chain of Custody)의 중요성
증거가 수집된 시점부터 법정 제출까지 누가(Who)·언제(When)·어떻게(How) 취급했는지 끊김 없이 기록해야 법적 증거 능력을 인정받는다. 한 번이라도 연계보관성이 단절되면 법정에서 증거 효력이 상실되며, 디지털 증거의 가장 취약한 고리로 작용한다.

Google이 제안한 공급망 보안 성숙도 프레임워크. 빌드 출처(Provenance) 보장 수준을 4단계로 정의:
Level 0: 보장 없음 → Level 1: 빌드 출처 문서화(Provenance 생성) → Level 2: 출처 서명(인증된 빌드 서비스) → Level 3: 강화된 격리(Hermetic Build, 재현 가능 빌드, 감사 로그)
Sigstore를 이용한 소프트웨어 아티팩트 서명 체계 [서명 단계 — 개발자] 1. Fulcio CA에 OIDC 토큰(GitHub/Google)으로 단기 인증서 요청 2. 아티팩트(컨테이너 이미지·바이너리) 서명 → cosign sign 3. 서명·인증서 → Rekor(불변 투명 로그)에 기록 [검증 단계 — 배포·운영] 1. cosign verify → Rekor에서 서명·출처 검증 2. 아티팩트 다이제스트·서명자 신원·빌드 출처 일치 확인 ✓ 장점: PKI 인프라 없이 OIDC 신원으로 서명. 키 관리 불필요. 투명 로그로 공개 감사 가능. Ⅴ. 결론

2023년 개인정보보호법 전면 개정으로 국내 개인정보보호 규제는 GDPR과 동등한 수준으로 강화되었다. 특히 이동권·자동화결정 거부권 신설과 매출액 기준 과징금 상한 폐지는 기업의 개인정보 처리 전 과정에 걸친 법령 준수 필요성을 높였다. ISMS-P 인증은 개인정보보호법 준수의 실효적 증명 수단이자 감리 기준으로서, 조직의 데이터 거버넌스 체계 구축과 연계하여 추진되어야 한다.
