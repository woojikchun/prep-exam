Ⅰ. 혼잡의 개념, 혼잡 감지 방법 및 핵심 변수
1. 혼잡(Congestion)의 정의

혼잡(Congestion)이란 네트워크 링크·라우터 버퍼에 처리 가능 용량 이상의 트래픽이 집중되어 패킷 손실·지연 급증·처리율(Throughput) 저하가 발생하는 상태이다. 혼잡이 방치되면 Congestion Collapse(모두가 재전송하는 악순환)로 네트워크가 사실상 불통이 된다.

TCP는 종단간(End-to-End) 네트워크 상태를 추론하여 송신 속도를 자율 조절한다. 라우터는 혼잡 알림을 명시적으로 제공하지 않으며(ECN 제외), TCP 스스로 손실·지연 신호를 해석한다.

2. 혼잡 감지 방법
감지 신호내용심각도
패킷 손실 — 타임아웃(RTO) 재전송 타임아웃(RTO) 만료: ACK가 RTO 내에 수신되지 않음. 심각한 혼잡 신호 높음
패킷 손실 — 3 중복 ACK 동일 ACK를 3번 연속 수신 → 특정 세그먼트 손실 추정. 부분 손실 신호 중간
RTT 증가 (BBR 방식) 왕복 지연(RTT) 증가를 버퍼 포화의 조기 신호로 탐지. 손실 전 감지 낮음(선제)
ECN (Explicit Congestion Notification) 라우터가 IP 헤더에 ECN 비트 설정 → 수신자가 ECE 플래그로 송신자에 명시 통보 명시적

3. 핵심 변수
변수정의역할
cwnd
(Congestion Window)		 송신자가 ACK 없이 전송 가능한 최대 데이터 크기 (바이트 또는 MSS 단위). 혼잡 제어가 동적으로 조절 혼잡 상태에 따른 전송 속도 제한
rwnd
(Receiver Window)		 수신자의 버퍼 여유 공간. ACK 헤더에 포함하여 송신자에 전달 수신자 처리 속도에 따른 흐름 제어
ssthresh
(Slow Start Threshold)		 Slow Start → Congestion Avoidance 전환 임계값. 혼잡 이벤트 발생 시 갱신 단계 전환 기준
실효 전송 윈도우 실제 전송 가능 크기 = min(cwnd, rwnd) 혼잡 제어 + 흐름 제어의 통합 상한
【표 2】 TCP 혼잡 제어 핵심 변수
Ⅱ. 혼잡 제어 4단계 알고리즘
① Slow Start (느린 시작)
초기 또는 RTO 타임아웃 후 적용
· cwnd = 1 MSS 에서 시작
· ACK 1개 수신 시 cwnd += 1 MSS
  → 매 RTT마다 cwnd 2배 증가 (지수)
· cwnd ≥ ssthresh 시 CA 단계로 전환
목적: 초기 네트워크 용량을 빠르게 탐색
② Congestion Avoidance (혼잡 회피)
cwnd ≥ ssthresh 구간에서 적용
· ACK 1개 수신 시 cwnd += MSS²/cwnd
  → 매 RTT마다 cwnd += 1 MSS (선형)
· 손실 이벤트 발생 시 반응
목적: 혼잡 직전까지 선형적·안전하게 증가
③ Fast Retransmit (빠른 재전송)
3 중복 ACK(Triple Duplicate ACK) 수신 시
· RTO 만료를 기다리지 않고 즉시 손실 세그먼트 재전송
· RTO 방식 대비 재전송 지연 대폭 감소
목적: 타임아웃 없이 빠른 손실 복구
④ Fast Recovery (빠른 복구)
Fast Retransmit 직후 진입 (Reno 기준)
· ssthresh = cwnd / 2
· cwnd = ssthresh + 3 MSS
· 추가 중복 ACK 수신 시 cwnd += 1 MSS
· 새 ACK 수신 시 cwnd = ssthresh → CA 전환
목적: SS로 되돌아가지 않고 CA 단계 유지
【표 3】 TCP 혼잡 제어 4단계 — 초록(SS)·파랑(CA)·주황(FR)·보라(FRec)
혼잡 이벤트별 cwnd · ssthresh 조정 규칙
혼잡 이벤트ssthresh 갱신cwnd 갱신다음 단계
RTO 타임아웃 ssthresh = cwnd / 2 cwnd = 1 MSS Slow Start 재시작
3 중복 ACK ssthresh = cwnd / 2 cwnd = ssthresh + 3 MSS Fast Recovery → CA
새 ACK 수신 (SS 중) 변경 없음 cwnd += 1 MSS SS 지속 or CA 전환
새 ACK 수신 (CA 중) 변경 없음 cwnd += MSS²/cwnd (≈+1 MSS/RTT) CA 지속
【표 4】 혼잡 이벤트별 cwnd · ssthresh 조정 규칙
AIMD 원리 (Additive Increase Multiplicative Decrease)

TCP 혼잡 제어의 핵심은 AIMD(Additive Increase / Multiplicative Decrease) 원리이다. 정상 구간에서는 선형 증가(+1 MSS/RTT), 혼잡 감지 시 절반으로 감소(×1/2)하여 시스템 전체의 공정한 대역폭 공유를 달성한다.

정상: cwnd(t+1) = cwnd(t) + α  (α=1 MSS)
혼잡: cwnd(t+1) = cwnd(t) × β  (β=0.5)
TCP Reno cwnd 변화 다이어그램
【그림 1】 TCP Reno cwnd 변화 — SS(지수)→CA(선형)→3DupACK(절반+FR)→CA 재개→RTO(cwnd=1 초기화)
Ⅲ. TCP Reno vs. TCP CUBIC vs. BBR 알고리즘 비교
1. TCP Reno (표준 기준선)

1990년대 표준화된 기준 혼잡 제어 알고리즘. AIMD 기반으로 패킷 손실을 혼잡 신호로 사용한다. 고대역폭·고지연(High BDP) 환경에서는 window가 가득 차기까지 너무 오래 걸려 대역폭 저활용 문제가 발생한다.

BDP(Bandwidth-Delay Product) = 대역폭 × RTT — 고BDP 환경에서 cwnd가 BDP에 도달하는 데 수백 RTT 소요

2. TCP CUBIC (리눅스 기본값, RFC 8312)

고BDP 환경을 위해 설계된 알고리즘. cwnd 증가를 3차 함수(Cubic Function)로 모델링하여 이전 최대 cwnd(Wmax) 근처에서는 천천히 증가(안전 탐색), 멀리 있을 때는 빠르게 증가(공격적 회복)한다.

W(t) = C × (t − K)³ + Wmax

C: 스케일링 계수, K: 포화점까지 남은 시간, Wmax: 이전 혼잡 직전 cwnd

장점: Reno 대비 고BDP에서 대역폭 활용률 대폭 향상. RTT 독립적(여러 흐름 간 공정성 개선)

3. BBR (Bottleneck Bandwidth and Round-trip propagation time, Google 2016)

BBR은 패킷 손실이 아닌 병목 대역폭(BtlBw)과 최소 RTT(RTprop)를 직접 측정하여 혼잡 없는 최적 전송률을 추정한다. 버퍼가 채워지기 전에 적정 속도를 유지하므로 불필요한 버퍼링과 패킷 손실을 줄인다.

전송 목표율 = BtlBw × RTprop

장점: 고손실·위성·LTE 등 비이상적 환경에서 Reno·CUBIC 대비 처리율 수배. YouTube·Google CDN에 배포

한계: 일부 환경에서 다른 알고리즘과 공정성(Fairness) 문제, RTT 측정 부정확성

2. 실무 고려사항

① 초기 cwnd(IW) 증가: RFC 6928은 초기 cwnd를 10 MSS로 정의(기존 1~4). 단거리 소규모 HTTP 응답의 완료 속도를 크게 향상시킴. Google의 실험에서 평균 페이지 로드 10% 개선

② TCP BBR과 버퍼 팽창(Bufferbloat): CUBIC은 버퍼를 채울 때까지 전송 → 지연 급증(Bufferbloat). BBR은 최소 RTT 기반으로 버퍼링 최소화 → 게임·VoIP 등 지연 민감 서비스에 유리

③ 공정성(Fairness) 이슈: BBR과 CUBIC이 같은 병목을 공유할 때 BBR이 더 많은 대역폭을 점유하는 불공정 현상. 동일 알고리즘 사용 환경에서는 문제 없음

④ QUIC과 혼잡 제어: HTTP/3 기반 QUIC 프로토콜은 UDP 위에서 응용 계층이 혼잡 제어를 구현(Cubic·BBR 플러그인 방식). TCP의 HOL(Head-Of-Line) 블로킹 없이 스트림별 독립 혼잡 제어 가능

⑤ 데이터센터 내부: DCTCP(Data Center TCP) — ECN을 활용하여 버퍼 점유율에 비례해 cwnd를 정밀 조정. 마이크로초 수준의 지연 제어

Ⅴ. 결론

TCP 혼잡 제어는 Slow Start의 지수 탐색 → Congestion Avoidance의 선형 증가 → Fast Retransmit·Recovery의 신속 복구라는 4단계 구조로 네트워크 자원의 공정하고 효율적인 이용을 달성한다. Reno의 단순 AIMD에서 CUBIC의 3차 함수 증가, BBR의 대역폭 직접 측정으로 진화하면서 고BDP·고손실 환경에서의 처리율이 비약적으로 향상되었다. 현대 네트워크에서는 워크로드 특성과 네트워크 환경에 맞는 알고리즘의 선택적 적용이 핵심 설계 포인트이다.

★ 기술사 핵심 코멘트
TCP 혼잡 제어의 본질은 종단간에서 네트워크 상태를 추론하여 자율적으로 전송률을 조절하는 분산 제어 메커니즘이며, 시스템 감리 시 다음을 중점 검토해야 한다.
① 초기 cwnd 설정: 클라우드 서버·CDN에서 IW=10 미적용 시 단발성 소형 HTTP 응답의 불필요한 RTT 추가 소요 — /proc/sys/net/ipv4/tcp_init_cwnd 설정 확인
② Bufferbloat 유발 여부: CUBIC 환경에서 과도한 버퍼(수백 ms 지연)가 발생하면 BBR 전환 또는 AQM(Active Queue Management — fq_codel, PIE) 적용 검토
③ 알고리즘 혼재 공정성: 같은 병목을 BBR+CUBIC 플로우가 공유할 때 불공정 대역폭 점유 발생 가능 — 동일 환경 내 알고리즘 통일 정책 수립 여부 확인
④ QUIC/HTTP3 전환 시 혼잡 제어 가시성: QUIC는 암호화된 UDP이므로 중간 네트워크 장비의 트래픽 모니터링·QoS 적용이 어려움 — DSCP 마킹 및 관측 가능성(Observability) 설계 여부 점검
